# Мета: платформа задач для людей і ШІ

> Нормативний документ «навіщо». Архітектура ([architecture/](architecture/index.md)) — це «як»; кожне архітектурне рішення має підтримувати цю мету.

## Суть

Ця платформа реалізує інверсію делегування, переносячи фокус управління з постановки задач на визначення кінцевої мети та цінностей. ШІ-оркестратор автоматично декомпозує цю мету, використовуючи взаємозамінні виконавці — людей та агенти. Людина залишається джерелом стратегічного мандата та контролю на критичних "гейтах" системи. Платформа забезпечує крос-мовність та самопокращення через ретроспективний аналіз, роблячи людину власником рішень, а не просто диспетчером.

## Теза: інверсія делегування

Для управління проєктами створено багато прекрасних продуктів — від Trello до Jira. Усі вони проєктувались у світі, де задачі ставили люди і виконували люди; агентні надбудови (Linear agents, GitHub Agent HQ) лише додали ШІ-асистентів **усередину людського процесу**.

MT будується для протилежного світу: **засновник задає мету і цінності — ШІ досягає їх**. ШІ-оркестратор декомпозує мету в самомодифікований граф задач, запускає ШІ-виконавців, оцінює результати і перебудовує граф. Людина в цьому процесі — не диспетчер задач, а:

1. **Джерело мети.** «Навіщо», обмеження і пріоритети задає людина; оркестратор їх не перевизначає. У термінах [mandates.md](architecture/mandates.md) це кореневий мандат.
2. **Підписант гейтів.** Критичні переходи (деструктивні дії, план, аудит) вимагають криптографічного підпису людини — контроль зміщується з постановки задач на гейти.
3. **Власник рішень і призначуваний виконавець.** Там, де ШІ сам визнає потребу в людині (розвилка поза порогами модельного каскаду, кваліфікація, фізична дія), він ескалює упаковану розвилку власнику мандата або **призначає людину відповідної кваліфікації** — з тим самим контрактом вузла, що й для агента ([mandates.md](architecture/mandates.md)).

Взаємозамінність людини й агента на вузлі (той самий життєвий цикл, бюджети, аудит) — не мета, а **передумова** інверсії: щоб оркестратор міг призначити будь-кого, контракт має бути спільним.

Git/GitHub — насамперед інструмент розробників. Для нового світу він лишається **субстратом** (надійність, офлайн, audit trail, відсутність lock-in), але не інтерфейсом: зручність роботи людей і машин має бути понад ним.

## П'ять крос-вимірів

Інверсія делегування вимагає пʼяти крос-властивостей платформи:

1. **Крос-виконавча.** Людина і агент — взаємозамінні виконавці одного вузла задачі; той самий життєвий цикл, бюджети, аудит, ескалація.
2. **Крос-девайсова.** Мак / сервер / телефон — рівноправні точки участі: виконання там, де є ресурси; контроль і approve — з будь-якого пристрою, включно без git-доступу.
3. **Крос-програмкова.** Не один універсальний Codex/Claude, а **спеціалізовані тули** (designer, writer, cli, …), які працюють у рамках певної задачі/процесу — через профілі поверхонь, skills і провайдерів на вузол.
4. **Крос-людська.** Кілька людей у одній задачі: membership, ролі, запрошення, мультипартійні підписані approvals.
5. **Крос-мовна.** Раніше все приводилось до англійської — незручно для не носіїв, але багатомовність була «дорогою». Тепер ШІ робить контент багатомовним автоматично: канонічний вміст — мовою автора, поверхня рендерить кожному учаснику його мовою. Це сприяє крос-національній взаємодії.

## Ніша: чим це не існуючі продукти

- **Jira / Trello / Asana** — трекери для виконавців-людей: картка описує роботу, але не є контрактом, який може виконати машина. У MT `task.md` + `## Done when` + `## Check` — машинно-виконуваний контракт, спільний для людини й агента.
- **Linear (agents), GitHub Agent HQ** — агенти як асистенти всередині чужого vendor-hosted продукту, прив'язані до його домену (розробка ПЗ) і його хмари: ШІ-внесок усередині людського процесу. MT — навпаки, **людський внесок усередині агентного процесу**; local-first і вендор-нейтральний: стан у вашому git, моделі будь-які (локальні чи хмарні), домен задач не обмежений кодом.
- **LangGraph / CrewAI / AutoGen** — бібліотеки оркестрації для розробників: графи агентів у коді, без людей як рівноправних виконавців, без продукту поверх. MT — платформа, де граф живе у файлах, людина — повноцінний вузол, а контроль (approvals, аудит) — вбудований і криптографічний.
- Спільна риса, якої немає ні в кого з них: **крос-мовність як властивість платформи** — кожен учасник працює своєю мовою над одним каноном.

## Людина як власник, не exception-handler

Знімати вартість людської координації можна двома способами. Перший — людина отримує щільний потік чужих нерозв'язаних проблем без контексту (все, що ШІ не подужав); чим краще працює автомат, тим гірше вона підготовлена до моменту, коли він здається. MT свідомо будує другий: **фрактальне власництво** — на кожному рівні, від виконавця вузла до власника задачі, людина володіє «що і навіщо» свого горизонту (мандат), а система забирає «як» і транспорт інформації між рівнями. До людини не приходить «зламалось» — приходить розвилка її рівня: контекст, варіанти, ціна зволікання, рекомендація агента (`decision-request`); усе інше система зобов'язана вичерпати сама (retry-before-escalate). Влада (мандат) не слідує за скілом (компетенція) автоматично — інакше найкращий інженер стає менеджером мимоволі; мандат розширюється лише свідомим підписаним актом делегування. Механіка — [mandates.md](architecture/mandates.md).

## Мета-цикл: ретроспективне самопокращення

Людям (і агентам) складно системно покращувати власний процес: ретроспектива вимагає окремих зусиль, тому її майже не роблять. Платформа має ці дані задарма — audit trail графа фіксує, хто, що і чим вирішував. Тому окрема мета: ШІ ретроспективно аналізує виконані задачі, шукає кращі рішення/skills/інструменти і пропонує виконавцю: «задачу X ти вирішував через Y; з Z1 або Z2 вийшло б швидше/якісніше». Це замикає крос-програмковий вимір: вибір тулів на вузол живиться даними, а не лише статичною конфігурацією.

Принцип довіри: цикл працює **на виконавця** — пропозиції адресовані йому особисто, opt-in; це не інструмент нагляду для менеджменту.

## Наслідки для фокусу

- Пріоритет — ядро графа + одна поверхня, доведені **dogfooding-ом**: розробка MT ведеться через MT. Порядок кроків — [roadmap.md](roadmap.md).
- Перший продукт — **автономне досягнення мети з людиною на гейтах**; mission control (мак + сервер + телефон-approver) — його компонент, не окремий продукт. Масштаб «заміни Jira» (команди, крос-мовність) — наступне кільце.
- Git лишається Шаром 0 (істина), але жоден сценарій для не-розробника не має вимагати роботи з git напряму.
