## ADR Уніфікований `run_NNN.md` для всіх акторів замість окремих `error.md` / `repair_history.md`

## Context and Problem Statement
Первісний дизайн мав окремі файли для різних типів виконання: `outputs.md` (append-only), `error.md` (append-only), `repair/NNN.md` (для інженера), `repair_context.md` (бюджет). Усі файли під час роботи мали бути immutable, але append-only суперечить цьому принципу, а різні структури для різних акторів плодять складність.

## Considered Options
* Уніфікований `run_NNN.md` — один immutable файл на спробу, незалежно від актора
* Окремі файли `error.md` / `repair_history.md` / `repair_context.md` з різними форматами

## Decision Outcome
Chosen option: "Уніфікований `run_NNN.md`", because єдина схема для агента, інженера, людини та аудитора спрощує сканування стану й усуває суперечність append-only з принципом immutability.

### Consequences
* Good, because transcript фіксує очікувану користь: нуль append-only файлів; стан вузла визначається скануванням `run_*.md` і `outputs_*.md`; `## Reasoning` агента та інженера в одній секції без дублювання.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Фронтматер: `created_at`, `actor: agent | engineer | human | auditor`, `result: success | failed`, опціонально `worktree:` (шлях до залишеного ворктрі при `failed`).
- Секції: `## Reasoning` (обов'язкова), `## Script` (wrapper, якщо впав), `## Ref` (якщо є що посилатись).
- Видалені файли: `error.md`, `repair/NNN.md`, `repair_context.md`, `outputs.md` (append-only).
- Нові файли: `run_NNN.md` (immutable, один на спробу), `outputs_NNN.md` (immutable, один на успішний запуск).
- Файл документу: `npm/docs/mt.md`.

## ADR Усі імена файлів і директорій — англійська

## Context and Problem Statement
Система оброблятиме файли скриптами (сканування, підрахунок `run_*.md`, стан-детектор). Деякі директорії мали назви українською (`операції/`, `патчі/`, `підграф/`), а також `вхідні.md` / `вихідні.md`. Змішані мови в іменах ускладнюють glob-паттерни і shell-команди.

## Considered Options
* Усі імена файлів і директорій — англійська (код/скрипти), текст — довільна мова
* Змішані назви (частина українська, частина англійська)

## Decision Outcome
Chosen option: "Усі імена файлів і директорій — англійська", because файли обробляються скриптами, тому машинозчитувані імена мають бути однозначними й безпечними в shell.

### Consequences
* Good, because transcript фіксує очікувану користь: glob-паттерни (`run_*.md`, `outputs_*.md`) однозначні; shell-команди не потребують квотування Unicode.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Перейменування: `операції/` → `ops/`, `патчі/` → `patches/`, `підграф/` → `subgraph/`, `вхідні.md` → `inputs.md` (пізніше злито в `task.md`), `вихідні.md` → `outputs_NNN.md`.
- Зафіксовано в контракті: імена файлів і директорій — англійська; тіло Markdown-файлів — довільна мова.
- Файл документу: `npm/docs/mt.md`.

## ADR Злиття `inputs.md` у `task.md` як секція `## Inputs`

## Context and Problem Statement
Первісний дизайн передбачав два окремих файли на вузол: `task.md` (місія) і `inputs.md` (посилання на дані від батька). Агент при запуску мав читати обидва файли; обидва immutable після створення ворктрі.

## Considered Options
* Злиття `inputs.md` у `task.md` як секцію `## Inputs`
* Два окремих файли `task.md` і `inputs.md`

## Decision Outcome
Chosen option: "Злиття в `task.md`", because агент читає один файл замість двох; місія і вхідні дані завжди разом; менше файлів при spawn. Зміна inputs до старту ворктрі — редагування `task.md` напряму; після старту — kill ворктрі, редагування, restart.

### Consequences
* Good, because transcript фіксує очікувану користь: цілісний контекст в одному файлі; простіший spawn.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Секція `## Inputs` опціональна (відсутня якщо немає `deps` і батько нічого не передає).
- Підсекції `### name` — довільні назви, `ref:` якщо є файл, inline якщо немає.
- Кореневий вузол: `## Inputs` заповнює людина вручну перед першим запуском.
- Файл документу: `npm/docs/mt.md`.

## ADR Append-only інваріант діє лише після створення ворктрі

## Context and Problem Statement
Первісне формулювання "файли тільки створюються, ніколи не змінюються" не уточнювало коли саме діє це правило. До запуску ворктрі жоден агент ще не читає файли вузла, тому зміна або видалення безпечні.

## Considered Options
* Append-only/immutable після створення ворктрі (межа = `git worktree add`)
* Append-only/immutable з моменту створення файлу (без винятків)

## Decision Outcome
Chosen option: "Immutable після створення ворктрі", because ворктрі є природньою та observable межею старту роботи; до цього моменту файли вільно редагуються або видаляються без наслідків.

### Consequences
* Good, because transcript фіксує очікувану користь: інженер може редагувати `task.md` після kill ворктрі без версіонування файлів (`task-v2.md` не потрібен).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Протокол: немає ворктрі → редагуй вільно; є ворктрі → `mt kill` → редагуй вільно → restart.
- `running.lock` у директорії вузла зберігає PID активного процесу; `mt kill` читає PID з нього.
- Файл документу: `npm/docs/mt.md`.

## ADR Git `post-merge` hook як тригер наступних вузлів

## Context and Problem Statement
Після завершення вузла (merge ворктрі) система має автоматично запустити наступників у графі. Потрібно визначити де живе ця логіка: у самій команді `mt run`, у git-інфраструктурі, або у людини.

## Considered Options
* `mt run` як демон — після merge сам сканує і запускає наступників
* Git `post-merge` hook → `mt run --auto`
* Людина вручну запускає кожен вузол після merge

## Decision Outcome
Chosen option: "Git `post-merge` hook", because `mt run` залишається одноразовою командою; git-інфраструктура є природнім місцем для тригеру після merge; wrapper авто-мержить після успіху агента.

### Consequences
* Good, because transcript фіксує очікувану користь: кожен компонент робить одне; граф рухається автоматично від кореня до кінця без участі людини.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- `.git/hooks/post-merge`: `#!/bin/sh\nmt run --auto`.
- `mt run --auto` сканує граф, знаходить всі `deps=resolved` вузли, запускає паралельно в межах `max_worktrees`.
- Wrapper-кроки: агент → `run_NNN.md` → (аудитор якщо `audit: true`) → `git merge` → hook.
- Файл документу: `npm/docs/mt.md`.
