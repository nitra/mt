## ADR Повне переформатування `flow` під архітектуру `npm/docs/mt.md`

## Context and Problem Statement
Існуючий `mt` є «Пасивним Турнікетом»: людина-агент пише код, `flow` ізолює (worktree), оцінює якість і релізить. `npm/docs/mt.md` описує автономний ОАГ на основі `tasks/<node>/task.md` з агентами-виконавцями у git-worktrees. Потрібно поєднати обидві системи в єдину архітектуру, де `graph` є зовнішнім оркестратором, а `flow` — внутрішнім протоколом вузла.

## Considered Options
* Повне переформатування `flow` під контракт npm/docs/mt.md, видалення застарілих команд
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Повне переформатування `flow` під контракт npm/docs/mt.md", because `flow` має обслуговувати один вузол зсередини, а `graph` оркеструє весь DAG ззовні — це природній поділ відповідальностей.

### Consequences
* Good, because transcript фіксує очікувану користь: уніфікована файлова модель (`task.md`/`outputs_NNN.md`) замінює роздроблені `docs/specs/`, `docs/plans/`, MT file-presence state.
* Bad, because `mt init`, `mt init`, `mt done`, MT file-presence state, `docs/specs/`, `docs/plans/` повністю зникають — потребує міграції існуючих flows.

## More Information
Файли що зникають: `mt init`, `mt init`, `mt done`, MT file-presence state, `docs/specs/<date>.md`, `docs/plans/<date>.md`. Нова файлова структура вузла: `tasks/<name>/task.md`, `plan_001.md`, `run_001.md`, `outputs_001.md`, `invalidated` (sentinel). Правило `n-flow.mdc` потребує повного переписування.

## ADR Двостадійний протокол вузла: `mt plan` + виконання

## Context and Problem Statement
Всередині кожного вузла DAG агент має вирішити: розбити задачу на дочірній граф чи виконати атомарно. Ці дві дії мають різну природу (декомпозиція vs виконання) і потребують окремих кроків.

## Considered Options
* Stage 1 (`mt plan`) — design + decompose; Stage 2 — виконання + `mt verify` + сигнали
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Stage 1 + Stage 2", because планування і виконання — різні режими роботи агента; Stage 1 може завершитись spawn без переходу до виконання.

### Consequences
* Good, because composite-шлях (spawn) завершує вузол після Stage 1 без зайвого виконання.
* Good, because `mt verify` агент викликає сам (не wrapper) — агент контролює момент перевірки якості.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Stage 1 вихід — атомарний: `plan_001.md` (numbered, immutable); composite: дочірні `task.md` + явний виклик `mt spawn` агентом (Variant B). Stage 2 протокол: агент пише `outputs_NNN.md` → `mt verify` (перевіряє `## Done when` з `task.md`) → `mt done | mt audit | mt failed`. `mt init` поглинається в `mt plan` — один крок замість двох. Атрибут `mode: human|agent` у front-matter `task.md` керує інтерактивністю Stage 1, дефолт `human`.

## ADR Оркестратор черги: post-merge hook + `mt watch` демон

## Context and Problem Statement
Після завершення вузла (merge або spawn) потрібно автоматично визначити наступні готові вузли і запустити їх з урахуванням обмеження на кількість паралельних worktrees.

## Considered Options
* Тільки post-merge hook (реактивний, одноразовий)
* Тільки watch-демон (постійний процес)
* Обидва разом: hook для синхронної реакції на merge, демон для continuous monitoring

## Decision Outcome
Chosen option: "Обидва разом", because hook забезпечує реакцію на git-події, а демон (`mt watch`) — моніторинг файлових змін і watchdog для stale worktrees.

### Consequences
* Good, because transcript фіксує очікувану користь: жоден тригер не пропущений — ні git merge, ні прямі файлові зміни.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`mt run --auto` — одноразовий скан DAG: знаходить ready-вузли → черга → spawn worktrees (обмеження `max_worktrees` з `.n-cursor.json`). post-merge hook викликає `mt run --auto` після кожного merge. `mt watch` — демон: слідкує за файловими подіями у `tasks/` та за активністю worktrees, перезапускає `mt run --auto` при змінах.
