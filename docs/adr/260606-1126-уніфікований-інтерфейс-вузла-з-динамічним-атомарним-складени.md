## ADR Уніфікований інтерфейс вузла з динамічним атомарним/складеним розбиттям

## Context and Problem Statement
Оркестратор керує графом задач де вузли можуть мати різну внутрішню складність. Потрібно визначити, як батьківський граф взаємодіє з вузлом незалежно від того, чи є він атомарним або містить власний підграф.

## Considered Options
* Уніфікований інтерфейс: вузол завжди має однаковий зовнішній контракт (`state`, `inputs`, `outputs`)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Уніфікований інтерфейс вузла", because батько чекає лише `state = resolved` та `outputs` — не знаючи, чи вузол атомарний чи є Compound з власним підграфом. Розбиття на `Atomic` або `Compound` визначається **динамічно під час виконання** на основі `inputs`.

### Consequences
* Good, because transcript фіксує очікувану користь: атомарний вузол можна "розкрити" в підграф без змін у батьківському графі (Substitutability).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Запропонована схема у сесії:
```
execute(node):
if node.isAtomic → run fn(inputs) → outputs
else → execute(node.graph), wait for exit-nodes
→ node.state = resolved
```
Compound-вузол: `entry` отримує `inputs` батька, `exits[]` виробляють `outputs` батька.

## ADR Dynamic Dataflow DAG з рантаймовим уточненням вузлів

## Context and Problem Statement
Граф задач потребує визначення типу зв'язків між вузлами та моменту, коли визначається структура підграфу складеного вузла.

## Considered Options
* Dynamic Dataflow DAG: орієнтований ациклічний граф, edges несуть дані, розбиття визначається в рантаймі
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Dynamic Dataflow DAG з рантаймовим уточненням вузлів", because структура обрана явно: DAG (directed, acyclic), edges несуть результат (`output` одного вузла → `input` наступного), а рішення Atomic/Compound приймається при запуску вузла на основі його `inputs`.

### Consequences
* Good, because transcript фіксує очікувану користь: граф може адаптувати складність у рантаймі без змін у топології батьківського рівня.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Термін з сесії: **Dynamic Dataflow DAG** з рекурсивним уточненням вузлів. Аналоги: Dask, Prefect dynamic tasks, LangGraph. Compound-вузол може мати **кілька exit-вузлів**; їх `outputs` merge-уються в `outputs` батька.

## ADR Файлова система як розподілене сховище стану графа

## Context and Problem Statement
Граф з вузлами на кількох рівнях вкладеності потребує персистентного зберігання стану. Треба визначити де і ким зберігається стан.

## Considered Options
* Файлова система: кожен вузол самостійно пише свої файли
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Файлова система як розподілене сховище стану графа", because кожен task/вузол сам пише свій файл; централізованого orchestrator-сховища немає. Структура графу є **implicit у вкладеності директорій**.

### Consequences
* Good, because transcript фіксує очікувану користь: граф відновлюється scan-ом файлів; структура природньо підтримує `MutationLog` (`patches/` поряд з `meta.json`).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Запропонована файлова структура у сесії:
```
tasks/
<nodeId>/
meta.json      ← тип, стан, edges
inputs.json
outputs.json   ← при resolved
error.json     ← при failed
graph/         ← якщо Compound
<childId>/
...
```

## ADR LLM-інженер як мета-рівневий супервізор з необмеженим доступом до мутацій графа

## Context and Problem Statement
Вузол може завершитись з помилкою. Потрібно визначити механізм відновлення та хто і в якому обсязі може модифікувати граф для виправлення.

## Considered Options
* LLM-інженер як мета-рівень поза графом з повним доступом вгору по ієрархії
* LLM-інженер як вузол усередині графа
* Мета-рівень з обмеженим локальним доступом (лише поточний підграф)

## Decision Outcome
Chosen option: "LLM-інженер як мета-рівень поза графом з повним доступом вгору по ієрархії", because інженер є supervisor поза об'єктним рівнем графа, може аналізувати повний `path від кореня до вузла що впав` і вирішувати на якому рівні втрутитись — аж до `root`. Обмежень на мутації немає.

### Consequences
* Good, because transcript фіксує очікувану користь: інженер може обирати рівень втручання — від `patch node_7` до `redesign project_A subgraph` — залежно від природи помилки.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Семантика мутацій: **mutable with invalidation** — якщо інженер модифікує граф вище вузла що впав, раніше `resolved` вузли інвалідуються. Система не є append-only. Можливі операції `GraphPatch`: `replace node`, `insert nodes`, `rewire edges`, `modify inputs`.
