## ADR Рекурсивний складений ОАГ як модель задачного оркестратора

## Context and Problem Statement
Потрібно описати і зафіксувати архітектуру системи оркестрації задач, де директорії містять проєкти, проєкти — динамічні графи задач, а вузли графа можуть рекурсивно розкладатися на підграфи під час виконання. Постала потреба у точній назві та формальній моделі структури.

## Considered Options
* Recursive Compound DAG (рекурсивний складений ОАГ)
* 3D-граф (відхилено: описує просторове відображення, а не структуру зв'язків)
* Multi-level Graph
* Hierarchical Task Network (HTN)

## Decision Outcome
Chosen option: "Recursive Compound DAG / Рекурсивний складений ОАГ", because вузол або є атомарним (листом), або розкладається на підграф — і обидва варіанти мають однаковий інтерфейс для батька (`стан = вирішено` + вихідні дані); структура природньо відображається на файлову систему з вкладеними директоріями.

### Consequences
* Good, because transcript фіксує очікувану користь: `Substitutability` — атомарний вузол можна замінити підграфом без змін у батьківському графі; executor однаковий на всіх рівнях.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Зафіксовано у `npm/docs/mt.md`. Псевдокод моделі:
```
Вузол
├── вхідні: Map<портId, Значення>
├── вихідні: Map<портId, Значення>
├── стан: очікує | виконується | вирішено | помилка
└── реалізація (визначається динамічно):
├── Атомарний — fn(вхідні) → вихідні
└── Складений — Граф{ вхідний, вузли, ребра, вихідні[] }
```

## ADR Файлова система як розподілене сховище стану графа

## Context and Problem Statement
Необхідно визначити де і як зберігається стан графа (вузли, ребра, вхідні/вихідні дані, помилки) при динамічному розкладі вузлів під час виконання.

## Considered Options
* Централізоване сховище (оркестратор керує усім графом)
* Файлова система — кожен вузол пише свій файл самостійно

## Decision Outcome
Chosen option: "Файлова система — кожен вузол пише свій файл самостійно", because граф із усіма атрибутами пишеться у файли; кожен таск/вузол пише свій самостійно; структура вкладеності директорій відображає структуру підграфів.

### Consequences
* Good, because transcript фіксує очікувану користь: граф відновлюється скануванням файлів; вкладена структура директорій природньо моделює рекурсивні підграфи; `repair_history.json` зберігається на вузлі — пам'ять подорожує разом із даними.
* Bad, because Neutral, because transcript не містить підтвердження наслідку.

## More Information
Файлова структура вузла:
```
<nodeId>/
meta.json         — тип, стан, ребра
inputs.json
outputs.json      — з'являється при стан=вирішено
error.json        — з'являється при стан=помилка
repair_history.json
graph/            — якщо Складений
<childId>/...
```
Граф також зберігає `MutationLog` (послідовність снапшотів та патчів).

## ADR Мета-рівневий АгентІнженер із мутацією графа та часовим бюджетом

## Context and Problem Statement
Потрібно визначити механізм самовідновлення системи при збої вузла: хто діагностує, чи може змінювати граф, які обмеження накладаються на кількість спроб і час, та як відбувається ескалація.

## Considered Options
* Інженер як вузол у графі (рекурсивний supervisor)
* Інженер як мета-рівень поза графом з необмеженим доступом
* Обмеження за кількістю спроб (`max_attempts: N`)
* Обмеження за часом (`time_budget`)

## Decision Outcome
Chosen option: "Інженер як мета-рівень поза графом + обмеження за часом", because інженер може підніматись вгору і змінювати батьківський граф на будь-якому рівні; кількість спроб не обмежена, але виділено фіксований часовий бюджет (наприклад 10 хвилин) — підхід дозволяє адаптувати стратегію залежно від залишкового часу.

### Consequences
* Good, because transcript фіксує: `АгентІнженер` може виконати `замінити вузол`, `вставити вузли`, `перепідключити ребра`, `змінити вхідні`; батько отримує свіжий часовий бюджет при ескалації; `repair_history.json` на вузлі дає інженеру контекст попередніх невдалих спроб, унеможливлюючи повторення.
* Bad, because максимальний час до людини: `глибина × бюджет` — при глибокій вкладеності може бути тривалим; transcript не містить підтверджених інших негативних наслідків.

## More Information
Ієрархія ескалації:
```
збій вузла
→ АгентІнженер (бюджет: 10хв, необмежені спроби)
→ вичерпано → ескалація
→ АгентІнженер батька (свіжий бюджет: 10хв)
→ ...
→ root timeout
→ повідомлення СтаршомуІнженеру
```
`repair_context.json` містить `deadline`, `started_at`, `time_budget_sec`, `attempts[]`. `СтаршомуІнженеру` передається `senior_report.json` з повним ланцюгом ескалації та `suggested_next_steps`.
