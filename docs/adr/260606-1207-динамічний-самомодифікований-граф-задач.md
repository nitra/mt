## ADR Динамічний самомодифікований граф задач

## Context and Problem Statement
В системі, де задачі оркестратором розбиваються на динамічний workflow, виникла потреба описати структуру поєднання задач/підзадач, визначити формат файлового сховища стану, правила паралельного виконання через git-ворктрі та протокол відновлення після помилок з часовим бюджетом.

## Considered Options
* Recursive Compound DAG (рекурсивний складений ОАГ) із file-based state store
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Recursive Compound DAG із файловим сховищем стану та мета-рівнем інженера", because структура природньо підтримує ієрархічне виконання, append-only файли дають атомарність без транзакцій, git-ворктрі усувають гонки через ізоляцію, а інженер як stateless мета-агент з пам'яттю на вузлі дозволяє самовідновлення без централізованого стану.

### Consequences
* Good, because transcript фіксує очікувану користь: merge-межа ворктрі є єдиною точкою рішення оркестратора, що усуває race conditions між паралельними агентами.
* Good, because `repair_history.md` як append-only журнал дозволяє LLM читати і продовжувати природно без парсингу структурованих форматів.
* Good, because time budget замість лічильника спроб дає реалістичне обмеження з адаптивною стратегією (багато часу → складна спроба, мало → швидкий фікс).
* Bad, because ретроактивна зміна вже змердженого вузла вимагає топологічного кілу всіх залежних ворктрі перед патчем — складна координація.
