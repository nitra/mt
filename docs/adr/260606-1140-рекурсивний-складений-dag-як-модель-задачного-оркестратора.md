## ADR Рекурсивний складений DAG як модель задачного оркестратора

## Context and Problem Statement

Потрібно описати і зафіксувати архітектуру системи, де задачі розбиваються оркестратором на dynamic workflow, що формує граф, вузли якого самі можуть бути графами або атомарними задачами, а метарівневий агент-інженер може змінювати граф у runtime.

## Considered Options

* Recursive Compound DAG з динамічним розкладом вузлів
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Recursive Compound DAG з динамічним розкладом вузлів", because це єдина структура, яка точно описує вимоги: вузол або є атомарним, або динамічно розкладається в підграф; батько бачить лише `state = resolved/failed`; граф орієнтований (DAG); ребра несуть дані (outputs одного вузла → inputs наступного); вузол може мати кілька exit-вузлів у підграфі.

### Consequences

* Good, because батько не знає про внутрішню реалізацію вузла — атомарний і складений вузол мають однаковий інтерфейс (`state`, `inputs`, `outputs`), що дає substitutability без змін у батьківському графі.
* Good, because transcript фіксує очікувану користь: рекурсивний `execute(node)` однаковий на всіх рівнях, що спрощує execution engine.
* Bad, because повна mutable invalidation (вже resolved вузли стають stale при патчі) ускладнює відстеження консистентності; transcript фіксує цю складність явно.

## More Information

**Файлова структура (кожен вузол — свої файли):**
```
<nodeId>/
meta.json           # тип, state, edges
inputs.json
outputs.json        # з'являється при resolved
error.json          # з'являється при failed
repair_history.json # пам'ять інженера на вузлі
graph/              # якщо Compound
<childId>/...
```

**Ключові рішення з transcript:**

| Аспект | Рішення |
|---|---|
| Декомпозиція | Динамічна — при запуску вузла |
| Edges | Data-flow: `outputs → inputs` між вузлами |
| Exit nodes | Кілька; merge в `parent.outputs` |
| Storage | Файли — кожен вузол пише сам, граф відновлюється scan-ом |
| Self-repair | `EngineerAgent` — мета-рівень поза графом, необмежений доступ |
| Invalidation | Cascade вниз по successors при будь-якому патчі |
| Convergence | Time budget (не кількість спроб); дефолт — 10 хвилин |
| Пам'ять інженера | `repair_history.json` на тому вузлі, де внесено зміну |
| Escalation | Кожен батьківський рівень отримує свіжий time budget |
| Root halt | `notify SeniorEngineer` з повним `senior_report.json` |

**`repair_context.json`** (встановлюється при першому виклику інженера):
```json
{
"deadline": "<ISO timestamp>",
"started_at": "<ISO timestamp>",
"time_budget_sec": 600,
"attempts": []
}
```

}
