## ADR Рекурсивний складений ОАГ з динамічним розкладом вузлів

## Context and Problem Statement
Потрібна архітектура для системи задач, де проєкти мають ієрархічні підзадачі, підзадачі розбиваються на динамічні підграфи, а ті в свою чергу — теж можуть розбиватись. Виникло питання: як правильно описати цю структуру і які проєктні рішення забезпечать її реалізацію.

## Considered Options

* Рекурсивний складений ОАГ (Recursive Compound DAG) з динамічним розкладом
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Рекурсивний складений ОАГ з динамічним розкладом вузлів", because кожен вузол або є атомарним, або розкладається на підграф — рішення приймається динамічно при запуску на основі вхідних даних, а батьківський рівень бачить однаковий інтерфейс (`стан = вирішено`) незалежно від внутрішньої складності.

### Consequences

* Good, because transcript фіксує очікувану користь: інкапсуляція чорної скриньки дозволяє замінювати атомарні вузли складеними без змін у батьківському графі; топологія розподілена по `місія.md` кожного дочірнього — жодного центрального файлу графу не потрібно.
* Bad, because transcript фіксує: LLM-недетермінізм означає що той самий вузол може розкладатись по-різному при перезапуску, ускладнюючи debugging.

## More Information

Файл дизайну: `npm/docs/mt.md`. CLI: `mt done|failed|spawn <шлях-вузла>`.

## ADR Файловий стан вузла: implicit через наявність файлів

## Context and Problem Statement
Потрібно відстежувати стан вузла (очікує / виконується / вирішено / помилка / знедійснений) без центрального реєстру, щоб будь-який агент і скрипт моніторингу могли відновити стан скануванням файлової системи.

## Considered Options

* Implicit-стан через наявність файлів (які файли існують = який стан)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Implicit-стан через наявність файлів", because це природно поєднується з append-only підходом: файли лише створюються або видаляються атомарно, а наявність `вихідні.md` є одночасно і сигналом про завершення і самим результатом.

### Consequences

* Good, because transcript фіксує очікувану користь: ворктрі-межа забезпечує атомарність — наступник стартує лише після merge, гонки неможливі за архітектурою; `repair_history.md` як append-only журнал природно підтримується тим самим принципом.
* Bad, because transcript фіксує: scan для відновлення стану без індексу при великих графах — дорога операція.

## More Information

Таблиця стану у `npm/docs/mt.md`: `місія.md` + `вхідні.md` = очікує; `вихідні.md` = вирішено; `помилка.md` = помилка; `знедійснений` (sentinel) = знедійснений; активний ворктрі = виконується.

## ADR Принцип план → дія → факт як write-ahead log

## Context and Problem Statement
Потрібен механізм відновлення після збою агента або інженера в будь-якій точці виконання — запуск підграфу, кілл залежних, патч вузла. Без нього часткові операції залишають граф у невалідному стані.

## Considered Options

* Write-ahead log через файли: `*-план.md` пишеться до дії, `*-факт.md` — після
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Write-ahead log через файли (`*-план.md` / `*-факт.md`)", because наявність плану без факту є однозначним сигналом незавершеної операції — оркестратор scan-ить `tasks/**/*-план.md` і відновлює будь-яку незавершену операцію після збою.

### Consequences

* Good, because transcript фіксує очікувану користь: протокол патчу залежного вузла явно вимагає записати `патчі/001-план.md` першим кроком — збій між кілом залежних і самим патчем відновлюється детерміновано.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Таблиця операцій у `npm/docs/mt.md`. Команда відновлення: `scan tasks/**/*-план.md → якщо відповідний *-факт.md відсутній → відновити`.

## Context and Problem Statement
`вхідні.md` і `вихідні.md` не повинні дублювати дані з інших файлів. Потрібен єдиний синтаксис посилань що буде зрозумілий агентам і скрипту моніторингу.

## Considered Options

* Посилання на рівні секцій Markdown (`#назва-секції`, `#батьківська/вкладена`)
* Посилання з діапазонами рядків (`шлях/до/файлу.md рядки 10-50`)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Посилання на рівні секцій Markdown", because рядки не потрібні — посилання на рівні секцій є достатнім і більш стабільним (секції не зсуваються при редагуванні, рядки — зсуваються).

### Consequences

* Good, because transcript фіксує очікувану користь: `вхідні.md` залишається легким файлом з посиланнями, без копій даних; синтаксис природний для LLM.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Синтаксис у `npm/docs/mt.md`: `посилання: шлях/до/файлу.md#назва-секції` або `посилання: шлях/до/файлу.md#батьківська/вкладена`. Шляхи відносні від поточної директорії вузла.

## ADR Часовий бюджет замість ліміту спроб для агента-інженера

## Context and Problem Statement
Потрібен механізм збіжності для агента-інженера що виправляє вузли — без нього можливий нескінченний цикл ремонту. Класичний `max_attempts` обрізає хороші спроби довільно.

## Considered Options

* Часовий бюджет (необмежена кількість спроб протягом фіксованого часу, наприклад 10хв)
* Ліміт кількості спроб (`max_attempts: N`)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Часовий бюджет", because це реалістичніше обмеження — інженер адаптує стратегію до залишку часу (багато часу → складна спроба, мало → швидке виправлення), і максимальний час до людини стає передбачуваним: `глибина × бюджет`.

### Consequences

* Good, because transcript фіксує очікувану користь: кожен рівень ієрархії ескалації отримує свіжий бюджет, тому загальний час до сповіщення старшого інженера детерміновано обмежений.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Схема `repair_context.md` у `npm/docs/mt.md`: поля `крайній-термін`, `розпочато-о`, `бюджет-сек`. Ескалація до людини через `звіт-старшому.md`.
