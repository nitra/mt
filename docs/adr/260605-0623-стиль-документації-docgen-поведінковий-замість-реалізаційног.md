## ADR Стиль документації docgen: поведінковий замість реалізаційного

## Context and Problem Statement
Скіл `/n-docgen` генерував документацію з секціями «Залежності» (`node:fs`, `node:path`…), «Функції» (таблиці сигнатур і типів), «Rebuild Test» (перелік із прив'язкою до внутрішніх імен). Такі доки описують реалізацію, а не поведінку й задачу файлу — що робить їх важкими для читача й нестабільними при рефакторингу. Під час рев'ю `npm/rules/abie/lib/docs/enabled.md` виник запит: «не хотілось бачити тех. деталі, які не впливають на бізнес-задачу».

## Considered Options
* Залишити поточний «реалізаційний» стиль (перелік залежностей, таблиці типів, сигнатури)
* Перейти на «поведінковий» стиль (секції Огляд / Поведінка / Де використовується / Гарантії поведінки, без stdlib і внутрішніх імен)

## Decision Outcome
Chosen option: "поведінковий стиль", because користувач схвалив запропонований варіант `enabled.md` — секції без `## Залежності`, без таблиць типів, без прив'язки до імен хелперів — і попросив адаптувати скіл під нього.

### Consequences
* Good, because transcript фіксує очікувану користь: доки описують «що і навіщо», а не «як», тому стабільніші при рефакторингу й зрозуміліші без читання коду.
* Good, because секція `## Публічний API` пропускається для тривіальних leaf-модулів (предикати, константи) і додається лише за нетривіальної зовнішньої поверхні — менше шуму.
* Good, because `## Помилки` замінено на `## Гарантії поведінки` (read-only, fail-safe, ігнор поганих вхідних даних) — описує інваріанти, а не виключення stdlib.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Змінені файли: `npm/skills/docgen/SKILL.md`, `.cursor/skills/n-docgen/SKILL.md`, `.pi/skills/n-docgen/SKILL.md` (стаб).
Слово «вичерпну» у `description:` і цілі скіла замінено на «лаконічну поведінкову».
Еталонний файл, за яким верифікувалися правки: `npm/rules/abie/lib/docs/enabled.md`.
Перевірка на 15 файлах `npm/rules/abie` — 3 батчі по 5 Claude-субагентів.

## ADR Вибір локальної LLM для Tier 1 docgen на 8 GB M2: gemma3:4b vs qwen2.5-coder:3b

## Context and Problem Statement
Tier 1 скіла docgen генерує ~26 M хмарних токенів Claude на 1042 файли (~$300–500 за прогін). Виникла ідея переключити Tier 1 на локальну Ollama-модель ($0), залишивши Claude тільки для Tier 2/3 (~2.6 M токенів). Машина — Apple M2 8 GB unified memory; попередньо встановлена `gemma4:e4b` (9.6 GB) давала 0.4 tok/s через своп.

## Considered Options
* gemma3:4b (~3.3 GB, 100% GPU на 8 GB)
* qwen2.5-coder:3b (~2.4 GB, 100% GPU)
* qwen2.5:7b, qwen2.5-coder:7b, llama3.1:8b (~5.1–5.7 GB, CPU/GPU офлоад)
* llama3.2:3b, gemma2:2b, phi3.5 (~2.0–4.0 GB, 100% GPU)
* qwen3:4b (~3.6 GB) — відхилено через некерований режим «думання» (ігнорує `/no_think`, генерує рос.)

## Decision Outcome
Chosen option: "gemma3:4b як якість-first, qwen2.5-coder:3b як швидкість-first", because об'єктивний бенчмарк (чистий старт, `ollama ps` RAM, замір tok/s, порівняння виходу з еталоном) показав: (1) лише моделі ≤4.2 GB повністю влазять у GPU (100%) без офлоаду; (2) `gemma3:4b` дає найкращу українську серед тих, що влазять; (3) `qwen2.5-coder:3b` у 2.6× швидша за ціни незначних мовних дефектів.

### Consequences
* Good, because transcript фіксує очікувану користь: Tier 1 (~26 M токенів) виноситься на локальну машину за $0; Claude лишається тільки для Tier 2/3 (~2.6 M) і хвоста quality-gate.
* Good, because 7B-клас на 8 GB відхилено на підставі виміряних даних (`ollama ps` split 9%/91%), а не припущень.
* Bad, because на 8 GB локальний Tier 1 — послідовний (concurrency = 1): один потік насичує GPU/пам'ять. Батчі по 5 зі скіла зникають → повний прогін 1042 файлів ≈ 22–38 год (`gemma3:4b`) або ≈ 9–16 год (`qwen2.5-coder:3b`). Повний потенціал розкривається на 32 GB+.
* Bad, because `qwen2.5-coder:3b` дає «чорнові» доки — потребує cleanup (зрізати markdown-fence, виправити англіцизми) перед подачею на Tier 2/3.

## More Information
Команди для вимірювань: `ollama pull <model>`, `ollama stop <model>`, `ollama ps`, `POST http://localhost:11434/api/generate`.
Скрипти: `/tmp/docgen-bench/run.py`, `/tmp/docgen-bench2/run.py`.
Ключова умова придатності на 8 GB: **розмір моделі ≤ ~4.2 GB** (лишає місце під KV-cache і ОС без свопу).
Архітектурний наслідок: для гібридного режиму потрібно винести Tier-1 промпт у `docgen-prompt.mjs` (єдине джерело правди) і додати CLI `docgen gen --engine ollama --concurrency 1 --max-bytes … --strip-fences`.
