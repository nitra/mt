# Контракт файлів: Динамічний Граф Задач

## Інваріанти

- Формат: Markdown + YAML-фронтматер
- Атрибути фронтматеру: англійські, snake_case
- Заголовки секцій що парсить скрипт: англійські
- Секції з довільними даними: будь-яка мова
- Всі імена файлів і директорій: англійські
- Всі файли immutable після створення ворктрі
- До ворктрі: вільне редагування і видалення
- Перше поле у фронтматері завжди: `created_at` (ISO 8601)
- `ref:` якщо дані є у файлі, inline якщо немає деінде

```yaml
---
created_at: <ISO 8601>

deps:                       # відсутній якщо немає попередників
- <sibling-node-id>
---
```

```markdown
## Task
<обов'язково>

<обов'язково>

## Inputs          # відсутній якщо батько нічого не передає

```

Хто створює: батько-агент при spawn (кореневий — людина вручну).
`deps` — лише id сусідів (siblings). id вузла = назва директорії.

## ADR Уніфікований `run_NNN.md` для всіх спроб виконання вузла

## Context and Problem Statement
В архітектурі динамічного графу задач потрібно зберігати журнал спроб виконання вузла. Спочатку розглядалися окремі структури для звичайного агента (`error.md`, `outputs.md`) і інженера (`repair_history.md`, `repair_context.md`). Необхідно було уніфікувати ці структури, оскільки обидва типи актора роблять одне й те саме: спробу вирішити вузол з часовим бюджетом.

## Considered Options
* Окремі файли `error.md` (append-only), `outputs.md` (append-only), `repair_history.md` (append-only), `repair_context.md`
* Єдиний immutable `run_NNN.md` (один файл на спробу) для всіх акторів

## Decision Outcome
Chosen option: "Єдиний immutable `run_NNN.md`", because і звичайний агент, і інженер є акторами що роблять спробу вирішити вузол — різниця лише в полі `actor:` (`agent | engineer | human | auditor`) і в тому що вони описують у `## Reasoning`. Уніфікація усуває дублювання структур і узгоджується з принципом "всі файли immutable" (нові факти = нові файли, не append).

### Consequences
* Good, because жодних append-only файлів — всі файли immutable, система проста і консистентна.
* Good, because transcript фіксує очікувану користь: history читається як послідовність `run_001.md`, `run_002.md`, … незалежно від того хто виконував.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл: `npm/docs/mt.md`, секція `### run_NNN.md`. Поля: `created_at`, `actor`, `result: success | failed`. Секції: `## Reasoning` (обов'язкова), `## Script` (wrapper-скрипт, необов'язкова), `## Ref` (необов'язкова). Успішний запуск також створює окремий `outputs_NNN.md`.

## ADR Видалення `patches/` — зміни інженера описуються у `## Reasoning`

## Context and Problem Statement
Для фіксації змін інженера розглядалася директорія `patches/` з файлами `001-plan.md` / `001-fact.md` за патерном plan→action→fact. Ці файли слугували б для відновлення після збою при незавершеному патчі і як аудит-трейл змін.

## Considered Options
* `patches/NNN-plan.md` + `patches/NNN-fact.md` — окремі файли для кожного патчу інженера
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Видалити `patches/`", because recovery-сценарій (обрив посеред патчу) вирішено ігнорувати на поточному етапі; вся інформація про намір і зміни вміщується у `## Reasoning` секції `run_NNN.md` інженера — окрема директорія є надлишковою.

### Consequences
* Good, because файлова структура вузла мінімальна: `task.md`, `run_NNN.md`, `outputs_NNN.md`, `invalidated`.
* Bad, because детальний аудит-трейл конкретних змін у графі (що саме змінено, до/після) не зберігається структуровано — лише вільний текст у `## Reasoning`.

## More Information
Файл: `npm/docs/mt.md`. Команда `mt scan` оновлена: замість "детектує незавершені patch-plan без patch-fact" тепер `exit 1` якщо є вузли у стані `failed`.

## ADR Видалення `ops/spawn-plan` і `ops/kill-plan` — recovery-сценарій ігнорується

## Context and Problem Statement
В архітектурі передбачалися файли `ops/spawn-plan-<ts>.md` / `ops/spawn-fact-<ts>.md` і `ops/kill-plan-<ts>.md` / `ops/kill-fact-<ts>.md` для відновлення після збою при незавершеному spawn або kill залежних вузлів. Ці файли реалізовували патерн plan→action→fact на рівні операцій оркестратора.

## Considered Options
* Тримати `ops/` з plan/fact файлами для WAL-відновлення
* Видалити `ops/` — ігнорувати recovery-сценарій

## Decision Outcome
Chosen option: "Видалити `ops/`", because spawn відбувається всередині ворктрі агента — якщо ворктрі не змержено, граф залишається чистим без часткових вузлів; сценарій обриву посеред spawn вирішено ігнорувати на поточному етапі як малоймовірний.

### Consequences
* Good, because структура вузла спрощується — директорія `ops/` відсутня.
* Bad, because Neutral, because transcript не містить підтвердження наслідку для виробничої надійності.

## More Information
Файл: `npm/docs/mt.md`. Директорія `ops/` повністю видалена зі структури і схем.

## ADR Злиття `inputs.md` у `task.md` — єдиний вхідний файл вузла

## Context and Problem Statement
Спочатку планувалися два окремі файли: `task.md` (місія агента) і `inputs.md` (посилання на вхідні дані від батька). Агент при старті мав би читати обидва файли.

## Considered Options
* Окремі `task.md` і `inputs.md`
* Секція `## Inputs` всередині `task.md`

Chosen option: "Секція `## Inputs` всередині `task.md`", because агент читає один файл замість двох; місія і вхідні дані природно пов'язані і змінюються разом (при kill + edit обидва б редагувалися однаково).

### Consequences
* Good, because transcript фіксує очікувану користь: агент отримує повний контекст з одного читання.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл: `npm/docs/mt.md`, секція `### task.md`. Поле `budget_sec` також додано у фронтматер `task.md` замість окремого `repair_context.md`.

## ADR Англійські імена всіх файлів, директорій і атрибутів фронтматеру

## Context and Problem Statement
Архітектурний документ `npm/docs/mt.md` і схеми файлів починалися з українських назв (`місія.md`, `вхідні.md`, `вихідні.md`, `помилка.md`, `операції/`, `патчі/`, `підграф/`) і українських атрибутів (`батько:`, `залежності:`, `створено_о:`). Файли обробляються скриптами-оркестраторами.

## Considered Options
* Українські імена файлів і атрибутів
* Англійські імена файлів і атрибутів (snake_case для фронтматеру)

## Decision Outcome
Chosen option: "Англійські імена файлів і атрибутів", because файли і директорії обробляються скриптами — англійські імена уникають проблем з кодуванням, shell-екрануванням і сумісністю. Атрибути фронтматеру машинозчитувані → англійські, snake_case. Секції з довільними даними (тіло документу) залишаються будь-якою мовою.

### Consequences
* Good, because transcript фіксує очікувану користь: надійна обробка скриптами без проблем з кодуванням.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл: `npm/docs/mt.md`. Перейменовано: `місія.md`→`task.md`, `вхідні.md`→`inputs.md` (потім злито в `task.md`), `вихідні.md`→`outputs_NNN.md`, `помилка.md`→`error.md` (потім видалено), `операції/`→`ops/` (потім видалено), `патчі/`→`patches/` (потім видалено), `підграф/`→`subgraph/` (структура замінена на flat).
