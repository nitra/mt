## ADR Рекурсивний складений ОАГ з динамічним розкладом

## Context and Problem Statement
У системі є директорії з проєктами, всередині яких оркестратор розбиває задачі на dynamic workflow, формуючи граф задач. Вузли цього графу самі можуть розкладатись на підграфи, а задачі можуть існувати й на кореневому рівні директорії. Потрібно визначити, як точно описати цю структуру і яку архітектуру покласти в її основу.

## Considered Options
* Recursive Compound DAG із динамічним розкладом вузлів
* 3D-граф (просторова метафора)
* Multi-level Graph (акцент на рівнях, без рекурсії)
* Hierarchical Task Network (HTN)

## Decision Outcome
Chosen option: "Recursive Compound DAG із динамічним розкладом вузлів", because кожен вузол пред'являє батьку однаковий інтерфейс (`вирішено`/`помилка`) незалежно від того, атомарний він чи складений, а рішення про тип вузла приймається динамічно під час виконання на основі вхідних даних.

### Consequences
* Good, because transcript фіксує очікувану користь: замінюваність атомарного вузла складеним без змін у батьківському графі; executor однаковий на всіх рівнях.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Фінальна схема записана у `npm/docs/mt.md`. Структура вузла: `місія.md` + `вхідні.md` + `вихідні.md` + `помилка.md` + `repair_history.md` + `операції/` + `патчі/`. Ребра — data-flow (outputs → inputs). Кілька exit-вузлів у підграфі, їх outputs зливаються в outputs батька.

## ADR Файловий сховище стану — append-only Markdown із YAML-фронтматером

## Context and Problem Statement
Система потребує персистентного сховища стану графу задач, зручного для LLM-агентів і придатного для відновлення після збоїв. Обговорювались формати файлів і семантика запису.

## Considered Options
* Append-only Markdown-файли з YAML-фронтматером (кожна операція = новий файл)
* JSON-файли з мутацією in-place
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Append-only Markdown-файли з YAML-фронтматером", because LLM природно читає і продовжує Markdown-документи; `repair_history.md` як append-only журнал дозволяє агенту-інженеру читати всю історію спроб і писати далі без парсингу JSON; стан вузла визначається наявністю файлів, а не їх вмістом, що дає атомарність через файлову систему.

### Consequences
* Good, because transcript фіксує очікувану користь: `repair_history.md` — єдиний документ, LLM читає і продовжує; атомарність через `file create`; merge-конфлікти неможливі бо кожен ворктрі лише створює файли.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Стан вузла визначається набором наявних файлів: `місія.md`+`вхідні.md` = spawned; `вихідні.md` = resolved; `помилка.md` = failed; `знедійснений` (sentinel) = invalidated. `вхідні.md`/`вихідні.md` містять лише посилання на файли або рядки, не копії даних. `repair_history.md` — append-only. Нові патчі: `патчі/001-план.md` + `патчі/001-факт.md`.

## ADR Агент-інженер як мета-рівень із повним доступом і часовим бюджетом

## Context and Problem Statement
При помилці вузла потрібен механізм самовідновлення. Система повинна автономно виправляти збої, включно зі змінами структури графу, при цьому не зациклюватись нескінченно.

## Considered Options
* Агент-інженер як мета-рівень поза графом із необмеженим доступом і часовим бюджетом
* Агент-інженер як вузол у графі (рекурсивний)
* Retry з фіксованою кількістю спроб

## Decision Outcome
Chosen option: "Агент-інженер як мета-рівень поза графом із необмеженим доступом і часовим бюджетом", because інженер повинен мати змогу підніматись вгору і змінювати батьківські графи; time budget (не лічильник спроб) дає реалістичне обмеження з необмеженою кількістю спроб у межах вікна.

### Consequences
* Good, because transcript фіксує очікувану користь: memory на вузлі (`repair_history.md`) дозволяє інженеру не повторювати невдалі підходи; ескалація вгору по ієрархії з кожного рівня отримує свіжий time budget.
* Bad, because transcript фіксує ризик: cascade — інженер на кореневому рівні може інвалідувати весь граф і не вкластись у бюджет.

## More Information
`repair_context.json` у вузлі: `deadline`, `started_at`, `time_budget_sec`. Ескалаційний ланцюг: `вузол → батько → … → root → SeniorEngineer`. Кожен рівень отримує свіжий бюджет. `senior_report.json` формується при root-timeout. Протокол перед патчем батьківського вузла: записати `патч-план.md` → kill залежних ворктрі (топологічний порядок, від листів) → застосувати патч → записати `патч-факт.md`.

## ADR Паралельне виконання через git worktree із merge як єдиною точкою рішення

## Context and Problem Statement
Кілька агентів можуть одночасно працювати над різними вузлами графу. Потрібно забезпечити ізоляцію виконання і безконфліктне злиття результатів.

## Considered Options
* Git worktree на кожен вузол із merge як єдиною точкою рішення оркестратора
* Централізований лок-файл для координації
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Git worktree на кожен вузол із merge як єдиною точкою рішення оркестратора", because кожен вузол пише лише у свою директорію, тому merge-конфлікти між незалежними вузлами неможливі; ворктрі-межа є межею атомарності — до merge ізоляція, після — незворотній факт.

### Consequences
* Good, because transcript фіксує очікувану користь: гонка запис/скасування закрита архітектурно — merge ніколи не відбувається для скасованого ворктрі; git history = безкоштовний time-travel debugging.
* Bad, because transcript фіксує обмеження: жорсткі ліміти на кількість паралельних ворктрі через обмеження MacBook.

## More Information
Агенти запускаються wrapper-скриптом, який відловлює timeout і може kill процес. Freeze-сигнал при cascade кореня реалізується через kill процесу wrapper-скриптом. Кожен патч інженера виконується у власному ворктрі, який видаляється після завершення.

## ADR Топологія підграфу розподілена між місія.md дочірніх вузлів

## Context and Problem Statement
При розкладанні складеного вузла на підграф потрібно вирішити де зберігати топологію (ребра між дочірніми вузлами) — централізовано у батьку чи розподілено.

## Considered Options
* Топологія розподілена: кожен дочірній у своїй `місія.md` декларує власні залежності (`залежності:`)
* Централізований `граф.md` у батьківському вузлі

## Decision Outcome
Chosen option: "Топологія розподілена: кожен дочірній у своїй `місія.md` декларує власні залежності", because це забезпечує універсальність — оркестратор відновлює повний граф скануванням `місія.md` без центрального файлу, а динамічний spawn зводиться до появи нового `місія.md` без оновлення жодного існуючого файлу.

### Consequences
* Good, because transcript фіксує очікувану користь: динамічний spawn = новий файл без мутації існуючих; оркестратор підхоплює при наступному скануванні.
* Bad, because Neutral, because transcript не містить підтвердження наслідку.

## More Information
Агент-автор пише `місія.md` кожного дочірнього при spawn. Поле `залежності:` у YAML-фронтматері: список `id` попередників у межах батька. `вхідні.md` містить посилання (`посилання: ../збір-даних/вихідні.md#результати`), не копії. Динамічний spawn дозволений: агент може породжувати нові вузли після отримання часткових результатів.

## ADR Принцип план → дія → факт як universal write-ahead log

## Context and Problem Statement
При збої агента або оркестратора між двома операціями система може опинитись у невизначеному стані (наприклад, залежні вузли вбиті, але патч не застосований). Потрібен механізм відновлення.

## Considered Options
* Файловий write-ahead log: для кожної операції спочатку `*-план.md`, потім `*-факт.md`
* Зовнішня транзакційна БД
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Файловий write-ahead log: для кожної операції спочатку `*-план.md`, потім `*-факт.md`", because це консистентно з append-only файловою моделлю всієї системи; при відновленні достатньо порівняти план і факт щоб визначити де зупинились.

### Consequences
* Good, because transcript фіксує очікувану користь: оркестратор при старті сканує `операції/**/*-план.md` без відповідного `*-факт.md` → список незавершених операцій для відновлення.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Патерн застосовується до операцій: spawn (`spawn-план.md`/`spawn-факт.md`), kill залежних (`kill-план.md`/`kill-факт.md`), патч інженера (`патч-план.md`/`патч-факт.md`). Місія агента (`місія.md`) = план; `вихідні.md` = факт. Відновлення: `наявність *-план.md без *-факт.md` → операція не завершена → продовжити.
