## Суть проблеми #2

Є функція `resolveChangedScope` в `npm/rules/test/coverage/coverage.mjs:209-220`:

```js
function resolveChangedScope(cwd) {
let base = null
try {
const state = readState(flowStatePath(cwd))
base = state?.metadata?.base_commit ?? null   // ← ось де проблема
} catch {
console.error('coverage --changed: стан flow нечитабельний — scope від HEAD…')
base = null
}
return { base, files: collectChangedFilesSince(base, cwd) }
}
```

Вона читає MT file-presence state щоб дістати `base_commit` — коміт, від якого брати diff для gate.

**Але:** `readState` на **відсутньому** файлі повертає `null` — не кидає помилку. Тому:
- `catch`-блок **не** спрацьовує → warn-повідомлення **не** виводиться
- `base = null` → тихо

Далі `collectChangedFilesSince(null, cwd)` → fallback на `git diff HEAD` = лише uncommitted vs HEAD.

## Де це ламається

Flow-executor **комітить кожен крок** перед `coverage --changed`. Тому після коміту uncommitted diff = `[]` → scope = `[]`. Оркестратор (`coverage.mjs:250-255`):

```js
if (opts.changed && scope.files.length === 0) {
console.log('✓ coverage --changed: немає змінених файлів → pass')
return 0
}
```

**Результат:** при відсутньому MT file-presence state (або втраченому стані) гейт **мовчки проходить** після коміту, не перевіривши закомічені зміни. Це пряма протилежність обіцянці у `flow.mdc`.

## Чому не просто throw

HEAD-fallback **навмисний** для запуску `coverage --changed` **поза flow** — наприклад, якщо розробник хоче запустити coverage по uncommitted змінах без `mt init`. JSDoc це описує. Тобто:

- Поза flow → `base=null` → diff HEAD → ✅ правильна поведінка
- В flow (стан є) → `base=commit_id` → diff від base → ✅ правильна поведінка
- В flow, але **стан зник** → `base=null` → тихий fallback → ❌ неочевидний баг

Нинішній код не розрізняє «поза flow (ОК)» і «в flow, але стан зник (не ОК)».

## Три варіанти фіксу

**Варіант A — мінімальний (warn завжди):**
Додати `console.error` у `catch`-блок і ще один warn коли `readState` повернув `null` (не тільки коли кинув). Поведінка не змінюється, але стає гучною:

```js
const state = readState(flowStatePath(cwd))
if (!state) console.error('coverage --changed: стан flow відсутній — scope від HEAD (uncommitted)')
base = state?.metadata?.base_commit ?? null
```
Переваги: нічого не ламає, зовнішній запуск поза flow продовжує працювати. Недолік: гейт все ще не падає — проблема "в тиші" стає проблемою "вголос", але не виправляється.

---

**Варіант B — location-aware (рекомендований):**
Якщо `cwd` лежить під `.worktrees/` (тобто ми всередині flow-ізоляції) — fail-closed при відсутньому стані. Поза `.worktrees/` — лишити HEAD-fallback.

```js
const state = readState(flowStatePath(cwd))
if (!state && cwd.includes('/.worktrees/')) {
throw new Error('coverage --changed у worktree без flow-стану — запусти `mt init` спочатку')
}
base = state?.metadata?.base_commit ?? null
```
Переваги: зберігає «зовнішній» use-case, правильно fail-closed у flow. Недолік: прив'язаний до `.worktrees/` шляху — крихкий якщо worktree в іншому місці.

---

**Варіант C — fail-closed завжди:**
Якщо нема стану — кидаємо помилку незалежно від місця. Поза flow треба явно `coverage` (без `--changed`) або `coverage --changed --no-flow`.

Переваги: максимально суворо. Недолік: ламає зручний dev-workflow поза flow.

---

**Моя рекомендація — варіант B** (location-aware). Він вирішує реальний баг, не ламає зовнішній use-case і відповідає вже наявній конвенції `.worktrees/` у проєкті.
