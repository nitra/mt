## ADR Уніфікований `run_NNN.md` — єдиний запис будь-якої спроби вузла

## Context and Problem Statement
Під час ітеративного проєктування файлової схеми для системи Динамічного Самомодифікованого Графу Задач виявилося, що задля відстеження стану вузла передбачалися окремі файли: `error.md` (аварійні завершення), `outputs.md` (append-only результати), `repair_history.md` (журнал спроб інженера), `repair_context.md` (часовий бюджет). Це плодило кілька різних структур для семантично однакової концепції — «одна спроба виконати вузол».

## Considered Options
* Окремі файли: `error.md` (append-only), `outputs.md` (append-only), `repair_history_NNN.md`, `repair_context.md`
* Єдиний `run_NNN.md` з полем `actor` — один файл на спробу для будь-якого виконавця

## Decision Outcome
Chosen option: "Єдиний `run_NNN.md` з полем `actor`", because і звичайний агент, і інженер, і людина — це «спроба вирішити вузол»; спільна структура усуває дублювання та спрощує скани стану.

### Consequences
* Good, because `mt scan` може читати стан одним патерном `run_*.md` / `outputs_NNN.md` без розгалуження по типах файлів.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл `run_NNN.md` — frontmatter: `created_at`, `actor: agent|engineer|human|auditor`, `result: success|failed`, опційно `worktree:`; секції `## Reasoning` (обов'язково), `## Script` (wrapper при збої), `## Ref` (посилання на `outputs_NNN.md` або відсутнє). `outputs_NNN.md` і `run_NNN.md` — окремі immutable файли; нумерація NNN (zero-padded 3 цифри) — wrapper рахує `count(run_*.md) + 1` перед стартом. Схема зафіксована у `npm/docs/mt.md`.

## ADR Append-only інваріант починається з моменту створення worktree

## Context and Problem Statement
Система будується на принципі immutable файлів для забезпечення безпечного паралелізму через git worktrees. Виникло питання: чи поширюється заборона на зміну файлів також на період до запуску агента — тобто чи можна редагувати `task.md` до того як worktree створено?

## Considered Options
* Append-only / immutable з моменту створення файлу
* Append-only / immutable лише після створення worktree для вузла

## Decision Outcome
Chosen option: "Append-only / immutable лише після створення worktree", because до моменту `git worktree add` жоден агент ще не читає файли вузла; межа чітка і відповідає межі атомарності.

### Consequences
* Good, because інженер може вільно редагувати `task.md` після `mt kill` (знищення worktree), перезапускаючи вузол без версіонування файлів.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Протокол зміни: якщо worktree відсутній — редагуй `task.md` вільно; якщо worktree існує — `mt kill` → редагуй вільно → `mt run`. Зафіксовано у розділі «Інваріант незмінності» `npm/docs/mt.md`.

## ADR Автоматичний тригер наступника через git post-merge hook

## Context and Problem Statement
Після завершення агента і запису `run_NNN.md` / `outputs_NNN.md` система має запустити наступників у графі. Постало питання про механізм тригера: хто і як запускає `mt run` на вузли, чиї залежності щойно стали `resolved`.

## Considered Options
* Людина вручну запускає кожен вузол після merge
* `mt run` як демон-цикл, що сам оркеструє весь граф
* Wrapper авто-мержить після успіху → git `post-merge` hook → `mt run --auto`

## Decision Outcome
Chosen option: "Wrapper авто-мержить → git post-merge hook → `mt run --auto`", because кожен компонент робить одне; `mt run` залишається одноразовою командою; hook — стандартний git-механізм без зовнішніх залежностей.

### Consequences
* Good, because transcript фіксує очікувану користь: людина запускає лише кореневий вузол, решту система доводить до кінця сама.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Ланцюг: агент завершує → wrapper пише `run_NNN.md` → wrapper робить `git merge` worktree → git `post-merge` hook → `mt run --auto` (сканує граф, запускає всіх вузлів зі статусом `ready`). Зафіксовано у wrapper-скрипті та CLI-контракті `npm/docs/mt.md`.

## ADR Видалення `ops/` та `patches/` директорій

## Context and Problem Statement
Початковий дизайн включав директорії `ops/` (файли `spawn-plan-<ts>.md`, `spawn-fact-<ts>.md`, `kill-plan-<ts>.md`, `kill-fact-<ts>.md`) і `patches/` (`NNN-plan.md`, `NNN-fact.md`) для реалізації патерну plan→action→fact та відновлення після збою при перерваних операціях.

## Considered Options
* Зберегти `ops/` і `patches/` для crash-recovery через порівняння plan/fact пар
* Видалити обидві директорії; інженер документує зміни в `## Reasoning` секції `run_NNN.md`

## Decision Outcome
Chosen option: "Видалити `ops/` і `patches/`", because сценарій обриву при spawn відбувається тільки якщо spawn іде напряму в main без worktree — а в поточному дизайні весь spawn відбувається всередині worktree, тому незавершений spawn просто не мержиться; patches дублюють інформацію яка вже є в `run_NNN.md`.

### Consequences
* Good, because файлова структура вузла мінімальна: `task.md`, `invalidated`, `run_NNN.md`, `outputs_NNN.md`, `artifacts/`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Після видалення crash-recovery для `mt scan` спрощується до: вузли без `outputs_*.md` при наявності `run_*.md` = `failed` → перезапустити. Зафіксовано у `npm/docs/mt.md`.
