## ADR Рекурсивний складений ОАГ як структура для динамічного графу задач

## Context and Problem Statement
У системі є директорія з проєктами, де оркестратор розбиває задачі на dynamic workflow, що формує граф задач. Вузли всередині графу самі можуть розкладатись на підграфи, а задачі існують і на рівні кореневої директорії. Потрібно точно назвати і формалізувати цю структуру.

## Considered Options
* Recursive Compound DAG (Рекурсивний складений ОАГ)
* 3D-граф
* HTN (Hierarchical Task Network)
* Multi-level Graph

## Decision Outcome
Chosen option: "Рекурсивний складений ОАГ", because кожен вузол або є атомарним, або розкладається на підграф — інтерфейс для батька однаковий в обох випадках; батько чекає лише на `resolved`-стан, не знаючи що всередині.

### Consequences
* Good, because transcript фіксує очікувану користь: атомарний вузол можна "розкрити" в підграф без змін у батьківському графі (substitutability).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`npm/docs/mt.md`; виконавець: `execute(node) → if isAtomic → run, else → execute(node.graph), wait exit-nodes`.

## ADR Файлова система як append-only сховище стану вузлів

## Context and Problem Statement
Вузли графу задач потребують персистентного сховища для стану, вхідних/вихідних даних та журналу відновлення. Вимога — атомарність запису та LLM-first формат.

## Considered Options
* Append-only файлова система (файли лише створюються, не змінюються)
* Mutable JSON-файли зі станом

## Decision Outcome
Chosen option: "Append-only файлова система", because `create file` атомарна на більшості ФС; стан вузла визначається наявністю файлів, а не їх вмістом; два ворктрі створюють лише нові файли з унікальними іменами — `git merge` завжди чистий.

### Consequences
* Good, because transcript фіксує очікувану користь: відсутність конфліктів при злитті ворктрі, безкоштовна персистентність і аудит.
* Bad, because для визначення поточного стану треба сканувати файли без індексу — при великих графах дорого.

## More Information
Файлова структура у `npm/docs/mt.md`: `місія.md`, `вхідні.md`, `вихідні.md`, `помилка.md`, `операції/*-план.md`, `операції/*-факт.md`, `патчі/*-план.md`, `патчі/*-факт.md`; формат — Markdown з YAML-фронтматером.

## ADR Markdown з YAML-фронтматером як LLM-first формат файлів вузлів

## Context and Problem Statement
Файли вузлів мають читатись і записуватись як LLM-агентами (інженерами), так і скриптами-оркестраторами. Потрібен формат зручний для обох.

* Markdown з YAML-фронтматером
* JSON

## Decision Outcome
Chosen option: "Markdown з YAML-фронтматером", because фронтматер несе машинозчитувані поля (тип, ребра, мітки часу), тіло — вільний Markdown для LLM; `repair_history.md` як append-only журнал — інженер читає єдиний документ і продовжує природно, без парсингу JSON-масиву.

### Consequences
* Good, because transcript фіксує очікувану користь: `repair_history.md` — LLM читає і дописує як текст без реконструкції контексту.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

## ADR Git-ворктрі як межа атомарності для паралельного виконання вузлів

## Context and Problem Statement
Кілька агентів виконують незалежні вузли паралельно. Потрібно уникнути конфліктів запису та гонок між записом `вихідні.md` і сигналом скасування від інженера.

## Considered Options
* Git-ворктрі як межа атомарності (merge = єдина точка рішення)
* Атомарна перевірка-запис у shared filesystem

## Decision Outcome
Chosen option: "Git-ворктрі як межа атомарності", because наступники не стартують поки ворктрі не змерджено; інженер опрацьовує стан на етапі merge — не принципово чи встигне агент записати стан до сигналу скасування.

### Consequences
* Good, because transcript фіксує очікувану користь: паралельне виконання без конфліктів (різні директорії `tasks/<вузолId>/`), race condition закрито архітектурно.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`npm/docs/mt.md`, розділ "Паралельне виконання через ворктрі".

### Consequences
* Good, because transcript фіксує очікувану користь: оркестратор при старті сканує `tasks/**/*-план.md` і відновлює незавершені операції.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

## ADR АгентІнженер як мета-рівень із правом топологічного kill залежних перед патчем

## Context and Problem Statement
АгентІнженер може змінювати граф на будь-якому рівні, в т.ч. вузли що вже мають запущені залежні вузли у ворктрі. Потрібне правило яке запобігає інконсистентному стану.

## Considered Options
* Спочатку kill залежних (від листів до цілі), потім патч
* Патч з одночасним cascade-скасуванням

## Decision Outcome
Chosen option: "Спочатку kill залежних (топологічний порядок), потім патч", because не можна модифікувати вузол поки залежний від нього вузол виконується у ворктрі; kill від листів до цілі → патч → записати факт → рестарт каскаду.

### Consequences
* Good, because transcript фіксує очікувану користь: жоден залежний агент не продовжує роботу з застарілими вхідними після патчу.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`npm/docs/mt.md`, розділ "Протокол патчу залежного вузла"; порядок: записати `патч-план.md` → kill від листів → застосувати патч → записати `патч-факт.md` → рестарт.

## ADR Часовий бюджет (не лічильник спроб) як механізм збіжності АгентаІнженера

## Context and Problem Statement
АгентІнженер може робити безліч спроб відновлення вузла. Потрібен механізм що зупиняє нескінченний цикл і при цьому не обмежує кількість корисних спроб.

## Considered Options
* Time budget (фіксований час на весь процес відновлення)
* Max attempts (ліміт кількості спроб)

Chosen option: "Time budget", because часовий бюджет — реалістичне обмеження; інженер адаптує стратегію залежно від залишку часу (багато часу → складна спроба, мало → швидкий fix); кожен рівень ієрархії отримує свіжий бюджет при ескалації.

### Consequences
* Good, because transcript фіксує очікувану користь: максимальний час до SeniorEngineer передбачуваний: `глибина × бюджет`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`npm/docs/mt.md`, розділ "Збіжність: часовий бюджет"; `repair_context.json`: поля `deadline`, `started_at`, `time_budget_sec`.
