## ADR Рекурсивний складений ОАГ як основна структура графу задач

## Context and Problem Statement
Потрібно описати структуру системи оркестрації, де задачі розбиваються на динамічний workflow з вкладеними підграфами; вузли можуть містити власні підграфи або бути атомарними; і те саме стосується будь-якого рівня ієрархії, включно з кореневим.

## Considered Options
* Recursive Compound DAG (рекурсивний складений ОАГ)
* Hierarchical Task Network (HTN)
* Multi-level Graph
* "3D graph" (відкинуто одразу — описує просторове відображення, а не структуру зв'язків)

## Decision Outcome
Chosen option: "Рекурсивний складений ОАГ", because кожен вузол або є атомарним (`fn(вхідні) → вихідні`), або розкладається на підграф; батьківський рівень бачить лише `стан: вирішено | помилка` незалежно від внутрішньої складності — це чиста інкапсуляція.

### Consequences
* Good, because transcript фіксує очікувану користь: атомарний вузол можна "розкрити" в підграф без змін у батьківському графі (substitutability); executor однаковий на всіх рівнях.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Зафіксовано у `npm/docs/mt.md`. Назви рівнів у схемі: `Атомарний`, `Складений`, `Граф`, `ребра`, `виходи[]`.

## ADR Динамічний розклад вузла при запуску (агент вирішує сам)

## Context and Problem Statement
Вузол може виявитись як простим, так і складним залежно від вхідних даних; тип не можна визначити наперед при побудові графу.

## Considered Options
* Статичне оголошення типу в `meta.md` (тип відомий наперед)
* Динамічне рішення агента при запуску на основі `inputs.md`

## Decision Outcome
Chosen option: "Динамічне рішення агента при запуску", because агент читає `inputs.md` і самостійно визначає чи задача атомарна чи потребує підграфу — поле `тип` відсутнє у `meta.md`.

### Consequences
* Good, because transcript фіксує очікувану користь: граф не потребує переоголошення при зміні складності вузла; батьківський граф не зачіпається.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`npm/docs/mt.md`, розділ "Вузол як агент". Схема виконання: `агент читає inputs.md → виконує fn() або створює graph/<дочірній>/{meta.md, inputs.md}`.

## ADR Markdown + YAML фронтматер як формат файлів вузла

## Context and Problem Statement
Файли вузла мають читатись як агентами (LLM), так і оркестратором (машина); потрібен формат зручний перш за все для LLM.

## Considered Options
* JSON (`.json` файли)
* Markdown + YAML фронтматер (`.md` файли)

## Decision Outcome
Chosen option: "Markdown + YAML фронтматер", because фронтматер містить машинозчитувані поля (стан, ребра, мітки часу), а тіло — вільний Markdown для LLM; `repair_history.md` як append-only документ дозволяє інженеру читати весь контекст і продовжувати природно без парсингу JSON-масиву.

### Consequences
* Good, because transcript фіксує очікувану користь: `repair_history.md` читається як єдиний документ — не треба реконструювати контекст з JSON-масиву.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`npm/docs/mt.md`, розділ "Формати файлів (LLM-first)". Файли: `meta.md`, `inputs.md`, `outputs.md`, `error.md`, `repair_history.md`. Поле `застарілі: true` у фронтматері `outputs.md` при каскаді інвалідації.

## ADR Протокол агента через system prompt оркестратора + CLI хуки

## Context and Problem Statement
Кожен вузол — запуск Claude; потрібно визначити де живе протокол взаємодії (як агент сигналізує про результат) і як оркестратор отримує сповіщення.

## Considered Options
* Протокол у `meta.md` (агент self-contained, читає протокол зі своїх файлів)
* Протокол у `system_prompt` оркестратора + Push через CLI хуки

## Decision Outcome
Chosen option: "Протокол у system_prompt оркестратора + Push через CLI хуки", because файли вузла містять лише місію та дані; агент не бачить протокол у своїх файлах — це відповідальність оркестратора. Push-модель: агент сам викликає CLI після запису файлів.

### Consequences
* Good, because transcript фіксує очікувану користь: файли вузла залишаються чистими (місія + дані), не змішуються з інфраструктурним протоколом.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`npm/docs/mt.md`, розділ "Вузол як агент". CLI команди: `mt done <шлях>`, `mt spawn <шлях>`, `mt failed <шлях>`. Порядок: спочатку запис файлів (`outputs.md` / дочірні `meta.md+inputs.md` / `error.md`), потім CLI хук.

## ADR АгентІнженер як мета-рівень з необмеженим доступом до графу

## Context and Problem Statement
При помилці вузла потрібен механізм відновлення; питання — чи інженер є вузлом у графі або окремим мета-рівнем, і чи може він змінювати батьківські графи.

## Considered Options
* Інженер як вузол у графі (рекурсивний виклик)
* Інженер як мета-рівень поза графом з необмеженим доступом

## Decision Outcome
Chosen option: "Мета-рівень поза графом з необмеженим доступом", because інженер stateless, може підніматись до кореня і змінювати будь-який рівень ієрархії; граф при цьому — mutable with invalidation (змінений вузол інвалідує наступників).

### Consequences
* Good, because transcript фіксує очікувану користь: можна усувати причини помилок на будь-якому рівні абстракції, а не лише локально.
* Bad, because необмежений доступ + mutable invalidation потребує `repair_history.md` та `MutationLog` для відстеження; без них можливий нескінченний цикл патчів.

## More Information
`npm/docs/mt.md`, розділи "Самовідновлення: агент-інженер" та "Каскад інвалідації". `outputs.md` фронтматер: `застарілі: true` при патчі. `repair_history.md` записується на вузлі де відбулась зміна; дочірній вузол фіксує `triggered_parent_patch`.

## ADR Часовий бюджет замість ліміту спроб для збіжності відновлення

## Context and Problem Statement
Без обмежень інженер може нескінченно намагатись виправити вузол; потрібен convergence guard.

## Considered Options
* Ліміт кількості спроб (`max_attempts: N`)
* Часовий бюджет (`budget_sec: 600`) з необмеженою кількістю спроб

## Decision Outcome
Chosen option: "Часовий бюджет", because реальне обмеження — час (вартість, wall clock), а не довільна кількість спроб; інженер може адаптувати стратегію (складна спроба при великому залишку, швидкий патч при малому).

### Consequences
* Good, because transcript фіксує очікувану користь: інженер не обрізається штучно при правильному напрямку; адаптивна стратегія.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`npm/docs/mt.md`, розділ "Збіжність: часовий бюджет". Поля у фронтматері `repair_history.md`: `бюджет_сек`, `дедлайн`, `розпочато_о`. При `залишок ≤ 0` → `стан = невирішуваний` → `mt failed`.

## ADR Ієрархічна ескалація зі свіжим бюджетом на кожному рівні

## Context and Problem Statement
Після вичерпання часового бюджету на рівні вузла потрібно вирішити що відбувається далі і чи отримує батьківський рівень той самий або новий бюджет.

## Considered Options
* Батько успадковує залишок від дочірнього бюджету
* Кожен рівень отримує свіжий бюджет при ескалації; корінь → сповіщення СтаршогоІнженера

## Decision Outcome
Chosen option: "Свіжий бюджет на кожному рівні", because максимальний час до людини передбачуваний: `глибина_графу × бюджет`; корінь після timeout викликає `mt escalate root` і сповіщає СтаршогоІнженера (людина або вищий агент) через `senior_report.md`.

### Consequences
* Good, because transcript фіксує очікувану користь: передбачуваний worst-case час до залучення людини.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`npm/docs/mt.md`, розділи "Ієрархія ескалації" та "Сповіщення старшого інженера". `senior_report.md` містить: `вузол_що_впав`, таблицю ланцюга ескалації (рівень / витрачено / спроб), поточний стан графу, пропозиції.
