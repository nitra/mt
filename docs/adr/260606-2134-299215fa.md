## Stage 1 — `mt plan`: детальний дизайн

### Вхід

Агент вже всередині worktree (попав туди через `mt run`). Запускає `mt plan`.

**Читає:**
- `task.md` поточного вузла (`## Task`, `## Done when`, `## Inputs`, metadata)
- `.n-cursor/system-prompt.md` (контекст проекту)

---

### Атрибути task.md

```yaml
---
actor: agent | human   # default: human
budget_sec: 3600
created_at: 2026-06-06T18:00:00Z
---
```

**Питання 1:** `actor` контролює обидві точки огляду (plan + output), чи потрібні окремі поля?

```yaml
# Варіант A — один actor
actor: human  # огляд і plan, і output

# Варіант B — роздільно
plan_review: true   # людина апрувить plan.md перед solve
output_review: true # людина апрувить outputs перед merge (= mt audit)
```

Ти сказав що «деякий % задач на людину чекати не потрібно» — це більше схоже на варіант B, де задача точно вказує **де** потрібен огляд.

### LLM-рішення (atomic vs composite)

```
mt plan
↓
prompt: "читаєш task.md → composite чи atomic?"
↓
composite → writes tasks/<current>/<child-A>/task.md
writes tasks/<current>/<child-B>/task.md
→ mt spawn   (wrapper пише run_NNN.md outcome=spawned)

atomic    → writes plan.md
(якщо plan_review:true → сигнал awaiting-plan → пауза)
→ Stage 2
```

**Питання 2:** Коли LLM вирішує composite — чи дочірні задачі отримують `plan_review`/`output_review` від батька автоматично (inherit), чи LLM сам прописує їх в кожному дочірньому `task.md`?
