## ADR Teleport як SSH gateway для доступу розробників до k8s dev pods

## Context and Problem Statement
Dev pods із `tasks-pvc` живуть у k8s, але розробники не мають прав `kubectl`. Потрібен gateway що контролює доступ через бекенд `nitra/task`, а не через k8s RBAC напряму.

## Considered Options
* `kubectl port-forward` + SSH у поді (відхилено: потребує `kubectl` у розробника)
* Teleport Auth+Proxy з node-agent у dev pod
* Власний SSH gateway з кастомною авторизаційною логікою
* Tailscale / WireGuard у поді

## Decision Outcome
Chosen option: "Teleport Auth+Proxy з node-agent у dev pod", because вирішує саме задану умову — бекенд контролює доступ через RBAC-label `owner: {{internal.logins}}`, розробник не має `kubectl`, аудит з коробки, Zed підключається через стандартний SSH без патчів.

### Consequences
* Good, because transcript фіксує очікувану користь: короткоживучі X.509/SSH сертифікати (TTL 8–24 год), per-task RBAC через labels, один `ProxyCommand` у `~/.ssh/config` для всіх редакторів.
* Bad, because потрібно задеплоїти Teleport Auth+Proxy один раз (Helm chart); transcript фіксує це як складність.

## More Information
Компоненти: Teleport Auth Server + Proxy (один Deployment), Teleport node-agent (sidecar у dev pod), Teleport Operator (CRD), SSO provider (GitHub OAuth або Google).

SSH config:
```
Host *.teleport.nitra.com
ProxyCommand tsh proxy ssh --cluster=nitra %h:%p
User dev
```

Pod lifecycle: spawn on request → grace period після закриття сесії → auto-delete після timeout без активності.

## ADR On-demand spawn dev pods через бекенд `nitra/task`

## Context and Problem Statement
Dev pods мають з'являтися динамічно коли розробник запитує доступ до конкретного task-node, а не бути постійно запущеними. Бекенд повинен оркеструвати весь цикл.

## Considered Options
* Попередньо запущені (pre-warm) dev pods
* On-demand spawn через бекенд при запиті розробника

## Decision Outcome
Chosen option: "On-demand spawn через бекенд при запиті розробника", because користувач явно підтвердив цей паттерн; бекенд перевіряє права → `kubectl apply dev-pod.yaml` → чекає Ready → Teleport реєстрація → повертає connection string.

### Consequences
* Good, because transcript фіксує очікувану користь: ресурси не витрачаються на idle pods; lifecycle керується автоматично (grace period + timeout).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Задача `nitra/task/tasks/open-in-editor/task.md` описує повний цикл реалізації. Pod label: `task=<node-name>`, `owner=<email>`. Команда запуску: `mt run tasks/<name>`.

## ADR Підтримка VS Code, Cursor і Zed у кнопці "Open in Editor"

## Context and Problem Statement
Задача `open-in-zed` спочатку охоплювала лише Zed. Але з'ясувалося що VS Code і Cursor підтримують URI deep link що автоматично відкриває редактор із SSH-підключенням — без ручних кроків. Потрібна єдина кнопка для всіх трьох редакторів.

## Considered Options
* Підтримка лише Zed
* Підтримка VS Code, Cursor і Zed (обрано)

## Decision Outcome
Chosen option: "Підтримка VS Code, Cursor і Zed", because користувач підтвердив («так») після того як transcript зафіксував перевагу URI deep link для VS Code і Cursor (`vscode://vscode-remote/...`, `cursor://vscode-remote/...`), а також обмеження Zed (відсутність URI-протоколу).

### Consequences
* Good, because transcript фіксує очікувану користь: VS Code і Cursor відкриваються одним кліком через URI; Zed — через копіювання hostname з інструкцією.
* Bad, because Zed не підтримує URI deep link — UX для Zed гірший ніж для двох інших редакторів (transcript це фіксує явно).

## More Information
URI схеми:
```

# Cursor
cursor://vscode-remote/ssh-remote+<hostname>.teleport.nitra.com/tasks
```
