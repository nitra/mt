## ADR Рекурсивний складений ОАГ як структура для динамічного графу задач

## Context and Problem Statement
Система оркеструє задачі у вигляді ієрархічних проєктів. Задачі можуть рекурсивно розбиватись на підграфи під час виконання, і потрібна чітка назва та формальна модель цієї структури для подальшого проєктування.

## Considered Options
* Recursive Compound DAG (рекурсивний складений ОАГ)
* 3D-граф
* Flat hierarchical task network

## Decision Outcome
Chosen option: "Рекурсивний складений ОАГ", because кожен вузол або є атомарним (виконується безпосередньо), або розкладається у внутрішній ОАГ — і для батьківського рівня ці два варіанти неrozрізнені: батько чекає лише на `стан = вирішено`. 3D-граф відкинуто, бо описує просторовий вимір, а не структуру вкладеності.

### Consequences
* Good, because батько не знає про внутрішній устрій вузла — атомарний можна замінити складеним без змін у батьківському графі (substitutability).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Фінальна модель: `Вузол { стан, вхідні, вихідні, реалізація: Атомарний | Складений{Граф} }`. Ребра несуть дані (вихідні одного вузла → вхідні наступного). Складений вузол може мати кілька вихідних вузлів, чиї результати зливаються у `вихідні` батька. Зафіксовано у `npm/docs/mt.md`.

## ADR Інженер як мета-рівень поза графом з необмеженим доступом

## Context and Problem Statement
При помилці вузла потрібен механізм відновлення. Питання: чи є агент-інженер вузлом у графі, чи існує поза ним — і який рівень доступу до структури графу він повинен мати?

## Considered Options
* Інженер як мета-рівень поза графом з необмеженим доступом
* Інженер як звичайний вузол у графі

## Decision Outcome
Chosen option: "Інженер як мета-рівень поза графом з необмеженим доступом", because він повинен мати можливість підніматись вгору і змінювати батьківський граф, що унеможливлює його розміщення всередині самого графу.

### Consequences
* Good, because transcript фіксує очікувану користь: `EngineerAgent` бачить повний шлях від кореня до вузла що впав і сам вирішує на якому рівні втручатись.
* Bad, because необмежений доступ створює ризик каскадних перебудов — без convergence guard можливий нескінченний цикл перепроєктування.

## More Information
`EngineerAgent(помилка, повний_шлях_від_кореня, repair_history)` → `GraphPatch`: замінити вузол / вставити вузли / перепідключити ребра / змінити вхідні. При патчі батька memory пишеться на батька, дочірній зберігає `{ triggered_parent_patch: patch_id }`. Зафіксовано у `npm/docs/mt.md`.

## ADR Часовий бюджет замість ліміту спроб для відновлення вузла

## Context and Problem Statement
Без обмеження на роботу інженера можливий нескінченний цикл виправлень. Потрібно визначити convergence guard.

## Considered Options
* Часовий бюджет (N хвилин, необмежена кількість спроб)
* Ліміт кількості спроб (max_attempts: N)

## Decision Outcome
Chosen option: "Часовий бюджет", because він дає інженеру свободу у виборі стратегії (багато часу → складна спроба, мало часу → швидкий fix) без штучного обрізання корисних спроб.

### Consequences
* Good, because максимальний час до ескалації передбачуваний: `глибина × бюджет`. При ескалації кожен батьківський рівень отримує свіжий бюджет.
* Good, because transcript фіксує очікувану користь: `залишок_часу = дедлайн - now()` визначає стратегію інженера.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`repair_context.json`: `deadline`, `started_at`, `time_budget_sec`, `attempts[]`. Root-рівень timeout → повідомлення `SeniorEngineer` з `senior_report` (ланцюг ескалації, знімок графу, suggested_next_steps). Зафіксовано у `npm/docs/mt.md`.

## ADR Пам'ять інженера зберігається на вузлі (файл `repair_history.md`)

## Context and Problem Statement
Інженер stateless — між викликами він не пам'ятає що вже пробував на конкретному вузлі. Без пам'яті він неминуче повторюватиме невдалі підходи.

## Considered Options
* Пам'ять живе на вузлі (файл поряд із `meta.md`)
* Пам'ять живе в стані самого агента-інженера

## Decision Outcome
Chosen option: "Пам'ять живе на вузлі", because знання про вузол повинні мандрувати разом із вузлом, а не залежати від конкретного екземпляру агента.

### Consequences
* Good, because кілька викликів `EngineerAgent` на одному вузлі (навіть різних агентів) бачать спільну історію і не повторюють невдалі стратегії.
* Good, because `repair_history.md` — append-only журнал: LLM читає як єдиний документ і природно дописує наступну спробу без JSON-парсингу.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Структура: `## Спроба N — ISO8601 / **Міркування:** / **Патч:** / **Результат:**`. Файлова структура вузла: `meta.md`, `inputs.md`, `outputs.md`, `error.md`, `repair_history.md`, підкаталог `граф/` для складених вузлів. Зафіксовано у `npm/docs/mt.md`.

## ADR Markdown + YAML-фронтматер як формат файлів стану вузла

## Context and Problem Statement
Файли стану вузлів читаються і записуються насамперед LLM-агентами. Потрібний формат що є зручним для LLM і водночас машинозчитуваним для планувальника.

## Considered Options
* Markdown з YAML-фронтматером
* Чистий JSON

## Decision Outcome
Chosen option: "Markdown з YAML-фронтматером", because YAML-фронтматер містить структуровані поля (стан, тип, ребра, мітки часу), а тіло Markdown дозволяє LLM читати і писати опис, міркування та контекст природною мовою.

### Consequences
* Good, because `repair_history.md` є append-only журналом: інженер читає як єдиний документ і продовжує природно, без реконструкції контексту з JSON-масиву.
* Good, because паттерн вже використовується у цьому проєкті (`.mdc`-файли, ADR).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`meta.md`: фронтматер `id`, `тип`, `стан`, `ребра.від[]`, `ребра.до[]`; тіло — опис призначення. `error.md`: фронтматер `час`, `тип`; тіло — опис помилки і що вже пробували. JSON-ключі у прикладах файлів залишаються англійськими (майбутній реальний код). Зафіксовано у `npm/docs/mt.md`.

## ADR Паралельне виконання вузлів через git-worktrees з агентом-медіатором для конфліктів

## Context and Problem Statement
Незалежні вузли ОАГ можуть виконуватись паралельно різними агентами. Потрібно визначити механізм ізоляції та злиття.

## Considered Options
* Паралельне виконання через окремі git-worktrees, злиття через git merge
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "git-worktrees з агентом-медіатором", because щасливий шлях (незалежні вузли пишуть у різні директорії `tasks/<вузол_id>/`) не породжує конфліктів за визначенням; складний шлях (інженер у ворктрі A патчить спільний батьківський `meta.md` поки агент у ворктрі B ще виконується) вирішується окремим агентом-медіатором — та сама симетрія що й відновлення вузла.

### Consequences
* Good, because transcript фіксує очікувану користь: паралельне виконання = git-гілки, злиття = `git merge`, конфлікти = нова задача на відновлення — всі шари системи симетричні.
* Bad, because Neutral, because transcript не містить підтвердження наслідку для продуктивності або складності реалізації агента-медіатора.

## More Information
Кожен агент-вузол працює у своєму worktree (`.worktrees/<назва>/`). Після завершення — merge до основної гілки. Конфлікт при злитті на `meta.md` батька → spawn агента-медіатора. Зафіксовано у `npm/docs/mt.md`.
