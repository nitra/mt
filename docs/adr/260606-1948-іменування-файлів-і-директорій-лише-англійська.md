## Context and Problem Statement

## Considered Options
* Зберегти українські назви де вони є
* Усі імена файлів і директорій — лише англійська; вміст файлів — довільна мова

## Decision Outcome
Chosen option: "Усі імена файлів і директорій — лише англійська", because скрипти обробляють шляхи напряму і користувач зафіксував це як контракт системи.

### Consequences
* Good, because скрипти і CLI-команди безпечно обробляють шляхи без escape-логіки.
* Bad, because transcript не містить підтверджених негативних наслідків.

## Considered Options
* Зберегти `outputs.md`, `error.md`, `repair_history.md` як append-only файли
* Уніфікувати всі спроби в immutable numbered файли: `run_NNN.md` (спроба) + `outputs_NNN.md` (результат)

## Decision Outcome
Chosen option: "Уніфікувати всі спроби в immutable numbered файли", because всі дії в системі — агент, інженер, людина, аудитор — є "спробами вирішити вузол" з однаковою структурою; жоден файл не повинен мінятись після створення.

### Consequences
* Good, because transcript фіксує очікувану користь: жодних append-only файлів, одна схема для всіх акторів, стан вузла однозначно виводиться зі сканування файлів.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`run_NNN.md` — frontmatter: `created_at`, `actor: agent|engineer|human|auditor`, `result: success|failed`, `worktree` (опціонально, тільки якщо `failed`). Секції: `## Reasoning` (обов'язкова), `## Script` (wrapper, опціонально), `## Ref` (опціонально). NNN нумерація: wrapper рахує існуючі `run_*.md` перед стартом (`count + 1`), zero-padded до 3 цифр. ## Considered Options
* Зберегти plan/fact файли для відновлення після збою
* Видалити `ops/` і `patches/` повністю, ігнорувати crash-recovery на поточному етапі

Chosen option: "Видалити `ops/` і `patches/` повністю", because spawn відбувається у ворктрі і при збої весь ворктрі просто не застосовується — граф залишається чистим; recovery-сценарій визнано неактуальним на поточному етапі. Вміст patches тепер живе в секції `## Reasoning` файлу `run_NNN.md`.

### Consequences
* Good, because transcript фіксує очікувану користь: суттєво спрощена файлова структура, жодних залишкових WAL-файлів.
* Bad, because crash під час атомарного spawn поза ворктрі залишиться без автоматичного відновлення — прийнятий як відомий ризик.

## More Information

## ADR Каскадна інвалідація через mt kill --cascade

## Context and Problem Statement
Коли інженер патчить вузол, всі downstream-наступники стають застарілими і мають бути зупинені та позначені як знедійснені. Питання — чи це одна команда чи дві окремі операції.

## Considered Options
* Розділити: `mt kill` (kill + delete worktree) і окремо `mt invalidate --cascade`
* `mt kill --cascade` виконує все: kill процесу, delete worktree, рекурсивно створює `invalidated` sentinel у downstream-нащадках

## Decision Outcome
Chosen option: "`mt kill --cascade`", because для інженера природно зробити все в одному виклику; порядок обходу — листи → корінь (топологічний kill).

### Consequences
* Good, because transcript фіксує очікувану користь: одна команда замість двох, атомарніший workflow для інженера.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`mt kill <path> --cascade`: вбиває процес (знаходить за `cwd` = worktree path), видаляє worktree, рекурсивно створює файл `invalidated` у всіх downstream-нащадках. PID tracking: `mt kill` шукає процеси що мають worktree path як `cwd` — без окремого lock-файлу. Зафіксовано в `/Users/vitaliytv/www/nitra/cursor/npm/docs/mt.md`.
