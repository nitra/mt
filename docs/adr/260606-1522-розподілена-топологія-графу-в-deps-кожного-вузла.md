## ADR Розподілена топологія графу в `deps:` кожного вузла

## Context and Problem Statement
При проектуванні Dynamic Self-Modifying Task Graph постало питання де зберігати топологію DAG — ребра між вузлами. Система підтримує динамічний spawn дочірніх вузлів під час виконання, тому структура графу заздалегідь невідома.

## Considered Options
* Центральний файл `граф.md` на рівні батьківського вузла
* Розподілена топологія: кожен дочірній вузол декларує своїх попередників у полі `deps:` власного `task.md`

## Decision Outcome
Chosen option: "Розподілена топологія в `deps:` кожного дочірнього `task.md`", because це забезпечує Universal схему — кожен вузол самодостатній; оркестратор відновлює повний граф скануванням усіх `task.md`, центральний файл оновлювати не потрібно.

### Consequences
* Good, because динамічний spawn зводиться до створення нового `task.md` з `deps:` — жодного центрального стану оновлювати не треба.
* Bad, because оркестратор мусить сканувати всі `task.md` щоб реконструювати граф, натомість центральний файл давав би O(1) читання.

## More Information
Поле `deps:` містить список `id` сиблінгів (вузлів у межах того ж батька). Шлях резолвиться як `../<dep-id>/`. Файл: `task.md`. Видалено: `граф.md`, `граф-розш-NNN.md`.

## ADR Злиття `inputs.md` у `task.md`

## Context and Problem Statement
Початково планувався окремий файл `inputs.md` для посилань на вхідні дані вузла. Виникло питання чи доцільно тримати два файли, якщо обидва immutable, пишуться батьком при spawn і читаються агентом на старті.

## Considered Options
* Окремий `inputs.md` поряд із `task.md`
* Злиття: секція `## Inputs` всередині `task.md`

## Decision Outcome
Chosen option: "Злиття в `task.md`", because агент читає один файл замість двох, task і дані завжди разом як цілісний контекст; автор і life cycle обох файлів ідентичні.

### Consequences
* Good, because менше файлів при spawn, агент отримує повний контекст одним читанням.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`task.md` структура: frontmatter (`created_at`, `parent`, `deps`, `budget_sec`) + `## Task` + `## Done when` + `## Inputs` (підсекції з `ref:` або inline). Секції `## Task` і `## Done when` — англійські (парсяться скриптом). Файл: `npm/docs/mt.md`, розділ `### task.md`.

## ADR Уніфікований `run_NNN.md` для всіх типів виконавців

## Context and Problem Statement
Система мала окремі файли для різних сценаріїв: `outputs.md` (append-only результати), `error.md` (append-only помилки), `repair/NNN.md` (спроби інженера), `repair_context.md` (бюджет). Постало питання уніфікації, оскільки звичайний агент і інженер-агент мають однакову структуру спроби.

## Considered Options
* Окремі файли: `outputs.md`, `error.md`, `repair_history.md`, `repair_context.md`
* Уніфікований `run_NNN.md` + `outputs_NNN.md`

## Decision Outcome
Chosen option: "Уніфікований `run_NNN.md`", because і звичайний агент, і інженер — це "спроба вирішити вузол"; спільні поля `actor`, `result`, `## Reasoning`, `## Ref`; усуває всі append-only файли — кожен `run_NNN.md` immutable.

### Consequences
* Good, because transcript фіксує очікувану користь: жодних append-only файлів, однаковий формат для всіх типів виконавців, спрощена observability.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`run_NNN.md` frontmatter: `created_at`, `actor: agent|engineer|human|auditor`, `result: success|failed`. Секції: `## Reasoning` (обов'язкова), `## Script` (wrapper, опціонально), `## Ref` (опціонально). `outputs_NNN.md` — окремий immutable файл з `## Summary` (обов'язкова) + довільні секції-порти. Файл: `npm/docs/mt.md`, розділ `### run_NNN.md`.

## ADR Append-only інваріант обмежений lifecycle ворктрі

## Context and Problem Statement
При проектуванні append-only системи виникло питання чи поширюється заборона на модифікацію файлів на весь час існування вузла, зокрема до старту роботи агента.

## Considered Options
* Абсолютний append-only: файли ніколи не змінюються після створення
* Обмежений append-only: тільки після створення git worktree для вузла

## Decision Outcome
Chosen option: "Обмежений append-only: діє з моменту `git worktree add`", because до створення ворктрі жоден агент не читає файли; ворктрі є чіткою спостережуваною межею; дозволяє вільно редагувати `task.md` до запуску.

### Consequences
* Good, because інженер може змінювати `task.md` без версіонування — достатньо kill ворктрі → редагувати → restart.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Протокол: `немає ворктрі → редагуй вільно; є ворктрі → kill → редагуй вільно → restart`. Зафіксовано в `npm/docs/mt.md` розділ "Контракт файлової системи", підрозділ "Інваріант незмінності".

## ADR Всі імена файлів і директорій — англійська

* Всі імена файлів і директорій — англійська

### Consequences
* Good, because transcript фіксує очікувану користь: надійна обробка скриптами без encoding edge-cases.
