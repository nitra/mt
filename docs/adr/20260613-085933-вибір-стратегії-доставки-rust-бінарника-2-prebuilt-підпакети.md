## ADR Вибір стратегії доставки Rust-бінарника: 2 prebuilt-підпакети через `optionalDependencies`

## Context and Problem Statement

`mt-scanner` — Rust-бінарник, що виконує сканування задач. Пакет `@7n/mt` є CLI-інструментом, що публікується в npm. Бінарник не може лежати в `target/` (gitignored) і не постачається npm-пакетом за замовчуванням — потрібна стратегія доставки, яка дозволяє `npm install @7n/mt` отримати платформний бінарник без Rust-тулчейну на машині користувача.

## Considered Options

* **Postinstall `cargo build --release`** — збирати бінарник при `npm install`.
* **Prebuilt бінарники через `optionalDependencies`** — платформні підпакети за моделлю esbuild/swc.
* **Бінарник у `files` головного пакета** — покласти скомпільований бінарник прямо в репо/пакет.

## Decision Outcome

Chosen option: "Prebuilt бінарники через `optionalDependencies`", because цей варіант не вимагає Rust-тулчейну у кінцевого користувача (на відміну від postinstall-варіанту); бінарник не потрапляє в git і покриває різні платформи чисто через npm-механізм `os`/`cpu`.

Стартовий набір — **2 підпакети**: `@7n/mt-darwin-arm64` (macOS Apple Silicon, `aarch64-apple-darwin`) та `@7n/mt-linux-x64` (весь Linux x64 — musl-static без поля `libc`, покриває і glibc, і Alpine). Linux-бінарник будується через `cargo-zigbuild` (один `ubuntu-latest` runner; дешеве додавання `linux-arm64` у майбутньому).

### Consequences

* Good, because платформні підпакети встановлюються вибірково (`npm`/`bun` ставить лише той, що збігається з хостом); архітектура дозволяє додати `win32-x64`, `linux-arm64`, `darwin-x64` як "+1 підпакет +1 CI-job" без зміни коду.
* Good, because transcript фіксує очікувану користь: musl-static-бінарник без `libc`-поля скорочує кількість Linux-підпакетів вдвічі (один пакет покриває glibc і musl).
* Bad, because нові підпакети (`@7n/mt-darwin-arm64`, `@7n/mt-linux-x64`) потребують окремого налаштування trusted-publishing (OIDC) або `NPM_TOKEN` на npmjs.com — перша публікація блокується `ENEEDAUTH` через відсутність прав на нові scope-пакети.

## More Information

* Платформні підпакети: `packages/mt-darwin-arm64/package.json`, `packages/mt-linux-x64/package.json`.
* `optionalDependencies` у `npm/package.json`.
* CI-матриця у `.github/workflows/npm-publish.yml`: jobs `build-binaries` (strategy.matrix), `release-publish` з кроками `JS-DevTools/npm-publish@v4.1.5` для кожного підпакету.
* Резолвер шляху до бінарника: `npm/lib/core/scanner-bin.mjs` — пошук: `MT_SCANNER_BIN` → `require.resolve('@7n/mt-<platform>-<arch>/mt-scanner[.exe]')` → `target/release|debug` (dev-fallback).
* `cargo-zigbuild` обрано як musl-tooling для Linux: один runner крос-збирає `x86_64-unknown-linux-musl`; при додаванні `linux-arm64` достатньо додати `aarch64-unknown-linux-musl` до матриці без нового тулчейну.
* Windows (`.exe`): резолвер уже враховує суфікс `.exe` на `win32`; Rust-код path-портабельний (`\\`→`/` нормалізація наявна в `lib.rs:273`).
