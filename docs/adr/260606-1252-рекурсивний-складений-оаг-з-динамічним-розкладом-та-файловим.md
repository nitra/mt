## ADR Рекурсивний складений ОАГ з динамічним розкладом та файловим сховищем

## Context and Problem Statement
Потрібно спроектувати систему для виконання задач в AI-оркестраторі, де задачі можуть динамічно розкладатись на підзадачі під час виконання, підтримувати паралельне виконання через git worktree, та відновлюватись після збоїв. Відкрите питання: як описати і реалізувати структуру поєднання задач/підзадач та dynamic workflow.

## Considered Options
* Звичайний статичний DAG з наперед визначеною топологією
* Рекурсивний складений ОАГ (Recursive Compound DAG) з динамічним розкладом вузлів під час виконання

## Decision Outcome
Chosen option: "Рекурсивний складений ОАГ з динамічним розкладом", because вузол повинен вирішувати під час виконання чи він атомарний, чи потребує розкладання на підграф — це є вимогою системи.

### Consequences
* Good, because батьківський вузол бачить однаковий інтерфейс (`resolved`/`failed`) незалежно від того, атомарний дочірній вузол чи складений — замінюваність без змін у батьківському графі.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Зафіксовано в `npm/docs/mt.md`. Команди: `mt ...` (CLI контракт в розробці). Ключові властивості: data-flow edges (`вхідні.md`/`вихідні.md`), кілька exit-вузлів з merge до parent outputs, файловий стан за принципом append-only.

## ADR Файловий сховище стану — append-only через Markdown+YAML

## Context and Problem Statement
Вузли графу потребують персистентного стану, доступного як LLM-агентам (для читання і запису), так і CLI-скриптам (для машинного парсингу). Проблема атомарності: агент міг записати `outputs.md` але не змінити стан в центральному сховищі.

## Considered Options
* Централізована база даних зі схемою
* Файли JSON на кожен вузол
* Append-only Markdown файли з YAML-фронтматером, де стан кодується наявністю файлів

## Decision Outcome
Chosen option: "Append-only Markdown файли з YAML-фронтматером", because worktree-межа є межею атомарності — successors не стартують поки worktree не змердженого вузла не злито, тому атомарність гарантована merge-gate, а не файловими блокуваннями.

### Consequences
* Good, because transcript фіксує очікувану користь: `git merge` завжди чистий коли два агенти створюють лише нові файли з унікальними іменами в різних директоріях; `repair_history.md` як append-only журнал — LLM читає і продовжує природно.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Структура директорії вузла: `місія.md`, `вхідні.md`, `вихідні.md`, `помилка.md`, `операції/spawn-план.md`, `операції/spawn-факт.md`, `патчі/001-план.md`, `патчі/001-факт.md`, `підграф/<child-node-id>/`. Атрибути фронтматеру — англійська snake_case: `id`, `parent`, `deps`, `created_at`, `occurred_at`, `type`, `nodes`, `kill_order`, `target_node`, `result`. Синтаксис посилань: `ref: path/to/file.md`, `ref: path/to/file.md#section-name`, `ref: path/to/file.md lines 10-50`.

## ADR Принцип "план → дія → факт" як universal write-ahead log

## Context and Problem Statement
Будь-яка операція системи (spawn, kill, patch) може перерватись на півдорозі через збій агента або процесу. Без механізму відновлення система не може відрізнити "ще не зроблено" від "зроблено але не зафіксовано".

## Considered Options
* Центральний transaction log
* Принцип "план → дія → факт" через окремі файли на кожну операцію

## Decision Outcome
Chosen option: "Принцип план → дія → факт через файли", because при відновленні після збою достатньо сканувати `operations/*-plan.md` і перевіряти наявність відповідних `*-fact.md` — незавершені операції відновлюються автоматично.

### Consequences
* Good, because transcript фіксує очікувану користь: алгоритм відновлення детерміністичний — наявність `*-plan.md` без `*-fact.md` однозначно визначає незавершену операцію.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Застосовується до всіх операцій: spawn, kill залежних, патч інженера. Протокол патчу: спочатку записати `патчі/001-plan.md` → kill залежних у топологічному порядку від листів до цілі → застосувати патч → записати `патчі/001-fact.md`. Правило: інженер не може модифікувати вузол поки є активні worktree залежних вузлів.

## ADR Топологія графу розподілена в `місія.md` кожного дочірнього вузла

## Context and Problem Statement
Необхідно зберігати топологію підграфу (ребра між вузлами) так, щоб оркестратор міг відновити повний граф у будь-який момент, а агент міг динамічно додавати нові вузли без оновлення центрального файлу.

## Considered Options
* Центральний `граф.md` у батьківському вузлі
* Топологія в `deps:` фронтматері `місія.md` кожного дочірнього вузла

## Decision Outcome
Chosen option: "Топологія в `deps:` кожного `місія.md`", because при динамічному spawn новий вузол просто пише своє `місія.md` з `deps:` — оркестратор підхоплює при наступному скануванні без оновлення будь-якого центрального файлу.

### Consequences
* Good, because transcript фіксує очікувану користь: universality — кожен вузол самодостатній, граф відновлюється скануванням `місія.md` всіх дочірніх без центральної точки відмови.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Поле `deps:` у `місія.md`: список об'єктів `{node: <node-id>, port: <port-name>}`. Динамічний spawn дозволений — агент може породити нові дочірні вузли під час виконання після отримання часткових результатів. Дані передаються виключно через посилання (`ref:`), без копіювання.

## ADR Мета-рівень АгентІнженер з необмеженим доступом до графу та time-budget

## Context and Problem Statement
Вузли графу можуть відмовляти в роботі. Потрібен механізм самовідновлення, який може не просто перезапустити вузол, а реструктурувати граф (додати вузли, змінити ребра, замінити реалізацію) і підніматись до батьківських рівнів при потребі.

## Considered Options
* Supervisor у межах графу (вузол-агент з обмеженим доступом)
* АгентІнженер на мета-рівні поза графом з необмеженим доступом до будь-якого рівня ієрархії

## Decision Outcome
Chosen option: "АгентІнженер на мета-рівні з необмеженим доступом", because інженер повинен мати можливість підніматись до кореневого вузла і змінювати батьківський граф — обмежений доступ не дозволив би вирішувати системні помилки в архітектурі задачі.

### Consequences
* Good, because transcript фіксує очікувану користь: `repair_history.md` на вузлі дає контекст для LLM — інженер stateless, знання зберігаються у файлах вузла, тому будь-який виклик інженера продовжує попередні спроби.
* Bad, because каскадна інвалідація: якщо інженер патчить вузол на будь-якому рівні, всі successor-вузли стають `invalidated` і потребують повторного виконання (зафіксовано в transcript як відомий наслідок).

## More Information
Ескалація: при timeout кожен батьківський рівень отримує свій свіжий time budget (не залишок від дочірнього). При timeout кореневого рівня — генерується `senior_report.md` для людини. Memory інженера: `repair_history.md` — append-only журнал на вузлі де відбувались зміни; при зміні батька записується в батька, дочірній отримує `triggered_parent_patch:`. Time budget: поле `repair_deadline` в `repair_context.json` — встановлюється один раз при першому виклику інженера, кількість спроб необмежена в межах часу.

## ADR Паралельне виконання через git worktree з merge-gate

## Context and Problem Statement
Незалежні вузли графу мають виконуватись паралельно для ефективності, при цьому ізоляція між агентами і атомарність переходу стану є критичними.

## Considered Options
* Паралельні процеси з блокуваннями файлів
* Кожен вузол у своєму git worktree, merge як gate перед стартом successors

## Decision Outcome
Chosen option: "Паралельне виконання через git worktree", because merge-gate є природною точкою рішення: оркестратор (або людина вручну) оцінює стан при merge і вирішує чи дозволити successor-вузлам стартувати — race condition між агентом і інженером вирішується архітектурно.

### Consequences
* Good, because transcript фіксує очікувану користь: `git merge` завжди чистий для незалежних вузлів (різні директорії `tasks/<node-id>/`), конфлікти виникають лише при одночасній модифікації спільних батьківських файлів і вирішуються агентом-медіатором.
* Bad, because жорсткі ліміти на кількість паралельних worktree через обмеження MacBook (конкретні числа ще не визначені).

## More Information
Naming convention worktree: `.worktrees/<node-id>/`. Патч інженера — також окремий worktree, видаляється після завершення. Freeze сигнал при каскаді кореня: wrapper-скрипт, що запускає агентів, може kill їх процеси — агенти запускаються виключно через скрипти.
