## ADR Фінальна схема `outputs.md`

## Context and Problem Statement
Потрібно визначити формат файлу `outputs.md` — результату виконання вузла. Питання: мати одну секцію чи дві (`## Summary` + `## Results`), і коли використовувати inline-дані vs посилання.

## Considered Options
* Одна секція `## Results` — агент пише результати у довільному форматі
* Дві секції `## Summary` + `## Results` — `Summary` для observability, `Results` для даних
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Дві секції + правило ref-first", because `## Summary` потрібен observability-скрипту (читає без парсингу даних), `## Results` — для агентів-наступників; посилання замість копіювання — вже встановлений інваріант системи.

### Consequences
* Good, because скрипт може читати `## Summary` з будь-якого `outputs.md` без розуміння специфіки вузла.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл: `tasks/<node-id>/outputs.md`. Правило: якщо дані живуть у файлі — `ref:`, якщо агент сформував текст без окремого артефакту — inline. `created_at` — перше поле фронтматеру (спільний контракт усіх файлів).

## ADR Файловий граф задач: LLM-first формат файлів

## Context and Problem Statement
Система динамічного графу задач зберігає стан кожного вузла у файловій системі. Потрібно визначити формат файлів що одночасно зручний для читання LLM-агентів і машинного парсингу оркестратором.

## Considered Options
* JSON-файли — структуровані, але незручні для LLM контексту
* Markdown з YAML-фронтматером — машинозчитуваний фронтматер + вільний текст для LLM
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Markdown з YAML-фронтматером", because фронтматер парситься скриптом, тіло читається LLM без реконструкції контексту. Секції що парсить скрипт — англійські заголовки; секції з довільними даними — будь-яка мова.

### Consequences
* Good, because `repair_history.md` як append-only журнал — агент читає і дописує природно, без парсингу JSON-масиву.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Контракт у `npm/docs/mt.md`. Поле `created_at` — перше у фронтматері всіх файлів. Всі імена файлів і директорій — англійська. Атрибути фронтматеру — snake_case англійська.

## ADR Append-only файлова система як state store

## Context and Problem Statement
Потрібна атомарність запису при паралельному виконанні вузлів у git-worktree. Класичний підхід — модифікація файлів стану — створює race conditions і потребує блокувань.

## Considered Options
* Mutable файли з sentinel-флагами та CLI-хуками для зміни стану
* Append-only: файли тільки створюються, стан визначається наявністю файлів
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Append-only: файли тільки створюються", because `file create` атомарна операція на більшості ФС; наявність `outputs.md` сама по собі є сигналом "resolved" без окремого CLI-хука; два ворктрі ніколи не конфліктують бо пишуть лише нові файли з унікальними іменами.

### Consequences
* Good, because merge двох ворктрі завжди чистий — конфлікт архітектурно неможливий.
* Good, because transcript фіксує очікувану користь: відновлення після збою через scan `*-plan.md` без `*-fact.md`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Інваріант: до створення ворктрі файли вільно редагуються; після — тільки нові. Щоб змінити після старту: kill ворктрі → редагуй → restart. Контракт у `npm/docs/mt.md`.

## ADR plan → action → fact як universal WAL

## Context and Problem Statement
Агент або інженер може впасти між виконанням операції та записом її результату — система опиниться у невизначеному стані без можливості відновлення.

## Considered Options
* Записувати лише факт після завершення
* Write-ahead log: спочатку план (`*-plan.md`), потім дія, потім факт (`*-fact.md`)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Write-ahead log через план/факт файли", because оркестратор при відновленні сканує `*-plan.md` без відповідного `*-fact.md` — знаходить незавершені операції і відновлює без ambiguity.

### Consequences
* Good, because transcript фіксує очікувану користь: детермінований алгоритм відновлення після збою на будь-якому рівні.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Застосовується до: `ops/spawn-plan/fact`, `ops/kill-plan/fact`, `patches/<ts>-plan/fact`. Контракт у `npm/docs/mt.md`.

## ADR Топологія графу в `task.md` кожного вузла

## Context and Problem Statement
Треба зберігати топологію підграфу — які вузли існують і які між ними залежності. Питання: центральний файл `graph.md` на рівні батька чи розподілена топологія в кожному дочірньому вузлі.

## Considered Options
* Центральний `graph.md` у батьківському вузлі — авторитетне джерело топології
* Розподілена топологія: кожен `task.md` знає своїх попередників через `deps:`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Розподілена топологія в `task.md`", because немає центрального файлу який потрібно оновлювати при динамічному spawn; оркестратор відновлює повний граф скануванням — нічого центрального не потрібно підтримувати у консистентному стані.

### Consequences
* Good, because динамічний spawn = просто новий `task.md` з `deps:` без оновлення будь-якого центрального файлу.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`deps:` містить лише siblings (вузли в межах того ж `subgraph/`). Шлях резолвиться як `../<dep-id>/`. Контракт у `npm/docs/mt.md`.

## ADR `task.md` як єдиний вхідний файл вузла

## Context and Problem Statement
Вузол потребує двох типів даних: опис місії (що робити) і посилання на вхідні дані (де взяти). Питання: два окремих файли `task.md` + `inputs.md` чи один об'єднаний.

## Considered Options
* Два файли: `task.md` (місія) + `inputs.md` (refs на дані)
* Один файл `task.md` з секцією `## Inputs`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Один файл `task.md`", because агент читає один файл замість двох; місія і дані завжди разом — контекст цілісний; менше файлів при spawn.

### Consequences
* Good, because transcript фіксує очікувану користь: агент отримує повний контекст з одного читання.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Секція `## Inputs` містить підсекції `### <port-name>` з `ref:` або inline-текстом. Контракт у `npm/docs/mt.md`.

## ADR Time budget замість лічильника спроб для відновлення

## Context and Problem Statement
Потрібен convergence guard для агента-інженера — механізм що зупиняє нескінченні спроби відновлення вузла.

## Considered Options
* `max_attempts: N` — жорсткий ліміт кількості спроб
* Time budget — фіксований часовий бюджет (наприклад 10 хвилин), необмежена кількість спроб
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Time budget", because реалістичне обмеження — час відповідає реальним витратам (compute, гроші, wall clock); інженер може адаптувати стратегію залежно від залишку часу (багато часу → складна спроба, мало → швидкий fix).

### Consequences
* Good, because передбачуваний максимальний час до ескалації: `depth × budget`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`repair_context.md` містить `budget_sec` і `deadline`. При timeout → `error.md` з `type: timeout` → escalate. Кожен рівень ескалації отримує свіжий budget. Контракт у `npm/docs/mt.md`.
