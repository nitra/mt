## ADR Заміна JS-реалізації сканера на виклик Rust-бінарника `mt-scanner`

## Context and Problem Statement
У проєкті існують дві паралельні реалізації одного сканера задач: `npm/lib/core/scanner.mjs` (JS/Node.js) та `scanner/src/` (Rust, бінарник `mt-scanner`). Користувач поставив вимогу: JS-реалізації не повинно існувати — замість неї `scanner.mjs` має бути тонким шимом, що викликає Rust-бінарник через `spawnSync`.

## Considered Options
* Замінити `scanner.mjs` на шим, що викликає `mt-scanner scan <mtDir>` через `spawnSync` і адаптує JSON-вихід
* Залишити обидві реалізації паралельно
* Перейти на NAPI `.node`-аддон замість окремого CLI-бінарника

## Decision Outcome
Chosen option: "Шим `scanner.mjs` + `spawnSync` до `mt-scanner scan`", because контракти двох реалізацій відрізняються (плоский список vs вкладене дерево, kebab vs snake стани, наявність поля `dir`, composite/children), тому потрібен JS-адаптер зі збереженою сигнатурою функцій; водночас NAPI не розглядався як варіант.

### Consequences
* Good, because transcript фіксує очікувану користь: єдина реалізація логіки скану, швидкість Rust-обходу ФС, усунення дублювання.
* Bad, because `spawnSync` на кожен скан додає латентність форку процесу; для команди `mt watch`, що сканує часто, це може бути відчутним (зазначено в spec, поза scope рефакторингу).

## More Information
Файли: `npm/lib/core/scanner.mjs`, `scanner/src/lib.rs`, `scanner/src/main.rs`, `docs/spec-scanner-rust-integration.md`. Адаптер виконує: `flatten(tree, mtDir)`, `state.replace('_', '-')`, відновлення `dir = join(mtDir, path)`, `is_composite → composite`, `children[]` — масив шляхів.

## ADR Принцип «уся ФС-робота → Rust»

## Context and Problem Statement
Під час визначення, де реалізувати логіку `worktree → running` (визначення стану `running` за наявністю активного git-worktree), постало питання: лишати post-process у JS-шимі чи переносити в Rust. Користувач сформулював загальну директиву.

## Considered Options
* Worktree→running post-process у JS-шимі (поверх виходу бінарника)
* Worktree→running у Rust: `mt-scanner scan` сам викликає `git worktree list --porcelain` і підвищує стан до `running`

## Decision Outcome
Chosen option: "Worktree→running у Rust", because користувач встановив директиву: «усе, що стосується роботи з файловою системою, повинно бути Rust» — JS не торкається ФС взагалі.

### Consequences
* Good, because transcript фіксує очікувану користь: єдина точка відповідальності за стан задачі; JS-шим стає чисто адаптером без FS-логіки.
* Bad, because `sanitizeTaskName` (конвенція іменування worktree) потрібно продублювати в Rust для матчингу; JS-копія лишається лише для *створення* worktree в `mt run`. Вимагає синхронізації двох реалізацій спільними тест-векторами.

## More Information
Rust отримує опцію `--worktrees <list>` — лише для детермінованих тестів; у продакшені опускається, Rust сам виконує discovery. Файли: `scanner/src/lib.rs`, `npm/lib/core/scanner.mjs`, `npm/lib/commands/run.mjs`, `docs/spec-scanner-rust-integration.md §5`.

## ADR Доставка платформних бінарників через `optionalDependencies`

## Context and Problem Statement
`target/` директорія в `.gitignore`, бінарник `mt-scanner` не потрапляє в опублікований npm-пакет. Без рішення по доставці пакет `@7n/mt` падатиме у користувача з відсутнім виконуваним файлом.

## Considered Options
* (A) `postinstall cargo build --release` — вимагає Rust-тулчейн у кожного
* (B) Prebuilt бінарники через `optionalDependencies` (як esbuild/swc) — платформні підпакети
* (C) Бінарник безпосередньо у `files` головного пакету

## Decision Outcome
Chosen option: "(B) optionalDependencies з платформними підпакетами", because правильний прод-шлях для публічного npm-CLI; npm/bun ставить лише підпакет, що збігається з хостом; решта тихо ігноруються.

### Consequences
* Good, because transcript фіксує очікувану користь: не вимагає Rust на машині користувача; стандартна модель (esbuild/swc); додавання нових платформ — add-only без зміни логіки.
* Bad, because потребує CI-матриці з окремими job'ами і публікації кількох npm-пакетів з lockstep-версіями.

## More Information
Структура: `packages/mt-darwin-arm64/`, `packages/mt-linux-x64/` з полями `os`, `cpu` (без `libc` — musl-static покриває увесь Linux). Резолвер у `npm/lib/core/scanner-bin.mjs`: `MT_SCANNER_BIN` → `require.resolve('@7n/mt-' + key + '/mt-scanner[.exe]')` → dev-fallback `target/release/`. Файли: `docs/spec-scanner-rust-integration.md §4.1, §6`.

## ADR Початкова матриця платформ: `darwin-arm64` + `linux-x64` (musl-static)

## Context and Problem Statement
Потрібно визначити початковий набір платформних підпакетів для `optionalDependencies`. Повна матриця (10 платформ) є надлишковою для старту; неповна — може залишити частину команди без підтримки.

## Considered Options
* Minimal (3): `darwin-arm64`, `darwin-x64`, `linux-x64-musl`
* 2 пакети: `darwin-arm64` + `linux-x64` (musl-static, без поля `libc`)
* Standard (5): + `linux-arm64-musl`, `win32-x64`

## Decision Outcome
Chosen option: "2 пакети: `@7n/mt-darwin-arm64` + `@7n/mt-linux-x64` (musl-static)", because команда працює на Apple Silicon Mac + Linux x64 CI/Docker; musl-static бінарник без поля `libc` покриває весь Linux x64 (Alpine, Ubuntu, Docker) одним підпакетом.

### Consequences
* Good, because мінімум CI-job'ів (1 `macos-14` + 1 `ubuntu`); менше підпакетів для публікації і підтримки.
* Bad, because Intel Mac (`darwin-x64`), Linux arm64 і Windows не покриті. Зазначено явно: при відсутності збігу резолвер дає зрозумілу помилку, а не падає мовчки. Windows і linux-arm64 — add-only, без зміни коду шима.

## More Information
Rust triple для linux: `x86_64-unknown-linux-musl` (static). CI: `macos-14` (native arm64) + `ubuntu-latest` з `cargo-zigbuild`. Файли: `docs/spec-scanner-rust-integration.md §6`.

## ADR Вибір `cargo-zigbuild` для збірки Linux musl у CI

## Context and Problem Statement
Для збірки `x86_64-unknown-linux-musl` у CI є кілька підходів. Вибір впливає на те, наскільки дешево буде додати `linux-arm64` у майбутньому.

## Considered Options
* `musl-tools` + `rustup target add` (стандартний підхід, лише x64)
* `cargo-zigbuild` (zig як лінкер, крос-компіляція всіх Linux-таргетів з одного раннера)

## Decision Outcome
Chosen option: "`cargo-zigbuild`", because користувач підтвердив цей варіант після пояснення, що він дозволяє додати `linux-arm64` без нового тулчейну — лише зміна `--target`.

### Consequences
* Good, because transcript фіксує очікувану користь: додавання `linux-arm64` = `+1 job`, без нових тулчейнів; пінінг старого glibc через zig не потрібен (musl-static).
* Bad, because `cargo-zigbuild` — додаткова залежність; Windows-таргет через zigbuild не збирається (Windows потребує окремого native `windows-latest` раннера незалежно від вибору).

## More Information
CI-job Linux: `ubuntu-latest`, `cargo install cargo-zigbuild`, `cargo zigbuild --release --target x86_64-unknown-linux-musl`. Файли: `docs/spec-scanner-rust-integration.md §6.3, §9.2`.
