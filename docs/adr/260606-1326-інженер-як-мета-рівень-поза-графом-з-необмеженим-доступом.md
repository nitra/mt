## ADR Інженер як мета-рівень поза графом з необмеженим доступом

## Context and Problem Statement
При збої вузла в рекурсивному складеному ОАГ потрібна сутність здатна аналізувати причину і виправляти граф. Постало питання: чи інженер є вузлом всередині графу (рекурсивна структура), чи існує поза ним як supervisor.

## Considered Options
* Інженер як вузол у графі (рекурсивний виклик)
* Інженер як мета-рівень поза графом з повним доступом до всіх рівнів

## Decision Outcome
Chosen option: "Інженер як мета-рівень поза графом з повним доступом", because інженер повинен бачити повний `path від кореня до вузла що впав` і вирішувати де саме втручатись — на рівні вузла, батька або кореня.

### Consequences
* Good, because transcript фіксує очікувану користь: інженер може патчити будь-який рівень ієрархії, включно з кореневим, і вибирати оптимальну точку втручання.
* Bad, because необмежений доступ дає можливість каскадно перебудувати весь граф, що може тригерити масштабну інвалідацію.

## More Information
Файл `npm/docs/mt.md`. Патерн: `EngineerAgent → analyze(failure + full_path) → patch на найефективнішому рівні`. Ескалаційна ланцюжок: `node → parent → ... → root → notify SeniorEngineer`. `senior-report.md` зберігається у директорії вузла.

## ADR Граф мутабельний з каскадною інвалідацією

## Context and Problem Statement
При патчі вузла інженером потрібно визначити що відбувається з вже `resolved` наступниками — чи зберігають вони свої результати, чи потребують переобрахунку.

## Considered Options
* Append-only (нові вузли, старі незмінні)
* Mutable with invalidation (патч змінює вузол, наступники інвалідуються каскадно)

## Decision Outcome
Chosen option: "Mutable with invalidation", because інженер повинен мати можливість виправити помилкові результати на будь-якому рівні, а наступники що отримали хибні вхідні дані мають бути переобраховані.

### Consequences
* Good, because транскрипт фіксує очікувану користь: граф сходиться до коректного стану навіть після виявлення помилок у вже виконаних вузлах.
* Bad, because патч глибокого вузла може інвалідувати великі частини графу, перезапускаючи значну кількість роботи.

## More Information
Файл `npm/docs/mt.md`. При патчі вузла-X: `X.outputs = invalidated → всі successors = invalidated → до листів`. Sentinel-файл `invalidated` у директорії вузла. Протокол: спочатку kill всіх залежних worktree (топологічний порядок від листів до цілі), потім патч, потім рестарт каскаду.

## ADR Пам'ять інженера зберігається на вузлі (repair_history.md)

## Context and Problem Statement
Інженер викликається кілька разів для одного вузла. Потрібно щоб він бачив попередні спроби і не повторював невдалі підходи. Питання: де зберігається ця пам'ять — в інженері чи у вузлі.

## Considered Options
* Пам'ять в інженері (stateful engineer)
* Пам'ять у вузлі (`repair_history.md`), інженер stateless

## Decision Outcome
Chosen option: "Пам'ять у вузлі, інженер stateless", because вузол накопичує діагностичне знання про себе, і ця пам'ять доступна будь-якому майбутньому виклику інженера без централізованого стану.

### Consequences
* Good, because transcript фіксує очікувану користь: `repair_history.md` є append-only журналом — інженер читає його цілком і природно продовжує, не потребуючи зовнішньої пам'яті.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл `npm/docs/mt.md`. Схема `repair_history.md`: YAML-фронтматер з `created_at` і `deadline`, тіло — `## Attempt N` секції з `Reasoning`, опціональним `Patch:`, `Result`, `Failure reason`. Поле `deadline` встановлюється при першому створенні і не змінюється. Після `deadline` — фінальний запис + `error-<ts>.md (type: unresolvable)`.

## ADR Часовий бюджет замість ліміту спроб для інженера

## Context and Problem Statement
Потрібен механізм що запобігає нескінченному циклу виправлень інженером. Варіант з лімітом кількості спроб може обрізати вирішення посередині складного процесу.

## Considered Options
* Ліміт кількості спроб (`max_attempts: N`)
* Часовий бюджет (`budget_sec`, необмежена кількість спроб)

## Decision Outcome
Chosen option: "Часовий бюджет (`budget_sec`)", because часовий бюджет є реалістичним обмеженням яке відповідає реальним витратам (compute, wall clock) і дозволяє інженеру адаптувати стратегію — складні спроби при великому залишку, швидкі — при малому.

### Consequences
* Good, because transcript фіксує очікувану користь: інженер не обрізається посередині перспективного підходу через довільний лічильник спроб.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл `npm/docs/mt.md`. Поле `budget_sec` у `task.md` (опціональне, дефолт з конфігу). Wrapper обчислює `deadline = start_time + budget_sec`. При ескалації кожен батьківський рівень отримує **свіжий** бюджет (не залишок від дочірнього). Root timeout → `senior-report.md` + WARN у `mt status`.

## ADR Worktree як межа атомарності (append-only інваріант)

## Context and Problem Statement
Файлова система як state store не дає транзакційних гарантій. Агент може записати `outputs.md` але не оновити стан, або вузол може бути скасований поки агент ще пише результати.

## Considered Options
* Sentinel-файли + атомарний rename для управління станом
* Git worktree як межа атомарності — наступники стартують лише після merge worktree

## Decision Outcome
Chosen option: "Git worktree як межа атомарності", because наступники не стартують поки worktree не змерджено — merge є єдиною точкою рішення оркестратора, що усуває race condition між завершенням агента і скасуванням інженером.

### Consequences
* Good, because transcript фіксує очікувану користь: конфлікти при паралельних worktree неможливі якщо незалежні вузли пишуть у різні директорії; append-only інваріант починається в момент `git worktree add`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл `npm/docs/mt.md`. Інваріант: до `git worktree add` — файли вузла можна вільно змінювати; після — тільки нові файли, нічого не змінюється. Worktree: `.worktrees/<node-id>`. При конфлікті merge (інженер модифікував спільний батьківський файл) — агент-медіатор вирішує конфлікт.

## ADR Принцип план → дія → факт (write-ahead log через файли)

## Context and Problem Statement
Операції (spawn, kill, patch) можуть перерватись посередині. Потрібен механізм відновлення що дозволяє визначити що саме не завершилось і продовжити з правильного місця.

## Considered Options
* Транзакційні операції з rollback
* Write-ahead log через пари `*-plan.md` / `*-fact.md`

## Decision Outcome
Chosen option: "Write-ahead log через пари `*-plan.md` / `*-fact.md`", because файлова система append-only, тому план пишеться до будь-якої дії, факт — після завершення; наявність плану без факту однозначно ідентифікує незавершену операцію.

### Consequences
* Good, because transcript фіксує очікувану користь: `mt recover` сканує `*-plan.md` без `*-fact.md` і відновлює незавершені операції.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл `npm/docs/mt.md`. Застосовується для: `ops/spawn-plan/fact-<ts>.md`, `ops/kill-plan/fact-<ts>.md`, `patches/patch-plan/fact-<ts>.md`. Протокол патчу: 1) записати `patch-plan.md` → 2) kill залежних worktree → 3) застосувати патч → 4) записати `patch-fact.md`.

## ADR Топологія графу розподілена в task.md кожного вузла (поле deps)

## Context and Problem Statement
Підграф агента потребує опису топології — хто від кого залежить. Питання: зберігати топологію централізовано (окремий `graph.md`) чи у кожному вузлі.

## Considered Options
* Централізований `graph.md` у батьківському вузлі
* Розподілена топологія: кожен дочірній вузол зберігає своїх попередників у `deps:` поля `task.md`

## Decision Outcome
Chosen option: "Розподілена топологія в `task.md` (поле `deps`)", because забезпечує універсальність — динамічний spawn просто додає новий дочірній з `task.md` без оновлення центрального файлу; оркестратор відновлює повний граф скануванням.

### Consequences
* Good, because transcript фіксує очікувану користь: відсутній центральний файл який потрібно атомарно оновлювати при кожному spawn; сканування `task.md` достатньо для повної реконструкції графу.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл `npm/docs/mt.md`. Поле `deps: [node-id, ...]` у YAML-фронтматері `task.md`. `deps` — завжди sibling-вузли в межах того самого `subgraph/`. Скрипт валідує: кожен dep має ≥1 `ref:` у `## Inputs` секції.

## ADR Агент інтерпретує повні outputs семантично (без типізованих портів)

## Context and Problem Statement
При передачі даних між вузлами потрібно вирішити: чи агент отримує конкретні named sections попередника (`port: results`), чи повний файл і сам вирішує що релевантно.

## Considered Options
* Типізовані порти — наступник оголошує яку секцію очікує з попередника
* Семантична інтерпретація — агент отримує повні `outputs.md` всіх deps і сам вирішує що використовувати

## Decision Outcome
Chosen option: "Семантична інтерпретація повних outputs", because агент не може заздалегідь знати що поверне попередник; єдина тверда специфікація — кореневий вузол у `## Done when`.

### Consequences
* Good, because transcript фіксує очікувану користь: гнучкість — кожен агент адаптується до того що реально отримав, без жорстких контрактів між вузлами.
* Bad, because при великих outputs агент отримує весь контекст навіть якщо більша частина нерелевантна; `ref:` з конкретними секціями є лише опціональною оптимізацією.

## More Information
Файл `npm/docs/mt.md`. Resolver (`mt resolve`) конкатенує повні `outputs.md` всіх deps → `context.md`. Секції в `outputs.md` — для читабельності, не для програмного парсингу. `ref: <node>/outputs.md#section` — опціональна оптимізація коли батько знає структуру.

## ADR Inputs merged у task.md; context.md генерується resolver-ом

## Context and Problem Statement
Вузол мав два окремі файли: `task.md` (місія) і `inputs.md` (маппінг вхідних даних). Обидва пишуться батьком при spawn одночасно і є незмінними.

## Considered Options
* Окремі `task.md` і `inputs.md`
* Єдиний `task.md` з секцією `## Inputs`

## Decision Outcome
Chosen option: "Єдиний `task.md` з секцією `## Inputs`", because обидва файли мають однаковий lifecycle (пишуться батьком при spawn, незмінні після worktree), тому розділення — зайва складність без користі.

### Consequences
* Good, because transcript фіксує очікувану користь: агент читає один файл замість двох; структура вузла простіша.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл `npm/docs/mt.md`. `task.md` містить секції: `## Task`, `## Done when`, `## Inputs` (з `ref:` рядками), опціональний `## Контекст`. Wrapper запускає `mt resolve` → генерує `context.md` (зберігається як аудит того що бачив агент). Агент читає тільки `task.md` + `context.md`.
