## ADR Переформатування `flow` під дворівневий протокол вузла графу

## Context and Problem Statement
Існуюча система `mt` ("попередній MT workflow") виконувала всі ролі всередині одного процесу: ізоляція worktree, планування, виконання коду, перевірка, реліз. Нова архітектура (`npm/docs/mt.md`) вводить зовнішній оркестратор `mt`, який бере на себе lifecycle worktree. Це зробило `flow` надлишково товстим і суперечливим із зовнішнім DAG-шаром.

## Considered Options
* Залишити `flow` як є, дублювати lifecycle логіку в `graph`
* Повністю замінити `flow` командами `graph`
* Переформатувати `flow` як внутрішній протокол вузла (два чіткі Stage), делегувавши worktree lifecycle до `graph`

## Decision Outcome
Chosen option: "Переформатувати `flow` як внутрішній протокол вузла", because `graph` керує зовнішнім lifecycle (worktree, deps, merge, cascade), а `flow` залишається протоколом всередині одного вузла і ділиться на Stage 1 (planning) та Stage 2 (execution).

### Consequences
* Good, because transcript фіксує очікувану користь: чітке розмежування "зовнішній DAG" vs "внутрішній протокол вузла" без дублювання lifecycle-логіки.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Нові ролі команд після рефакторингу: `mt plan` (Stage 1), `mt verify` (Stage 2 quality gate), решта видаляється. Зафіксовано в `npm/docs/mt.md` секція "Інтеграція з `mt`".

## ADR `mt plan` — об'єднання spec і decompose в один крок

## Context and Problem Statement
У старому `flow` існували окремі команди `mt init` (brainstorm, панель персон, elicitation) та `mt plan` (декомпозиція на кроки). Людина могла зупинитись між ними для перегляду дизайну. Питання: чи зберегти цей розрив як явний крок або об'єднати.

## Considered Options
* **Два кроки**: `mt init` → дизайн → `mt plan` → декомпозиція (явна пауза між ними)
* **Один крок**: `mt plan` = spec + decompose разом, `mt init` зникає

## Decision Outcome
Chosen option: "Один крок (`mt plan`)", because у сесії підтверджено "спільний крок поєднуємо їх" — поглинення `mt init` до `mt plan` спрощує протокол без втрати змісту.

### Consequences
* Good, because transcript фіксує очікувану користь: менше кроків для агента, простіший контракт.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`mt plan` контролюється атрибутом `mode: human | agent` у `task.md` (default: `human`). Вихід атомарного шляху: `plan_001.md`. Вихід composite шляху: дочірні `task.md` файли.

## ADR Явний виклик `mt spawn` агентом (не bundled у `mt plan`)

## Context and Problem Statement
Після того як `mt plan` у composite-режимі створює дочірні `task.md`, потрібно вирішити: хто ініціює `mt spawn` — сам `mt plan` автоматично, чи агент явно після перегляду результату.

## Considered Options
* **Варіант A (bundled):** `mt plan` сам викликає `mt spawn` в кінці
* **Варіант B (explicit):** `mt plan` створює дочірні `task.md`, агент окремо викликає `mt spawn`

## Decision Outcome
Chosen option: "Варіант B (explicit)", because transcript підтверджує цей вибір ("Варіант B") і додає вимогу окремого скрипту-оркестратора, який сканує весь граф і вирішує порядок виконання вузлів через чергу.

### Consequences
* Good, because агент має можливість переглянути і скоригувати дочірні `task.md` до виклику `mt spawn`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Оркестратор черги (`mt watch` + post-merge hook → `mt run --auto`) підхоплює нові вузли при наступному скані після `mt spawn`.

## ADR Аудит як файловий стан черги з нумерацією `pending-audit_NNN.md`

## Context and Problem Statement
Стара система `mt audit`/`mt verify` виконувала adversarial review синхронно та блокувала виконання. Нова архітектура потребує асинхронного аудиту: агент сигналізує "готово до перевірки", оркестратор ставить у чергу, auditor-агент обробляє окремо. Питання: як зберігати запит на аудит і як пов'язати його з конкретною версією `outputs_NNN.md`.

## Considered Options
* **Порожній sentinel:** `.pending-audit` — сигналізує стан, але без прив'язки до версії
* **Overwrite-файл:** `.pending-audit` з `ref:` полем — один файл, перезаписується
* **Numbered immutable:** `pending-audit_NNN.md` — NNN дзеркалює NNN відповідного `outputs_NNN.md`

## Decision Outcome
Chosen option: "Numbered immutable (`pending-audit_NNN.md`)", because NNN в імені файлу є одночасно й посиланням, і immutable записом — `pending-audit_003.md` однозначно відповідає `outputs_003.md` без потреби у `ref:` полі.

### Consequences
* Good, because transcript фіксує очікувану користь: аудитор точно знає яку версію перевіряти; нумерація не губиться на всіх етапах; immutable-властивість зберігається як у `run_NNN.md` і `outputs_NNN.md`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Нова таблиця станів вузла: `waiting` / `running` / `pending-audit` / `resolved` / `failed` / `invalidated`. `run_NNN.md` має незалежний лічильник (всі актори). `outputs_NNN.md` і `pending-audit_NNN.md` — спільний ключ NNN. `mt audit` і `mt verify` зникають повністю. Auditor-агент пише `run_NNN.md` з `actor: auditor, result: success|failed`.

## ADR `mt verify` — гібридна перевірка: скрипт + LLM

## Context and Problem Statement
Стара `mt verify` перевіряла структуровані кроки з MT file-presence state. У новій архітектурі критерій завершення — `## Done when` у `task.md` — написаний природньою мовою. Постало питання: хто приймає рішення PASS/FAIL — детермінований скрипт чи LLM-агент?

## Considered Options
* **A — скрипт:** перевіряє наявність і непорожність `outputs_NNN.md`. Швидко, але не розуміє змісту `## Done when`.
* **B — LLM:** читає `task.md ## Done when` + `outputs_NNN.md` + `plan_001.md`, семантично оцінює. Розуміє зміст, але недетермінований.
* **C — гібрид:** скрипт перевіряє структурні інваріанти (outputs існує, непорожній) + LLM перевіряє семантику `## Done when`. PASS тільки якщо обидва OK.

## Decision Outcome
Chosen option: "C (гібрид)", because transcript підтверджує вибір варіанту C — детермінована структурна перевірка зменшує зайві LLM-виклики, семантична перевірка LLM покриває зміст критерію, який скрипт не може проаналізувати.

### Consequences
* Good, because transcript фіксує очікувану користь: надійність структурних інваріантів поєднується з семантичним розумінням довільного критерію завершення.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Обговорення завершилось на підтвердженні варіанту C; деталі реалізації (формат виходу `verify_NNN.md`, exit-коди, інструменти LLM-агента) у transcript не зафіксовані.
