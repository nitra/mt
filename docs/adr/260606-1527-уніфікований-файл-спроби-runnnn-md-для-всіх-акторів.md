## ADR Топологія графу розподілена по `deps:` кожного дочірнього вузла

## Context and Problem Statement
Граф задач — ОАГ з динамічним розкладом вузлів. Потрібно десь зберігати ребра між дочірніми вузлами підграфу. Розглядалось центральне сховище (`граф.md` у батьківському вузлі) проти розподіленого (кожен дочірній описує своїх попередників).

## Considered Options
* Центральний `граф.md` у директорії батька — авторитетний список вузлів і ребер
* Розподілена топологія: `deps:` у `task.md` кожного дочірнього вузла

## Decision Outcome
Chosen option: "Розподілена топологія в `deps:`", because мета — універсальність: кожен вузол самодостатній, оркестратор відновлює повний граф скануванням `task.md` без центрального файлу.

### Consequences
* Good, because transcript фіксує очікувану користь: динамічний spawn нового вузла не вимагає оновлення жодного центрального файлу — досить створити дочірній `task.md` з `deps:`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Поле `deps:` у `task.md` містить лише id братів-сусідів (siblings) — deps завжди в межах одного підграфу. Файл `граф.md` і `граф-розш-*.md` видалено. `ops/spawn-plan-NNN.md` дублює `deps:` для кожного дочірнього у фронтматері — для атомарного підтвердження наміру до spawn.

## ADR `task.md` як єдиний вхідний контракт (merged з inputs)

## Context and Problem Statement
Вузол потребував двох файлів при старті: `task.md` (опис місії) та `inputs.md` (посилання на вхідні дані від попередників). Агент мав читати два файли щоб почати роботу.

## Considered Options
* Окремі `task.md` + `inputs.md`
* Об'єднаний `task.md` зі секцією `## Inputs`

## Decision Outcome
Chosen option: "Об'єднаний `task.md` зі секцією `## Inputs`", because агент читає один файл замість двох; task і дані завжди разом — контекст цілісний.

### Consequences
* Good, because transcript фіксує очікувану користь: менше файлів при spawn, агент не мусить шукати окремий inputs файл.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Секція `## Inputs` містить підсекції `### <port-name>` з `ref: path/to/file.md#section` або inline-текстом. Правило: `ref:` якщо дані є у файлі, inline тільки якщо немає деінде. Зміна inputs до старту ворктрі — вільне редагування `task.md`; після старту — `mt kill` → редагування → restart.

## ADR Immutable файли після старту ворктрі + plan→fact WAL патерн

## Context and Problem Statement
Система потребувала надійного відновлення після збою: знати що було заплановано, що виконано, і що потрібно відновити. Паралельно — уникнути race conditions при роботі кількох агентів у ворктрі.

## Considered Options
* Mutable файли зі станом (поле `стан:` у фронтматері)
* Append-only файли (нові секції дописуються в кінець)
* Повністю immutable файли (тільки створюються, ніколи не змінюються) + WAL через plan/fact пари

## Decision Outcome
Chosen option: "Immutable файли + plan→fact пари", because межа ворктрі вже є атомарною межею: наступники не стартують поки ворктрі не змерджено, тому race condition відсутній. Стан вузла визначається виключно наявністю файлів, не полями.

### Consequences
* Good, because transcript фіксує очікувану користь: `graph resume` може точно знайти незавершені операції порівнявши `*-plan-NNN.md` з відповідними `*-fact-NNN.md` без жодної додаткової логіки.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Інваріант: до створення ворктрі файли вільно редагуються/видаляються; після — тільки нові файли. Стан через наявність: `task.md` = waiting; активний ворктрі = running; `outputs_NNN.md` = resolved; `run_NNN.md result:failed` без outputs = failed; `invalidated` sentinel = invalidated. WAL застосовується до всіх операцій: `ops/spawn-plan/fact`, `ops/kill-plan/fact`, `patches/NNN-plan/fact`.
