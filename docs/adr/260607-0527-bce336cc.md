## Що є основою для UI

Архітектура в npm/docs/mt.md визначає **два ролі людини** щодо UI:
1. **Моніторинг** — бачити де і в якому стані граф прямо зараз
2. **Розслідування інцидентів** — коли вузол `failed`, зрозуміти **чому** і вирішити що робити

UI — суто read-only переглядач. Усі дії (`mt kill`, `mt run`, патчі) — тільки через CLI, тому що архітектура файл-центрична і CLI-first.

---

## Flow 1 — Щоденний моніторинг

```
Відкрити UI
→ бачить дерево вузлів із кольоровими станами
waiting   (grey)
running   (blue)
resolved  (green)
failed    (red)    ← виділено
invalidated (orange) ← виділено
→ автооновлення (polling mt scan --json)
→ нічого критичного → закрити
```

**Чому саме polling** а не SSE: стан вузла визначається наявністю **файлів** (не подіями). Сервер не може пушити зміни — він сам не знає про них до сканування. Polling `mt scan --json` — найпростіший і найпрямолінійніший варіант.

---

## Flow 2 — Розслідування інциденту

```
Помітив failed/invalidated вузол (виділений)
→ клік на вузол
→ панель деталей:
task.md:       що мав зробити вузол
run_001.md:    actor, result, created_at + ## Reasoning
run_002.md:    ...
→ читаю Reasoning → розумію причину
→ іду в CLI: mt kill / mt run / патч
```

**Чому tree-view а не DAG-граф із стрілками**: топологія `deps` → зберігається у фронтматері кожного `task.md`, але фізично вузли живуть у **вкладених директоріях**. Дерево файлової системи і є основна навігаційна структура. DAG зі стрілками — складніший рендеринг без додаткової користі для read-only переглядача.

---

## Flow 3 — Прогрес виконання задачі

```
Хочу побачити наскільки просунулась задача
→ розгортаю кореневий вузол
→ бачу дочірні (складений вузол)
collect-data  → resolved ✓
analyze       → running  ⟳
synthesize    → waiting  ○
→ розумію де фронт роботи
```

## Чого НЕ буде у цьому UI

| Функція | Чому ні |
|---|---|
| Кнопки `kill` / `run` | UI — observability tool, не control plane; CLI контролює |
| Редагування `task.md` | Immutable після worktree; навіть до — це CLI-операція |
| Повний вміст `outputs_NNN.md` | Файли можуть бути великими і містити `ref:` на інші файли; краще посилання |
| Real-time stream логів агента | Агент пише тільки `run_NNN.md` після завершення — немає потоку |

---

## Висновок: яка flow і чому

**Три flow** — моніторинг / розслідування / прогрес — всі зводяться до одного циклу:

```
scan файлів → відобразити стани → людина читає → людина діє через CLI
```

Це пряме відображення контракту моніторингу з npm/docs/mt.md (розділ "Контракт для моніторингу"): скрипт сканує, показує дерево зі станами, failed-вузли, активні worktree. UI — це той самий контракт, але у браузері замість термінала.
