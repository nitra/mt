## ADR Топологія графу розподілена в `task.md` кожного дочірнього вузла

## Context and Problem Statement
Система будує орієнтований ациклічний граф задач де кожен вузол може динамічно розкластись у підграф. Потрібно вирішити де зберігається топологія (ребра між вузлами) — в центральному файлі чи розподілено.

## Considered Options
* Центральний `graph.md` у батьківському вузлі — авторитетний список вузлів і ребер
* Кожен дочірній вузол у власному `task.md` оголошує своїх попередників через поле `deps:`

## Decision Outcome
Chosen option: "кожен дочірній вузол у `task.md` оголошує своїх попередників", because мета — універсальність: кожен вузол самодостатній, оркестратор відновлює повний граф скануванням усіх `task.md`, динамічний spawn не потребує оновлення центрального файлу.

### Consequences
* Good, because динамічний spawn — просто новий дочірній з `task.md`, оркестратор підхоплює при наступному скані без оновлення центрального стану.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Поле `deps:` у фронтматері `task.md` — список `id` siblings (вузлів-попередників у тому самому `subgraph/`). `id` вузла = назва директорії (не зберігається у фронтматері). Шлях: `tasks/<parent>/subgraph/<child-id>/task.md`.

## ADR Інваріант immutability прив'язаний до моменту створення git-worktree

## Context and Problem Statement
Система зберігає стан у файлах. Потрібно визначити коли файли вузла стають незмінними — одразу при створенні чи з якоїсь межі.

## Considered Options
* Файли незмінні з моменту створення (повна immutability)
* Файли вільно редагуються до старту роботи; після старту — тільки нові файли

## Decision Outcome
Chosen option: "файли вільно редагуються до старту роботи; після — тільки нові файли", because межа чітка і вже є в системі: `git worktree add` — це старт роботи агента. До worktree ніхто не читає файли вузла; після — append-only (тільки нові файли, нічого не змінюється).

### Consequences
* Good, because інженер може редагувати `task.md` без версіонування — просто kill worktree → редагуй → restart. Немає потреби в `task-v2.md`.
* Good, because transcript фіксує очікувану користь: відсутність edge cases "чи не змінився файл поки я читав?" — перевірка стану зводиться до "є worktree чи ні".
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Протокол зміни після старту: `kill worktree → редагуй вільно → restart`. Файли без worktree — вільні; з активним worktree — immutable.

## ADR Злиття `inputs.md` у `task.md`

## Context and Problem Statement
При spawn батько пише дочірньому два файли: `task.md` (місія і критерій) та `inputs.md` (посилання на дані). Питання — чи тримати їх окремо чи об'єднати.

## Considered Options
* Окремі файли `task.md` і `inputs.md`
* Єдиний `task.md` з секцією `## Inputs`

## Decision Outcome
Chosen option: "єдиний `task.md` з секцією `## Inputs`", because агент читає один файл замість двох; менше файлів при spawn; task і дані завжди разом — контекст цілісний.

### Consequences
* Good, because transcript фіксує очікувану користь: агент отримує повний контекст (місія + дані) з одного читання.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Секція `## Inputs` у `task.md` містить підсекції `### <port-name>` з `ref:` посиланнями — ніякого копіювання даних. Синтаксис `ref:`: `ref: path/to/file.md`, `ref: path/to/file.md#section-name`, `ref: path/to/file.md lines 10-50`. Зміна inputs до worktree: редагуй `task.md` вільно. Після worktree: kill → редагуй → restart.

## ADR Уніфікований `run_NNN.md` для всіх типів спроб на вузлі

## Context and Problem Statement
Система мала окремі файли для кожного типу активності: `error.md`, `repair/NNN.md`, `repair_context.md`. Паралельно виникла потреба у часових лімітах і для звичайних агентів — не тільки для інженера. Постало питання уніфікації.

## Considered Options
* Окремі файли: `error.md`, `repair/NNN.md`, `repair_context.md`
* Єдиний `run_NNN.md` з полем `actor:` для всіх типів спроб

## Decision Outcome
Chosen option: "єдиний `run_NNN.md` з полем `actor:`", because і звичайний агент, і інженер — це "спроба вирішити вузол"; спільне: reasoning, result, часовий ліміт, посилання на продукт. Різниця тільки в actor і в тому куди вказує `## Ref`.

### Consequences
* Good, because `budget_sec` переїхав у `task.md` — єдине місце для часового ліміту вузла; `repair_context.md` видалено.
* Good, because transcript фіксує очікувану користь: однакова схема для agent / engineer / human / auditor — оркестратору не потрібна окрема логіка для кожного типу.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Фронтматер `run_NNN.md`: `created_at`, `actor: agent | engineer | human | auditor`, `result: success | failed`. Секції: `## Reasoning` (обов'язкова), `### Script` (пише wrapper-скрипт, опціональна), `## Ref` (опціональна — `ref: outputs_001.md` для агента, `ref: patches/001-plan.md` для інженера). `budget_sec` у `task.md` фронтматері.

Тіло всіх `ops/` і `patches/` файлів — опціональне; фронтматер достатній. Recovery-логіка: `*-plan.md` без пари `*-fact.md` → операція перервана.

## ADR Усі назви файлів і директорій англійською

## Context and Problem Statement
Система активно використовує скрипти для сканування файлової системи і відновлення стану графу. Початково частина директорій мала кириличні назви (`операції/`, `патчі/`, `підграф/`, `вхідні.md`, `вихідні.md`, `помилка.md`).

## Considered Options
* Кириличні назви файлів і директорій для читабельності
* Англійські назви для всіх файлів і директорій

## Decision Outcome
Chosen option: "англійські назви для всіх файлів і директорій", because файли і директорії обробляються скриптами, тому назви повинні бути безпечними для shell без екранування.

### Consequences
* Good, because transcript фіксує очікувану користь: скрипти оркестратора і observability-інструменти не потребують обробки Unicode у шляхах.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Перейменування: `операції/` → `ops/`, `патчі/` → `patches/`, `підграф/` → `subgraph/`, `вхідні.md` → (злито в `task.md`), `вихідні.md` → `outputs_NNN.md`, `помилка.md` → (`run_NNN.md` з `result: failed`). Правило контракту: секції що парсить скрипт/оркестратор — англійські заголовки; секції з довільними даними — будь-яка мова.
