## ADR Рекурсивний складений ОАГ із динамічним розкладом вузлів та файловим сховищем стану

## Context and Problem Statement
Користувач проектував систему оркестрації задач, де директорії містять проекти, проекти містять задачі, а задачі можуть динамічно розкладатися на підграфи. Потрібно було вибрати правильну структуру даних та назву для цієї рекурсивної композиції, де кожен вузол або є атомарним, або містить власний граф задач.

## Considered Options
* Recursive Compound DAG (рекурсивний складений орієнтований ациклічний граф)
* 3D graph (просторова тривимірна структура)
* Hierarchical Task Network (HTN)
* Multi-level Graph

## Decision Outcome
Chosen option: "Recursive Compound DAG / Рекурсивний складений ОАГ", because структура задовольняє всі вимоги: вузол або атомарний, або розкладається на підграф; батько бачить лише інтерфейс (resolved/not); ребра несуть дані (outputs → inputs); декомпозиція динамічна (під час виконання).

### Consequences
* Good, because transcript фіксує очікувану користь: інкапсуляція — батько не знає що всередині вузла, будь-який вузол замінюється складеним без змін у батьківському графі.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл: `npm/docs/mt.md`. Команди: `mt`. Структура зберігається у файловій системі: `tasks/<node-id>/task.md`, `вхідні.md`, `вихідні.md`.

## ADR Файлова система як append-only сховище стану графу

## Context and Problem Statement
Потрібно було вибрати механізм зберігання стану графу задач, який підтримував би відновлення після збоїв, аудит, паралельне виконання через git-worktrees і був зручний для читання LLM-агентами.

## Considered Options
* Append-only файли (файли лише створюються, видаляються атомарно)
* Mutable JSON files (файли змінюються при зміні стану)

## Decision Outcome
Chosen option: "Append-only файли", because наявність файлу = стан вузла; `create file` атомарна операція; два ворктрі не можуть конфліктувати при merge, бо пишуть лише нові файли з унікальними іменами.

### Consequences
* Good, because transcript фіксує очікувану користь: ворктрі-межа вирішує race condition — merge є єдиною точкою рішення; аудит-трейл безкоштовний через git history.
* Bad, because transcript фіксує: scan для відновлення стану без індексу дорогий на великих графах.

## More Information
Стан вузла визначається наявністю файлів: `task.md`+`вхідні.md` = spawned; ворктрі існує = running; `вихідні.md` = resolved; `помилка.md` = failed. Append-only інваріант діє з моменту створення ворктрі: до цього моменту файли вузла можна вільно редагувати або видаляти.

## ADR Мета-рівень агент-інженер із необмеженим доступом до графу

## Context and Problem Statement
Потрібно було визначити механізм самовідновлення системи: як обробляти помилки вузлів, хто має право змінювати граф, і наскільки широким має бути scope цих змін.

## Considered Options
* Агент-інженер як мета-рівень поза графом із необмеженим доступом (може підніматися вгору і змінювати батьківські графи)
* Агент-інженер як вузол у графі (рекурсивний, обмежений локальним підграфом)

## Decision Outcome
Chosen option: "Агент-інженер як мета-рівень поза графом із необмеженим доступом", because потрібна можливість змінювати граф на будь-якому рівні ієрархії, включно з кореневим; рекурсивна природа агента-вузла ускладнила б supervision.

### Consequences
* Good, because transcript фіксує очікувану користь: агент може замінити атомарний вузол складеним, вставити проміжні вузли, перепідключити ребра або змінити вхідні дані.
* Bad, because transcript фіксує: інженер що змінює кореневий вузол інвалідує весь граф і може не вкластися в time budget — система стоїть.

## More Information
`GraphPatch` операції: replace node, insert nodes, rewire edges, modify inputs. Зберігається у `патчі/001-план.md` + `001-факт.md`. Перед патчем вузла інженер зобов'язаний спочатку кілнути всі залежні запущені ворктрі в топологічному порядку (від листів до цілі).

## ADR Часовий бюджет замість ліміту спроб для агента-інженера

## Context and Problem Statement
Потрібно було визначити механізм convergence guard для агента-інженера: як запобігти нескінченному циклу відновлення.

## Considered Options
* Time budget (виділений час на відновлення, необмежена кількість спроб)
* Max attempts (обмеження кількості спроб)

## Decision Outcome
Chosen option: "Time budget", because реалістичніше обмеження; агент може адаптувати стратегію залежно від залишку часу (багато часу → складна спроба, мало → швидкий fix).

### Consequences
* Good, because transcript фіксує очікувану користь: максимальний час до людини передбачуваний — `depth × budget`; кожен рівень ієрархії отримує свіжий бюджет при ескалації.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Поля у `repair_context`: `deadline`, `started_at`, `time_budget_sec`, `attempts`. При timeout: `node.state = unresolvable`, ескалація вгору. Батько при ескалації отримує свіжий (не залишок) time budget. Root timeout → notify SeniorEngineer.

## Decision Outcome

## ADR Топологія підграфу розподілена по `task.md` кожного дочірнього вузла

## Context and Problem Statement
Потрібно було вирішити де зберігати топологію підграфу (ребра між дочірніми вузлами): в центральному файлі батька чи розподілено по дочірніх.

## Considered Options
* Розподілена топологія — кожен дочірній вузол у своєму `task.md` оголошує `deps:` (список попередників)
* Центральний `граф.md` у батьківському вузлі

## Decision Outcome
Chosen option: "Розподілена топологія в `task.md` кожного дочірнього", because забезпечує універсальність: динамічний spawn нового вузла = просто новий `task.md` з `deps:`, без оновлення центрального файлу.

### Consequences
* Good, because transcript фіксує очікувану користь: оркестратор відновлює повний граф скануванням `task.md` всіх дочірніх; динамічний spawn не потребує оновлення жодного існуючого файлу.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`deps:` містить лише `id` сусідніх вузлів (siblings в межах того ж `підграф/`). Скрипт резолвить `../<dep-id>/`. Поле `id` у фронтматері відсутнє — читається з назви директорії. Поле `parent:` — відносний шлях від `tasks/`.
