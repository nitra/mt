title: flow як внутрішній протокол вузла під graph-оркестратором
description: `graph` керує DAG, worktree lifecycle і merge, а `flow` стає внутрішнім двостадійним протоколом одного вузла.
---

**Status:** Accepted
**Date:** 2026-06-06

## Context and Problem Statement

Існують дві паралельні системи: ранній `mt` workflow з init/spec/plan/verify/release, MT file-presence state і `docs/`, та нова архітектура `npm/docs/mt.md` з autonomous DAG: `tasks/<node>/task.md`, file-based state, git worktree і post-merge hook. Вони перекриваються у worktree lifecycle та кроках виконання, але мають несумісні формати і різні рівні автономії. Потрібно поєднати їх в єдину систему без дублювання відповідальності.

## Considered Options

- `graph` як оркестратор ззовні, `flow` як протокол всередині вузла.
- Повне злиття: `mt init` = `mt init`, `mt done` = merge + cascade.
- Зберегти обидві системи паралельно без злиття.
- Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "`graph` як зовнішній оркестратор, `flow` як внутрішній протокол вузла", because `graph` управляє worktree lifecycle, залежностями, merge і каскадом, а `flow` обслуговує логіку одного запуску зсередини worktree; межа відповідальності чітка і не дублюється.

### Consequences

- Good, because transcript фіксує очікувану користь: `flow` стає легшим, зникають MT file-presence state, `docs/specs/` і `docs/plans/`.
- Good, because `graph` отримує повний контроль над станом та паралелізмом.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

- Артефакти, що зникають: MT file-presence state, `docs/specs/`, `docs/plans/`.
- Нові артефакти: `task.md`, `plan_001.md`, `outputs_NNN.md`.
- Команди `mt init` і старий init-flow видаляються.
- `mt done` замінюється на сигнали `mt done|audit|failed`.
- Виконання вузла поділяється на Stage 1 Planning і Stage 2 Execution.
- Stage 1 entry point: `mt plan` усередині worktree після `mt run`.
- Stage 1 повертає або `plan_001.md` для atomic-вузла, або дочірні `task.md` і `mt spawn` для composite-вузла.
- Stage 2: агент виконує роботу, `mt verify` перевіряє критерії з `## Done when`, далі пишеться `outputs_NNN.md` і сигналізується `mt done|audit|failed`.
- `mt init` і `mt plan` з попередньої системи об'єднані в один крок `mt plan` з атрибутом `mode:` у frontmatter `task.md`.
- `mode: human` є default і означає інтерактивний діалог у терміналі.
- `mode: agent` означає автономне планування без участі людини.
- Preflight-перевірка виконання в `.worktrees/` переноситься з `mt init` у `mt plan`.

## Update 2026-06-06

- Драфт фіксує вихідну проблему: `flow` змішував planning (`spec`/`plan`) і execution (`run`/`verify`/`release`) в одну лінію без чіткої межі.
- Запропоновано два режими під контрактом `npm/docs/mt.md`: Stage 1 Planning для побудови DAG і Stage 2 Resolution для атомарного виконання вузла.
- Старі `mt verify` / `mt done` розглядалися як частина критерію `## Done when` у `task.md` або як сигнали `done/failed/spawn/audit`.

## Update 2026-06-06

- `mt plan` запускається агентом усередині worktree після `mt run`.
- `mt plan` читає `task.md` поточного вузла та `.n-cursor/system-prompt.md`.
- Stage 1 має вирішити, чи вузол atomic, чи composite: composite створює дочірні `tasks/<current>/<child>/task.md`, atomic створює `plan.md` або numbered plan artifact.
- Обговорювалися review-поля для плану й output: один `actor` або роздільні `plan_review` / `output_review`.
- Для `plan.md` обговорювалися секції `## Approach`, `## Files to touch`, `## Risks / open questions`.

## Update 2026-06-07

Уточнено двоетапний контракт flow як внутрішнього протоколу вузла:

- Stage 1 — `mt plan`: поєднує spec і decompose, читає `mode: human|agent` з `task.md`, пише `plan_001.md` або дочірні `task.md`.
- `mode: human` після Stage 1 переводить вузол у стан `plan-pending`; `mode: agent` дозволяє переходити далі без ручної паузи.
- Stage 2 — `mt verify`: окремий quality gate, що читає `task.md ## Done when`, `outputs_NNN.md` і `plan_001.md`, та пише `verify_001.md`.
- Додано ідею `flow audit` як спеціаліст-агента з JSON verdict `{ verdict: PASS|FAIL, reason }`.
- Застарілі команди старого flow (`mt init`, `spec`, `run`, `cancel`, `resume`, `repair`, `review`, `gate`, `release`) мають бути вилучені або замінені новим контрактом.

Transcript не містить підтверджених негативних наслідків, окрім breaking change через видалення старих команд.

## Update 2026-06-07

Уточнено межу між `graph` і `flow`:

- `graph` відповідає за DAG, deps, merge, worktrees і cascade.
- `flow` лишається execution protocol всередині одного вузла: plan, verify і agent signals.
- `npm/docs/mt.md` визначено living spec і джерелом правди; runtime-prompts і правила мають деривуватись із нього.
- Для агентів вводиться capability manifest `.n-cursor/actors.md`, а `task.md` може містити `actors: [...]` для task-level access control.
- Асинхронний аудит має бути представлений файловою чергою `pending-audit_NNN.md`, яку обробляє `mt watch`.

Додаткові файли з transcript: `.n-cursor/system-prompt.md`, `.n-cursor/engineer-prompt.md`, `.n-cursor/actors.md`, `memory/project_graph_flow_design.md`.

## Update 2026-06-07

Зафіксовано деталізацію переходу від монолітного flow до двох етапів:

- `mt init` поглинається у `mt plan`; `docs/specs/` зникає.
- `mt plan` створює `plan_001.md` для атомарного вузла або дочірні `task.md` для composite-вузла.
- Після composite planning `mt plan` не викликає `mt spawn` автоматично; агент має явно викликати `mt spawn` після перевірки дочірніх задач.
- Для аудиту обрано numbered файли `pending-audit_NNN.md`, де NNN відповідає `outputs_NNN.md`; `ref:` у frontmatter не потрібен, бо посилання закодоване в імені файлу.
- Старий facade B (`run/resume/cancel/repair`) має бути видалений, бо стан тепер повністю відновлюється з файлів вузла.

## Update 2026-06-07

Уточнено межу між зовнішньою оркестрацією DAG і протоколом всередині вузла.

- Stage 1: `mt plan` читає `task.md`, визначає atomic/composite шлях і створює або `plan_001.md`, або дочірні `task.md`.
- Stage 2: агент виконує роботу, пише `outputs_NNN.md` і сигналізує `mt done`, `mt audit` або `mt failed`.
- `mode: human` є дефолтом для інтерактивного планування; `mode: agent` дозволяє автономне планування.
- Аудит-черга використовує `pending-audit_NNN.md`, де `NNN` відповідає конкретному `outputs_NNN.md`.
- `mt watch` і post-merge hook доповнюють одне одного: hook реактивно запускає scan після merge, daemon покриває audit-loop, stale worktrees і budget-related стани.

Transcript не містить підтверджених негативних наслідків цих уточнень.

## Update 2026-06-07

Уточнено реалізаційну роль `mt plan` у двох режимах.

- Для `mode: human` обрано модель тонкого helper-а: `mt plan` робить preflight, читає `task.md`, форматує контекст, а IDE-агент разом із людиною пише `plan_001.md` або дочірні `task.md`; `mt plan --finalize` лише валідує результат.
- Для `mode: agent` `mt plan` може запускати subagent subprocess через наявний `subagent-runner` і обмежувати planning-фазу часткою `budget_sec`.
- Для composite-шляху обрано explicit spawn: `mt plan` створює дочірні `task.md`, але агент окремо викликає `mt spawn`.
- `mt verify` у цьому transcript описувався як hybrid check: скрипт перевіряє наявність/непорожність `outputs_NNN.md`, LLM оцінює `## Done when`; пізніші clean-рішення можуть замінювати цей механізм audit-чергою.

Згадані файли: `npm/scripts/dispatcher/index.mjs`, `npm/scripts/dispatcher/lib/commands.mjs`, `npm/scripts/dispatcher/lib/plan.mjs`, `npm/scripts/dispatcher/lib/spec.mjs`, `npm/scripts/dispatcher/lib/subagent-runner/`, `npm/docs/mt.md`.

## Update 2026-06-07

Зафіксовано спробу радикального спрощення namespace: прибрати окремий `flow`-шар і уніфікувати lifecycle задач під командами `mt`/graph.

- Мотивація: окремий namespace не виправданий, якщо більшість старих команд або замінюються graph-командами, або зникають через файловий стан вузла.
- Stage 1/Stage 2 лишаються ключовим протоколом вузла: planning через `mt plan`, execution через `outputs_NNN.md` і сигнали завершення.
- Аудит-черга через `pending-audit_NNN.md` замінює синхронну self-check модель.
- `pending-audit_NNN.md` дзеркалює `outputs_NNN.md`, щоб ім'я файлу саме було посиланням на версію виходу.

Цей update фіксує design-намір transcript; пізніші драфти цього батча уточнюють реалізацію через `flow`-рівневі wrappers над graph-командами.

## Update 2026-06-07

Після реалізаційного рефакторингу зафіксовано конкретний двоетапний протокол всередині вузла.

- Видалено старі dispatcher-компоненти навколо MT file-presence state: `active`, `spec`, `review`, `gate`, `commands`, `plan`, `state-store`, `flow-lock`, `level`, `capability`, `budget`, `artifact`, `planner`, `snapshot`, `reviewer`, `events`.
- Додано `lib/flow-plan.mjs`: `mt plan` читає `task.md`, поля `mode` і `hint`, створює numbered `plan_NNN.md` template.
- Додано `lib/flow-verify.mjs`: структурний check і stdout-контекст; exit code відображає лише структурний результат.
- Додано `lib/flow-signals.mjs`: `flow done`, `flow audit`, `flow failed`, `flow spawn` резолвлять node path через `MT_NODE_PATH` або `.n-cursor/current-node` і делегують у graph-команди.
- Для Stage 1 обрано гібридне поле `hint: atomic|composite`: людина може підказати напрям, але агент не заблокований цим полем.
- `mode: human` є default; `mode: agent` дозволяє автономну planning-фазу.
- `mt plan` не викликає spawn автоматично; після створення дочірніх `task.md` агент явно викликає `flow spawn`.
- `pending-audit_NNN.md` лишається async-чергою для зовнішнього аудитора, де `NNN` відповідає `outputs_NNN.md`.

Результат transcript: оновлено `index.mjs`, `tests/index.test.mjs`, `worktree-cli.mjs`; 97 тестів green.
