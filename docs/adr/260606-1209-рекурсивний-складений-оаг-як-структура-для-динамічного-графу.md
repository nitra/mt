## ADR Рекурсивний складений ОАГ як структура для динамічного графу задач

## Context and Problem Statement
Потрібно описати та обрати структуру даних для системи де задачі, що керуються оркестратором, розбиваються на динамічні workflow-графи, а вузли графу можуть рекурсивно містити власні підграфи. Питання полягало в тому, як правильно назвати і формалізувати цю структуру.

## Considered Options
* Recursive Compound DAG (рекурсивний складений ОАГ)
* 3D-граф (відкинуто — описує просторове відображення, а не структуру зв'язків)
* Multi-level Graph
* Hierarchical Task Network (HTN)

## Decision Outcome
Chosen option: "Recursive Compound DAG", because батько бачить лише інтерфейс вузла (`state = resolved`/`failed`), не знаючи чи він атомарний чи складений; будь-який атомарний вузол можна замінити складеним без змін у батьківському графі (substitutability).

### Consequences
* Good, because transcript фіксує очікувану користь: рекурсивний executor однаковий на всіх рівнях — `execute(node)` перевіряє тип і делегує.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл `npm/docs/mt.md` у корені репозиторію. Реалізація схожа на Dask, Prefect dynamic tasks, LangGraph.

## ADR Динамічна декомпозиція вузла під час виконання

## Context and Problem Statement
Питання: коли вузол вирішує стати складеним — статично (структура відома наперед) чи динамічно (під час виконання, залежно від вхідних даних)?

## Considered Options
* Статична декомпозиція — структура графу відома до запуску
* Динамічна декомпозиція — вузол вирішує при запуску на основі `inputs`

## Decision Outcome
Chosen option: "Динамічна декомпозиція", because рішення приймається `inspect(inputs)` при запуску вузла; батько не знає заздалегідь чи буде вузол атомарним чи складеним.

### Consequences
* Good, because transcript фіксує очікувану користь: граф адаптується до даних, а не до заздалегідь відомої структури.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл `npm/docs/mt.md`. Паттерн: `execute(node): if node.isAtomic → run it; else → execute(node.graph)`.

## ADR Data-flow ребра (outputs → inputs між вузлами)

## Context and Problem Statement
Потрібно визначити семантику ребер графу: вони несуть лише залежності (порядок виконання) чи і дані між вузлами?

## Considered Options
* Ребра як залежності (лише порядок)
* Ребра як data-flow (результат вузла = вхідні дані наступника)

## Decision Outcome
Chosen option: "Ребра як data-flow", because значення "тече" по ребру коли `from.state = resolved`; ребро має `from: NodeId+portId` та `to: NodeId+portId`.

### Consequences
* Good, because transcript фіксує очікувану користь: вузол може мати кілька вхідних/вихідних портів з явно названими даними.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл `npm/docs/mt.md`. Множинні exit-вузли підграфу merge свої outputs в outputs батьківського вузла.

## ADR Файлова система як append-only state store (Markdown+YAML)

## Context and Problem Statement
Потрібно обрати формат зберігання стану графу, зручний насамперед для LLM-агентів, та усунути проблему атомарності запису при мутації файлів.

## Considered Options
* JSON-файли з мутацією (`meta.json`, `inputs.json`, `outputs.json`) — початкова пропозиція
* Markdown з YAML-фронтматером, append-only (тільки створення і атомарне видалення sentinel-файлів)

## Decision Outcome
Chosen option: "Markdown+YAML, append-only", because стан вузла визначається наявністю файлів, а не їх вмістом; `create file` атомарна операція на більшості ФС; merge ворктрі без конфліктів за визначенням (кожен агент створює лише нові файли з унікальними іменами).

### Consequences
* Good, because transcript фіксує очікувану користь: `repair_history.md` як append-only журнал — LLM читає і дописує природно; git merge ворктрі завжди чистий.
* Bad, because transcript фіксує: визначення поточного стану потребує scan файлів (немає індексу).

## More Information
Файл `npm/docs/mt.md`. Sentinel-файли: `виконується`, `скасовано`, `знедійснений`. Версіонування виходів: `вихідні-v2.md` замість мутації `вихідні.md`. Аналог: git object store, log-structured FS.

## ADR АгентІнженер як мета-рівень з повним доступом до графу

## Context and Problem Statement
Потрібно визначити роль і рівень доступу агента що відновлює помилки: він є вузлом всередині графу, чи supervisor поза графом? І наскільки широкий його доступ до модифікації?

## Considered Options
* Інженер як вузол у графі (рекурсивний supervisor)
* Інженер як мета-рівень поза графом з необмеженим доступом

## Decision Outcome
Chosen option: "Мета-рівень поза графом, необмежений доступ", because інженер може підніматись вгору по ієрархії і змінювати будь-який рівень включно з коренем; він не є вузлом і не може бути викликаний рекурсивно тим самим механізмом.

### Consequences
* Good, because transcript фіксує очікувану користь: інженер бачить повний `path від кореня до вузла що впав` і вибирає оптимальний рівень втручання.
* Bad, because transcript фіксує: необмежений доступ дозволяє інвалідувати весь граф від кореня; можливий cascade інвалідації.

## More Information
Файл `npm/docs/mt.md`. `GraphPatch` включає: `replace node`, `insert nodes`, `rewire edges`, `modify inputs`. Архітектурний патерн: Reflective Architecture (мета-рівень інспектує і модифікує об'єктний рівень).

* Пам'ять на вузлі де відбуваються зміни (`repair_history.md`)

## ADR Time budget замість лічильника спроб для відновлення

## Context and Problem Statement
Потрібен механізм convergence guard щоб інженер не циклився нескінченно. Варіанти: обмежити кількість спроб або виділити часовий бюджет.

## Considered Options
* Максимальна кількість спроб (`max_attempts: N`)
* Часовий бюджет (`repair_deadline`, наприклад 10 хвилин, необмежена кількість спроб)

## Decision Outcome
Chosen option: "Часовий бюджет", because кількість спроб штучно обмежує — одна складна спроба еквівалентна десяти швидким; час відповідає реальним витратам і дозволяє інженеру адаптувати стратегію залежно від залишку часу.

### Consequences
* Good, because transcript фіксує очікувану користь: `максимальний час до людини = depth × budget` — передбачувано; інженер може вибрати складну або швидку стратегію залежно від `remaining = deadline - now()`.
* Bad, because Neutral, because transcript не містить підтвердження наслідку.

## More Information
Файл `npm/docs/mt.md`. Поле `repair_context.json`: `deadline`, `started_at`, `time_budget_sec`. При `remaining ≤ 0` → `node.state = "unresolvable"` → escalate.

## ADR Ієрархічна ескалація зі свіжим бюджетом на кожному рівні

## Context and Problem Statement
Коли інженер вичерпує time budget на одному рівні — що відбувається далі? Батьківський рівень отримує залишок часу чи власний свіжий бюджет?

## Considered Options
* Батько отримує залишок бюджету від дочірнього рівня
* Кожен рівень отримує свіжий незалежний бюджет при ескалації
* Root timeout → halt системи

## Decision Outcome
Chosen option: "Кожен рівень — свіжий бюджет; root timeout → notify SeniorEngineer", because кожен рівень ієрархії незалежний у своїх спробах відновлення; root передає контроль людині з повним звітом про всі спроби.

### Consequences
* Good, because transcript фіксує очікувану користь: `senior_report.json` містить `escalation_chain` з усіма спробами і часом — повний контекст для людини.
* Bad, because transcript фіксує: максимальний час до людини = `depth × budget`; глибокий граф може довго не ескалувати.

## More Information
Файл `npm/docs/mt.md`. `senior_report.json` включає: `failed_node` (path від root), `escalation_chain[]`, `current_graph_snapshot`, `suggested_next_steps`.

## ADR Правило: kill залежних перед патчем вже змердженого вузла

## Context and Problem Statement
Якщо інженер вирішує пропатчити вузол що вже в `main` (після merge), а залежні від нього вузли вже виконуються у своїх ворктрі — в якому порядку діяти?

## Considered Options
* Патч вузла → cascade інвалідація залежних
* Спочатку kill всіх залежних ворктрі (топологічний порядок від листів), потім патч

## Decision Outcome
Chosen option: "Спочатку kill залежних (від листів до цілі), потім патч", because інженер не може модифікувати вузол поки є запущені ворктрі залежних — це гарантує що залежні не завершаться з застарілими входами.

### Consequences
* Good, because transcript фіксує очікувану користь: жоден залежний вузол не merge з застарілими `вхідні.md`.
* Bad, because Neutral, because transcript не містить підтвердження наслідку.

## More Information
Файл `npm/docs/mt.md`. Порядок: kill leaf-ворктрі → kill проміжних → патч цільового вузла → рестарт каскаду.

## ADR Принцип "план → дія → факт" як universal recovery pattern

## Context and Problem Statement
При збої інженера між кроком "kill залежних" і кроком "патч вузла" система опиняється в невизначеному стані. Потрібен загальний механізм відновлення для будь-якої операції.

## Considered Options
* Ідемпотентні операції (повторне виконання безпечне)
* Write-ahead log через файли: `*-план.md` пишеться до дії, `*-факт.md` після

## Decision Outcome
Chosen option: "Write-ahead log через файли (`*-план.md` → дія → `*-факт.md`)", because принцип застосовується до всіх операцій системи: spawn вузла, виконання агента, kill залежних, патч інженера; при старті оркестратор сканує `*-план.md` без відповідного `*-факт.md` → незавершена операція → відновити.

### Consequences
* Good, because transcript фіксує очікувану користь: оркестратор при старті: `scan tasks/**/операції/*-план.md` — якщо `*-факт.md` відсутній → відновити операцію.
* Bad, because Neutral, because transcript не містить підтвердження наслідку.

## More Information
Файл `npm/docs/mt.md`. Таблиця відповідностей: `місія.md` (план агента) → `вихідні.md` (факт); `kill-план.md` → `kill-факт.md`; `патч-план.md` → `патч-факт.md`. Аналог: WAL в базах даних, Saga pattern у розподілених системах.
