## ADR Файлова система як сховище стану графу задач

## Context and Problem Statement
Система динамічного графу задач потребує механізму зберігання стану вузлів, що є стійким до збоїв, доступним для LLM-агентів і не потребує зовнішньої бази даних.

## Considered Options
* Файлова система (Markdown + YAML-фронтматер)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Файлова система (Markdown + YAML-фронтматер)", because стан кожного вузла повністю відновлюється скануванням файлів (`task.md`, `run_<epoch>.md`, `outputs_<epoch>.md`, `running.lock`, `invalidated`) без зовнішніх сервісів; фронтматер читається скриптами, тіло — LLM.

### Consequences
* Good, because transcript фіксує очікувану користь: граф відновлюється при будь-якому збої простим `ls`-скануванням директорій.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли вузла: `tasks/<node-id>/task.md`, `run_<epoch>.md`, `outputs_<epoch>.md`, `running.lock`, `invalidated`, `artifacts/`. Стан виводиться з наявності файлів: тільки `task.md` → `waiting`; є `running.lock` → `running`; є `outputs_<epoch>.md` → `resolved`; є `invalidated` → `invalidated`.

## ADR Append-only інваріант з межею ворктрі

## Context and Problem Statement
Система потребує чіткого правила незмінності файлів, щоб дозволити безпечне паралельне виконання і відновлення після збоїв, одночасно не блокуючи підготовку вузла до старту.

## Considered Options
* Файли immutable з моменту створення (абсолютна незмінність)
* Файли immutable тільки після створення ворктрі

## Decision Outcome
Chosen option: "Файли immutable тільки після створення ворктрі", because до `git worktree add` жоден агент ще не читає файли вузла — вільне редагування безпечне; після створення ворктрі агент активний і будь-яка зміна порушує консистентність.

### Consequences
* Good, because transcript фіксує очікувану користь: інженер може вільно змінити `task.md` якщо спочатку вбиває ворктрі — один атомарний протокол (`kill → edit → restart`) без версіонування файлів.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Протокол зміни: `немає ворктрі → редагуй вільно`; `є ворктрі → mt kill → редагуй → mt run`. Ворктрі створюється командою `git worktree add .worktrees/<sanitized-path>-<epoch> main`.

## ADR Об'єднання `inputs.md` у `task.md`

## Context and Problem Statement
Початкова архітектура мала два окремих файли: `task.md` (місія) і `inputs.md` (посилання на вхідні дані). Потрібне рішення щодо їх розподілу або об'єднання.

## Considered Options
* Окремі файли `task.md` і `inputs.md`
* Один файл `task.md` з секцією `## Inputs`

## Decision Outcome
Chosen option: "Один файл `task.md` з секцією `## Inputs`", because агент читає один файл замість двох; задача і дані завжди в одному місці; змінити inputs можна тим самим протоколом (`kill → edit → restart`).

### Consequences
* Good, because transcript фіксує очікувану користь: менше файлів при spawn, цілісний контекст для агента.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Фронтматер `task.md`: `created_at`, `parent` (відносний, відсутній у кореневого), `deps` (список siblings), `budget_sec`. Обов'язкові секції: `## Task`, `## Done when`, `## Inputs`.

## ADR Уніфікований `run_<epoch>.md` для всіх акторів

## Context and Problem Statement
Початковий дизайн мав окремі файли: `outputs.md` (append-only), `error.md` (помилки), `repair/NNN.md` (інженер). Це ускладнювало систему і порушувало загальний принцип immutability.

## Considered Options
* Окремі файли для різних типів запусків (`error.md`, `repair/NNN.md`)
* Уніфікований `run_<epoch>.md` з полем `actor`

## Decision Outcome
Chosen option: "Уніфікований `run_<epoch>.md` з полем `actor`", because і звичайний агент, і інженер, і людина — це "спроба вирішити вузол" з однаковою структурою; `actor: agent | engineer | human | auditor` розрізняє тип без окремих файлів.

### Consequences
* Good, because transcript фіксує очікувану користь: одна схема замість трьох, всі immutable, граф стану простіший.

## More Information
Схема `run_<epoch>.md`: фронтматер (`created_at`, `actor`, `result: success|failed`, опц. `worktree`), `## Reasoning` (вільний текст, для інженера включає опис змін), `## Script` (опц., wrapper), `## Ref` (опц., `ref: outputs_<epoch>.md`). Видалені: `error.md`, `repair/NNN.md`, `repair_context.md`, `patches/`.

## ADR Git post-merge hook як тригер наступника

## Context and Problem Statement
Після завершення вузла і злиття ворктрі потрібен механізм що автоматично запускає наступників у топологічному порядку.

## Considered Options
* Ручний запуск людиною (`mt run <path>`)
* `mt run` як живий процес-демон (B)
* Git `post-merge` hook тригерить `mt run --auto` (C)

## Decision Outcome
Chosen option: "Git `post-merge` hook тригерить `mt run --auto`", because кожен компонент робить одне: wrapper завершує і мержить, git hook тригерить наступний цикл; `mt run` залишається одноразовою командою без стану.

### Consequences
* Good, because transcript фіксує очікувану користь: людина запускає тільки кореневий вузол — система доходить до кінця (або до `failed`) автоматично.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Ланцюг: агент завершує → wrapper пише `run_<epoch>.md` → wrapper робить `git merge` → `post-merge` hook → `mt run --auto` сканує граф, запускає всіх розблокованих наступників з урахуванням ліміту `warn_worktrees_above`.

## ADR Видалення `ops/` і `patches/` — ігнорування crash recovery при spawn/kill

## Context and Problem Statement
Початкова архітектура включала `ops/spawn-plan-<ts>.md`, `ops/spawn-fact-<ts>.md`, `ops/kill-plan-<ts>.md`, `ops/kill-fact-<ts>.md` і `patches/NNN-plan.md` / `NNN-fact.md` для відновлення після збою під час spawn або патчингу.

## Considered Options
* Зберегти `ops/` і `patches/` (повне crash recovery)
* Видалити (ігнорувати сценарій)

## Decision Outcome
Chosen option: "Видалити", because spawn відбувається всередині ворктрі і мержиться атомарно — якщо ворктрі не замержено, дочірні вузли просто не з'являються в main; сценарій часткового spawn в main без ворктрі не є актуальним на поточному етапі.

### Consequences
* Good, because transcript фіксує очікувану користь: спрощена файлова структура, менше файлів на кожен вузол.
* Bad, because відсутнє crash recovery при spawn в main (але transcript вважає сценарій неактуальним зараз).

## More Information
Додаткової інформації в transcript не зафіксовано.

## ADR Англійська для всіх імен файлів і директорій

* Виключно англійська для всіх імен файлів і директорій

## Decision Outcome
Chosen option: "Виключно англійська для всіх імен файлів і директорій", because всі файли і директорії опрацьовуються скриптами — єдиний безпечний варіант без проблем з кодуванням і glob-паттернами.

### Consequences
* Good, because transcript фіксує очікувану користь: скрипти і CLI працюють без обробки Unicode в шляхах.

Правило в контракті: тіло секцій — будь-яка мова; секції що парсить скрипт — англійські заголовки; імена файлів і директорій — виключно англійська.

## ADR Вільний текст у `## Summary` замість структурованих портів

## Context and Problem Statement
`outputs_<epoch>.md` міг мати іменовані секції-порти (наприклад `## anomalies`, `## report`) що дозволяють наступникам робити точні `ref:` посилання. Потрібне рішення між структурованим API і вільним текстом.

## Considered Options
* Іменовані секції-порти (`## anomalies ref: ...`, `## report ref: ...`)
* Одна вільна секція `## Summary`

## Decision Outcome
Chosen option: "Одна вільна секція `## Summary`", because система LLM-first: агент-наступник читає весь `outputs_<epoch>.md` і розуміє що є з prose; він може попросити всі дані або вказати в своєму `## Inputs` що саме йому потрібно — без жорстких обмежень на структуру виводу.

### Consequences
* Good, because transcript фіксує очікувану користь: агент не прив'язаний до фіксованої схеми портів, може описати будь-які артефакти в довільному форматі.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Схема `outputs_<epoch>.md`: `created_at` (обов'язково), `## Summary` (обов'язково, вільний текст). Великі артефакти зберігаються в `tasks/<node-id>/artifacts/`, посилання на них — в тексті `## Summary`.
