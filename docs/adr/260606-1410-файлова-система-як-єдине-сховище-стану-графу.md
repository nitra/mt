## ADR Файлова система як єдине сховище стану графу

## Context and Problem Statement
Система динамічного графу задач потребує надійного сховища стану, що витримує збої агентів та оркестратора. Потрібно вирішити, де і як зберігати стан кожного вузла та топологію графу.

## Considered Options
* Файлова система як state store (стан = наявність файлів)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Файлова система як state store", because стан вузла визначається виключно наявністю файлів у його директорії — жодної окремої БД. Відновлення після збою = scan файлів.

### Consequences
* Good, because transcript фіксує очікувану користь: безкоштовна персистентність, аудит-трейл, відновлення без зовнішніх залежностей.
* Bad, because scan для відновлення стану може бути повільним на великих графах — відсутній індекс.

## More Information
Таблиця станів зафіксована в `npm/docs/mt.md`: `task.md` → pending; активний worktree → running; `outputs.md` → resolved; `error.md` без worktree → failed; sentinel `invalidated` → invalidated.

## ADR Append-only інваріант з межею на git worktree

## Context and Problem Statement
Система потребує гарантії цілісності файлів під час паралельного виконання агентів у ворктрі, водночас залишаючи можливість редагувати задачі до їх запуску.

## Considered Options
* Append-only з межею на момент створення worktree
* Повна незмінність файлів після spawn
* Дозволити зміни до завершення обробки (до `outputs.md`)

## Decision Outcome
Chosen option: "Append-only з межею на момент створення worktree", because до створення worktree ніякий агент не читає файли — редагування безпечне; після створення worktree ізоляція гарантує цілісність. Користувач уточнив: "якщо робота у вузлі почалась [...] до цього моменту, це не актуально".

### Consequences
* Good, because merge = єдина точка рішення оркестратора; race condition між записом outputs і скасуванням усувається архітектурно (не через блокування).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Правило зафіксоване в `npm/docs/mt.md` у розділі `## Invariants`: "Append-only після ворктрі — файли тільки створюються. Виняток: до старту ворктрі файли вільно редагуються."

## ADR Топологія графу розподілена по task.md кожного вузла

## Context and Problem Statement
При розкладанні вузла на підграф необхідно зберігати топологію (залежності між дочірніми вузлами). Питання: центральний файл чи розподілений підхід.

## Considered Options
* Топологія в `task.md` кожного дочірнього (поле `deps:`)
* Центральний файл `graph.md` у батьківському вузлі

## Decision Outcome
Chosen option: "Топологія в `task.md` кожного дочірнього (поле `deps:`)", because "мета в універсальності" — кожен вузол знає своїх попередників, центральний файл не потрібен. Оркестратор відновлює повний граф скануванням `task.md` всіх вузлів.

### Consequences
* Good, because transcript фіксує очікувану користь: динамічний spawn — просто новий вузол зі своїм `task.md` з `deps:`, не потрібно оновлювати центральний файл; можливий concurrent spawn без конфліктів.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Поле `deps:` у `task.md` містить список siblings (id, унікальний в межах батька). Повний шлях до будь-якого dep = `../<dep-id>/`. Детальна схема в `npm/docs/mt.md`, розділ `### task.md`.

## ADR task.md як єдиний вхідний файл вузла (злиття місії та inputs)

## Context and Problem Statement
Спочатку передбачалося два окремих файли: `місія.md` (опис задачі) і `вхідні.md` / `inputs.md` (посилання на вхідні дані). Постало питання чи доцільне таке розділення.

## Considered Options
* Один файл `task.md` з секціями `## Task`, `## Done when`, `## Inputs`
* Два окремих файли: `task.md` + `inputs.md`

## Decision Outcome
Chosen option: "Один файл `task.md`", because "агент читає один файл замість двох", місія і дані завжди разом — контекст цілісний. Проблема "не можна редагувати після старту" вирішена протоколом kill-before-edit, а не версіонуванням.

### Consequences
* Good, because transcript фіксує очікувану користь: менше файлів при spawn, менша кількість file reads агентом при старті.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Обов'язкові секції: `## Task`, `## Done when`, `## Inputs` — заголовки англійські (парсяться скриптом). Підсекції `### <name>` у `## Inputs` містять `ref:` або inline-текст. Детальна схема в `npm/docs/mt.md`, розділ `### task.md`.

## ADR Патерн plan→action→fact як write-ahead log для всіх операцій

## Context and Problem Statement
Агенти та інженер можуть падати в середині багатокрокових операцій (spawn підграфу, kill залежних вузлів, патч). Потрібен механізм відновлення без централізованого координатора.

## Considered Options
* Write-ahead log через пари файлів `*-plan.md` / `*-fact.md`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Write-ahead log через пари файлів", because "завжди пишемо план, а потім діємо, а потім факт, і якщо збій то звіряємо план та факт". Наявність `*-plan.md` без `*-fact.md` = незавершена операція → відновити.

### Consequences
* Good, because transcript фіксує очікувану користь: `mt recover` сканує `ops/*-plan.md` без відповідного `*-fact.md` і відновлює незавершені операції.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Операції з парами plan/fact: `ops/spawn-plan-<ts>.md` / `ops/spawn-fact-<ts>.md`, `ops/kill-plan-<ts>.md` / `ops/kill-fact-<ts>.md`, `patches/001-plan.md` / `patches/001-fact.md`. Формат timestamp у назвах файлів: `YYYYMMDDTHHmmssZ`. Детально в `docs/graph-cli.md` і `npm/docs/mt.md`, розділ `## план → дія → факт`.

## ADR Протокол kill-before-patch для ретроактивних змін вузлів

## Context and Problem Statement
Інженер може виявити що вже завершений і змерджений вузол повернув некоректні дані. Залежні вузли вже можуть виконуватись. Потрібен безпечний порядок дій.

## Considered Options
* Kill залежних у топологічному порядку перед патчем цілі
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Kill залежних у топологічному порядку перед патчем", because "інженер не може скасувати/модифікувати вузол_1 якщо вже йде у своєму вузлі вузол_2 залежний від нього, він зпочатку повинен кільнути агента з його вузлом2 і ворктрі, і тільки тоді патчити вузол 1".

### Consequences
* Good, because transcript фіксує очікувану користь: запобігає ситуації де залежний вузол завершується зі стейлими вхідними після патчу попередника.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Повний протокол: 1) записати `patches/001-plan.md` у вузлі-цілі; 2) `mt kill --cascade` від листів до цілі; 3) застосувати патч; 4) записати `patches/001-fact.md`; 5) рестарт каскаду. Детально в `npm/docs/mt.md`, розділ "Протокол patch залежного вузла".
