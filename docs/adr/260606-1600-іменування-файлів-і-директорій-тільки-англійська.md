## ADR Іменування файлів і директорій — тільки англійська

## Context and Problem Statement
Файлова система є state store системи; імена файлів і директорій обробляються скриптами-оркестраторами. Початкові імена були українськими (`вхідні.md`, `підграф/`, `операції/`, `патчі/`).

## Considered Options
* Залишити українські імена
* Перейти на англійські імена

## Decision Outcome
Chosen option: "Перейти на англійські імена", because всі файли та директорії обробляються скриптами, тому їхні назви мають бути безпечними у будь-якому shell і утиліті.

### Consequences
* Good, because скрипти не мають проблем з Unicode в шляхах і glob-патернах.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Замінено: `вхідні.md` → `inputs.md`, `підграф/` → `subgraph/`, `операції/` → `ops/`, `патчі/` → `patches/`. Зафіксовано в контракті `npm/docs/mt.md`: "Секції що парсить скрипт/оркестратор → англійські заголовки; секції з довільними даними → будь-яка мова."

## ADR YAML-атрибути фронтматеру — англійська, snake_case

## Context and Problem Statement
Фронтматер файлів системи містив атрибути українською мовою (`батько:`, `залежності:`, `створено_о:`). Ці поля читаються агентами й скриптами.

## Considered Options
* Зберегти українські атрибути
* Перейти на англійські атрибути (snake_case)

## Decision Outcome
Chosen option: "Перейти на англійські атрибути", because атрибути є машинозчитуваними полями для скриптів і агентів; англійська + snake_case є стандартом для YAML у програмному середовищі.

### Consequences
* Good, because уніфіковано стиль атрибутів із загальноприйнятими YAML-конвенціями.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Відображення: `батько` → `parent`, `залежності` → `deps`, `створено_о` → `created_at`, `час` → `occurred_at`, `вузли-створено` → `nodes_created`. Файл `npm/docs/mt.md`, розділ "Схеми файлів".

## ADR `id` вузла — не поле фронтматеру, читається зі шляху

## Context and Problem Statement
Вузли мали поле `id:` у фронтматері `task.md`. Структура файлової системи вже однозначно визначає ідентифікатор вузла через назву директорії.

## Considered Options
* Зберігати `id:` у фронтматері
* Прибрати `id:` — читати зі шляху директорії

## Decision Outcome
Chosen option: "Прибрати `id:` — читати зі шляху директорії", because дублювання між назвою директорії і полем `id:` створює ризик розбіжності; скрипт завжди може отримати id з шляху до файлу.

### Consequences
* Good, because transcript фіксує очікувану користь: усунення ризику розбіжності між `id` у фронтматері і назвою директорії.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Правило: `id` вузла = назва директорії вузла. `task.md` не містить поля `id:`. Файл `npm/docs/mt.md`.

## ADR Злиття `inputs.md` у `task.md`

## Context and Problem Statement
Початкова схема передбачала окремий файл `inputs.md` — контракт даних від батька до дитини. Агент при запуску читав два файли: `task.md` і `inputs.md`.

## Considered Options
* Зберегти окремий `inputs.md`
* Злити вхідні дані в `task.md` як секцію `## Inputs`

## Decision Outcome
Chosen option: "Злити вхідні дані в `task.md` як секцію `## Inputs`", because агент отримує цілісний контекст з одного файлу; місія і дані завжди пов'язані між собою.

### Consequences
* Good, because агент читає один файл замість двох; менше файлів при spawn.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Секція `## Inputs` (обов'язковий заголовок англійською) з підсекціями `### <port-name>` містить `ref:` або inline-текст. Файл `npm/docs/mt.md`, схема `task.md`.

## ADR Межа append-only інваріанту — створення worktree

## Context and Problem Statement
Архітектура передбачає append-only підхід: файли тільки створюються. Виникло питання: чи можна редагувати `task.md` до початку роботи агента?

## Considered Options
* Глобальний append-only (ніколи не змінювати після створення)
* Append-only тільки після старту worktree

## Decision Outcome
Chosen option: "Append-only тільки після старту worktree", because до створення worktree жоден агент не читає файли; редагування безпечне. Межа чітка і вже закладена в системі через git worktree.

### Consequences
* Good, because інженер може коригувати `task.md` без re-spawn: якщо worktree існує — kill → edit → restart.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Правило: до `git worktree add` — файли вільно редагуються і видаляються; після — тільки нові файли, нічого не змінюється. Файл `npm/docs/mt.md`, розділ "Інваріант незмінності".

## ADR Уніфікований `run_NNN.md` замість `error.md` і `repair_history.md`

## Context and Problem Statement
Система мала окремі файли для різних типів "спроб": `error.md` (wrapper-скрипт), `repair_history.md` (інженер). Обидва описували спробу виконати вузол. Крім того, user зауважив що і звичайний агент може мати кілька підходів з часовим лімітом.

## Considered Options
* Окремі файли за типом (`error.md`, `repair_history.md`)
* Єдиний нумерований файл `run_NNN.md` з полем `actor`

## Decision Outcome
Chosen option: "Єдиний нумерований файл `run_NNN.md` з полем `actor`", because будь-яка спроба — агента, інженера, людини чи аудитора — структурно ідентична; уніфікація прибирає дублювання схем.

### Consequences
* Good, because одна схема для всіх типів спроб; history вузла — відсортований список `run_*.md`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Схема: frontmatter `created_at`, `actor: agent | engineer | human | auditor`, `result: success | failed`; секції `## Reasoning` (обов'язкова), `## Script` (wrapper, опціонально), `## Ref` (опціонально). `## Reasoning` для інженера містить і що зробив, і що змінив. Файл `npm/docs/mt.md`.

## ADR Видалення `ops/` і `patches/` — ігнорування crash-recovery сценарію

## Context and Problem Statement
Директорії `ops/` (`spawn-plan/fact`, `kill-plan/fact`) і `patches/` (`NNN-plan/fact`) реалізували патерн plan→action→fact для відновлення після збою при spawn/kill/patch. Ці файли становили більшу частину структури вузла.

## Considered Options
* Зберегти `ops/` і `patches/` для crash recovery
* Видалити — ігнорувати сценарій обриву

## Decision Outcome
Chosen option: "Видалити — ігнорувати сценарій обриву", because якщо spawn відбувається в worktree (а не напряму в main), обрив означає просто незамерджений worktree — граф залишається чистим без частково-створених вузлів. Crash-recovery сценарій неактуальний для поточного контексту.

### Consequences
* Good, because файлова структура вузла мінімальна: `task.md`, `invalidated`, `run_NNN.md`, `outputs_NNN.md`.
* Bad, because Neutral, because transcript не містить підтвердження наслідку — crash recovery при прямому spawn у main не охоплено.

## More Information
Інформація з `patches/` (reasoning, зміни) перенесена до `## Reasoning` секції в `run_NNN.md`. `mt scan` тепер визначає стан за наявністю `outputs_*.md`, а не за парами `plan/fact`. Файл `npm/docs/mt.md`, розділ "Файлова структура вузла".
