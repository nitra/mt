## ADR Рекурсивний складений ОАГ з динамічним розкладом

## Context and Problem Statement
Необхідно описати архітектуру системи для виконання задач, де завдання розбиваються на dynamic workflow, формуючи граф задач. Вузли всередині графу можуть рекурсивно розкладатись на підграфи, а на кореневому рівні також можуть існувати окремі задачі. Потрібна точна назва та формальний опис цієї структури.

## Considered Options
* Recursive Compound DAG (рекурсивний складений орієнтований ациклічний граф)
* Hierarchical Task Network (HTN)
* Multi-level Graph / Graph of Graphs
* 3D Graph

## Decision Outcome
Chosen option: "Recursive Compound DAG (Рекурсивний складений ОАГ)", because структура точно відповідає термінології теорії графів (compound node — вузол що містить підграф), і батьківський рівень бачить лише інтерфейс вузла (стан resolved/failed) незалежно від внутрішньої реалізації.

### Consequences
* Good, because будь-який вузол замінюваний: атомарний може стати складеним без змін у батьківському графі (substitutability).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Зафіксовано у `npm/docs/mt.md`. "3D graph" відхилено — це просторова візуалізація, а не структурна властивість.

## ADR Dynamic Dataflow з рекурсивним уточненням вузлів

## Context and Problem Statement
Визначення семантики ребер та типу декомпозиції у рекурсивному складеному ОАГ: чи несуть ребра дані між вузлами, і чи визначається тип вузла (атомарний / складений) статично чи динамічно.

## Considered Options
* Data-flow edges + динамічна декомпозиція при запуску
* Dependency-only edges (лише порядок, без передачі даних) + статична декомпозиція

## Decision Outcome
Chosen option: "Data-flow edges + динамічна декомпозиція при запуску", because outputs одного вузла є inputs наступного (explicit port mapping), а рішення "атомарний чи складений" приймається агентом при запуску на основі вхідних даних.

### Consequences
* Good, because transcript фіксує очікувану користь: агент може адаптувати стратегію виконання до конкретного набору вхідних даних.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Близькі реалізації: Dask, Prefect dynamic tasks, LangGraph. Зафіксовано у `npm/docs/mt.md`.

## ADR Файлова система як append-only state store

## Context and Problem Statement
Потрібен механізм зберігання стану графу задач, безпечний для паралельного запису з кількох git worktree і придатний для LLM-агентів як основних читачів і авторів.

## Considered Options
* Append-only файли: файли лише створюються, стан вузла визначається наявністю певних файлів
* Mutable JSON files (перезапис `meta.json` при зміні стану)

## Decision Outcome
Chosen option: "Append-only файли з Markdown+YAML фронтматером", because `git merge` між worktree завжди чистий (два worktree ніколи не модифікують один файл), атомарність забезпечується на рівні `create file`, а Markdown природний для LLM-читання та генерації.

### Consequences
* Good, because worktree-межа є межею атомарності: наступні вузли не стартують поки worktree не змерджено, що усуває race condition між агентом і інженером.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Стан вузла кодується наявністю файлів: `місія.md` = spawned, `вихідні.md` = resolved, `помилка.md` = failed, `скасовано` = cancelled. Атрибути фронтматеру — англійська snake_case (`id`, `parent`, `deps`, `created_at`, `occurred_at`). Синтаксис посилань: `ref: path/to/file.md`, `ref: path/to/file.md#section`, `ref: path/to/file.md lines 10-50`. Зафіксовано у `npm/docs/mt.md`.

## ADR LLM-інженер як мета-рівень з необмеженим доступом до графу

## Context and Problem Statement
Потрібен механізм відновлення після помилок у вузлах: хто діагностує, чи може модифікувати граф, і до якого рівня ієрархії має доступ.

## Considered Options
* LLM-інженер як мета-рівень поза графом з повним доступом (може змінювати будь-який рівень включно з root)
* LLM-інженер як вузол у графі (рекурсивна самолагодження)
* LLM-інженер з обмеженим доступом (лише локальний підграф)

## Decision Outcome
Chosen option: "LLM-інженер як мета-рівень поза графом з повним доступом", because батьківські вузли можуть бути першопричиною помилки дочірнього, і інженер повинен мати можливість restructure на будь-якому рівні.

### Consequences
* Good, because transcript фіксує очікувану користь: інженер може аналізувати повний path від root до failed node і втручатись на оптимальному рівні.
* Bad, because cascade інвалідація при патчі кореня може зупинити весь граф.

## More Information
Перед патчем будь-якого вузла: записати `патчі/N-план.md` → kill залежних у топологічному порядку (від листів до цілі) → застосувати патч → записати `патчі/N-факт.md`. Memory інженера зберігається у `repair_history.md` на вузлі де відбулись зміни. Зафіксовано у `npm/docs/mt.md`.

## ADR Time budget замість лічильника спроб для агента-інженера

## Context and Problem Statement
Потрібен convergence guard щоб запобігти нескінченному циклу відновлення. Вибір між обмеженням кількості спроб і обмеженням часу.

## Considered Options
* Time budget (фіксований час, необмежена кількість спроб)
* Max attempts (фіксована кількість спроб, необмежений час)

## Decision Outcome
Chosen option: "Time budget", because штучний ліміт спроб може перервати успішний процес відновлення, тоді як час — реальне обмеження ресурсу; інженер адаптує стратегію залежно від залишку часу.

### Consequences
* Good, because transcript фіксує очікувану користь: при ескалації кожен батьківський рівень отримує свіжий time budget; максимальний час до людини = `глибина × budget` — передбачувано.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`repair_context.json` (або відповідний розділ у `repair_history.md`) містить поля: `deadline` (ISO timestamp), `started_at`, `time_budget_sec`. При вичерпанні бюджету → `type: timeout` у `помилка.md` → ескалація до батька. Root timeout → notify SeniorEngineer через `senior_report.md`. Зафіксовано у `npm/docs/mt.md`.

## ADR Принцип план → дія → факт як universal WAL

## Context and Problem Statement
Потрібна стратегія відновлення системи після збоїв (crash агента або інженера в середині операції) без центрального координатора.

## Considered Options
* Write-ahead plan + fact file для кожної операції (план → дія → факт)
* Ідемпотентні операції без явного WAL

## Decision Outcome
Chosen option: "План → дія → факт через окремі файли", because файлова система append-only, і наявність `*-план.md` без відповідного `*-факт.md` однозначно ідентифікує незавершену операцію для відновлення.

### Consequences
* Good, because transcript фіксує очікувану користь: оркестратор при старті сканує `tasks/**/*-план.md` і відновлює незавершені операції без зовнішнього стану.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Застосовується до всіх операцій: `spawn-план.md`/`spawn-факт.md`, `kill-план.md`/`kill-факт.md`, `патчі/N-план.md`/`патчі/N-факт.md`. Атрибути фронтматеру англійські (`created_at`, `nodes`, `kill_order`, `target_node`, `result`). Зафіксовано у `npm/docs/mt.md`.

## ADR Топологія підграфу розподілена по `місія.md` дочірніх вузлів

## Context and Problem Statement
Потрібно визначити де зберігається топологія підграфу (ребра між дочірніми вузлами) щоб оркестратор міг визначити порядок запуску, а динамічний spawn не потребував оновлення центрального файлу.

## Considered Options
* Топологія в `deps:` кожного `місія.md` дочірнього (розподілена)
* Центральний `граф.md` у батьківському вузлі

## Decision Outcome
Chosen option: "Розподілена топологія в `deps:` кожного `місія.md`", because центральний `граф.md` потрібно оновлювати при кожному динамічному spawn, що порушує append-only принцип; розподілена топологія дозволяє додавати вузли без модифікації існуючих файлів.

### Consequences
* Good, because transcript фіксує очікувану користь: оркестратор реконструює повний граф скануванням `місія.md` файлів; динамічний spawn = лише новий файл без змін у існуючих.
* Bad, because для реконструкції повного стану потрібне сканування всієї файлової ієрархії.

## More Information
Поле `deps:` у `місія.md` містить список `{ node: <id>, port: <port-name> }`. Агент-автор записує `deps:` при spawn. Динамічний spawn дозволений: новий дочірній може посилатись на вже існуючі вузли у `deps:`. `ref:` у `вхідні.md` посилається на конкретні файли або секції попередників без копіювання даних. Зафіксовано у `npm/docs/mt.md`.
