## ADR Рекурсивний складений ОАГ з файловим сховищем стану та мета-інженером

## Context and Problem Statement
Потрібна архітектура для виконання складних задач агентами, де задачі динамічно розкладаються на підзадачі, можуть виконуватись паралельно через git worktree, і система здатна автономно відновлюватись після збоїв шляхом самомодифікації графу.

## Considered Options
* Рекурсивний складений ОАГ з файловим сховищем і мета-рівневим агентом-інженером
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Рекурсивний складений ОАГ з файловим сховищем і мета-рівневим агентом-інженером", because файлова система забезпечує природну персистентність, git worktree дає паралельність без додаткової інфраструктури, а мета-рівневий `EngineerAgent` може модифікувати граф на будь-якому рівні ієрархії.

### Consequences
* Good, because append-only інваріант (файли лише створюються після старту worktree) усуває race conditions і спрощує відновлення після збоїв.
* Good, because `repair_history.md` як append-only журнал дозволяє `EngineerAgent`-у бачити попередні спроби і не повторювати їх.
* Good, because топологія графу розподілена по `task.md` кожного вузла через поле `deps:` — жодного центрального файлу графу, оркестратор відновлює стан скануванням.
* Good, because патерн `plan → action → fact` для всіх операцій (spawn, kill, patch) забезпечує детерміноване відновлення після збою: скрипт знаходить `*-plan.md` без відповідного `*-fact.md` і продовжує виконання.
* Bad, because `EngineerAgent` без обмежень може підніматись до кореневого рівня і інвалідувати весь граф — cascade скасування потребує топологічного kill-order залежних worktree перед будь-яким патчем.
