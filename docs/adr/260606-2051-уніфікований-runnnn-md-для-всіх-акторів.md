## ADR Уніфікований `run_NNN.md` для всіх акторів

## Context and Problem Statement
Система мала окремі файли `error.md`, `repair_history.md`, `repair_context.md` для звичайного агента та інженера. Це плодило різні структури для одного й того самого поняття — "спроба вирішити вузол".

## Considered Options
* Окремі файли на актора: `error.md`, `repair_history.md`, `repair_context.md`
* Єдиний `run_NNN.md` для всіх спроб незалежно від актора

## Decision Outcome
Chosen option: "Єдиний `run_NNN.md`", because і звичайний агент, і інженер — це "спроба вирішити вузол" з тими самими атрибутами (reasoning, result, ref).

### Consequences
* Good, because transcript фіксує очікувану користь: менше різних структур, консистентний формат, `actor: agent | engineer | human | auditor` відрізняє акторів без окремих схем.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Фронтматер: `created_at`, `actor`, `result: success | failed`. Секції: `## Reasoning` (обов'язкова), `## Script` (wrapper, лише при збої), `## Ref` (необов'язкова). Файл `npm/docs/mt.md`, секція `### run_NNN.md`.

## ADR Повністю immutable numbered файли замість append-only

## Context and Problem Statement
Початковий дизайн мав `outputs.md`, `repair_history.md` як append-only — нові секції дописувались до існуючого файлу. Принцип "всі файли під час роботи immutable" і логіка відновлення після збою вимагали перегляду.

## Considered Options
* Append-only файли (`outputs.md`, `repair_history.md`)
* Numbered immutable файли (`outputs_001.md`, `run_001.md`, …)

## Decision Outcome
Chosen option: "Numbered immutable файли", because один файл на одну спробу/результат відповідає загальному принципу immutability після старту і усуває append взагалі.

### Consequences
* Good, because transcript фіксує очікувану користь: жодних append-only файлів, recovery зводиться до сканування `run_*.md` / `outputs_*.md` без порівняння plan/fact.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
NNN генерується wrapper-скриптом: `count(run_*.md) + 1`, zero-padded до 3 цифр (`001`, `002`, …). Файл `npm/docs/mt.md`, секції `### run_NNN.md`, `### outputs_NNN.md`, таблиця `Стан вузла`.

## ADR Видалення `ops/` і `patches/` директорій

## Context and Problem Statement
Директорії `ops/spawn-plan-<ts>.md`, `ops/kill-plan-<ts>.md` і `patches/NNN-plan.md` / `NNN-fact.md` існували для відновлення після збою при spawn/kill/patch. При аналізі з'ясувалось що spawn відбувається у ворктрі, тому обрив не залишає неповного стану в main.

## Considered Options
* Зберегти `ops/` і `patches/` для crash recovery
* Видалити — ігнорувати сценарій обриву при spawn/kill на поточному етапі

## Decision Outcome
Chosen option: "Видалити", because spawn у ворктрі атомарний (незамерджений ворктрі просто не застосовується), тому crash recovery для spawn і kill не потрібен зараз.

### Consequences
* Good, because transcript фіксує очікувану користь: мінімальна файлова структура вузла, жодних зайвих директорій.
* Bad, because якщо в майбутньому з'явиться spawn напряму в main — crash recovery не буде.

## More Information
Reasoning для patches: вся інформація про що інженер змінив вже є в `## Reasoning` секції `run_NNN.md`. Файл `npm/docs/mt.md`, секція `## Файлова структура вузла`.

## ADR Git post-merge hook як тригер оркестратора

## Context and Problem Statement
Після завершення агента в ворктрі потрібно автоматично запустити наступні вузли графу. Були варіанти: ручний запуск людиною, `mt run` як демон, git hook.

## Considered Options
* Людина вручну запускає `mt run` після кожного merge
* `mt run` як живий процес / демон що оркеструє сам
* Git `post-merge` hook що тригерить `mt run --auto`

## Decision Outcome
Chosen option: "Git `post-merge` hook", because кожен компонент робить одне: wrapper запускає агента і мержить ворктрі, git hook тригерить оркестрування, `mt run --auto` сканує і запускає розблокованих наступників.

### Consequences
* Good, because transcript фіксує очікувану користь: `mt run` залишається одноразовою командою; людина тільки запускає кореневий вузол, далі система сама до кінця або до `failed`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Ланцюг: агент завершує → wrapper пише `run_NNN.md` → wrapper `git merge` ворктрі → git `post-merge` hook → `mt run --auto`. Файл `npm/docs/mt.md`, секції `## Wrapper-скрипт`, `Вирішені (Рівень 1)` п.1.

## ADR Агент самостійно тригерить аудит через `mt audit`

## Context and Problem Statement
Потрібен якісний гейт між виконанням агента і станом `resolved`. Питання: хто і коли вмикає аудит — поле `audit: true` у `task.md`, батьківський вузол, або сам агент.

## Considered Options
* Поле `audit: true` у `task.md` — батько встановлює при spawn
* Агент сам вирішує після виконання ("я не впевнений, краще перевір") через `mt audit`
* Аудит для всіх вузлів завжди

## Decision Outcome
Chosen option: "Агент самостійно викликає `mt audit`", because агент краще за все знає чи він впевнений у результаті; поле у `task.md` — зайва семантика.

### Consequences
* Good, because transcript фіксує очікувану користь: аудитор бачить контекст задачі (`task.md`) і всі зміни у ворктрі, дає оцінку виконання і зауваження; при провалі агент отримує зворотній зв'язок.
* Bad, because після 3 поспіль failed циклів аудиту система зупиняється і чекає людину.

## More Information
Аудитор пише `run_(NNN+1).md` з `actor: auditor`. Ліміт: 3 поспіль `result: failed` від аудитора → зупинка. CLI: `mt audit <path>` vs `mt done <path>`. Конфіг: `audit_model`. Файл `npm/docs/mt.md`, секції `## Аудитор`, `mt audit` у CLI контракті.

## ADR `mt watch` як pull-модель сигналізації людині

## Context and Problem Statement
Система повинна повідомляти людину коли вузол потребує втручання (вичерпаний бюджет на рівні кореня, 3 поспіль failed аудити, завислий ворктрі). Розглядалась push-модель через `graph notify`.

## Considered Options
* `graph notify` — push-повідомлення (stdout + `notify_cmd` у конфігу)
* `mt watch` — pull-сканування: скрипт сам знаходить проблемні вузли

## Decision Outcome
Chosen option: "`mt watch`", because pull-підхід простіший: людина або система запускає `watch`, скрипт сканує граф і репортить проблеми без зовнішньої інфраструктури для push.

### Consequences
* Good, because transcript фіксує очікувану користь: запускається автоматично після кожного merge (post-merge hook поруч з `mt run --auto`) і як watchdog при відсутності активності від ворктрі довше `stale_worktree_min` хвилин.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Репортить: ≥ 3 поспіль `actor: auditor, result: failed`; `actor: engineer, result: failed` на кореневому рівні; stale ворктрі (нема змін > `stale_worktree_min` хв). Конфіг: `stale_worktree_min`. Файл `npm/docs/mt.md`, секція `## mt watch`.
