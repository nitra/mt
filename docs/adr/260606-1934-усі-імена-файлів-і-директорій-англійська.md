## ADR Усі імена файлів і директорій — англійська

## Context and Problem Statement
У системі динамічного графа задач файли читаються і LLM-агентами, і bash/Node.js скриптами. Початкові назви (`вхідні.md`, `вихідні.md`, `операції/`, `патчі/`, `підграф/`) були українськими.

## Considered Options
* Залишити українські назви файлів і директорій
* Перейти на англійські назви для всіх файлів і директорій

## Decision Outcome
Chosen option: "Англійські назви", because файли опрацьовуються скриптами — українські назви ускладнюють glob-паттерни, path-matching і portability між системами.

### Consequences
* Good, because transcript фіксує очікувану користь: скрипти не потребують екранування Unicode у шляхах.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Замінено: `вхідні.md` → `inputs.md` (→ merged до `task.md`), `вихідні.md` → `outputs_NNN.md`, `операції/` → `ops/` (→ видалено), `патчі/` → `patches/` (→ видалено), `підграф/` → `subgraph/`. Зафіксовано в контракті `npm/docs/mt.md`: "Усі імена файлів і директорій — англійська".

## ADR `id` вузла читається з шляху, не зберігається у frontmatter

## Context and Problem Statement
Кожен вузол потребує унікального ідентифікатора. Спочатку передбачалось поле `id:` у frontmatter `task.md`.

## Considered Options
* Зберігати `id:` явно у frontmatter
* Читати `id` з назви директорії (directory name = node id)

## Decision Outcome
Chosen option: "Читати з шляху", because назва директорії вже є унікальним ідентифікатором у межах батька; дублювання у frontmatter створює ризик розсинхронізації.

### Consequences
* Good, because transcript фіксує очікувану користь: frontmatter коротший, немає можливості мати `id` що не збігається з реальним шляхом.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Іменування вузлів: kebab-case, унікальні в межах батька. Повний шлях: `tasks/<parent>/subgraph/<node-id>/task.md`. `parent:` у frontmatter — відносний шлях до tasks/, відсутній у кореневого вузла. `deps:` — список `id` сиблінгів (тільки назви, без шляху); скрипт резолвить `../<dep-id>/`.

## ADR Межа append-only інваріанта — момент створення worktree

## Context and Problem Statement
Система будується на принципі "файли тільки створюються, ніколи не змінюються". Питання: чи ця вимога діє від моменту створення файлу, чи тільки з моменту початку обробки вузла?

## Considered Options
* Абсолютна незмінність: файл immutable відразу після запису
* Відносна незмінність: файл вільно редагується до створення worktree; immutable після

## Decision Outcome
Chosen option: "Відносна незмінність (межа = worktree)", because до worktree жоден агент ще не читає файли; зміна не порушує консистентності.

### Consequences
* Good, because transcript фіксує очікувану користь: не потрібно re-spawn для виправлення помилок у `task.md` до старту — достатньо прямого редагування.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Протокол зміни після старту: `kill worktree → редагуй вільно → restart`. Межа чітка і вже існує в системі: `git worktree add ...` = старт роботи. Зафіксовано в `npm/docs/mt.md` розділ "Інваріант незмінності".

## ADR `inputs.md` злитий у `task.md` як секція `## Inputs`

## Context and Problem Statement
Агент при запуску потребував читати два файли: `task.md` (місія) і `inputs.md` (посилання на вхідні дані від батька). Обидва файли immutable, обидва пишуться батьком при spawn.

## Considered Options
* Окремий файл `inputs.md` для посилань на вхідні дані
* Секція `## Inputs` всередині `task.md`

## Decision Outcome
Chosen option: "Секція `## Inputs` у `task.md`", because агент отримує всю необхідну інформацію з одного читання; батько пише один файл замість двох при spawn.

### Consequences
* Good, because transcript фіксує очікувану користь: цілісний контекст в одному файлі, менше церемонії при spawn.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Фінальна схема `task.md` frontmatter: `created_at`, `parent` (відносно `tasks/`, відсутній у кореневого), `deps` (список sibling id), `budget_sec`. Секції: `## Task` (обов'язкова), `## Done when` (обов'язкова), `## Inputs` з підсекціями `### <name>` — `ref:` якщо є файл, inline якщо немає деінде. Усі заголовки секцій що парсить скрипт — англійські; секції з довільними даними — будь-яка мова.

## ADR Видалення `ops/` (spawn-plan/fact, kill-plan/fact)

## Context and Problem Statement
Файли `ops/spawn-plan-<ts>.md` / `spawn-fact-<ts>.md` і аналогічні для kill призначались для відновлення після збою при spawn або kill залежних вузлів — WAL-патерн.

## Considered Options
* Зберегти `ops/` для crash recovery при spawn/kill
* Видалити `ops/` — ігнорувати сценарій обриву spawn/kill

Chosen option: "Видалити `ops/`", because spawn відбувається у ворктрі і при збої весь ворктрі просто не застосовується — граф залишається чистим; окремі WAL-файли не потрібні.

### Consequences
* Good, because transcript фіксує очікувану користь: мінімальна файлова структура без допоміжних директорій.
* Bad, because сценарій обриву spawn-без-ворктрі залишається без механізму відновлення (transcript явно зафіксував: "ігноруємо на зараз цей сценарій").

## ADR Git post-merge hook як тригер наступних вузлів

## Context and Problem Statement
Після завершення вузла і ручного merge ворктрі система має запустити наступні готові вузли (чиї `deps` перейшли у `resolved`). Питання: хто і як тригерить `mt run` на наступників?

## Considered Options
* A: Людина вручну запускає кожен наступний вузол
* B: `mt run` є довготривалим демоном що сам відстежує граф
* C: Git post-merge hook тригерить `mt run --auto`

## Decision Outcome
Chosen option: "C — git post-merge hook", because це "максимально чистий автоматичний варіант" (цитата з transcript): `mt run` залишається одноразовою командою, автоматизація через git-інфраструктуру.

### Consequences
* Good, because transcript фіксує очікувану користь: `mt run` простий і одноразовий; граф рухається автоматично через стандартний git-механізм.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Конкретна реалізація hook не зафіксована в transcript. `mt run --auto` сканує граф, знаходить вузли де всі `deps` → `resolved` і `outputs_*.md` існує, запускає їх з урахуванням ліміту паралельних ворктрі.
