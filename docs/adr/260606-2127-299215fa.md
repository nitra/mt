Розумію. Ось що я чую:

**Поточна проблема**: `flow` змішує планування (spec/plan) і виконання (run/verify/release) в одну лінію без чіткої межі.

**Нова ідея**: два виразних режими під одним `npm/docs/mt.md` контрактом.

---

## Пропоную таке розділення для обговорення

### Stage 1 — Planning (побудова графу)

Людина ставить задачу → AI/людина декомпозує в DAG:

```
mt plan <task-md>    # або mt init --decompose
```

- Input: один рядок або `task.md` з high-level описом
- Output: ієрархія `tasks/<node>/task.md` з deps/budget/done-when
- Може бути інтерактивним (людина править граф) або автономним
- Завершується коли людина каже "граф готовий"

### Stage 2 — Resolution (виконання всередині вузла)

Агент отримує один `task.md` → вирішує атомарно:

```
mt run <node>       # worktree → agent → outputs_NNN.md
```

- Старі `mt verify` / `mt done` стають частиною критерію "Done when" в `task.md`
- Сигнали `done/failed/spawn/audit` замість окремих команд

## Питання для уточнення

1. **Планування** — хто декомпозує граф? AI автоматично, людина вручну через `mt init`, чи разом (AI пропонує → людина редагує)?

2. **`flow` як термін** — він залишається (як назва протоколу), чи повністю замінюється на `mt plan` + `mt run`?

3. **Verify** — залишається окремим кроком в resolution, чи вбудовується в умову `Done when`?
