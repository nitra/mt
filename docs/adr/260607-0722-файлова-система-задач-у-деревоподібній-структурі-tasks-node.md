## ADR Teleport як SSH gateway для доступу розробників до k8s dev pods

## Context and Problem Statement
Розробники `nitra/task` потребують SSH-доступу до dev pods у k8s для редагування файлів задач у Zed/VS Code/Cursor, але не мають прав kubectl і не повинні мати прямого доступу до кластера. Авторизація має визначатися бекендом, а не k8s RBAC.

## Considered Options
* `kubectl port-forward` + SSH у поді
* Teleport — identity-aware SSH proxy з RBAC через labels
* Tailscale / WireGuard у поді
* Власний SSH gateway на бекенді
* Інші варіанти в transcript не обговорювалися повністю, але `kubectl port-forward` відкинуто явно.

## Decision Outcome
Chosen option: "Teleport — identity-aware SSH proxy з RBAC через labels", because `kubectl port-forward` відхилено явно (розробники без kubectl); Teleport дозволяє бекенду контролювати доступ через label `owner: email` без видачі розробнику будь-яких k8s credentials; короткоживучі сертифікати (TTL 8–24 год) усувають управління ключами.

### Consequences
* Good, because transcript фіксує очікувану користь: бекенд реєструє dev pod з label `owner: email` → розробник автоматично бачить тільки свої поди без ручної видачі прав; Zed/VS Code/Cursor підключаються через стандартний SSH (`~/.ssh/config` + `ProxyCommand tsh proxy ssh`) без патчів.
* Bad, because Teleport Auth+Proxy потребує окремого Deployment у кластері (один компонент для всіх клієнтів); transcript позначає `replicas: 1` з коментарем про SQLite як обмеження для HA.

## More Information
Маніфести: `k8s/teleport/configmap.yaml`, `deployment.yaml`, `service.yaml`, `ingress.yaml`, `pvc.yaml`, `rbac.yaml`, `roles.yaml`, `k8s/dev-pod/template.yaml`, `k8s/dev-pod/rbac.yaml`, `k8s/README.md` — у `/Users/vitaliytv/www/nitra/task/`. ## ADR "Open in Editor" — multi-editor deep link через URI scheme

## Context and Problem Statement
UI `nitra/task` планує кнопку для відкриття task-node у редакторі. Початковий scope — лише Zed; потім постало питання про VS Code і Cursor.

## Considered Options
* Підтримка тільки Zed (hostname copy + інструкція)
* Multi-editor: VS Code і Cursor через `vscode://` / `cursor://` URI deep link; Zed через ручне копіювання hostname

## Decision Outcome
Chosen option: "Multi-editor: VS Code, Cursor через URI deep link, Zed через hostname copy", because VS Code і Cursor підтримують `vscode-remote/ssh-remote+` URI що відкриває сесію одним кліком; Zed на момент сесії URI-протоколу не підтримує, тому для нього — fallback на копіювання. Задача перейменована з `open-in-zed` на `open-in-editor`.

### Consequences
* Good, because transcript фіксує очікувану користь: для VS Code і Cursor — zero-step підключення через `vscode://vscode-remote/ssh-remote+<host>/tasks` та `cursor://vscode-remote/ssh-remote+<host>/tasks`; всі три редактори використовують однаковий Teleport `~/.ssh/config` під капотом.
* Bad, because Zed не має URI deep link — вимагає додаткового UX (копіювання + інструкція) на відміну від одноклікового досвіду у VS Code/Cursor.

## More Information
Задача: `tasks/open-in-editor/task.md` у `/Users/vitaliytv/www/nitra/task/tasks/`. URI format: `vscode://vscode-remote/ssh-remote+<node>.teleport.nitra.com/tasks` (VS Code), `cursor://vscode-remote/ssh-remote+<node>.teleport.nitra.com/tasks` (Cursor). Попередня назва задачі: `open-in-zed`.

## ADR Dev pod спавниться по запиту через бекенд `nitra/task`

Dev pods для SSH-доступу розробників не мають бути попередньо створені — це витрачає ресурси кластера і ускладнює lifecycle management.

## Considered Options
* Pre-provisioned dev pods (один pod на задачу постійно)
* On-demand spawn через бекенд `nitra/task` API
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "On-demand spawn через бекенд `nitra/task` API", because розробник натискає "Open in Editor" → бекенд перевіряє права → `kubectl apply dev-pod.yaml` → чекає `Pod Ready` (~5–15с) → Teleport node-agent реєструється → повертає connection string. Паттерн аналогічний GitPod/Codespaces але спеціалізований для task-nodes DAG.

* Good, because transcript фіксує очікувану користь: ресурси кластера витрачаються лише під час активної сесії; lifecycle table у transcript описує auto-delete після timeout без активності.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Lifecycle: spawn при відкритті сесії → grace period після закриття SSH → auto-delete після timeout → pod видаляється при переході task-node у `resolved`.
