## ADR Файлова структура вузлів задач у `tasks/`

## Context and Problem Statement
Система `mt` потребує файлового сховища стану для вузлів рекурсивного DAG (описаного у `npm/docs/mt.md`). Не існувало жодної директорії `tasks/` — потрібно визначити схему файлів для нових вузлів, щоб CLI-команди (`mt run`, `mt scan`) та UI могли однозначно читати стан.

## Considered Options
* Файл `tasks/<name>/task.md` з YAML-frontmatter (`created_at`, `budget_sec`) та секціями `## Task`, `## Done when`, `## Inputs`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Файл `tasks/<name>/task.md` з YAML-frontmatter", because схема взята безпосередньо з `npm/docs/mt.md` де описано, що вузол у стані `waiting` — це директорія лише з `task.md` без `run_*.md` і `outputs_*.md`.

### Consequences
* Good, because transcript фіксує очікувану користь: CLI-команда `mt run tasks/<name>` розуміє стан без жодної додаткової конфігурації.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Створені вузли: `tasks/ui-task-view/task.md`, `tasks/coverage-skill-test/task.md`, `tasks/skills-orchestrator-migration/task.md`. Запуск: `mt run tasks/<name>`. Схема frontmatter: `created_at` (ISO 8601), `budget_sec` (integer).

## ADR Teleport як SSH-шлюз для доступу розробників до k8s pod-ів

## Context and Problem Statement
Розробники без прав `kubectl` потребують SSH-доступу до dev-pod-ів у k8s (де монтується `tasks-pvc`). Доступ має контролювати бекенд `nitra/task`, а не k8s RBAC безпосередньо — щоб гарантувати per-task авторизацію незалежно від кластерних прав розробника.

## Considered Options
* `kubectl port-forward` + SSH у поді
* Teleport (Auth Server + Proxy + node-agent sidecar)
* Власний SSH-gateway з кастомною auth-логікою
* Tailscale / WireGuard у поді

## Decision Outcome
Chosen option: "Teleport", because `kubectl port-forward` явно відкинуто (розробники без kubectl), а Teleport вирішує задачу бекенд-контрольованого доступу через RBAC із label-ами на pod-ах та короткоживучими SSH-сертифікатами без статичних ключів.

### Consequences
* Good, because transcript фіксує очікувану користь: Zed підключається через стандартний SSH (`~/.ssh/config` + `ProxyCommand tsh proxy ssh`), аудит-лог і SSO з коробки, розробник ніколи не бачить k8s.
* Good, because RBAC через label `owner: "{{internal.logins}}"` дає per-task контроль без ручної видачі прав — бекенд просто проставляє label при створенні pod.
* Bad, because потрібно задеплоїти Teleport Auth + Proxy (один раз, через Helm chart), що додає операційну складність.

## More Information
SSH-конфіг для Zed: `Host *.teleport.nitra.com` + `ProxyCommand tsh proxy ssh --cluster=nitra %h:%p`. Node label-и: `task`, `owner`, `project`. Role-правило: `node_labels.owner: "{{internal.logins}}"`. Компоненти: Teleport Auth+Proxy (k8s Deployment), node-agent (sidecar у dev pod), Teleport Operator (k8s CRD для декларативного керування).

## ADR On-demand spawning dev pod-ів через бекенд `nitra/task`

## Context and Problem Statement
Dev pod-и, до яких підключається розробник через Zed + Teleport, не можуть бути pre-created статично — бо прив'язані до конкретного task-вузла, монтують той самий `tasks-pvc`, і їхній lifecycle залежить від стану задачі (waiting/running/resolved).

## Considered Options
* On-demand spawn через бекенд `nitra/task` (бекенд робить `kubectl apply` за запитом розробника)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "On-demand spawn через бекенд `nitra/task`", because розробник запитує доступ через UI, бекенд перевіряє права у власній БД, створює pod із потрібними label-ами, чекає `Ready`, після чого Teleport node-agent реєструється автоматично і розробник отримує connection string.

### Consequences
* Good, because transcript фіксує очікувану користь: розробник у Zed бачить актуальні `run_NNN.md`, `outputs_NNN.md`, може патчити `task.md` з повним контекстом виконання.
* Good, because lifecycle (grace period після закриття SSH, auto-delete по timeout або при переході task-вузла у `resolved`) прибирає idle-ресурси автоматично.
* Bad, because Neutral, because transcript не містить підтвердження наслідку щодо часу cold-start (~5–15 с на spawn pod + реєстрація Teleport).

## More Information
Потік: UI-запит → перевірка прав (власний RBAC бекенду) → `kubectl apply dev-pod.yaml` (labels: `task=<name>`, `owner=<email>`) → `pod Ready` → Teleport реєстрація → connection string `<name>.teleport.nitra.com`. Pod монтує `tasks-pvc`. Паттерн аналогічний GitPod/Codespaces, але pod є точкою доступу до DAG-стану агентів, а не загальним dev-environment.
