## ADR Переосмислення `flow` під архітектуру `npm/docs/mt.md`

## Context and Problem Statement
Існуючий `mt` — монолітний протокол із 5-фазним циклом (`init/spec/plan/verify/release`) і MT file-presence state як джерелом стану. При інтеграції з автономним ОАГ із `npm/docs/mt.md` (файлова-система-як-стан, `tasks/<node>/task.md`) виникає суперечність: flow має внутрішній стан, граф — зовнішній. Потрібно переосмислити `flow` як протокол всередині вузла графу.

## Considered Options
* Залишити flow незмінним, запускати його всередині `mt run` як black-box
* Повністю переосмислити flow у два окремих етапи: planning + execution

## Decision Outcome
Chosen option: "Повністю переосмислити flow у два окремих етапи", because `graph` вже відповідає за worktree-lifecycle ззовні, а `flow` повинен обслуговувати лише те, що відбувається всередині одного вузла.

### Consequences
* Good, because transcript фіксує очікувану користь: MT file-presence state, `docs/specs/`, `docs/plans/` зникають, стан вузла повністю деривується з файлів у `tasks/<node>/`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Рішення зафіксовано в `npm/docs/mt.md` розділ "Інтеграція з `mt`". Нова команда: `npm/scripts/dispatcher/index.mjs`. Файли, що зникають: MT file-presence state, `docs/specs/`, `docs/plans/`.

## ADR Явний виклик `mt spawn` після `mt plan` (Variant B)

## Context and Problem Statement
Після того як `mt plan` у складеному шляху створив дочірні `task.md`, потрібно сигналізувати оркестратору про нові вузли. Питання: хто саме викликає `mt spawn` — `mt plan` автоматично, чи агент вручну.

## Considered Options
* Variant A — bundled: `mt plan` сам викликає `mt spawn` в кінці
* Variant B — explicit: `mt plan` тільки створює дочірні `task.md`, агент викликає `mt spawn` явно

## Decision Outcome
Chosen option: "Variant B — explicit", because transcript: "Варіант B — потрібен скрипт який буде проходити весь граф і визначати де потрібно запустити і згідно черги буде вирішувати де запустити."

### Consequences
* Good, because transcript фіксує очікувану користь: `mt spawn` потрапляє в чергу оркестратора, а не запускається негайно — це дозволяє `mt watch` керувати порядком і обмеженнями паралелізму.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Оркестратор черги реалізується через post-merge hook (`mt run --auto`) і демон `mt watch`. Обидва скануть граф і приймають рішення про наступний вузол для запуску.

## ADR Схема нумерації `pending-audit_NNN.md` — NNN дзеркалює `outputs_NNN.md`

## Context and Problem Statement
Після завершення роботи агент запитує аудит. Аудит ставиться у чергу (файловий sentinel). Оркестратор має знати яку саме версію `outputs_NNN.md` перевіряти, якщо між запитом і виконанням аудиту агент міг написати нові версії.

## Considered Options
* Порожній sentinel-файл `.pending-audit` (overwrite при новому запиті)
* Numbered файли `pending-audit_NNN.md`, де NNN = NNN відповідного `outputs_NNN.md`
* Lifecycle: `.pending-audit` → `.running-audit` → видалення

## Decision Outcome
Chosen option: "Numbered файли `pending-audit_NNN.md`, де NNN = NNN відповідного `outputs_NNN.md`", because "якщо агент завершив на `outputs_003.md` і саме цю версію він вважає єдиною доцільною аудиту, то створюється `pending-audit_003.md` і так далі — щоб нумерація не губилася на всіх етапах."

### Consequences
* Good, because transcript фіксує очікувану користь: ім'я файлу саме по собі є посиланням на версію (`ref:` у front-matter вже не потрібен), аудитор завжди знає яку версію перевіряти.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Схема нумерації по вузлу: `outputs_001.md` → `pending-audit_001.md` → `run_002.md (actor: auditor)`. `run_NNN.md` — незалежний лічильник (всі актори). `outputs_NNN.md` і `pending-audit_NNN.md` — спільний ключ.

## ADR Видалення Фасаду B (`mt run/resume/cancel/repair`)

* Залишити Фасад B для зворотної сумісності
* Видалити Фасад B повністю, передати відповідальність `graph`-шару

Chosen option: "Видалити Фасад B повністю", because `mt run` замінюється `mt run` → агент → `mt plan` → `mt verify`; `mt run` не потрібен (стан у файлах, повторний `mt run` читає наявні `run_NNN.md`); `mt kill` = `mt kill`; `mt invalidate` не потрібен (MT file-presence state зникає).

### Consequences
* Good, because transcript фіксує очікувану користь: менше команд, єдина точка входу для lifecycle вузла через `graph`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Видаляються: `mt run`, `mt run`, `mt kill`, `mt invalidate`, `mt audit`, `mt verify`. Залишаються: `mt plan` (Stage 1), `mt verify` (Stage 2). Додаються сигнали: `mt done`, `mt audit`, `mt failed`, `mt spawn`.
