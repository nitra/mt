## ADR Рекурсивний складений ОАГ з динамічним розкладом вузлів

## Context and Problem Statement
Потрібно описати структуру системи де задачі керуються оркестратором, розбиваються на динамічні workflow, формують граф задач, і кожен вузол цього графу може сам розбиватись на підграф — з можливістю самовідновлення, паралельного виконання та моніторингу стану в будь-який момент.

## Considered Options
* Recursive Compound DAG (Рекурсивний складений ОАГ)
* "3D граф"
* Hierarchical Task Network (HTN)

## Decision Outcome
Chosen option: "Рекурсивний складений ОАГ", because вузол або є атомарним (`fn(вхідні) → вихідні`), або розкладається на підграф — рішення приймається динамічно тільки на старті агента; батьківський рівень бачить лише `стан = вирішено` незалежно від внутрішньої складності. "3D граф" не описує структуру зв'язків, HTN не покриває file-based state та self-modifying поведінку.

### Consequences
* Good, because transcript фіксує очікувану користь: інкапсуляція чорної скриньки дозволяє замінювати атомарний вузол складеним без змін у батьківському графі.
* Good, because файловий стан (append-only Markdown + YAML) є одночасно LLM-friendly форматом і безкоштовним audit trail через git history.
* Bad, because transcript не містить підтверджених негативних наслідків — медіатор при конфлікті злиття може сам помилитись (SWOT зафіксовано як слабку сторону).

## More Information
- Файл дизайну: `npm/docs/mt.md`
- Стан вузла визначається наявністю файлів: `місія.md`+`вхідні.md` → очікує; активний ворктрі → виконується; `вихідні.md` → вирішено; `помилка.md` → помилка; sentinel `знедійснений` → інвалідований
- Ребра зберігаються в `граф.md` батька (авторитет для оркестратора) і в полі `залежності:` в `місія.md` дочірнього (локальний контекст)
- Динамічне розширення через `граф-розш-001.md` (append-only, не модифікація `граф.md`)
- Протокол відновлення: `scan tasks/**/*-план.md` — якщо відповідний `*-факт.md` відсутній → незавершена операція → відновити
- Агент-інженер — мета-рівень поза графом, time budget (не лічильник спроб), кожен патч у своєму ворктрі
- Перед патчем вже вирішеного вузла: спочатку кілл залежних у топологічному порядку, потім патч
- Всі агенти запускаються через скрипт-обгортку (детекція таймауту + кілл процесу)

## ADR Append-only файлова структура як state store

## Context and Problem Statement
Потрібно вирішити проблему атомарності запису стану вузла: агент міг записати `вихідні.md` але не викликати CLI хук — граф у невалідному стані. Також потрібно уникнути конфліктів при паралельному виконанні агентів у різних git-ворктрі.

## Considered Options
* Append-only файли (тільки `create` + `unlink`, без `modify`)
* Mutable JSON файли зі станом (`state` поле в `meta.json`)
* Централізована база даних

## Decision Outcome
Chosen option: "Append-only файли", because стан вузла визначається наявністю файлів, а не їх вмістом — операція `create file` атомарна на більшості файлових систем; окремий CLI хук для зміни стану не потрібен.

### Consequences
* Good, because transcript фіксує очікувану користь: два ворктрі ніколи не конфліктують при злитті — вони створюють лише нові файли з унікальними іменами в різних директоріях.
* Good, because `repair_history.md` як append-only журнал дозволяє агент-інженеру читати весь контекст і продовжувати природно, без парсингу JSON-масивів.
* Bad, because transcript не містить підтверджених негативних наслідків — scan для відновлення стану (пошук `*-план.md` без `*-факт.md`) може бути дорогим при великих графах (зафіксовано в SWOT).

## More Information
- Файл дизайну: `npm/docs/mt.md`, розділ "Формати файлів"
- `вхідні.md`/`вихідні.md` не дублюють дані — лише посилання на файли або конкретні рядки
- Динамічне розширення підграфу через нові файли `граф-розш-001.md`, не модифікацію `граф.md`
- Патчі інженера: нові файли `патчі/001-план.md` + `патчі/001-факт.md` на кожну спробу
- Ворктрі-межа є межею атомарності: наступник стартує лише після merge → race condition неможлива за архітектурою
