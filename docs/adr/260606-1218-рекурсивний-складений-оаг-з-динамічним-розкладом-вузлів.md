## ADR Рекурсивний складений ОАГ з динамічним розкладом вузлів

## Context and Problem Statement
Потрібно описати структуру системи в якій директорії містять проєкти, всередині яких оркестратор розбиває задачі на dynamic workflow, формуючи граф задач. Вузли графу можуть самі рекурсивно розкладатись на підграфи. Вузли також існують на кореневому рівні директорії. Поставало питання — як коректно назвати і формалізувати цю структуру.

## Considered Options
* Recursive Compound DAG (рекурсивний складений ОАГ)
* Hierarchical Task Network (HTN)
* 3D Graph

## Decision Outcome
Chosen option: "Recursive Compound DAG (рекурсивний складений ОАГ)", because батьківський вузол бачить лише інтерфейс (вирішено/не вирішено), не знаючи чи всередині атомарна задача чи повноцінний підграф; DAG (орієнтований ациклічний граф) точно описує напрямок і відсутність циклів; HTN має специфічний AI-planning контекст, а 3D Graph описує просторовість а не структуру зв'язків.

### Consequences
* Good, because transcript фіксує очікувану користь: substitutability — атомарний вузол можна «розкрити» в підграф без змін у батьківському графі; рекурсивний executor однаковий на всіх рівнях.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл з дизайном: `npm/docs/mt.md`. Близькі реалізації згадані в transcript: Dask, Prefect dynamic tasks, LangGraph.

## ADR Файлова система як state store (append-only, immutable files)

## Context and Problem Statement
Потрібно визначити де і як зберігати стан графу. Обговорювались питання атомарності запису, конфліктів при паралельному виконанні у ворктрі, та зручності для LLM-агентів.

## Considered Options
* Append-only файлова система (файли лише створюються, видаляються атомарно, ніколи не змінюються)
* Централізований state store (оркестратор зберігає граф)
* Mutable файли (кожен вузол перезаписує власні файли)

## Decision Outcome
Chosen option: "Append-only файлова система", because кожен вузол пише лише свої файли; `create file` — атомарна операція; наявність файлу = сигнал про стан без окремого CLI-хуку; конфлікти при merge ворктрі неможливі за визначенням (два ворктрі створюють лише нові файли з унікальними іменами).

### Consequences
* Good, because transcript фіксує очікувану користь: атомарність через файлову систему; природний аудит-трейл; ворктрі-межа = межа атомарності, наступники не стартують поки ворктрі не змерджено.
* Bad, because transcript зазначає: scan для відновлення стану без індексу при великих графах — дорого; зростання займаного місця на диску без GC.

## More Information
`npm/docs/mt.md`, розділи «Формати файлів», «Паралельне виконання». Стан вузла визначається наявністю файлів: `місія.md + вхідні.md` → spawned; `вихідні.md` → resolved; `помилка.md` → failed; `скасовано` → cancelled.

## Considered Options
* Git worktrees (кожен вузол у власному ізольованому ворктрі, злиття через git merge)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Git worktrees", because наступники не стартують поки ворктрі не змерджено; merge = єдина точка рішення оркестратора; при merge наявність конфліктів у спільних батьківських файлах вирішується агентом-медіатором.
