## ADR Append-only файловий стан із sentinel-файлами

## Context and Problem Statement
При модифікації файлів на місці виникає проблема атомарності запису: агент міг написати `вихідні.md` але не встигнути оновити стан вузла до збою.

## Considered Options
* Append-only: файли лише створюються, видаляються атомарно
* Mutable files з atomic rename
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Append-only: файли лише створюються", because `create file` є атомарною операцією на більшості ФС; поява файлу є одночасно і записом стану і сигналом для оркестратора; git merge двох worktree не матиме конфліктів, бо кожен worktree лише додає нові файли з унікальними іменами.

### Consequences
* Good, because transcript фіксує очікувану користь: merge worktree завжди чистий; окремий CLI-хук для зміни стану не потрібен.
* Bad, because диск зростає без garbage collection; поточний стан вузла вимагає перевірки набору файлів (`вихідні.md` є + `скасовано` нема → resolved).

## More Information
Стан вузла кодується присутністю файлів: `місія.md` = spawned; `виконується` (sentinel) = running; `вихідні.md` = resolved; `помилка.md` = failed; `скасовано` = cancelled (пріоритет над `вихідні.md`); `знедійснений` = invalidated. Патчі: `патчі/001-план.md`, `патчі/001-факт.md` (append-only послідовність).

## ADR Принцип план → дія → факт для всіх операцій

## Context and Problem Statement
При збої між двома операціями (наприклад kill залежних → patch цільового вузла) система не знала, що відновлювати.

## Considered Options
* Write-ahead log через файли (план → дія → факт)
* Транзакційна БД
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Write-ahead log через файли", because наявність `*-план.md` без відповідного `*-факт.md` однозначно вказує на незавершену операцію; оркестратор при старті сканує `tasks/**/операції/*-план.md` і відновлює незавершені.

### Consequences
* Good, because transcript фіксує: відновлення після збою = порівняти план з фактом без зовнішнього координатора.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Застосовується до всіх операцій: spawn (`spawn-план.md` / `spawn-факт.md`), kill (`kill-план.md` / `kill-факт.md`), patch (`патчі/001-план.md` / `патчі/001-факт.md`). Git worktree як межа атомарності: successors не стартують поки worktree не merged, тому race condition між записом `вихідні.md` і скасуванням архітектурно виключений.

## ADR Топологія графу розподілена по `місія.md` дочірніх вузлів

## Context and Problem Statement

## Considered Options
* Центральний `граф.md` у батьківському вузлі
* Кожен дочірній вузол декларує своїх попередників у власній `місія.md`

## Decision Outcome
Chosen option: "Кожен дочірній вузол декларує своїх попередників у власній `місія.md`", because підхід universally consistent — кожен вузол самодостатній; динамічний spawn = просто новий `місія.md` з `deps:` без оновлення жодного центрального файлу; оркестратор відновлює повний DAG скануванням.

## ADR Часовий бюджет замість лічильника спроб для агента-інженера

## Context and Problem Statement
Потрібен механізм convergence guard щоб уникнути нескінченного циклу відновлення вузла.

## Considered Options
* Ліміт кількості спроб (`max_attempts: N`)
* Часовий бюджет (`repair_deadline: timestamp`, необмежена кількість спроб)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Часовий бюджет", because реальне обмеження — час (compute, гроші, wall clock), а не кількість спроб; агент може адаптувати стратегію: складна спроба поки багато часу, швидкий fix коли мало.

### Consequences
* Good, because максимальний час до ескалації до людини передбачуваний: `depth × budget`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`repair_context.json` містить `deadline`, `started_at`, `time_budget_sec`. Кожен рівень ієрархії при ескалації отримує **свіжий** бюджет (не залишок). Root-рівень timeout → `senior_report.json` → SeniorEngineer. Агент-інженер є мета-рівнем поза графом, може змінювати граф на будь-якому рівні ієрархії включно з кореневим. Memory інженера зберігається як `repair_history.md` на **вузлі** (не на агенті) — append-only журнал спроб.

## ADR Обов'язковий kill залежних перед патчем вузла

## Context and Problem Statement
Інженер може вирішити що вже `resolved` вузол дав неправильні виходи і потребує виправлення, але залежні вузли можуть вже виконуватись у своїх worktree з даними від нього.

## Considered Options
* Дозволити паралельний патч і invalidation cascade
* Спочатку kill всіх залежних worktree (топологічний порядок від листів), потім патч
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Спочатку kill всіх залежних, потім патч", because дозволяє уникнути ситуації де залежний завершується з застарілими вхідними даними після патчу.

### Consequences
* Good, because transcript фіксує: операція атомарно фіксується через `kill-план.md` перед виконанням, збій між кроками відновлюється за принципом план → факт.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Протокол: 1) записати `патчі/001-план.md` у цільовий вузол; 2) записати `операції/kill-план.md`; 3) kill залежних від листів до цілі; 4) записати `операції/kill-факт.md`; 5) застосувати патч; 6) записати `патчі/001-факт.md`; 7) рестарт каскаду. Конвенція merge worktree: successors не стартують поки worktree не merged — тому race condition між kill і spawn наступника архітектурно виключений.
