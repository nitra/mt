## ADR Переосмислення `flow` як протоколу всередині вузла графу

## Context and Problem Statement
Існуючий `mt` діяв як самостійна система (ізолює → планує → верифікує → релізить). Паралельно розроблялась архітектура автономного DAG-оркестратора (`npm/docs/mt.md`). Обидва використовують `.worktrees/`, але з несумісними конвенціями і дублюючими ролями. Потрібно об'єднати системи без втрати наявної функціональності.

## Considered Options
* Зробити `flow` протоколом всередині вузла, а `graph` — зовнішнім оркестратором
* Залишити `flow` і `graph` повністю незалежними системами
* Повністю злити обидві системи в єдину

## Decision Outcome
Chosen option: "Зробити `flow` протоколом всередині вузла", because `graph` керує DAG і worktree lifecycle ззовні, а `flow` обслуговує один запуск зсередини — це зберігає чіткий поділ відповідальностей і дозволяє кожному вузлу мати свій ізольований execution protocol.

### Consequences
* Good, because transcript фіксує очікувану користь: чіткий поділ `graph` (зовнішній шар) / `flow` (внутрішній протокол) без дублювання worktree логіки.
* Bad, because `mt init`, `mt init`, `mt done`, MT file-presence state, `docs/specs/`, `docs/plans/` зникають — це breaking change для існуючих користувачів flow.

## More Information
Зміни: `npm/docs/mt.md` (новий розділ «Інтеграція з `mt`»). Файли: `npm/scripts/dispatcher/index.mjs` (буде змінено), `npm/scripts/dispatcher/lib/` (буде переглянуто).

## ADR Двоетапне виконання вузла: Stage 1 Planning + Stage 2 Execution

## Context and Problem Statement
Вузол DAG може бути атомарним (виконується напряму) або складеним (потребує декомпозиції в дочірній підграф). Потрібний єдиний протокол, який дозволяє агенту спочатку визначити природу вузла, а потім або виконати роботу, або розкласти її.

## Considered Options
* Два окремі кроки: `mt init` (design) → `mt plan` (decompose)
* Один спільний крок: `mt plan` охоплює і design, і decompose
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Один спільний крок `mt plan`", because користувач явно обрав об'єднання: «спільний крок поєднуємо їх».

### Consequences
* Good, because transcript фіксує очікувану користь: менше команд, менше стикувальних станів між spec і plan.
* Bad, because `mt plan` стає більш складним (один крок робить і дизайн, і декомпозицію), що може ускладнити тестування.

## More Information
Stage 1 output для атомарного шляху: `plan_001.md` (numbered, immutable). Stage 1 output для composite шляху: дочірні `tasks/<child>/task.md` файли (агент потім явно викликає `mt spawn`). Атрибут `mode: human | agent` у front-matter `task.md` контролює режим Stage 1; `human` — default.

## ADR `mode` атрибут у `task.md` для контролю інтерактивності Stage 1

## Context and Problem Statement
Stage 1 (`mt plan`) може виконуватись або в інтерактивному діалозі з людиною, або повністю автономно агентом. Потрібен механізм, який декларативно задає режим на рівні окремого вузла, а не глобально.

## Considered Options
* Атрибут `mode` у front-matter `task.md` (`human` | `agent`)
* Прапорець у команді `mt run --actor`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Атрибут `mode` у front-matter `task.md`", because рішення закріплено у специфікації вузла, а не в командному рядку — так воно не залежить від того, хто або що запускає `mt run`.

### Consequences
* Good, because transcript фіксує очікувану користь: default `mode: human` — безпечний за замовчуванням для всіх вузлів.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Default: `mode: human`. Значення: `human` (інтерактивний діалог), `agent` (повністю автономно). Поле задається у YAML front-matter `task.md` поряд з `budget_sec` і `deps`.

## ADR Явний виклик `mt spawn` агентом після `mt plan` (Composite шлях)

## Context and Problem Statement
Після того як `mt plan` у composite-режимі створив дочірні `task.md`, потрібно вирішити хто і коли сигналізує оркестратору про появу нових вузлів.

## Considered Options
* **Bundled (A):** `mt plan` сам викликає `mt spawn` в кінці
* **Explicit (B):** `mt plan` тільки створює дочірні `task.md`; агент потім викликає `mt spawn` вручну

## Decision Outcome
Chosen option: "Explicit (B)", because користувач явно обрав «Варіант B» — агент контролює момент сигналізації явно, що дає більше гнучкості (наприклад, агент може перевірити дочірні файли перед spawn).

### Consequences
* Good, because transcript фіксує очікувану користь: агент має явний контроль над переходом composite → spawn без прихованої автоматики в `mt plan`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Команда: `mt spawn <path>` (існуючий сигнал у `signals.mjs`). `mt plan` лише пише файли до `tasks/<child>/task.md`; wrapper-process отримує сигнал через `.ncursor-signal` після явного `mt spawn`.

## ADR Оркестратор графу: і post-merge hook, і `mt watch` демон

## Context and Problem Statement
Після завершення вузла (merge) або появи нових готових вузлів (spawn) система має автоматично визначати наступні вузли для запуску і управляти чергою з урахуванням ліміту `max_worktrees`.

## Considered Options
* Тільки реактивний post-merge hook
* Тільки постійний демон (`mt watch`)
* Обидва механізми паралельно

## Decision Outcome
Chosen option: "Обидва механізми паралельно", because користувач підтвердив: «і хук і демон». Hook дає гарантований тригер після merge; демон покриває edge-cases (spawn без merge, stale worktrees, audit) і є вже згаданим у `npm/docs/mt.md`.

### Consequences
* Good, because transcript фіксує очікувану користь: жоден тригер не пропущений — hook реагує на merge, watch покриває решту подій.
* Bad, because два оркестратори потребують координації щоб уникнути race condition при одночасному запуску.

## More Information
Hook: `.git/hooks/post-merge` → `mt run --auto` (one-shot scan → find ready → spawn up to `max_worktrees`). Daemon: `mt watch` — постійний процес, черга, watchdog для stale worktrees. Конфіг: `.n-cursor.json` (поле `max_worktrees`, `stale_worktree_min`). Зміни зафіксовані у `npm/docs/mt.md` у розділі «Інтеграція з `mt`».
