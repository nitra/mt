## ADR Вибір локальної моделі Tier 1 для docgen на 8 GB M2

## Context and Problem Statement
Потрібно обрати локальну LLM для docgen Tier 1 на машині з 8 GB RAM. Попередні виміри gemma4:e4b (9.6 GB) показали 0.4 tok/s через memory-thrashing під одночасним навантаженням, тому потрібна альтернатива.

## Considered Options
* `gemma3:4b` (3.3 GB, 100% GPU, ~20 tok/s)
* `gemma4:4b` / `batiai/gemma4-e4b:q4` (5.3 GB, 56%/44% CPU/GPU, ~11 tok/s)
* `qwen2.5-coder:3b` (виключено після попереднього benchmark-2: ~77%, витоки сигнатур/stdlib, мовні помилки)
* `gemma4:e4b` full (9.6 GB — memory-thrashing, 0.4 tok/s, непридатно)

## Decision Outcome
Chosen option: "обидві — `gemma3:4b` як швидкість-first, `gemma4:4b` як якість-first", because benchmark-2 і benchmark-3 (прямий `/api/chat` + system-prompt, 3 abie-файли проти еталона) показали: gemma3 ~85%/20 tok/s/3.3 GB (повністю в GPU), gemma4:4b q4 ~92%/11 tok/s/6.2 GB (частковий офлоад). Q4-перепак gemma4 вписується у 8 GB без катастрофічного свопу, тоді як full (9.6 GB) — ні.

### Consequences
* Good, because gemma3:4b дає повністю GPU-генерацію (~20 tok/s), екстраполяція на 1042 файли ≈14.5 год.
* Good, because gemma4:4b q4 підіймає якість на +7 п.п. відносно gemma3 one-shot, лишаючись в межах 8 GB (6.2 GB RAM, 56%/44% CPU/GPU).
* Bad, because gemma4:4b q4 ~вдвічі повільніша (11 vs 20 tok/s) і частково офлоадить у CPU, що чутливо до паралельного навантаження на 8 GB.

## More Information
Модель `batiai/gemma4-e4b:q4` скопійована як `gemma4:4b` через `ollama cp` (спільний blob `d682bf87e3a3`, 0 додаткового місця). Вибір між моделями залежить від пріоритету: чорновий масовий прогін → `gemma3:4b`; людино-орієнтована якість-first → `gemma4:4b`. Еталони: 3 файли `npm/rules/abie/` (`firebase_hosting.mjs`, `overlay-paths.mjs`, `k8s-tree.mjs`).

## ADR System-prompt як визначальний чинник якості docgen

## Context and Problem Statement
Порівняльний експеримент (A: прямий ollama без system, B: через pi, C: прямий + system) виявив великий розрив між A (~71%) і B/C (~85–87%). Потрібно було визначити справжню причину різниці, щоб не прив'язуватись до pi як інструмента.

## Considered Options
* Вважати pi кращим за природою (архітектурна перевага транспорту)
* Гіпотеза: різниця зумовлена наявністю system-prompt, а не самим pi

## Decision Outcome
Chosen option: "system-prompt — головний важіль якості, не транспорт", because контрольований прогін C (прямий `/api/chat` + system-prompt) наздогнав pi (85% vs 87%) — різниця 2 п.п. потрапляє в шум ручного оцінювання ±3 п.п. і пояснюється якістю system-prompt (відшліфований у pi vs чорновий у C).

### Consequences
* Good, because прямий `/api/chat` + system-prompt дає ту саму якість без pi-оверхеду (node-старт ~4 с/файл) і з повним контролем `num_ctx`/`num_predict`.
* Good, because system-prompt дає +14–16 п.п. над "голим" викликом — це більший приріст, ніж різниця між моделями в fit-класі.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Експеримент у `/tmp/docgen-bench3/` (потім `~/docgen-bench3/`): скрипти `run.py`, `confirm.py`. pi підключено до `gemma3:4b` через `~/.pi/agent/models.json` з `baseUrl: http://localhost:11434/v1`, `api: openai-completions`. Транспорт pi обраний за ергономікою (інструменти read/write, сесії), а не за якісною перевагою в one-shot docgen.

## ADR Оркестратор docgen як JS-скрипт із детермінованою екстракцією (інверсія керування)

## Context and Problem Statement
One-shot LLM-генерація доку дає нестабільну якість: витоки stdlib/сигнатур/внутрішніх імен, галюцинації в гарантіях, обрізання виводу на великих файлах. Причина — модель одночасно відповідає за факти, структуру і прозу.

## Considered Options
* Один one-shot промпт із жорстким system-prompt (поточний підхід)
* JS-оркестратор із детермінованою Stage 0 екстракцією + секційними промптами + детермінованою зборкою

## Decision Outcome
Chosen option: "JS-оркестратор — `docgen-gen.mjs` — як входна точка; локальна модель — лише сервіс перефразування", because інверсія керування усуває класи помилок без LLM-токенів: JS екстрагує факти детерміновано (Stage 0), будує точкові промпти (Stage 1), лінтить витоки (Stage 2), збирає структуру фіксованими `##`-заголовками (Stage 3). Benchmark v2 підтвердив: оркестрована `gemma3:4b` ~86% vs one-shot ~80% (+6 п.п.), часово конкурентна з one-shot (overlay 77с vs 61с, k8s 31с vs 45с).

### Consequences
* Good, because детермінований лінт (Stage 2) прибирає витоки сигнатур/внутрішніх імен без додаткових токенів; структура гарантована кодом, а не моделлю.
* Good, because `gemma3:4b` + оркестрація (~86%) наближається до `gemma4:4b` one-shot (~92%), лишаючись у GPU (3.3 GB) і без офлоаду.
* Bad, because v1 оркестрації (код у всіх секціях) була 3–5× повільніша за one-shot через повторний інгест коду в кожен секційний виклик — виправлено у v2 (код тільки в `Поведінку`).
* Bad, because Stage 0 JS-екстрактор прив'язаний до мови: `.mjs`/`.js` підтримані, `.vue`/`.py` — fallback до one-shot.

## More Information
Файли в `.worktrees/feat-docgen-orchestrator-pi/npm/skills/docgen/js/`: `docgen-extract.mjs` (Stage 0), `docgen-prompts.mjs` (Stage 1, секційно-мінімальний контекст), `docgen-gen.mjs` (входна точка, Stage 2–3). Виявлений баг: `gemma4:4b` через `/api/chat` повертає вихід у `j.response`, а не `j.message.content` — фікс: `j.message?.content || j.response || ''`. Гілка `feat/docgen-orchestrator-pi`.
