## ADR Рекурсивний складений DAG як базова модель графа задач

## Context and Problem Statement
Потрібно описати структуру оркестрованих задач, де проекти всередині директорій декомпозуються на динамічні workflow-графи, а вузли цих графів самі можуть містити вкладені workflow. Питання: яка структура даних точно відповідає такій ієрархії з необмеженою глибиною вкладеності.

## Considered Options
* "3D-граф" (відхилено під час розмови)
* Recursive Compound DAG (Hierarchical Workflow Graph)

## Decision Outcome
Chosen option: "Recursive Compound DAG", because кожен вузол або є атомарним листком, або сам містить DAG з entry/exit-вузлами; батьківський рівень бачить лише `state = resolved | failed` та outputs, не знаючи про внутрішню реалізацію — принцип substitutability.

### Consequences
* Good, because атомарний вузол можна "розкрити" у підграф без жодних змін у батьківському графі (однаковий інтерфейс).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Модель вузла:
```
Node
├── inputs:  Map<portId, Value>
├── outputs: Map<portId, Value>
├── state: pending | running | resolved | failed
└── impl (визначається динамічно):
Atomic   → fn(inputs) → outputs
Compound → Graph{ entry, nodes[], edges[], exits[] }
```
Edges несуть дані: `outputs[from] → inputs[to]`. Підграф має кілька exit-вузлів; їх outputs merge-уються в outputs батьківського Compound-вузла. Близькі реалізації: Dask, Prefect dynamic tasks, LangGraph.

## ADR Динамічна декомпозиція вузла під час виконання

## Context and Problem Statement
Необхідно вирішити, коли визначається, чи є вузол атомарним або compound: статично при побудові графа чи під час виконання на основі вхідних даних.

## Considered Options
* Статична декомпозиція (структура відома наперед)
* Динамічна декомпозиція (під час виконання)

## Decision Outcome
Chosen option: "Динамічна декомпозиція", because вузол аналізує власні `inputs` під час запуску і вирішує, чи будувати підграф чи виконати атомарну функцію — батьківський граф у цьому рішенні участі не бере.

### Consequences
* Good, because transcript фіксує очікувану користь: однаковий executor-алгоритм `execute(node)` на всіх рівнях рекурсії.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
```
execute(node):
inspect(inputs) →
"простий випадок" → Atomic, run fn()
"складний"        → Compound, build sub-DAG, wait exits
→ node.state = resolved | failed
```

## ADR Файлова система як розподілений state store графа

## Context and Problem Statement
Потрібно визначити, де зберігається стан вузлів і структура графа: централізований оркестратор або розподілено.

## Considered Options
* Централізоване зберігання (оркестратор керує всім графом)
* Розподілене файлове зберігання (кожен вузол пише свої файли)

## Decision Outcome
Chosen option: "Розподілене файлове зберігання", because кожен task/вузол самостійно записує свої файли; граф відновлюється через scan файлової системи — структура графа implicit у вкладеності директорій.

### Consequences
* Good, because transcript фіксує очікувану користь: файлова структура природньо підтримує MutationLog та патчі від EngineerAgent.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файлова структура:
```
tasks/<nodeId>/
meta.json       ← тип, стан, edges
inputs.json
outputs.json    ← з'являється при resolved
error.json      ← з'являється при failed
graph/          ← якщо Compound
<childId>/
...
```
Граф = DAG + MutationLog: `snapshot_v1 → patch_1 → snapshot_v2 → ...`. Patches записуються при втручанні EngineerAgent.

## ADR Meta-рівневий Engineering Supervisor поза графом з необмеженим scope змін

## Context and Problem Statement
При падінні вузла потрібен механізм відновлення. Питання: чи є агент-інженер вузлом у самому графі (рекурсивний), або існує поза графом з привілейованим доступом; і чи може він змінювати лише локальний підграф впалого вузла або і батьківські рівні.

## Considered Options
* EngineerAgent як вузол у графі (рекурсивна self-repair)
* EngineerAgent як meta-рівень поза графом (supervisor)

## Decision Outcome
Chosen option: "EngineerAgent як meta-рівень поза графом", because він не є вузлом, не має обмежень на scope — може підніматися вгору і змінювати батьківський граф на будь-якому рівні ієрархії.

### Consequences
* Good, because transcript фіксує очікувану користь: інженер може insert нові вузли, replace атомарний вузол compound-ним, rewire edges, patch inputs — GraphPatch застосовується на будь-якому рівні, після чого тригериться retry.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
```
on node.state = failed:
EngineerAgent:
├── читає error.json + graph slice (будь-який рівень вгору)
├── GraphPatch: replace node | insert nodes | rewire edges | modify inputs
└── тригерить retry
```
Формат GraphPatch (JSON diff, CRDT тощо), scheduler і конкурентність на рівні підграфу в transcript не визначені.
