## ADR Kubernetes як середовище виконання задач

## Context and Problem Statement
Задачі з `tasks/` потрібно виконувати в ізольованому середовищі. Користувач окреслив вимогу: "таски будуть жити в k8s". Водночас виникло питання як мати Zed-remote доступ до файлів задач під час розробки та налагодження.

## Considered Options
* Dev pod, що монтує той самий PVC що й worker-поди
* `kubectl port-forward` + SSH у поді (тимчасовий доступ)
* Tailscale / WireGuard у поді (постійна мережа)

## Decision Outcome
Chosen option: "Dev pod, що монтує той самий PVC що й worker-поди", because цей варіант позначений у transcript як "рекомендований" — забезпечує одночасний доступ до `tasks/` і для worker-подів, і для Zed remote через SSH.

### Consequences
* Good, because transcript фіксує очікувану користь: `tasks-pvc` одночасно монтується в worker-поди та dev pod; Zed бачить живий стан файлів задач.
* Bad, because Neutral, because transcript не містить підтвердження наслідку — рішення ще не реалізоване.

## More Information
Запропонована команда доступу: `kubectl port-forward pod/n-graph-dev 2222:22`. Dev pod потребує SSH-сервера та встановленого `zed --headless`. Діаграма з transcript: `tasks-pvc ←── n-cursor-graph (worker pods)` і `tasks-pvc ←── dev-pod (SSH + zed)`.

## ADR UI задач як окремий проект із REST/SSE-інтерфейсом

## Context and Problem Statement
Для перегляду стану задач та циклу виконання графа вирішено створити окремий веб-проєкт. Постало питання як UI отримуватиме дані — через пряме монтування файлів `tasks/` чи через API-шар.

## Considered Options
* UI читає стан через `mt scan --json` (REST або SSE)
* Пряме монтування `tasks/` у Zed remote (dev pod)

## Decision Outcome
Chosen option: "UI читає стан через `mt scan --json` (REST або SSE)", because для UI немає потреби монтувати `tasks/` локально — UI спілкується з API-сервером у k8s, що прибирає залежність від Zed remote.

### Consequences
* Good, because transcript фіксує очікувану користь: UI не залежить від Zed remote й безпосереднього доступу до файлів; достатньо HTTP/SSE до API-сервера у k8s.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Запропоновані назви проєкту: `n-graph` (рекомендовано — відповідає екосистемі `n-cursor`), `graphwatch`, `taskflow`. Остаточну назву в transcript не підтверджено — сесія завершилась уточнювальним питанням. Технічний стек UI: Vue 3 (згадано у `tasks/ui-task-view/task.md`).
