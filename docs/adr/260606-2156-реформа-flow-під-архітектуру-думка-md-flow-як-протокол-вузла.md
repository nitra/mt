## ADR Реформа `flow` під архітектуру `npm/docs/mt.md`: `flow` як протокол вузла, `graph` як оркестратор

## Context and Problem Statement
Існували два незалежних шари: `mt` (попередній MT workflow) і архітектура автономного ОАГ (`npm/docs/mt.md`). Обидва використовували git worktrees, але мали несумісні протоколи стану (MT file-presence state vs `tasks/<node>/task.md`). Потрібно було об'єднати їх в єдину архітектуру.

## Considered Options
* Зберегти обидва шари незалежними, тільки додати `graph` поряд з `flow`
* Повністю переписати `flow` під контракт `npm/docs/mt.md`: `flow` = протокол всередині вузла, `graph` = зовнішній оркестратор DAG

## Decision Outcome
Chosen option: "Повністю переписати `flow` під контракт `npm/docs/mt.md`", because `flow` і `graph` вже перекривались по відповідальності (обидва управляли worktrees і відстежували стан задачі); об'єднання усуває дублювання та дає чітку межу: `graph` — DAG зовні, `flow` — протокол всередині одного вузла.

### Consequences
* Good, because transcript фіксує очікувану користь: один джерело істини для стану вузла (`task.md` / `outputs_NNN.md` замість MT file-presence state + `docs/specs/` + `docs/plans/`).
* Bad, because видаляються чинні команди `mt init`, `mt init`, `mt done` та весь MT file-presence state-стан — потребує повного рефакторингу dispatcher і n-flow.mdc.

## More Information
Файли, що видаляються: `lib/spec.mjs`, `lib/plan.mjs`, `lib/plan-panel.mjs`, `lib/flow-lock.mjs`, `lib/state-store.mjs`, `lib/snapshot.mjs`, `lib/artifact.mjs`. Команди `init`, `spec`, `release` вилучаються з `commands.mjs`. Документ архітектури: `npm/docs/mt.md`.

## ADR Два етапи виконання вузла: Stage 1 (`mt plan`) і Stage 2 (solve/verify/signal)

## Context and Problem Statement
В попередньому дизайні `flow` мав лінійний ланцюг: `init → spec → plan → verify → release`. У новій архітектурі вузол може виявитись або атомарним (вирішується безпосередньо), або складеним (розкладається на дочірній граф) — і це рішення потрібно прийняти першим, до будь-якого виконання.

## Considered Options
* Один монолітний крок (агент сам вирішує атомарний/складений без явного gate)
* Два явних etapи: Stage 1 — planning (рішення про структуру), Stage 2 — execution (реалізація атомарного вузла)

## Decision Outcome
Chosen option: "Два явних етапи", because потрібен чіткий gate, де агент або людина підтверджує: це атомарна задача або граф. Stage 1 формує `plan_001.md` (атомарний шлях) або дочірні `task.md` (складений шлях); Stage 2 виконує лише атомарну роботу і сигналізує `mt done|audit|failed`.

### Consequences
* Good, because transcript фіксує очікувану користь: людина може зупинитись між Stage 1 і Stage 2 і переглянути декомпозицію до початку виконання.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Stage 1 вихід атомарного шляху: `tasks/<node>/plan_001.md` (numbered, immutable). Stage 1 вихід складеного шляху: дочірні `tasks/<node>/<child>/task.md` + явний виклик `mt spawn <шлях-вузла>` агентом. Stage 2: `mt verify` перевіряє `## Done when` з `task.md`, потім агент пише `outputs_NNN.md` і викликає `mt done|audit|failed`. Документ: `npm/docs/mt.md §Інтеграція з mt`.

## ADR `mt plan` — об'єднання `mt init` і `mt plan` в один крок

## Context and Problem Statement
Стара система мала `mt init` (brainstorm, панель персон, elicitation) і `mt plan` (декомпозиція на кроки → MT file-presence state) як окремі команди. В новій Stage 1 архітектурі потрібно вирішити: чи залишаються вони двома кроками (де людина зупиняється між дизайном і декомпозицією) або зливаються в один.

## Considered Options
* Варіант B (два кроки): `mt init` → дизайн/brainstorm → `mt plan` → декомпозиція
* Варіант A (один крок): `mt plan` поглинає і spec, і decompose

## Decision Outcome
Chosen option: "Варіант A — спільний крок, об'єднуємо їх", because user явно обрав: "спільний крок поєднуємо їх". `mt init` зникає; `mt plan` виконує дизайн і декомпозицію за один запуск і пише єдиний `plan_001.md`.

### Consequences
* Good, because менше команд для запам'ятовування; агент не чекає проміжного підтвердження між дизайном і декомпозицією.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`mt plan` читає `task.md` → поле `mode: human|agent` (default: `human`) → веде діалог або автономно вирішує → пише `plan_001.md` (атомарний) або дочірні `task.md` (складений). `mt init` вилучається повністю. Файл: `npm/scripts/dispatcher/lib/spec.mjs` — видалити; `npm/scripts/dispatcher/lib/plan.mjs` і `lib/plan-panel.mjs` — переписати.

## ADR `mode: human|agent` у `task.md` та явний `mt spawn` (Variant B)

## Context and Problem Statement
Stage 1 (`mt plan`) може бути інтерактивним (людина бере участь) або автономним (агент сам). Потрібно вирішити (а) де зберігається цей атрибут і (б) хто викликає `mt spawn` — автоматично всередині `mt plan` чи агент вручну після.

## Considered Options
* Атрибут mode як аргумент команди (`--mode human`)
* Атрибут `mode:` в `task.md` front-matter; за замовчуванням `human`
* Variant A для spawn: `mt plan` сам викликає `mt spawn` в кінці (bundled)
* Variant B для spawn: `mt plan` тільки створює дочірні `task.md`, агент викликає `mt spawn` вручну

## Decision Outcome
Chosen option: "`mode:` у `task.md`, default `human`; Variant B для spawn", because атрибут у файлі завдання дозволяє задавати режим ще до запуску агента та зберігається разом з задачею. Variant B обраний явно у transcript; явний `mt spawn` дає агенту можливість перевірити коректність дочірніх задач перед реєстрацією.

### Consequences
* Good, because transcript фіксує очікувану користь: агент контролює момент реєстрації дочірніх вузлів і може виправити `task.md` дочірніх вузлів перед `spawn`.
* Bad, because Neutral, because transcript не містить підтвердження наслідку — агент може забути викликати `mt spawn`, залишивши вузол у невизначеному стані.

## More Information
`task.md` front-matter: `mode: human` (default) або `mode: agent`. Після Stage 1 на складеному шляху агент явно викликає `mt spawn <шлях-вузла>`. Додатково user підтвердив потребу в окремому скрипті-оркестраторі, що обходить весь граф і розміщує вузли в чергу: реалізується як `mt run --auto` (post-merge hook) і `mt watch` (демон-черга).
