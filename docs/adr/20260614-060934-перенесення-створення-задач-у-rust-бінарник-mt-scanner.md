## ADR Перенесення створення задач у Rust-бінарник `mt-scanner`

## Context and Problem Statement
Наразі створення нової задачі реалізовано в JS-шарі (`npm/lib/commands/init.mjs`). Специфікація `docs/spec-task-create-rust-integration.md` (статус: «погоджено — готово до реалізації», 2026-06-13) вимагає симетрії з read-side: так само, як сканування ФС делеговано Rust-бінарнику `mt-scanner scan`, запис (створення `task.md`) теж має виконуватись у Rust через новий субкоманду `mt-scanner create`.

## Considered Options
* Залишити логіку створення в `npm/lib/commands/init.mjs` (поточний стан)
* Реалізувати `mt-scanner create` у Rust-крейті `scanner/` і делегувати виклик з JS-обгортки

## Decision Outcome
Chosen option: "Реалізувати `mt-scanner create` у Rust-крейті `scanner/`", because специфікація містить пряму директиву користувача: «усе, що стосується роботи з файловою системою, має бути в Rust», симетрично до вже реалізованого `mt-scanner scan`.

### Consequences
* Good, because transcript фіксує очікувану користь: єдина точка відповідальності за ФС-операції в Rust, узгодженість з вже делегованим сканом.
* Bad, because `scanner/Cargo.toml` не має залежностей для роботи з датою/часом (`chrono`/`time` відсутні в grep-перевірці), тож їх треба додати для генерації front-matter у `task.md`.

## More Information
- Специфікація: `docs/spec-task-create-rust-integration.md`
- Поточна JS-реалізація, яку замінює рішення: `npm/lib/commands/init.mjs`
- Rust-крейт для розширення: `scanner/` (workspace member у кореневому `Cargo.toml`)
- Вхідна точка бінарника: `scanner/src/main.rs` — вже парсить субкоманди `scan`; `create` треба додати за тим самим патерном
- Config-дефолти читаються у Rust із `.mt.json` (функція `read_agent_retry_max` у `scanner/src/lib.rs:484`)
- Залежності `chrono`/`time` відсутні в `scanner/Cargo.toml` — треба додати для генерації timestamp у front-matter
