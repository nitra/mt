## ADR Рекурсивний складений ОАГ з динамічним розкладом та файловим сховищем стану

## Context and Problem Statement
Потрібна структура даних і протокол виконання для системи, де оркестратор розбиває задачі на граф підзадач, кожна підзадача може рекурсивно розкладатися на власний підграф під час виконання, і вся ця ієрархія має зберігати стан між запусками та підтримувати самовідновлення через LLM-агента-інженера.

## Considered Options
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Рекурсивний складений ОАГ з файловим сховищем стану (файлова система як state store)", because автор сесії послідовно розкрив всі вимоги: динамічний розклад вузлів під час виконання, data-flow ребра, кілька exit-вузлів, LLM-агент-інженер на мета-рівні з повним доступом до графу та append-only файловий протокол з план/факт-парами для атомарного відновлення.

### Consequences
* Good, because кожен вузол пише лише свої файли (`task.md`, `inputs.md`, `outputs.md`, `error.md`, `repair_history.md`, `операції/`, `патчі/`) — git merge між ворктрі незалежних вузлів завжди чистий, гонки за атомарністю неможливі архітектурно.
* Good, because transcript фіксує очікувану користь: `repair_history.md` як append-only журнал дозволяє LLM-інженеру читати весь контекст ремонту і продовжувати природно; файлова структура = повний git-аудит.
* Bad, because transcript не містить підтверджених негативних наслідків; потенційна вартість скану `*-plan*.md` при великих графах позначена як слабка сторона в SWOT без кількісної оцінки.

## More Information
- Файл з повним дизайном: `npm/docs/mt.md`
- Структура вузла: `tasks/<node-id>/{task.md, inputs.md, outputs[-vN].md, error[-vN].md, invalidated, repair_history.md, repair_context.md, операції/, патчі/, підграф/}`

## ADR Топологія графу — розподілена по `task.md` дочірніх вузлів, без центрального файлу

## Context and Problem Statement
Потрібно вирішити де зберігається топологія підграфу (ребра між дочірніми вузлами) — в одному авторитетному файлі батька чи розподілено в метаданих кожного дочірнього вузла.

## Considered Options
* Центральний `граф.md` у батьківському вузлі
* Топологія в `task.md` кожного дочірнього (поле `deps:`)

## Decision Outcome
Chosen option: "Топологія в `task.md` кожного дочірнього", because автор аргументував «заради універсальності» — кожен вузол самодостатній, динамічний spawn не потребує оновлення жодного центрального файлу, оркестратор відновлює повний граф скануванням.

### Consequences
* Good, because динамічний spawn = просто новий `task.md` з `deps:` на вже існуючі вузли; ніяких оновлень батьківського файлу, жодних конфліктів при паралельних ворктрі.
* Bad, because transcript не містить підтверджених негативних наслідків; реконструкція топологічного порядку потребує повного скану `підграф/*/task.md`.

## More Information
- `task.md` формат: `deps: [sibling-id-1, sibling-id-2]` — лише `id` сусідів без path; `parent:` — шлях відносно `tasks/` для human-readability
- `id` вузла = назва директорії, не дублюється у фронтматері
- Команда реконструкції: `scan tasks/**/task.md` → збір `deps:` → побудова ОАГ

## ADR Append-only файловий протокол з версіонуванням через суфікс `-vN`

## Context and Problem Statement
Класична проблема атомарності: агент може впасти між записом файлу стану і викликом CLI-хука — граф опиняється у невалідному проміжному стані. Потрібен механізм що гарантує консистентність без транзакцій БД.

## Considered Options
* Мutable файли з atomic rename (write-to-temp + mv)
* Append-only: файли лише створюються, ніколи не перезаписуються; нова версія = новий файл з суфіксом `-v2`, `-v3`

## Decision Outcome
Chosen option: "Append-only з суфіксом `-vN` для версій та sentinel-файлами для стану", because автор зазначив що `create file` атомарна операція на більшості ФС — наявність файлу сама є сигналом стану, окремий CLI-хук для зміни стану не потрібен; ворктрі-межа закриває решту гонок.

### Consequences
* Good, because git merge між ворктрі незалежних вузлів завжди чистий — два ворктрі створюють лише нові файли з унікальними іменами, ніколи не змінюють існуючі.
* Good, because transcript фіксує очікувану користь: повна git-історія = безкоштовний time-travel debugging; `repair_history.md` append дозволяє LLM природно продовжувати журнал.
* Bad, because observability-скрипт повинен знаходити файл з найбільшим `-vN` суфіксом як актуальний — додаткова логіка сканування.

## More Information
- `outputs.md`, `outputs-v2.md`, `outputs-v3.md` — старі файли не видаляються
- `error.md`, `error-v2.md` — аналогічно
- `invalidated` — sentinel без розширення, наявність = знедійснений; при відновленні видаляється (єдина операція видалення)
- `repair_history.md` — єдиний виняток: append-only без версіонування, новий `## Attempt N` дописується в кінець

## ADR EngineerAgent як мета-рівень з необмеженим доступом до графу та time-budget замість attempt-limit

## Context and Problem Statement
При збої вузла потрібен механізм відновлення. Питання: де живе агент-інженер (всередині графу чи поза ним), який рівень ієрархії він може змінювати, і як запобігти нескінченному циклу ремонту.

## Considered Options
* EngineerAgent як вузол всередині графу
* EngineerAgent як мета-рівень поза графом з необмеженим доступом; convergence через attempt-limit
* EngineerAgent як мета-рівень поза графом; convergence через time-budget

## Decision Outcome
Chosen option: "EngineerAgent як мета-рівень поза графом з time-budget", because автор явно вказав «мета-рівень поза графом, може підніматись вгору і змінювати батьківський граф»; time-budget обрано замість attempt-limit бо «в тебе є 10 хвилин на вирішення з безмежною кількістю спроб» — реалістичніше обмеження.

### Consequences
* Good, because інженер stateless — знання і контекст живуть у файлах вузла (`repair_history.md`, `repair_context.md`), не в агенті; будь-який новий виклик підхоплює повний контекст.
* Good, because ієрархія ескалації симетрична: кожен рівень отримує свіжий `budget_sec`; максимальний час до людини = `depth × budget` — передбачувано.
* Bad, because transcript не містить підтверджених негативних наслідків; теоретично cascade при зміні кореня може вичерпати весь budget не вирішивши проблему.

## More Information
- `repair_context.md`: поля `deadline`, `started_at`, `budget_sec`
- `repair_history.md`: append-only, секції `**Reasoning:**`, `**Patch:**`, `**Result:**`
- Протокол патчу залежного вузла: спочатку `patch-plan-<ts>.md` (намір), потім kill залежних у топологічному порядку, потім патч, потім `patch-fact-<ts>.md`
- Root timeout → `звіт-старшому.md` → SeniorEngineer (людина)
