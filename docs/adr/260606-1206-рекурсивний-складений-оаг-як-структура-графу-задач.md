## ADR Рекурсивний складений ОАГ як структура графу задач

## Context and Problem Statement
Система оркестрації задач потребує структури що підтримує вкладені динамічні робочі процеси: вузол може або виконуватись атомарно, або розкладатись на підграф під час виконання, причому батьківський рівень не повинен знати деталей реалізації вузла.

## Considered Options
* Recursive Compound DAG — вузол реалізується або як атомарна функція, або як підграф із такими самими вузлами
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Recursive Compound DAG", because батько бачить лише інтерфейс вузла (`стан: вирішено / помилка` + вхідні/вихідні дані), незалежно від того чи вузол атомарний чи складений — це забезпечує substitutability і рекурсивне виконання одним і тим самим executor.

### Consequences
* Good, because атомарний вузол можна "розкрити" в підграф без змін у батьківському графі.
* Good, because transcript фіксує очікувану користь: executor однаковий на всіх рівнях — `execute(node): if atomic → run; else → execute(node.graph)`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Граф орієнтований (DAG), ребра несуть дані (`outputs` попереднього вузла → `inputs` наступного), розклад вузла визначається динамічно при запуску на основі вхідних даних. Декілька exit-вузлів у підграфі мерджать свої виходи в `outputs` батька. Близькі реалізації: Dask, Prefect dynamic tasks, LangGraph.

## ADR Append-only файлова система як сховище стану графу

## Context and Problem Statement
Граф задач потребує персистентного сховища стану що підтримує паралельне виконання агентів у git-ворктрі без конфліктів при записі та без потреби в транзакційному механізмі.

## Considered Options
* Append-only модель — файли лише створюються або видаляються атомарно, ніколи не змінюються
* Mutable JSON-файли (стан перезаписується при кожному переході)

## Decision Outcome
Chosen option: "Append-only модель", because `create file` є атомарною операцією на більшості файлових систем — наявність файлу і є сигналом зміни стану, окремий CLI-хук для зміни стану більше не потрібен; два ворктрі створюють тільки нові файли з унікальними іменами → `git merge` завжди чистий.

### Consequences
* Good, because конфлікти при злитті ворктрі неможливі за визначенням — кожен агент створює тільки нові файли.
* Good, because `repair_history/` реалізується як тека з окремим файлом на кожну спробу — природній append-only лог.
* Bad, because для визначення поточного стану вузла потрібно сканувати директорію (немає індексу); transcript фіксує це як потенційну проблему при великих графах.

## More Information
Sentinel-файли визначають стан: `виконується` (порожній файл) = running, `вихідні.md` = resolved, `помилка.md` = failed, `скасовано` = cancelled (має пріоритет над `вихідні.md`), `знедійснений` = invalidated. Ворктрі-межа є межею атомарності: наступники не стартують поки ворктрі попередника не змерджено — merge є єдиною точкою рішення оркестратора.

## ADR Markdown + YAML frontmatter як LLM-first формат файлів вузла

## Context and Problem Statement
Файли вузла (місія, вхідні, вихідні, помилка, журнал відновлення) читаються та записуються LLM-агентами — формат повинен бути зручним для мовних моделей у першу чергу, а не для програмного парсингу.

## Considered Options
* Markdown з YAML frontmatter — структуровані поля у frontmatter, вільний текст у тілі
* JSON — машинозчитуваний але неприродний для LLM-генерації та читання

## Decision Outcome
Chosen option: "Markdown з YAML frontmatter", because LLM-моделі натреновані на Markdown і природно продовжують документ; `repair_history.md` як append-only Markdown-журнал дозволяє інженеру читати попередні спроби і дописувати нову як продовження тексту — без парсингу JSON-масиву.

### Consequences
* Good, because transcript фіксує ключову перевагу: `repair_history.md` — LLM просто читає і продовжує природно.
* Good, because YAML frontmatter зберігає машинозчитувані поля (стан, тип, ребра, мітки часу) для CLI-оркестратора.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли вузла: `місія.md` (незмінна після spawn), `вхідні.md` (незмінна), `вихідні.md` (з'являється при resolved), `помилка.md` (при failed), `repair_history/NNN-timestamp.md` (новий файл на кожну спробу). Ребра у `meta.md` frontmatter вказують конкретні поля (`вузол_1.відповідь → вузол_2.запит`) для impact analysis при інвалідації.

## ADR Git worktree на вузол як одиниця паралельності з merge як точкою рішення

## Context and Problem Statement
Кілька вузлів графу можуть виконуватись паралельно незалежними агентами; потрібен механізм ізоляції що запобігає конфліктам стану та визначає момент коли наступники можуть стартувати.

## Considered Options
* Git worktree на кожен вузол — агент виконується в ізольованому worktree, merge = завершення
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Git worktree на кожен вузол", because merge ворктрі є природною атомарною операцією — до merge наступники не стартують, після merge стан зафіксований; LLM-оркестратор при merge оцінює `outputs.md` і вирішує чи приймати результат.

### Consequences
* Good, because гонки між агентами усуваються структурно: ізоляція через worktree, а не через локи.
* Good, because transcript фіксує: "merge = єдина точка рішення для оркестратора. До merge — ізоляція, після — факт."
* Bad, because масштаб: 100+ паралельних вузлів = 100+ worktree на диску (transcript фіксує як слабку сторону в SWOT).

## More Information
Worktree розміщуються в `.worktrees/<branch-name>/`. Паралельне виконання незалежних вузлів не породжує конфліктів при merge оскільки кожен агент пише тільки у свою директорію `tasks/<nodeId>/`. Конфлікт можливий лише якщо агент-інженер у ворктрі A і агент у ворктрі B одночасно модифікують `meta.md` спільного батьківського вузла — вирішується агентом-медіатором при merge.

## ADR Stateless LLM-оркестратор запускається подіями

## Context and Problem Statement
Після завершення вузла (merge ворктрі) система повинна визначити наступні готові до виконання вузли і запустити їх — потрібен механізм координації що відповідає природі LLM-агентів.

## Considered Options
* Stateless event-driven LLM — запускається подією (post-merge hook), читає ФС, приймає рішення, завершується
* Persistent daemon — постійно запущений процес що слідкує за ФС

## Decision Outcome
Chosen option: "Stateless event-driven LLM", because оркестратор є LLM-агентом — постійно запущений процес суперечить природі LLM; файлова система є єдиним джерелом правди, оркестратор реконструює стан при кожному виклику.

### Consequences
* Good, because transcript фіксує перевагу: LLM-оркестратор може читати `місія.md` кожного вузла і приймати контекстні рішення — не тільки "залежності виконані?" але і "чи варто паралельно чи послідовно враховуючи зміст задач?".
* Good, because відсутність persistent state спрощує відновлення після збоїв — при перезапуску оркестратор просто сканує ФС знову.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Тригери оркестратора: merge ворктрі → spawn наступників; `помилка.md` → spawn engineer agent; engineer timeout → escalate, spawn engineer на рівні вгору; root timeout → notify SeniorEngineer. CLI-точка входу: `n-cursor orchestrate`. Оркестратор знаходить готові вузли: всі попередники мають `вихідні.md` і не мають `знедійснений`.

## ADR Агент-інженер як мета-рівень із часовим бюджетом і пам'яттю на вузлі

## Context and Problem Statement
При помилці вузла система повинна автономно діагностувати і виправляти проблему — включно зі зміною структури графу на будь-якому рівні ієрархії — зберігаючи при цьому збіжність і уникаючи нескінченних циклів.

## Considered Options
* Engineer як мета-рівень поза графом з time budget і пам'яттю на вузлі
* Обмеження кількістю спроб (max attempts)
* Engineer як звичайний вузол у графі

## Decision Outcome
Chosen option: "Engineer як мета-рівень поза графом з time budget і пам'яттю на вузлі", because time budget (не лічильник спроб) є реалістичнішим обмеженням що дозволяє необмежену кількість спроб але з адаптацією стратегії залежно від часу що залишився; пам'ять зберігається на вузлі (не на engineer) — engineer stateless, знання в `repair_history/`.

### Consequences
* Good, because transcript фіксує: інженер може замінювати вузли, вставляти проміжні кроки, перепідключати ребра, змінювати вхідні дані — без обмежень по рівню ієрархії.
* Good, because `repair_history/` на вузлі забезпечує контекст для майбутніх викликів і накопичує діагностичне знання про конкретний вузол.
* Bad, because необмежений доступ інженера до батьківських графів може спричинити каскадну інвалідацію всього дерева (transcript фіксує як загрозу в SWOT).

## More Information
Ієрархія ескалації: `node fails → EngineerAgent (10хв) → timeout → parent.EngineerAgent (свіжий 10хв) → ... → root timeout → notify SeniorEngineer`. Максимальний час до людини: `глибина × бюджет`. При ескалації формується `senior_report.md` з ланцюгом спроб і `suggested_next_steps`. Impact analysis перед каскадом інвалідації: перевіряється чи зачіпає зміна конкретні вхідні порти наступника — якщо ні, наступник залишається валідним.
