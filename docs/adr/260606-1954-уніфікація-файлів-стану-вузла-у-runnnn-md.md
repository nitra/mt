## ADR Уніфікація файлів стану вузла у `run_NNN.md`

## Context and Problem Statement
В архітектурі динамічного графу задач існували окремі файли для різних типів виконань: `repair_history.md` для інженерних ремонтів, `error.md` для збоїв, і `outputs.md` для результатів. Виникло питання чи можна уніфікувати ці файли щоб уникнути множення структур.

## Considered Options
* Окремі файли: `error.md` / `repair_history_NNN.md` / `outputs.md` — спеціалізовані структури для кожного типу актора
* Єдиний `run_NNN.md` — один immutable файл на спробу з полем `actor: agent | engineer | human | auditor`

## Decision Outcome
Chosen option: "Єдиний `run_NNN.md`", because й звичайний агент, і інженер — це «спроба вирішити вузол», і обидва мають однакову структуру: reasoning, результат, посилання на вихідні дані; уніфікація усуває потребу в окремих схемах і дозволяє скрипту однаково читати повну історію вузла незалежно від актора.

### Consequences
* Good, because скрипт читає одну структуру (`run_*.md`) для всієї історії вузла; інженер отримує контекст усіх попередніх спроб в одному форматі.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл `run_NNN.md`: поля `created_at`, `actor`, `result`, `worktree` (опціонально); секції `## Reasoning`, `## Script`, `## Ref`. Нумерація zero-padded до 3 цифр, wrapper рахує існуючі `run_*.md` перед стартом. `outputs_NNN.md` — окремий immutable файл на успішний запуск. `npm/docs/mt.md` — основний архітектурний документ.

## ADR Всі імена файлів і директорій — англійська

## Context and Problem Statement
В архітектурному документі `npm/docs/mt.md` директорії та файли мали назви українською мовою (`операції/`, `патчі/`, `підграф/`, `вхідні.md`, `вихідні.md`). Оскільки файли оброблятимуться скриптами, виникло питання мови іменування.

## Considered Options
* Українські назви — узгоджується з мовою документації
* Англійські назви для всіх файлів і директорій — стандарт для машинного опрацювання

## Decision Outcome
Chosen option: "Англійські назви для всіх файлів і директорій", because файли будуть оброблятись скриптами, і англійська є стандартом для ідентифікаторів що обробляються програмно.

### Consequences
* Good, because transcript фіксує очікувану користь: сумісність зі скриптами без проблем з кодуванням.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Перейменовано: `операції/` → `ops/`, `патчі/` → `patches/`, `підграф/` → `subgraph/`, `вхідні.md` → `inputs.md` (згодом злито в `task.md`), `вихідні.md` → `outputs_NNN.md`. Правило зафіксовано в `npm/docs/mt.md` у секції «Naming convention вузлів».

## ADR Злиття `inputs.md` у `task.md`

## Context and Problem Statement
Початковий дизайн мав окремий файл `inputs.md` для посилань на вхідні дані вузла. Виникло питання чи доцільно тримати місію і вхідні дані в різних файлах.

## Considered Options
* Окремий `inputs.md` — чіткий поділ між місією і даними
* Секція `## Inputs` у `task.md` — агент читає один файл

## Decision Outcome
Chosen option: "Секція `## Inputs` у `task.md`", because агент отримує все необхідне з одного файлу, менше файлів при spawn, і task разом з даними є цілісним контекстом.

### Consequences
* Good, because агент читає один файл замість двох; менше файлів при spawn.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Зміна файлів: `inputs.md` прибрано зі структури. `task.md` тепер містить секції `## Task`, `## Done when`, `## Inputs` з підсекціями `### name` що є `ref:` або inline-текстом. Іммутабельність: до ворктрі — вільно редагується; після — kill ворктрі, редагуй, restart. `npm/docs/mt.md` — схема `task.md`.

## ADR Автоматичний тригер наступних вузлів через git post-merge hook

## Context and Problem Statement
Після завершення вузла та merge ворктрі потрібно запустити наступників у графі. Необхідно вирішити хто і як ініціює ці запуски.

## Considered Options
* Ручний запуск людиною після кожного merge
* `mt run` як живий демон що сам чекає merge і запускає наступників
* Git `post-merge` hook що тригерить `mt run --auto`

## Decision Outcome
Chosen option: "Git `post-merge` hook", because це найчистіший варіант: кожен компонент робить одне — wrapper мержить, git-інфраструктура тригерить продовження; `mt run` залишається простою одноразовою командою.

### Consequences
* Good, because transcript фіксує очікувану користь: повна автоматизація після першого `mt run`; людина тільки запускає кореневий вузол.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Wrapper-скрипт: після `result: success` → `git merge .worktrees/<...>`. `.git/hooks/post-merge`: `mt run --auto`. `mt run --auto`: сканує граф, знаходить усі вузли де `deps → resolved`, запускає паралельно в межах `max_worktrees`, черга в топологічному порядку. `npm/docs/mt.md` — секції «Wrapper-скрипт» та «Git hook».

## ADR Протокол інженера: один тригер, одна спроба, escalate до людини

## Context and Problem Statement
При збої вузла (`result: failed`) потрібна стратегія самовідновлення. Необхідно визначити коли залучається інженер, скільки спроб він має, і що відбувається при невдачі.

## Considered Options
* Інженер викликається при кожному `failed` незалежно від актора
* Інженер викликається лише коли попередня спроба була від `actor: agent`, і тільки одна спроба

## Decision Outcome
Chosen option: "Тільки після `actor:agent result:failed`, одна спроба", because це запобігає нескінченним петлям (агент → інженер → агент → ...) і чітко визначає ескалацію до людини.

### Consequences
* Good, because transcript фіксує очікувану користь: детермінована ескалація без нескінченних петель.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Конфіг: `auto_engineer: true` у `.n-cursor.json`. `budget_sec` у `task.md` — загальний на весь вузол (всі спроби всіх акторів). При `actor:engineer result:failed` → `graph notify <path>`. При вичерпанні `budget_sec` → негайний стоп + `graph notify`. `npm/docs/mt.md` — секція «Протокол агента-інженера».
