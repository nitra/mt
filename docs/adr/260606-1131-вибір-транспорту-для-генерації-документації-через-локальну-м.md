## ADR Вибір транспорту для генерації документації через локальну модель: прямий ollama vs pi

## Context and Problem Statement
Скіл `/n-docgen` Tier 1 викликає локальну модель (gemma3:4b, gemma4:4b) для генерації документації. Потрібно вибрати транспорт доступу до Ollama: прямий `POST /api/chat` або утиліта `pi` (Node.js CLI з власним agent-framework). Різниця у якості виходів між двома транспортами вимагала пояснення й верифікації.

## Considered Options
* Прямий `ollama /api/chat` без system-prompt (варіант A)
* Виклик через `pi CLI` з вбудованим coding-assistant system-prompt (варіант B)
* Прямий `ollama /api/chat` з явним system-prompt (варіант C)

## Decision Outcome
Chosen option: "Прямий `ollama /api/chat` з явним system-prompt", because експеримент показав, що якісна різниця між pi (87%) і прямим+system (85%) лежить у межах шуму (~2 п.п.), тоді як різниця між «без system» і «з system» — системна (+15 п.п.). pi не дає архітектурної переваги: його приріст над голим викликом дає лише вбудований system-prompt.

### Consequences
* Good, because усувається ~4 с фіксованого node-старту pi на кожен файл (≈+1.1 год на 1042 файли при паралельній оркестрації).
* Good, because є повний контроль над параметрами `num_ctx`/`num_predict` — усунення ризику обрізання (варіант A програв `overlay-paths` саме через відсутність контролю).
* Bad, because transcript не містить підтверджених негативних наслідків: pi лишається доречним для агентних сценаріїв (інструменти read/write, сесії), якщо вони знадобляться пізніше.

## More Information
* Бенч-скрипти: `/tmp/docgen-bench3/run.py`, `~/docgen-bench3/duel.py`, `~/docgen-bench3/confirm.py`.
* pi-конфіг провайдера: `~/.pi/agent/models.json` (провайдер `ollama`, ключ `compat.supportsDeveloperRole: false`).
* Виміряний pi node-start overhead: **~3.8–6.4 с/виклик**; інференс-throughput однаковий (~20 tok/s) в обох варіантах.
* pi RPC-режим (`--mode rpc`) для амортизації старту виявився непридатним для docgen: персистентна сесія між незалежними файлами накопичує контекст, що сповільнює наступні файли та несе ризик крос-забруднення.

## ADR Оркестрований JS-конвеєр замість one-shot LLM для генерації документації (Tier 1)

## Context and Problem Statement
One-shot виклик локальної моделі (`gemma3:4b`) для генерації `.md`-документації давав ~80% якості проти еталона: модель витікала в реалізацію (stdlib, regex, сигнатури), пропускала крайові деталі та іноді обрізалась. Потрібен спосіб підняти якість без переходу на важчу модель.

## Considered Options
* One-shot LLM (один промпт → повний документ)
* JS-оркестрований конвеєр: детермінована екстракція фактів (Stage 0) → секційні LLM-виклики з мінімальним контекстом (Stage 1) → детермінований пост-лінт (Stage 2) → зборка (Stage 3)

## Decision Outcome
Chosen option: "JS-оркестрований конвеєр", because детермінований Stage 0 (`docgen-extract.mjs`) заземлює модель точними фактами (список `export`-ів, JSDoc, маркери: `readOnly`, `catchesErrors`, `skips`, `caches`), прибирає найбільші класи помилок без токенів LLM, а Stage 2 пост-лінт механічно зрізає сигнатури та внутрішні імена.

### Consequences
* Good, because transcript фіксує очікувану користь: `gemma3:4b` піднялась з ~80% до ~86% (+6 п.п.) при time-competitive швидкості (orchestrated ≈ one-shot при секційно-мінімальному контексті v2).
* Good, because transcript фіксує очікувану користь: структура документа (порядок і наявність секцій, авто-пропуск `Публічний API` для файлів з однією функцією) — детермінована JS-кодом, а не моделлю.
* Bad, because секційно-мінімальний контекст (код тільки в секцію `Поведінка`) є модель-специфічним: `gemma4:4b` (Gemma 3n E4B) повертає порожній рядок для секцій без коду — оркестрація несумісна з нею.
* Bad, because залишковий дефект: `gemma3:4b` вигадує «кеш» у `Гарантіях` для файлів без кешування (4B-артефакт pattern-matching), який не усувається заборонним промптом і потребує явного негативного маркера `caches: false` у факт-листі.

## More Information
* Реалізація: `npm/skills/docgen/js/docgen-extract.mjs`, `docgen-prompts.mjs`, `docgen-gen.mjs` у worktree `.worktrees/feat-docgen-orchestrator-pi`.
* v1 (shared full-code context) дав часовий регрес ×3.8 (overlay: 310 с vs 57 с one-shot) — причина: кожна секція повторно інгестить весь код.
* v2 (секційно-мінімальний контекст) прибрав регрес: overlay 310→77 с, k8s 141→55 с (швидше за one-shot).
* `gemma4:4b` one-shot з Stage-2 лінтом = ~92%, time-competitive (~100 с/файл), але частковий CPU-офлоад на 8 GB RAM.
* `gemma3:4b` orchestrated = ~86%, ~20 tok/s, повністю в GPU.

## ADR Вибір моделі Ollama для локального Tier 1 docgen на машині з 8 GB RAM

## Context and Problem Statement
На Mac M2 з 8 GB RAM доступні дві Ollama-моделі для docgen Tier 1. Потрібно обрати між легшою швидкою і важчою якіснішою моделлю з урахуванням обмеження пам'яті.

## Considered Options
* `gemma3:4b` (3.3 GB, повністю в GPU)
* `gemma4:4b` alias `batiai/gemma4-e4b:q4` (5.3 GB, частковий CPU-офлоад)
* `gemma4:e4b` full (9.6 GB) — відкинуто одразу: своп до ~0.4 tok/s, непридатно

## Decision Outcome
Chosen option: "обидві моделі з різними сценаріями використання", because бенч показав чіткий trade-off: `gemma3:4b` (~85%, ~20 tok/s, 100% GPU) підходить для швидких/чорнових прогонів; `gemma4:4b` (~92%, ~11 tok/s, 56%/44% CPU/GPU) — для якість-first генерації, де документацію читатимуть люди.

### Consequences
* Good, because transcript фіксує очікувану користь: `gemma4:4b` слухає негативні обмеження (без сигнатур/stdlib), які `gemma3:4b` частково ігнорувала — стеля якості пробита на +7 п.п.
* Bad, because `gemma4:4b` при частковому офлоаді дає ~2× більший час (71/105/137 с vs 30/74/70 с) і нестабільний throughput при конкуренції за пам'ять (ризик свопу при будь-якому фоновому навантаженні на 8 GB).

## More Information
* Ollama alias для зручної назви: `ollama cp batiai/gemma4-e4b:q4 gemma4:4b`.
* pi-конфіг `~/.pi/agent/models.json`: `gemma4:4b` поставлено першою в провайдері `ollama`.
* Вимірювання: `~/docgen-bench3/g4.py` — RAM 6.2 GB, 56% CPU / 44% GPU, cold-load 14 с.
* Blob `d682bf87e3a3` спільний між `gemma4:4b` і колишнім `batiai/gemma4-e4b:q4` (видалено); диск не подвоювався.
