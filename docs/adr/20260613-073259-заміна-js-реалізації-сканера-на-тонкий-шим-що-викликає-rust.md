## ADR Заміна JS-реалізації сканера на тонкий шим, що викликає Rust-бінарник `mt-scanner`

## Context and Problem Statement
У проєкті існують дві паралельні реалізації однієї логіки сканування задач: `scanner/` (Rust) і `npm/lib/core/scanner.mjs` (Node.js `fs`). JS-версія — єдина, яка реально викликається npm-командами; Rust-версія не інтегрована. Прийнято рішення усунути дублювання й залишити єдину реалізацію.

## Considered Options
* Прибрати JS-реалізацію і замінити її тонким шимом у `scanner.mjs`, що викликає бінарник через `spawnSync` та адаптує вихід до очікуваного JS-контракту.
* Залишити JS-версію поруч із Rust (підтримувати дві реалізації).
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Тонкий JS-шим поверх Rust-бінарника", because user прямо зазначив «потрібно щоб js реалізації не існувало, а вона викликала rust варіант».

### Consequences
* Good, because transcript фіксує очікувану користь: єдина авторитетна реалізація логіки сканування, без ризику розходження JS/Rust поведінки.
* Bad, because шим вимагає нетривіального маппінгу: Rust повертає вкладене дерево зі snake_case-станами без поля `dir`, тоді як JS-споживачі (`run.mjs`, `scan.mjs` та ін.) очікують плаский список із kebab-станами, полем `dir` (абс. шлях) і `running`-статусом із git-worktree — всі ці трансформації треба реалізувати у шимі.

## More Information
- `npm/lib/core/scanner.mjs` — файл, що замінюється шимом; `deriveNodeState`/`isComposite` з `state.mjs` використовуються **виключно** ним і після заміни можуть бути видалені.
- Споживачі шиму: `scan.mjs`, `status.mjs`, `run.mjs`, `invalidate.mjs`, `watch.mjs`, `kill.mjs`; публічний експорт `findTasks`/`getActiveWorktrees`/`parseWorktreeList` через `npm/index.js`.
- Ключові розбіжності контрактів (Rust → JS-адаптація у шимі): `pending_audit` → `pending-audit`; `is_composite` → `composite`; відновлення `dir` як `join(mtDir, path)`; `running`-статус із активних git-worktrees (post-process у JS).
- `scanner/Cargo.toml` — залежності лише `serde`/`serde_json`, без C/FFI.
- Специфікація рефакторингу: `docs/spec-scanner-rust-integration.md`.

## ADR Доставка Rust-бінарника через `optionalDependencies` (prebuilt-підпакети)

## Context and Problem Statement
Npm-пакет `@7n/mt` потребує Rust-бінарника `mt-scanner` у рантаймі, але `target/` у `.gitignore` і жодного build-кроку в npm немає. Без вирішення цього блокера пакет не зможе запустити бінарник у користувача.

## Considered Options
* **(A) `postinstall cargo build --release`** — вимагає Rust-тулчейн у всіх, хто ставить пакет.
* **(B) Prebuilt бінарники через `optionalDependencies`** — платформні підпакети (`@7n/mt-darwin-arm64`, `@7n/mt-linux-x64`), npm/bun ставить лише відповідний хосту; аналог esbuild/swc.
* **(C) Покласти бінарник у `files`** — швидко, але крихко для чужих платформ.

## Decision Outcome
Chosen option: "Prebuilt бінарники через `optionalDependencies`", because user обрав варіант B як «правильний прод-шлях».

### Consequences
* Good, because transcript фіксує очікувану користь: не потрібен Rust-тулчейн у кінцевого користувача; npm/bun автоматично встановлює лише потрібний підпакет.
* Bad, because потребує CI-матриці для збірки та публікації кількох npm-підпакетів; `npm-publish.yml` доведеться розширити (зараз однопакетний).

## More Information
- Кожен підпакет матиме `package.json` із полями `os`, `cpu` (і `libc` за потребою); головний `@7n/mt` — лише `optionalDependencies`.
- Резолвер шляху до бінарника: `MT_SCANNER_BIN` env → `node_modules/@7n/mt-<platform>-<arch>/mt-scanner` → `target/release/mt-scanner` (dev-fallback); при відсутності збігу — зрозуміла помилка.
- Специфікація рефакторингу: `docs/spec-scanner-rust-integration.md`.

## ADR Матриця платформ: `darwin-arm64` + `linux-x64` (musl-static)

## Context and Problem Statement
Обравши модель prebuilt-підпакетів, треба зафіксувати, які саме платформи підтримуються на старті — від цього залежать кількість CI-jobs, кількість npm-підпакетів і поля `os`/`cpu`/`libc` у їхніх `package.json`.

## Considered Options
* **Minimal (darwin-arm64 + linux-x64)** — покриває Apple Silicon і весь Linux x64.
* **Standard (5 платформ)** — +linux-arm64, +win32-x64; як у більшості великих проєктів.
* **Full (7+)** — +win32-arm64, armv7, freebsd; нішево.
* Інші профілі в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Minimal (darwin-arm64 + linux-x64 musl-static)", because user підтвердив «для мак тільки 1» (Apple Silicon) і «будуть тільки 2 зараз».

### Consequences
* Good, because transcript фіксує очікувану користь: лише 2 CI-jobs і 2 npm-підпакети; musl-static бінарник без поля `libc` покриває весь Linux x64 — Alpine, Ubuntu, Docker — без проблем із версіями glibc.
* Bad, because Intel Mac (`darwin-x64`) і Linux arm64 (Graviton, CI на arm) не покриті; arm64-бінарник не запуститься на Intel Mac, якщо такий є в команди.

## More Information
- `@7n/mt-darwin-arm64`: Rust triple `aarch64-apple-darwin`, CI-раннер `macos-14` (native arm).
- `@7n/mt-linux-x64`: Rust triple `x86_64-unknown-linux-musl`, CI-раннер `ubuntu` + `musl-tools` (або `cargo-zigbuild`); поле `libc` у `package.json` **не виставляється** — статичний musl-бінарник працює на будь-якому Linux x64.
- Transcript зафіксував: хост розробника — `darwin arm64`; `linux-arm64` планується додати пізніше третім підпакетом без змін у резолвері.
- Розширення матриці у майбутньому: додати підпакет `@7n/mt-linux-arm64` (triple `aarch64-unknown-linux-musl`) — решта коду резолвера та npm-структури не змінюється.
- Специфікація рефакторингу: `docs/spec-scanner-rust-integration.md`.
