## ADR Рекурсивний складений ОАГ як структура для динамічного графу задач

## Context and Problem Statement
У сесії проектувалась система оркестрації задач, де оркестратор динамічно розбиває задачі на workflow-графи, при цьому вузли всередині графа також можуть бути розбиті на підграфи. Потрібно було визначити назву і формальну структуру цієї конструкції, а також описати поведінку вузлів, ребер, зберігання стану, самовідновлення та паралельного виконання.

## Considered Options
* Recursive Compound DAG / Рекурсивний складений ОАГ
* "3D граф" (запропонований користувачем)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Рекурсивний складений ОАГ", because це точний термін із теорії графів (compound graph, де кожен вузол може сам бути графом), а "3D граф" описує просторове відображення, а не структуру зв'язків.

### Consequences
* Good, because transcript фіксує очікувану користь: батько бачить однаковий інтерфейс незалежно від того, атомарний вузол чи складений — замінюваність без змін у батьківському графі.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Фінальна структура зафіксована у файлі `npm/docs/mt.md`. Ключові властивості: ОАГ, data-flow ребра (виходи → входи), кілька вихідних вузлів у підграфі, динамічна декомпозиція при запуску.

## ADR Файловий стан із форматом Markdown + YAML-фронтматер (LLM-first)

## Context and Problem Statement
Кожен вузол графа задач пише свій стан у файли. Постало питання формату: JSON або Markdown, з урахуванням що первинний читач файлів — LLM-агент (не людина і не типовий парсер).

## Considered Options
* Markdown із YAML-фронтматером
* JSON
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Markdown із YAML-фронтматером", because фронтматер зберігає машинозчитувані поля (стан, тип, ребра, мітки часу), а тіло — вільний Markdown для LLM; `repair_history.md` як append-only журнал дозволяє LLM читати і дописувати природно без реконструкції структури з JSON-масиву.

### Consequences
* Good, because transcript фіксує очікувану користь: `repair_history.md` — єдиний документ, який інженер читає і продовжує без парсингу.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файлова структура вузла: `meta.md`, `inputs.md`, `outputs.md`, `error.md`, `repair_history.md`, `patches/`, `graph/<дочірній>/`. Формат зафіксовано у `npm/docs/mt.md`.

## ADR Git-ворктрі як механізм паралельного виконання вузлів

## Context and Problem Statement
Незалежні вузли ОАГ повинні виконуватись паралельно різними агентами. Потрібен механізм ізоляції, злиття результатів і вирішення конфліктів при одночасній модифікації спільних файлів (наприклад `meta.md` батьківського вузла).

## Considered Options
* Git-ворктрі (один ворктрі на вузол)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Git-ворктрі", because це вже знайомий інструмент без нових абстракцій; `git worktree add` з іменем `<ідВузла>-виконання` атомарно провалюється при спробі запуску другого агента на той самий вузол — природний мютекс без додаткових локів (вирішує race condition типу TOCTOU).

### Consequences
* Good, because transcript фіксує очікувану користь: конфлікт злиття оброблюється агентом-медіатором за тим самим патерном що й помилка вузла (симетрія системи).
* Bad, because transcript фіксує обмеження: 100+ паралельних вузлів = 100+ ворктрі на диску.

## More Information
Ворктрі розміщуються у `.worktrees/<ідВузла>-виконання/`. При конфлікті злиття запускається `АгентМедіатор(версія_A, версія_B, контекст)`. Деталі у `npm/docs/mt.md`, розділ "Паралельне виконання: ворктрі".

## ADR Агент-інженер як stateless мета-рівень із повним доступом до графу

## Context and Problem Statement
При помилці у вузлі потрібен механізм самовідновлення. Визначалось: чи інженер є вузлом у графі чи мета-рівнем поза ним; чи може він змінювати батьківські рівні; де зберігається його пам'ять; як обмежити нескінченні спроби.

## Considered Options
* Інженер як вузол у графі (рекурсивний агент)
* Інженер як мета-рівень (supervisor) поза графом із повним доступом
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Інженер як мета-рівень поза графом із повним доступом", because інженер може підніматись вгору і змінювати батьківський граф, застосовувати `mutable with invalidation` (знедійснення вже вирішених вузлів), без обмежень по рівню.

### Consequences
* Good, because transcript фіксує очікувану користь: пам'ять живе на вузлі де інженер вносить зміни (`repair_history.md`) — інженер stateless, знання у файлах; майбутні виклики мають повний контекст попередніх спроб.
* Bad, because transcript фіксує ризик: drift намірів після N патчів; вирішення — незмінна секція `## Оригінальна місія` у `meta.md`.

## More Information
Convergence guard — часовий бюджет (не кількість спроб): `repair_context.md` містить `крайній_термін`, `розпочато_о`, `бюджет_сек`. Ієрархія ескалації: кожен рівень отримує свіжий бюджет; root timeout → `senior_report.md` → старший інженер (людина). Деталі у `npm/docs/mt.md`.
