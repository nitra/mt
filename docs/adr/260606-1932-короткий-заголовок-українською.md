## ADR Файловий контракт вузла динамічного графу задач

## Context and Problem Statement
Система "Динамічний Самомодифікований Граф Задач" потребує чіткого контракту: які файли живуть у директорії вузла, хто їх пише, в якому порядку, і якою мовою заголовки — щоб оркестратор, агент і людина-інженер могли незалежно читати стан без центрального реєстру.

## Considered Options
* Окремий файл `inputs.md` для вхідних даних і `task.md` для місії
* Єдиний `task.md` з секцією `## Inputs` (обраний)
* Append-only `repair_history.md` для журналу спроб інженера
* Окремий numbered файл `repair_history_NNN.md` / уніфікований `run_NNN.md` (обраний)
* Append-only `outputs.md` і `error.md`
* Numbered immutable файли `outputs_NNN.md` і `run_NNN.md` (обраний)
* `ops/spawn-plan` і `ops/kill-plan` для crash-recovery spawn/kill операцій
* Ігнорування цього сценарію і видалення `ops/` (обраний)
* `patches/NNN-plan.md / NNN-fact.md` для журналу патчів інженера
* Зберігання reasoning+changes прямо в `run_NNN.md` без окремої директорії (обраний)
* Окремий `repair_context.md` для часового бюджету → об'єднано в `budget_sec` у `task.md`

## Decision Outcome
Chosen option: "мінімальна immutable файлова структура з `task.md` + `run_NNN.md` + `outputs_NNN.md`", because кожен файл створюється один раз і не змінюється після старту ворктрі — це дозволяє відновлювати стан сканом без центрального реєстру і без перевірки часткових записів.

### Consequences
* Good, because стан графу повністю відновлюється скануванням `task.md` / `run_*.md` / `outputs_*.md` — crash-recovery без окремих plan/fact файлів.
* Good, because `run_NNN.md` уніфікує всі типи акторів (`agent | engineer | human | auditor`) в одній схемі — нема дублювання структури.
* Good, because видалення `ops/` і `patches/` скорочує кількість файлів до мінімуму при збереженні аудит-трейлу.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

**Фінальна схема `task.md`** (місія + inputs + бюджет, immutable після ворктрі):
```yaml
created_at: 2026-06-06T10:00:00Z
parent: research          # відносно tasks/; відсутній у кореневого
deps: [collect-data]      # siblings only
budget_sec: 600
```
Секції: `## Task`, `## Done when`, `## Inputs` — англійські (парсить оркестратор); довільні секції — будь-яка мова.

**Схема `run_NNN.md`** (immutable, NNN генерує wrapper):
```yaml
created_at: ..., actor: agent|engineer|human|auditor, result: success|failed
worktree: <path>          # тільки якщо failed
```
Секції: `## Reasoning` (обов'язкова), `## Script` (wrapper, якщо впав), `## Ref` (якщо є outputs).

**Схема `outputs_NNN.md`** (immutable):
```yaml
created_at: ...
```
Секції: `## Summary` (обов'язкова, один рядок), довільні named-порти для наступників.

**Правила контракту:**
- `created_at` — перше поле в усіх файлах, ISO 8601
- Усі імена файлів і директорій — англійська (обробляються скриптами)
- Секції що парсить скрипт/оркестратор → англійські заголовки; секції з довільними даними → будь-яка мова
- `ref:` якщо дані є у файлі, inline якщо немає деінде
- Append-only інваріант починається з моменту створення ворктрі; до цього файли вільно редагуються або видаляються
- `running.lock` (з PID) — єдиний тимчасовий файл; `mt kill` читає PID, вбиває і каскадить `invalidated`
- NNN нумерація: wrapper рахує `ls run_*.md | wc -l + 1`

Файли: `npm/docs/mt.md` — повний контракт архітектури.
