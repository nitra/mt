## ADR Вся робота з файловою системою — виключно в Rust

## Context and Problem Statement

Початковий дизайн шима `scanner.mjs` передбачав, що логіка `worktree→running` (підвищення стану задачі до `running` за наявності активного git-worktree) залишається в JS як post-process поверх результатів бінарника. При обговоренні специфікації `docs/spec-scanner-rust-integration.md` постало питання розподілу відповідальності між Rust і JS.

## Considered Options

* `worktree→running` лишається в JS-шимі (`scanner.mjs` викликає `listActiveWorktrees`, зіставляє результати бінарника з активними worktree)
* Вся робота з файловою системою — в Rust (`mt-scanner scan` сам запускає `git worktree list --porcelain` і підвищує стан до `running`)

## Decision Outcome

Chosen option: "Вся робота з файловою системою — в Rust", because користувач явно вказав: «все що стосується роботи з файловою системою повинно бути rust».

### Consequences

* Good, because JS-шим (`npm/lib/core/scanner.mjs`) більше не торкається файлової системи — він лише запускає бінарник і парсить JSON; деривація стану цілком у Rust і покрита `cargo test` (30 тестів).
* Bad, because функцію `sanitizeTaskName` (конвенція іменування worktree) довелося портувати в Rust для матчингу; JS-копія лишається тільки в `npm/lib/commands/run.mjs` для *створення* worktree, що означає дві реалізації одного алгоритму — spec §4.3 фіксує вимогу синхронності через спільні тест-вектори.

## More Information

Зміни: `scanner/src/lib.rs` (нова функція `discover_worktrees`, `sanitize_task_name`; параметр `active_worktrees` у `scan_dir`), `scanner/src/main.rs` (прапор `--worktrees <list>` для тестів; без прапора — self-discovery через `git worktree list --porcelain`). Специфікація: `docs/spec-scanner-rust-integration.md` §5 (переписано) + директива-принцип у §0.

## ADR `cargo-zigbuild` як CI-інструмент для musl-статичних Linux-збірок

## Context and Problem Statement

`npm-publish.yml` потребував способу зібрати статичний musl-бінарник `mt-scanner` для `x86_64-unknown-linux-musl` на GitHub Actions. Є два поширені підходи; вибір впливає на вартість додавання `linux-arm64` у майбутньому.

## Considered Options

* `cargo-zigbuild` — один Linux-раннер крос-компілює будь-який musl-таргет через `zig cc`; arm64 пізніше: `--target aarch64-unknown-linux-musl` без нових тулчейнів
* `musl-tools` + `rustup target add x86_64-unknown-linux-musl` — стандартний підхід, apt-пакети, лише x64; arm64 потребує окремого aarch64-тулчейна або окремого раннера

## Decision Outcome

Chosen option: "`cargo-zigbuild`", because користувач підтвердив вибір після явного порівняння: zigbuild дає дешеве додавання `linux-arm64` пізніше без нових тулчейнів; Windows при цьому ортогональний — він потребує нативного `windows-latest` незалежно від tooling.

### Consequences

* Good, because transcript фіксує очікувану користь: майбутній `linux-arm64` = `+1` пакет + `+1` CI-job без зміни тулчейна; матриця підтвердила роботу в CI (`ubuntu-latest`, build-binaries job, 2m10s).
* Bad, because `cargo-zigbuild` додає залежність від `zig` і є менш зрілим інструментом; на машині розробника для тестування musl-збірок локально довелося встановити `zig` через `brew install zig` та `cargo install --locked cargo-zigbuild`.

## More Information

Файл: `.github/workflows/npm-publish.yml`, job `build-binaries`, matrix `os: ubuntu-latest`, команда `cargo zigbuild --release --target x86_64-unknown-linux-musl`. Локальна верифікація: `cargo zigbuild` зібрав `ELF x86-64 statically linked` бінарник; `npm pack --dry-run` у `packages/mt-linux-x64/` показав коректний вміст.

## ADR Windows-сумісність закладена наперед у резолвері; підпакет відкладено

## Context and Problem Statement

Після фіксації двох платформних підпакетів (`darwin-arm64`, `linux-x64`) постало питання: чи закладати Windows-сумісність у поточний рефакторинг, чи лишити як повністю окремий крок.

## Considered Options

* Закласти `.exe`-суфікс у резолвер зараз; сам `@7n/mt-win32-x64` і CI-job — після рефакторингу (add-only)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Закласти `.exe`-суфікс у резолвер зараз; підпакет — пізніше", because аналіз показав: Rust-код уже path-портабельний (нормалізує `\`→`/` на рядку 273 `lib.rs`, `lines()` коректно обробляє `\r\n`); тому єдина зміна, яка потрібна *наперед* — суфікс у резолвері, решта = add-only.

### Consequences

* Good, because transcript фіксує очікувану користь: додавання Windows пізніше = `+1` підпакет (`@7n/mt-win32-x64`, поле `os:["win32"],cpu:["x64"]`) + `+1` CI-job `windows-latest`, нуль змін у логіці.
* Bad, because тестування `git worktree list` на Windows-раннері та worktree-матчинг потребуватимуть реальної перевірки при додаванні — цей ризик свідомо відкладено.

## More Information

Файл: `npm/lib/core/scanner-bin.mjs`, функція `binName(platform)` — повертає `'mt-scanner.exe'` коли `platform === 'win32'`, інакше `'mt-scanner'`. Spec §6.5: «Додавання Windows пізніше = add-only». Рядок `lib.rs:273`: `rel.to_string_lossy().replace('\\', "/")` — підтверджує наявну портабельність.
