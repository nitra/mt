## ADR `mt init` — ідемпотентна ініціалізація стану у вже існуючому worktree

## Context and Problem Statement
Worktree, створений через голий `worktree add` (без `mt init`), має ізоляцію, але не має flow-стану (MT file-presence state). Команди `mt audit`, `mt verify` та `mt done` читають стан першим кроком і повертають помилку без нього. Потрібен спосіб добрати стан до вже існуючого worktree без його видалення і повторного створення.

## Considered Options
* Вимагати видалення worktree і повторного запуску `mt init` з нуля
* Зробити `mt init` ідемпотентним: виявляти, що CWD вже є linked-worktree, пропускати `worktree add`, записувати лише MT file-presence state

## Decision Outcome
Chosen option: "ідемпотентний `mt init`", because `commands.mjs` реалізує перевірку `isLinkedWorktree(cwd)` (рядки 76–77) і пропускає `worktree add` коли вже всередині worktree, записуючи лише стан — без втрати незакомічених змін.

### Consequences
* Good, because transcript фіксує очікувану користь: `mt init` всередині worktree видав `flow: уже в worktree — не вкладаю новий`, після чого `mt audit` знайшов стан і відпрацював без помилок.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/scripts/dispatcher/lib/commands.mjs` (рядки 76–90, 99–117). Команда для відновлення стану: `cd .worktrees/<branch> && npx @nitra/cursor mt init <branch> "<опис>"`. Стан зберігається як sibling-файл `.worktrees/<branch>.mt-state.json` (не всередині worktree), що підтверджено `state-store.mjs`.

## ADR Scoped coverage gate через `git diff <base>` замість HEAD-diff

## Context and Problem Statement
Coverage gate у flow-турнікеті мав запускати повний Stryker/vitest-прогін по всіх файлах, що уповільнювало зворотній зв'язок при активній розробці. Потрібен режим `--changed`, що обмежує scope до файлів, змінених з моменту `mt init`.

## Considered Options
* `git diff HEAD` (нестейджені + стейджені) + `git diff --cached` (тільки стейджені)
* `git diff <base>` (working-tree vs base-commit без `..` чи `...`) + untracked через `git ls-files --others`

## Decision Outcome
Chosen option: "`git diff <base>` (working-tree vs base-commit) + untracked", because `git diff <base>` однаково ловить і закомічені від base, і незакомічені зміни — це гарантує правильний scope незалежно від того, чи executor вже закомітив кроки (рядки 35–57 `npm/scripts/lib/changed-files.mjs`).

### Consequences
* Good, because transcript фіксує очікувану користь: функція `collectChangedFilesSince` повертає однаковий scope при committed і uncommitted змінах, тест `'ловить і закомічене від base, і незакомічене — однаково'` проходить.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/scripts/lib/changed-files.mjs` (функції `collectChangedFiles`, `collectChangedFilesSince`). `collectChangedFilesSince` також перевіряє досяжність base через `git rev-parse --verify --quiet <base>^{commit}` і кидає з повідомленням `недосяжний` при помилці — fail-closed поведінка замість мовчазного порожнього scope. Використовується в `npm/rules/test/coverage/coverage.mjs:resolveChangedScope`.

## ADR `coverage --changed` не перезаписує COVERAGE.md

## Context and Problem Statement
У режимі `--changed` провайдери збирають метрики лише по підмножині файлів. Якщо ці часткові дані перезаписати в `COVERAGE.md`, повний звіт буде знищено — при наступному `mt verify` агент побачить хибні метрики.

## Considered Options
* Записувати partial-звіт у COVERAGE.md (і перезаписувати повний)
* Використовувати `--changed` лише як gate (exit-код), не чіпаючи COVERAGE.md

## Decision Outcome
Chosen option: "gate через exit-код без перезапису COVERAGE.md", because `npm/rules/test/coverage/coverage.mjs:runCoverageSteps` при `opts.changed === true` і успішному прогоні повертає `0` без запису файлу (рядки 262–266), зберігаючи повний звіт незмінним.

### Consequences
* Good, because transcript фіксує очікувану користь: тест `'changed + провайдер з даними → exit 0, але COVERAGE.md НЕ перезаписується'` проходить.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/rules/test/coverage/coverage.mjs` (функція `runCoverageSteps`, рядки 262–266), `npm/rules/test/coverage/tests/coverage.test.mjs` (describe `runCoverageSteps — --changed scope`). CLI-прапор: `npx @nitra/cursor coverage --changed`.
