## ADR Уніфікований `run_NNN.md` замість окремих файлів помилок і журналів

## Context and Problem Statement
Система потребувала окремих файлів для кожного типу події вузла: `error.md` для помилок, `repair_history.md` для спроб інженера, `outputs.md` для результатів. Це плодило різні структури і ускладнювало читання стану вузла.

## Considered Options
* Окремі файли: `error.md`, `repair_history.md`, `outputs.md`
* Єдиний `run_NNN.md` на кожну спробу будь-якого актора

## Decision Outcome
Chosen option: "Єдиний `run_NNN.md` на кожну спробу", because будь-яка спроба — агента, інженера, людини або аудитора — має ту саму структуру: фронтматер з `created_at`, `actor`, `result`; секція `## Reasoning` (вільний текст); опційні `## Script` (exit code, stderr) і `## Ref` (посилання на результат). Успішний запуск додатково породжує immutable `outputs_NNN.md`. Поле `actor: agent | engineer | human | auditor` забезпечує розрізнення без окремих файлів.

### Consequences
* Good, because transcript фіксує очікувану користь: одна схема для всіх типів акторів, стан вузла читається скануванням `run_*.md` без розбору різних форматів.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/docs/mt.md`. Фронтматер: `created_at`, `actor` (`agent|engineer|human|auditor`), `result` (`success|failed`), опційно `worktree`. NNN генерується wrapper-скриптом: `count(run_*.md) + 1`, zero-padded до 3 цифр.

## ADR Всі файли вузла immutable — жодних append-only

## Context and Problem Statement
Початковий дизайн мав append-only файли (`outputs.md`, `error.md`, `repair_history.md`), де нові секції дописувались до існуючих. Це суперечило загальному принципу що файлова система є незмінним WAL-сховищем стану.

## Considered Options
* Append-only файли (нові секції дописуються)
* Immutable numbered файли (новий файл на кожну подію)

## Decision Outcome
Chosen option: "Immutable numbered файли", because принцип "всі файли під час роботи immutable" стосується всіх файлів без винятку — кожна спроба (`run_001.md`, `run_002.md`) і кожен результат (`outputs_001.md`) є окремим незмінним файлом. Стан відновлюється скануванням файлів, а не читанням мутованого стану.

### Consequences
* Good, because transcript фіксує очікувану користь: crash recovery через сканування `run_*.md` без `outputs_*.md`; відсутність ризику корупції при паралельному записі.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/docs/mt.md`. Immutable після старту ворктрі: `task.md`, `run_NNN.md`, `outputs_NNN.md`. Sentinel `invalidated` — порожній файл, сама наявність = знедійснений вузол.

## ADR Видалення `ops/` і `patches/` — crash recovery ігнорується

## Context and Problem Statement
Дизайн містив `ops/spawn-plan-<ts>.md`, `ops/spawn-fact-<ts>.md`, `ops/kill-plan-<ts>.md`, `ops/kill-fact-<ts>.md` і `patches/NNN-plan.md`, `patches/NNN-fact.md` — для відновлення після збоїв за патерном plan→action→fact. Це добавляло директорії і файли без підтвердженої потреби.

## Considered Options
* Зберегти `ops/` і `patches/` для plan/fact crash recovery
* Видалити їх, ігноруючи сценарій часткового збою

## Decision Outcome
Chosen option: "Видалити `ops/` і `patches/`", because spawn відбувається всередині ворктрі і мержиться атомарно — якщо ворктрі не замерджений, граф залишається чистим. Сценарій часткового збою при spawn визнаний неактуальним для поточного контексту. Reasoning інженера тепер міститься в `## Reasoning` секції `run_NNN.md` замість `patches/NNN-plan.md`.

### Consequences
* Good, because transcript фіксує очікувану користь: мінімальна файлова структура вузла — лише `task.md`, `invalidated`, `run_*.md`, `outputs_*.md`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/docs/mt.md`. Видалені директорії: `ops/`, `patches/`. Каскад інвалідації перенесений у `mt kill` (kill процесів + delete worktree + `invalidated` для наступників).

## ADR Git post-merge hook як тригер оркестратора

## Context and Problem Statement
Після завершення вузла і merge ворктрі система має запускати наступні розблоковані вузли. Питання: хто тригерить `mt run --auto` — людина вручну, демон-процес, чи git hook?

## Considered Options
* Людина вручну запускає кожен вузол (`mt run <path>`) після кожного merge
* `mt run` як довготривалий демон що сам чекає merge і продовжує
* Git `post-merge` hook що тригерить `mt run --auto`

## Decision Outcome
Chosen option: "Git `post-merge` hook", because це найчистіший варіант: `mt run` залишається одноразовою командою, wrapper авто-мержить ворктрі після успіху агента, а git hook автоматично запускає `mt run --auto` для сканування і старту всіх розблокованих наступників. Людина тільки запускає кореневий вузол.

### Consequences
* Good, because transcript фіксує очікувану користь: повністю автоматичний ланцюг від кореневого вузла до завершення без ручних кроків.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/docs/mt.md`. Ланцюг: агент завершує → wrapper пише `run_NNN.md` → wrapper `git merge` → `post-merge` hook → `mt run --auto` (сканує граф, запускає вузли з `deps=resolved`). CLI: `mt run --auto`.

## ADR Механізм аудиту — агент-ініційований, 3 цикли до ескалації

## Context and Problem Statement
Потрібен механізм незалежної перевірки що вузол дійсно виконав своє завдання (`## Done when`) перед тим як граф рухається далі. Питання: хто вмикає аудит, що він перевіряє, і що відбувається при провалі.

## Considered Options
* `audit: true` в `task.md` — батько або людина вирішує при spawn
* Агент сам вирішує після виконання через `mt audit <path>`

## Decision Outcome
Chosen option: "Агент ініціює через `mt audit <path>`", because агент сам знає чи він впевнений у результаті — він викликає `mt audit` замість `mt done` коли хоче незалежну перевірку. Аудитор бачить `task.md` (включно з `## Done when`) і всі зміни агента у ворктрі, пише `run_(NNN+1).md` з `actor: auditor`. При `result: failed` агент отримує зауваження і перезапускається. Після **3 провалів аудиту поспіль** — STOP, чекає людського втручання.

### Consequences
* Good, because transcript фіксує очікувану користь: незалежна якісна перевірка без обов'язкового накладення на кожен вузол; 3-цикловий ліміт запобігає нескінченним петлям.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/docs/mt.md`. CLI: `mt audit <path>` — запускає аудитора в контексті поточного ворктрі. Лічильник провалів: кількість `run_*.md` з `actor: auditor` і `result: failed`. `audit_model` в `.n-cursor.json` — дешевша модель для аудиту.
