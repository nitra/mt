## ADR Рекурсивний складений ОАГ із динамічним розкладом вузлів

## Context and Problem Statement

Потрібна структура для системи де оркестратор розбиває задачі на підзадачі під час виконання. Вузли верхнього рівня не знають наперед чи буде підзадача атомарною операцією чи розкладеться у цілий підграф.

## Considered Options

* Рекурсивний складений ОАГ — вузол або атомарний або містить власний підграф; рішення динамічне при запуску
* Статичний ОАГ — структура задається наперед
* 3D-граф — термін описує просторове відображення, а не структуру зв'язків

## Decision Outcome

Chosen option: "Рекурсивний складений ОАГ", because батьківський вузол завжди бачить однаковий інтерфейс незалежно від внутрішньої складності дочірнього, а агент вирішує тип при запуску на основі вхідних даних.

### Consequences

* Good, because атомарний вузол можна "розкрити" у підграф без змін у батьківському графі — повна замінюваність.
* Good, because динамічний spawn дозволяє агенту додавати нові дочірні вузли під час виконання.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Ребра несуть дані (outputs → inputs). Кілька exit-вузлів — їх виходи зливаються в outputs батька. Топологія розподілена: кожен дочірній `task.md` містить `deps:` зі списком попередників-siblings. Центрального файлу графу немає. Файли: `tasks/<node-id>/task.md`, `вхідні.md`, `вихідні.md`.

## ADR Файловий стан, append-only інваріант і принцип план → дія → факт

## Context and Problem Statement

Система виконує задачі через LLM-агентів у розподіленому середовищі де агенти можуть падати або зависати. Потрібен механізм зберігання стану який є LLM-friendly, гарантує відновлення після збою і не вимагає транзакційної бази даних.

## Considered Options

* Файли як state store із append-only інваріантом і принципом план/факт
* Централізована база даних (PostgreSQL, SQLite)
* JSON-файли що змінюються

## Decision Outcome

Chosen option: "Файли як state store із append-only інваріантом і принципом план/факт", because файлова система дає безкоштовну персистентність і git-аудит, а append-only усуває race condition при паралельному виконанні у ворктрі.

### Consequences

* Good, because стан вузла визначається наявністю файлів — немає single point of failure.
* Good, because `*-план.md` + `*-факт.md` дозволяє відновити будь-яку незавершену операцію після збою.
* Good, because Markdown+YAML є природним форматом для LLM.
* Bad, because scan без індексу при великих графах дорого.

## More Information

Append-only: файли тільки створюються, ніколи не змінюються. Операція: (1) `*-план.md` → (2) виконання → (3) `*-факт.md`. Відновлення: `scan tasks/**/*-план.md` → якщо `*-факт.md` відсутній → відновити. Sentinel-файл `invalidated` визначає стан через наявність.

## ADR Ворктрі як межа атомарності для паралельного виконання агентів

## Context and Problem Statement

Незалежні вузли ОАГ мають виконуватись паралельно. Водночас потрібна гарантія що наступник не отримає частково записаний стан попередника.

## Considered Options

* Git worktree на кожного агента — ізоляція через git, merge як єдина точка рішення
* Файлові блокування (flock)
* Черговий запуск без паралельності

## Decision Outcome

Chosen option: "Git worktree на кожного агента", because ворктрі дає файлову ізоляцію без механізмів блокування, а merge є природною єдиною точкою де оркестратор оцінює результат.

### Consequences

* Good, because race condition неможливий за архітектурою: наступник стартує лише після merge попередника.
* Good, because незалежні вузли пишуть у різні директорії → git merge завжди чистий.
* Good, because конфлікт при злитті обробляється симетрично до помилки вузла: spawn АгентМедіатор.
* Bad, because жорсткий ліміт паралельних ворктрі на MacBook обмежує реальну паралельність.

## More Information

Naming convention: `.worktrees/<node-id>-run/` для виконання вузла. Протокол патчу залежного вузла: (1) `патчі/NNN-план.md`, (2) kill залежних топологічно, (3) патч, (4) `патчі/NNN-факт.md`, (5) рестарт. Кожен патч інженера — у своєму ворктрі, видаляється після завершення.
