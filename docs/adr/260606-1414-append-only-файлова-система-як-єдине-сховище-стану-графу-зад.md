## ADR Append-only файлова система як єдине сховище стану графу задач

## Context and Problem Statement
Система динамічного самомодифікованого графу задач потребує надійного механізму зберігання стану, що витримує збої агентів, паралельне виконання у git-ворктрі та відновлення без зовнішнього координатора.

## Considered Options
* Append-only файлова система: файли лише створюються, стан визначається наявністю файлів, ворктрі є межею атомарності
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Append-only файлова система з ворктрі-межею атомарності", because це усуває гонки архітектурно (наступник стартує лише після merge), дає безкоштовну персистентність і аудит через git history, і дозволяє відновлення через порівняння `*-plan.md` / `*-fact.md`.

### Consequences
* Good, because transcript фіксує очікувану користь: ворктрі-межа робить race condition неможливим за визначенням; `*-plan.md` без `*-fact.md` — надійний сигнал незавершеної операції для відновлення після збою.
* Bad, because scan усіх `*-plan.md` при великих графах — дорого без індексу; жорсткий ліміт паралельних ворктрі на MacBook обмежує реальну паралельність.

## More Information
Файлова структура вузла: `tasks/<node-id>/task.md`, `outputs.md`, `error.md`, `ops/spawn-plan-<ts>.md`, `ops/spawn-fact-<ts>.md`, `patches/001-plan.md`, `patches/001-fact.md`, `repair_history.md`, `repair_context.md`. Стан вузла — implicit через наявність файлів. Інваріант: до створення ворктрі файли редагуються вільно; після — лише нові файли.

## ADR Plan → Action → Fact як універсальний WAL-патерн

## Context and Problem Statement
Агенти, інженери та оркестратор виконують операції (spawn, kill, patch) які можуть перерватись на будь-якому кроці. Потрібен механізм відновлення після збою без зовнішньої транзакційної системи.

## Considered Options
* Write-ahead log через пару файлів `*-plan.md` / `*-fact.md`: намір записується до дії, результат — після
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Plan → Action → Fact через файлові пари", because відновлення зводиться до сканування `*-plan.md` без відповідного `*-fact.md` — незалежно від типу операції і рівня ієрархії.

### Consequences
* Good, because transcript фіксує очікувану користь: однаковий патерн для spawn, kill, patch і виконання агента — жодних спеціальних механізмів відновлення на тип операції.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Застосовується до: `ops/spawn-plan-<ts>.md` / `ops/spawn-fact-<ts>.md`, `ops/kill-plan-<ts>.md` / `ops/kill-fact-<ts>.md`, `patches/001-plan.md` / `patches/001-fact.md`. Агент як операція: `task.md` = plan, `outputs.md` або `error.md` = fact.

## ADR Топологія графу в `deps:` кожного дочірнього вузла

## Context and Problem Statement
Граф задач динамічно розширюється під час виконання (агент може додавати нові вузли в будь-який момент). Потрібно зберігати топологію так, щоб оркестратор міг відновити повний граф без центрального файлу, а динамічний spawn не вимагав оновлення жодного спільного стану.

## Considered Options
* Центральний `graph.md` у батьківському вузлі з переліком вузлів і ребер
* `deps:` поле у `task.md` кожного дочірнього: кожен вузол знає своїх попередників

## Decision Outcome
Chosen option: "`deps:` у `task.md` кожного дочірнього", because це дає універсальність — динамічний spawn просто створює новий `task.md` з `deps:`, жодного центрального файлу оновлювати не потрібно; оркестратор відновлює повний граф скануванням усіх `task.md`.

### Consequences
* Good, because transcript фіксує очікувану користь: динамічний spawn = новий файл без координації; оркестратор stateless при реконструкції графу.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
`deps` містить лише `id` siblings (сусідів у тій самій директорії батька). Поле `port` прибрано — маппінг даних відбувається через `ref:` у секції `## Inputs` файлу `task.md`. Агент-автор визначає структуру підграфу виключно самостійно.

## ADR Об'єднання `task.md` і `inputs.md` в єдиний файл вузла

## Context and Problem Statement
Спочатку планувались два окремих файли: `task.md` (місія) і `inputs.md` (посилання на вхідні дані). Розділення ускладнює читання агентом і збільшує кількість файлів при spawn.

## Considered Options
* Окремі файли `task.md` і `inputs.md`
* Єдиний файл `task.md` з секцією `## Inputs`

## Decision Outcome
Chosen option: "Єдиний `task.md` з секцією `## Inputs`", because агент читає один файл замість двох, менше операцій при spawn, task і дані завжди разом у цілісному контексті.

### Consequences
* Good, because transcript фіксує очікувану користь: спрощення spawn-протоколу і читання агентом.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Схема `task.md`: frontmatter (`created_at`, `parent`, `deps`), секції `## Task`, `## Done when`, `## Inputs` (підсекції `### name` з `ref:` або inline). Файл immutable після створення ворктрі. До ворктрі — вільно редагується; якщо ворктрі існує — kill → редагуй → restart.

## ADR Межа immutability: ворктрі як точка входу інваріанту

## Context and Problem Statement
Append-only інваріант ("файли тільки створюються") потрібен для коректності паралельного виконання, але надто жорсткий для фази планування, коли агент ще не запущений і файли можна вільно уточнювати.

## Considered Options
* Файли immutable з моменту створення (повний append-only завжди)
* Файли вільно редагуються до створення ворктрі, після — тільки нові файли

## Decision Outcome
Chosen option: "Межа immutability = створення git worktree", because ворктрі = старт роботи агента; до цього моменту жоден агент файли не читає, тому зміни безпечні та не ламають інваріант системи.

### Consequences
* Good, because transcript фіксує очікувану користь: інженер може виправити `task.md` без re-spawn якщо ворктрі ще немає; kill ворктрі → редагуй → restart — єдиний протокол для будь-якої зміни.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Протокол: до ворктрі — `task.md` вільно редагується або видаляється. Після ворктрі — kill worktree → вільне редагування → restart. `repair_context.md` і `ops/*-plan.md` immutable завжди (пишуться одразу перед дією).
