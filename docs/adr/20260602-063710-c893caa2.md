## Суть проблеми — рядок 111

```js
// trace.mjs:111 — ЗАРАЗ (баг)
const analysis = analyze(artifacts, target => exists(join(root, target)))
```

`analyze` отримує `target = fm[f]` — те, що записано у front-matter файлу. `exists(join(root, target))` резолвить цей рядок **відносно кореня репо**.

## Конкретний кейс, який падає

Файл:
```
docs/specs/2026-06-01-foo.md
```

Front-matter:
```yaml
plan: ../plans/2026-06-01-foo.md
```

Автор мав на увазі:
`docs/specs/` + `../plans/foo.md` = `docs/plans/foo.md` ✅

Але trace обчислює:
`join("/…/cursor", "../plans/foo.md")` = `/…/plans/foo.md` — **ВИЩЕ кореня репо** ❌

Файл фізично є, але trace каже «РОЗРИВ — файл відсутній».

## Чому `../` у лінках — норма, а не помилка

Лінки між `docs/specs/`, `docs/plans/`, `docs/adr/` природно file-relative — так їх рендерять GitHub/Obsidian/будь-який MD-переглядач. Вимагати repo-root-relative лінки (`docs/plans/…`) — означало б зламати превʼю у редакторах. Тому конвенція правильна, а `trace` просто не знає теки джерела.

## Поле `flow:` — окремий кейс

```yaml
flow: ../../.worktrees/flow-cwd-state-resolution.mt-state.json
```

Цей файл — **runtime-стан**, живе у `.worktrees/`, **gitignored**. Навіть із правильним резолвингом він буде відсутній у чистому checkout і в CI. Тому він завжди дасть «розрив» — незалежно від фіксу.

Звідси три варіанти, які я запропонував:
- **A** — `flow:` показувати у виводі, але не вважати відсутність розривом ланцюга (runtime-артефакт ≠ doc-артефакт).
- **B** — прибрати `flow:` із `LINK_FIELDS` зовсім.
- **C** — залишити як є, просто виправити відносний резолв — `flow:` і далі шумітиме в CI.
