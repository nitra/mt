## ADR Конвеєрна оркестрація docgen — детермінований JS-оркестратор + локальна LLM

## Context and Problem Statement
При one-shot генерації документації локальними моделями (gemma3:4b, gemma4:4b) виявлено три стабільних класи помилок: витік деталей реалізації (stdlib, regex, приватні імена), галюцинації у секції «Гарантії поведінки», та пропуск крайових деталей (наприклад, «корінь не перевіряється»). Ці помилки не залежали від транспорту (прямий ollama vs pi) — вони є стелею архітектури «один промпт».

## Considered Options
* One-shot промпт з посиленим system-prompt (варіанти A, B, C2 з бенчмарків)
* Конвеєрна оркестрація: детермінований JS-екстрактор (Stage 0) + точкові LLM-промпти на секцію (Stage 1) + детермінована зборка (Stage 3)

## Decision Outcome
Chosen option: "Конвеєрна оркестрація з JS-оркестратором як точкою входу", because JS-оркестратор усуває класи помилок, що виникають не через якість прози, а через неповний контекст — імена, skip-маркери, read-only/throws — які детерміновано витягуються без участі LLM (Stage 0), а модель отримує лише вузьку задачу «перефразуй ці факти».

### Consequences
* Good, because transcript фіксує очікувану користь: Stage 0 (`docgen-extract.mjs`) успішно витягнув JSDoc, класифікував stdlib vs internal, маркери `readOnly`/`catchesErrors`/`skips`/`caches` — заземлення, яке прибирає галюцинації й витоки без жодного токена LLM.
* Good, because інверсія керування дозволяє `gemma3:4b` (швидша, 100% GPU, ~20 tok/s) досягати якості `gemma4:4b` (~11 tok/s, частковий офлоад), уникаючи платежу за важку модель.
* Good, because KV-cache ollama на стабільному `system + код` префіксі амортизує вартість повторного інгесту між секційними викликами в межах одного файлу.
* Bad, because реалізація складніша за один промпт: `docgen-extract.mjs` + `docgen-prompts.mjs` + `docgen-gen.mjs` + зборка — більше точок відмови; Stage 0 прив'язаний до JS/MJS, `.vue`/`.py` деградують до fallback.
