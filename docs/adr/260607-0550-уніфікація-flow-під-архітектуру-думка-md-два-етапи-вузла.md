## ADR Уніфікація `flow` під архітектуру `npm/docs/mt.md` — два етапи вузла

## Context and Problem Statement
Існуючий `mt` (Dual-Mode Dispatcher) мав власний стан у MT file-presence state, команди `init/spec/plan/verify/release/review/gate/run/resume/cancel/repair` і артефакти в `docs/specs/`, `docs/plans/`. Архітектура `npm/docs/mt.md` вводить task-based DAG з файловим сховищем стану (`tasks/<node>/task.md`, `run_NNN.md`, `outputs_NNN.md`), де `graph` є зовнішнім оркестратором, а `flow` повинен стати внутрішнім протоколом вузла. Постала потреба привести `flow` у відповідність до нової архітектури та прибрати дублювання з `graph`.

## Considered Options
* Зберегти `flow` як є, додавши `graph` поряд (дублювання worktree lifecycle)
* Повністю видалити `flow`, замінивши командами `graph`
* Переформатувати `flow` під `npm/docs/mt.md`: розділити на Stage 1 (planning) і Stage 2 (execution), видалити команди що дублюються з `graph`

## Decision Outcome
Chosen option: "Переформатувати `flow` під `npm/docs/mt.md` з розділенням на два етапи", because `graph` бере оркестрування ззовні (worktrees, deps, merge), а `flow` залишається протоколом всередині вузла — це усуває дублювання і зберігає розподіл відповідальностей.

### Consequences
* Good, because transcript фіксує очікувану користь: усунення MT file-presence state і `docs/specs/`/`docs/plans/` на користь єдиного файлового контракту `tasks/<node>/`; єдина точка стану — файли, без ризику desync між MT file-presence state і реальністю.
* Good, because `mt plan` (mode: human/agent) з атрибутом у `task.md` дає гнучкий контроль над рівнем автономності без зміни коду.
* Bad, because transcript не містить підтверджених негативних наслідків; видалення 9 команд (`mt init/spec/run/cancel/resume/repair/review/gate/release`) є зворотньо несумісною зміною для існуючих workflow.

## More Information

**Stage 1 — `mt plan`:**
- `mode: human` (default) → людина вирішує atomic|composite; після Stage 1 стан вузла = `plan-pending`
- `mode: agent` → агент вирішує; після Stage 1 одразу Stage 2 (no `plan-pending`)
- Атомарний → `plan_001.md` (numbered, extensible front-matter)
- Складений → дочірні `task.md` + `plan_001.md` з `## Sub-tasks` → агент явно викликає `mt spawn`

**Stage 2 — `mt verify`:**
- Окремий LLM-процес (`mt verify`)
- Tools: `run_command(cmd)` + `flow_audit(criterion, files)`
- Читає: `task.md ## Done when` + `outputs_NNN.md` + `plan_001.md`
- Пише: `verify_001.md` (numbered); exit 0=PASS / 1=FAIL
- Після: виконавець-агент → `mt done | mt failed`

**`flow audit`** (новий): `mt audit --criterion "..." --file path` → stdout JSON `{verdict: PASS|FAIL, reason}`

**Новий стан вузла:** `plan-pending` — є `plan_*.md`, немає `outputs_*.md`, немає активного worktree, `mode: human`.

Файл з рішеннями: `npm/docs/mt.md` § "Інтеграція з `mt`".
