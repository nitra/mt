# Spec: інверсія делегування у vision.md

> Результат брейнштормінг-сесії 2026-07-09. Нормативні рішення нижче погоджені; чернетки текстів — пропозиція для правки [npm/docs/vision.md](../../../npm/docs/vision.md) і дотичних доків.

## Мотивація

Чинна теза vision.md побудована навколо **рівноправності**: «людина і агент — взаємозамінні виконавці одного вузла задачі». Це формулювання зі світу трекерів: хтось (людина) ставить задачі, виконавці (люди чи ШІ) їх виконують.

Нова місія інвертує напрям делегування:

> Не люди запускають ШІ для виконання задач, а **ШІ запускає ШІ** для досягнення **мети засновника**. Людина зʼявляється у виконанні лише там, де ШІ сам розуміє, що потрібна людина — і призначає людину **відповідної кваліфікації**.

Інверсія вже латентно присутня в архітектурі (самомодифікований граф; `retry ladder → engineer → людина` у graph.md), але не є центром тези. Spec закріплює її як центр.

## Рішення (погоджено на брейнштормінгу)

| # | Питання | Рішення |
| --- | --- | --- |
| 1 | Глибина зміни vision.md | **Нова теза** — переписати центр документа; пʼять крос-вимірів лишаються, але перепідпорядковуються інверсії |
| 2 | Де людина обовʼязкова by design | Усі три ролі: **джерело мети й цінностей**, **approval-гейти**, **вершина ескалації / кваліфікаційні вузли** |
| 3 | Retro-дані для вибору людини | **Вільно використовувати** — audit trail стає легітимним входом оркестрації; принцип довіри retro.md переписується |
| 4 | Джерело кваліфікації людини | **Гібрид**: декларований профіль (ролі/skills у membership) + уточнення фактичною retro-історією; симетрично до агентів (surface-профіль + impact-відбір) |
| 5 | Вплив на roadmap | **Перетрусити**: декомпозиція мети і оркестрація ШІ→ШІ піднімаються вище; частина чинних milestones відкладається |

## Нова теза (чернетка для vision.md)

Заміна розділу «Теза»:

> Для управління проєктами створено багато прекрасних продуктів — від Trello до Jira. Усі вони проєктувались у світі, де задачі ставили люди і виконували люди; агентні надбудови (Linear agents, GitHub Agent HQ) лише додали ШІ-асистентів **усередину людського процесу**.
>
> MT будується для протилежного світу: **засновник задає мету і цінності — ШІ досягає їх**. ШІ-оркестратор декомпозує мету в самомодифікований граф задач, запускає ШІ-виконавців, оцінює результати і перебудовує граф. Людина в цьому процесі — не диспетчер задач, а:
>
> 1. **Джерело мети.** «Навіщо», обмеження і пріоритети задає людина; оркестратор не перевизначає їх.
> 2. **Підписант гейтів.** Критичні переходи (деструктивні дії, план, аудит) вимагають криптографічного підпису людини — контроль зміщується з постановки задач на гейти.
> 3. **Призначуваний ресурс.** Там, де ШІ сам визнає, що потрібна людина (кваліфікація, фізична дія, вичерпаний retry ladder), він **призначає людину відповідної кваліфікації** — з тим самим контрактом вузла, що й для агента.
>
> Взаємозамінність людини й агента на вузлі (той самий життєвий цикл, бюджети, аудит) — не мета, а **передумова** цієї інверсії: щоб оркестратор міг призначити будь-кого, контракт має бути спільним.
>
> Git лишається субстратом (надійність, офлайн, audit trail, відсутність lock-in), але не інтерфейсом.

Пʼять крос-вимірів зберігаються; вступне речення розділу змінюється на: «Інверсія делегування вимагає пʼяти крос-властивостей платформи».

## Зміни по документах

### vision.md

- Теза — за чернеткою вище.
- Розділ «Ніша»: головний differentiator загострюється — «Linear agents / Agent HQ — агенти-асистенти всередині людського процесу; MT — **людський внесок всередині агентного процесу**». Рядки порівнянь лишаються.
- «Мета-цикл»: абзац про принцип довіри замінюється (див. retro.md нижче) — retro тепер і двигун матчингу «задача → виконавець», для людей і агентів симетрично.
- «Мапінг на архітектуру»: додати рядок «Інверсія делегування → graph.md (декомпозиція, retry ladder), access.md (кваліфікаційні профілі), retro.md (матчинг)».
- «Наслідки для фокусу»: перший продукт переформульовується з «mission control для агентів» на «автономне досягнення мети з людиною на гейтах»; mission control — компонент, не продукт.

### retro.md

Нормативні принципи 1–2 переписуються:

- **Було:** «працює на виконавця, opt-in; не інструмент нагляду, жодних персональних агрегатів для інших».
- **Стає:** audit trail — легітимний вхід оркестрації. Оркестратор вільно використовує retro-агрегати для матчингу «задача → виконавець» (людина чи агент). Пропозиції самопокращення лишаються адресованими виконавцю, але приватність метрик більше не є нормативною межею; межа тепер інша — **мету й цінності задає людина, а вимірюється виконання задач, не людина поза задачами**.
- Наявні компенсатори лишаються і поширюються на людей: порівняння лише в межах класу задач, `impact_min_runs` як поріг confidence, спірні зрізи — через штатний аудит (anti-gaming з retro.md).
- Розділ «Заохочення» для людей звужує розрив із агентами: заохочення = відбір діє в обидва боки (кращий підтверджений impact у класі задач → більше задач цього класу).

### access.md

- Membership розширюється **кваліфікаційним профілем** людини: декларовані skills/домени (аналог surface-профілю агента). API: поле у `invite`/`PATCH role` або окремий `PATCH profile`.
- Профіль — стартова точка для матчингу (cold start); далі уточнюється retro-історією. Оркестратор поєднує обидва джерела.

### graph.md (мінімально)

- Ескалація на людину описується не як «ручний фолбек», а як **призначення**: вузол отримує `assignee` з добіркою кандидатів за кваліфікацією + обґрунтуванням вибору в run-журналі (аудитовність призначення).

### roadmap.md — новий порядок

Принцип перетрусу: усе, що потрібно для циклу «мета → декомпозиція → ШІ запускає ШІ → ескалація на людину», йде поперед поверхонь і i18n. Retro підтягується вгору як джерело даних матчингу (MVP після M0 вже заявлений у чинному retro.md).

| Було | Стає | Зміст |
| --- | --- | --- |
| M0 | **M0** | Dogfood ядра — без змін, передумова всього |
| M1 | **M1** | agent-server локально — без змін: без нього ШІ не запускає ШІ інтерактивно |
| M5 | **M2** | **Оркестратор мети + retro-MVP**: `goal.md` → декомпозиція в граф → spawn ШІ→ШІ → ескалація-призначення людини (локально, гейти через `mt attach`); retro-аналіз на локальних даних як вхід матчингу. Demo: мета одним файлом → піддерево задач виконано агентами, один вузол ескальовано на людину з обґрунтуванням |
| M2 | **M3** | Mission control: relay + телефон-approver + **кваліфікаційні профілі в membership** — гейти й призначення людей на всіх пристроях |
| M3 | **M4** | Dashboard і поверхні |
| M4 | **M5** | Файловий шар i18n — відкладається: не критичний для інверсії |

Demo-критерії M0, M1, M3–M5 успадковуються від чинних; новий M2 отримує власний demo-критерій (вище).

## Ризики

1. **Репутаційний/командний.** «Вільне» використання персональних метрик — чутлива зміна для майбутніх команд-користувачів. Компенсатор: вимірюється виконання задач у класі, не людина; аудитовність кожного призначення; політика компенсацій лишається поза платформою.
2. **Передчасна автономія.** Оркестратор мети (новий M2) до relay-гейтів (M3) означає: у M2 гейти лише локальні. Компенсатор: M2 — single-user dogfood, деструктивні дії за замовчуванням вимагають локального підпису.
3. **Cold start матчингу.** До накопичення retro-даних призначення спирається лише на декларований профіль — якість вибору «відповідної кваліфікації» зростає з часом; це чесно зазначити у vision.

## Ревізія 2026-07-12 (пост-mandates.md)

Після цього spec у канон увійшла глава [npm/docs/architecture/mandates.md](../../../npm/docs/architecture/mandates.md) (PR #42, #44), яка дає точніші відповіді на частину рішень вище. Актуальний статус по пунктах:

| Рішення spec | Статус | Чим замінено |
| --- | --- | --- |
| 1. Нова теза vision.md (інверсія делегування) | **чинне, не реалізоване** | доповнюється: mandates.md формалізує «людина = власник рішень свого горизонту» — інверсія відповідає «хто ініціює», мандати — «хто вирішує»; чернетку тези варто оновити посиланням на карту мандатів |
| 2. Людина обовʼязкова у трьох ролях | **чинне** | узгоджується з mandates.md (джерело мети = кореневий мандат; гейти = ApprovalResponse; вершина ескалації = escalation_chain) |
| 3. Retro-дані — вільний вхід матчингу, приватність метрик не нормативна | **замінено** | синтез C+E у mandates.md: матчинг живиться **positive-only досьє підтверджень** в org-репо профілів (пише лише агрегатор, evidence-посилання, dispute через підпис власника мандата); негативних оцінок не існує ніде; принципи retro.md «не інструмент нагляду» збережені — переписувати їх не треба |
| 4. Кваліфікація = декларований профіль у membership + retro-історія | **замінено** | профілі живуть не в membership relay, а в org-репо `people-profiles`; декларована частина — `growth_edge` (пише людина), підтверджена — `competencies` (пише агрегатор); cold start закривається probe-квотою |
| 5. Перетрус roadmap (retro/оркестратор → M2) | **застаріло** | M2 (mission control) уже в активній реалізації за старою нумерацією (relay, approvals, handoff — див. npm/docs/log.md 2026-07-11..12); з'явився M6 (мандати, фазове впровадження). Оркестратор мети лишається кандидатом на окремий milestone — планувати від поточного стану, не перетрусом заднім числом |

Наступні кроки з попереднього розділу відповідно звужуються до кроку 1 (теза vision.md, з урахуванням mandates.md); кроки 2–4 не виконувати.
