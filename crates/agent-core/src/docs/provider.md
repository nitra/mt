## Огляд

Файл задає нейтральний контракт LLM-провайдера для запитів із історією повідомлень, tools, потоковими подіями, фінальною відповіддю та помилками транспорту без прив’язки публічного API до конкретного SDK: типи `async-openai` не протікають за межі provider-реалізації. Reference transport OpenAI-compatible Chat Completions через `async-openai` має бути окремою реалізацією trait-а в наступній задачі. Для офлайн-тестів і CI без мережі доступні `MockProvider` і `scripted`, які відтворюють відповіді провайдера. За певних помилок контракт допускає порожнє значення (`null`) замість винятку.

## Поведінка

- `Role` позначає роль повідомлення в історії розмови.
- `ChatMessage` представляє одне повідомлення історії з текстом і, за потреби, прив’язкою до результату tool call-а.
- `system` створює системне повідомлення для інструкцій моделі.
- `user` створює повідомлення користувача.
- `assistant` створює повідомлення асистента.
- `tool_result` створює повідомлення з результатом виконання tool call-а.
- `ToolSpec` описує доступний моделі tool і його схему вхідних даних.
- `CompletionRequest` збирає модель, історію повідомлень і доступні tools для одного запиту до провайдера.
- `ToolCallRequest` представляє запит моделі на виконання tool-а.
- `StreamEvent` передає потокові події провайдера для live-відображення відповіді.
- `Completion` містить фінальний текст відповіді та запити на виконання tools, якщо модель їх потребує.
- `ProviderError` описує помилки транспорту провайдера або вичерпаний сценарій тестового провайдера.
- `Provider` задає нейтральний контракт LLM-провайдера без витоку типів конкретного SDK у публічний API.
- `MockProvider` надає скриптований провайдер для офлайн-тестів і CI без мережі та зберігає отримані запити для перевірок.
- `scripted` створює тестовий провайдер із наперед заданою послідовністю відповідей.

## Публічний API

- Role — позначає автора або джерело повідомлення в історії.
- ChatMessage — зберігає один запис розмови для подальшого контексту.
- system — задає службові інструкції, які спрямовують поведінку моделі.
- user — фіксує запит або повідомлення від користувача.
- assistant — зберігає відповідь моделі в історії діалогу.
- tool_result — передає результат виконаного tool-а назад у розмову.
- ToolSpec — описує доступний tool для моделі та правила формату його виклику.
- CompletionRequest — збирає дані, потрібні провайдеру для продовження відповіді.
- ToolCallRequest — представляє намір моделі викликати tool.
- StreamEvent — передає проміжні події відповіді для live-оновлення сесії.
- Completion — містить підсумок одного завершеного ходу моделі.
- ProviderError — описує збій під час звернення до LLM-провайдера.
- Provider — уніфікує роботу з LLM і віддає як live-події, так і фінальний результат.
- MockProvider — імітує LLM у тестах та CI, повертаючи наперед задані відповіді без мережі.
- scripted — створює mock-провайдер із послідовністю готових відповідей.

## Гарантії поведінки

- Read-only: не виконує операцій запису (ФС/БД).
- За певних помилок повертає порожнє значення (напр. `null`) замість винятку.
