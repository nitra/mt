## Огляд

`Agent`, `new`, `history`, `run_turn` і `AgentError` описують read-only agent loop: зберігається історія повідомлень, виконується один хід у схемі user message → provider → tool calls → фінальний текст, а кожен крок емітить `agent-protocol` через callback; Envelope з `seq`/`ts`/`node_hash` збирає `agent-server`. Патерн відповідає `pi_agent_rust`, а подієвий контракт — `runtime.md`. Tool calls виконуються через реєстр, а їхні результати повертаються в історію повідомлень. Компонент fail-safe: перехоплює помилки, не кидає винятків назовні й у деяких випадках повертає порожнє значення замість помилки.

## Поведінка

- `AgentError` — описує помилки ходу агента: помилки provider або перевищення ліміту tool-циклу.
- `Agent` — тримає стан одного run-а: provider, tools, model і історію повідомлень.
- `new` — створює агента з початковим system-повідомленням у history.
- `history` — повертає поточну історію повідомлень без зміни стану.
- `run_turn` — виконує один хід: додає user message, запитує provider, обробляє tool calls через реєстр, оновлює history, емітить події та повертає фінальний текст або помилку.

## Публічний API

- AgentError — помилка, яку кидають під час збою роботи агента.
- Agent — описує одного агента на один run: провайдер, каталог tools і накопичена історія.
- new — збирає агента з `system_prompt` як правилом поведінки та додає вузлові файли як контекстні повідомлення.
- history — зберігає й відтворює послідовність повідомлень і викликів у межах run-а.
- run_turn — виконує один крок діалогу до фінальної відповіді асистента, видаючи текст, виклики tools і їхні результати.

## Гарантії поведінки

- Read-only: не виконує операцій запису (ФС/БД).
- Перехоплює помилки і не пропускає винятків назовні (fail-safe).
- За певних помилок повертає порожнє значення (напр. `null`) замість винятку.
