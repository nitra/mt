## Огляд

Файл задає контракт tools агента: кожен tool має опис, JSON-схему параметрів, виконання та повертає успішний або неуспішний результат без винятків назовні. JSON-схема параметрів має бути derive через `schemars`, щоб схема походила з контракту даних, а не підтримувалась вручну. Реєстр потрібен, щоб зберігати внутрішні tools, підключати зовнішні MCP-tools через `register_external`, знаходити їх за іменем і віддавати провайдеру список специфікацій. Власну MCP-реалізацію файл не містить: транспорт постачає окремий шар (`// use rmcp;`). Файл є read-only і не пише у ФС/БД.

## Поведінка

- `ToolOutput` представляє результат виконання tool-а з ознакою успішності, коротким людським підсумком і повним вмістом відповіді.
- `success` створює успішний результат tool-а для подальшого відображення в подіях агента.
- `failure` створює неуспішний результат tool-а з порожнім вмістом, щоб передати збій без винятку назовні.
- `ToolError` представляє помилку виконання tool-а, зокрема невалідні аргументи або внутрішній збій.
- `Tool` задає спільну поведінку внутрішніх tools агента: опис, схему параметрів і виконання.
- `ToolRegistry` зберігає зареєстровані tools і надає їх викликачам у детермінованому порядку.
- `new` створює порожній реєстр tools.
- `register` додає tool до реєстру або замінює наявний tool з таким самим іменем.
- `register_external` підключає зовнішній MCP-tool як звичайний tool; власний MCP-транспорт тут свідомо не реалізується.
- `get` повертає зареєстрований tool за іменем або порожній результат, якщо такого tool-а немає.
- `specs` формує специфікації tools для провайдера з описами та JSON-схемами параметрів, отриманими з реалізацій tool-ів.

## Публічний API

- ToolOutput — передає підсумок виконання tool-а як успішний або неуспішний результат для події відповіді.
- success — створює успішний результат із коротким описом виконаної дії.
- failure — створює неуспішний результат із поясненням причини збою.
- ToolError — описує збій запуску tool-а через непридатні аргументи або помилку всередині виконання.
- Tool — представляє внутрішній tool агента з назвою, описом, схемою аргументів і виконуваною дією.
- ToolRegistry — зберігає набір доступних tool-ів, уже обмежений профілем поверхні та sandbox-стелею вузла.
- new — створює порожній реєстр для подальшого наповнення tool-ами.
- register — додає внутрішній tool до реєстру за його назвою.
- register_external — додає зовнішній MCP-tool як звичайний запис реєстру без прив’язки до транспорту.
- get — повертає tool із реєстру за назвою для подальшого виклику.
- specs — віддає впорядкований перелік описів tool-ів для провайдера.

## Гарантії поведінки

- Read-only: не виконує операцій запису (ФС/БД).
- Перехоплює помилки і не пропускає винятків назовні (fail-safe).
- За певних помилок повертає порожнє значення (напр. `null`) замість винятку.
