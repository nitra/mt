# Мандати, профілі й людино-центрична ескалація

> Частина цільової архітектури **0.3.0-draft** — [зміст](index.md) · [огляд](overview.md). Статус: draft-розширення поверх [graph.md](graph.md) (`unresolvable`, retry ladder) і [access.md](access.md) (Ed25519 `ApprovalResponse`, push тип 3).

## Навіщо

MT відповідає «хто зараз працює» (git claim) і «чи схвалено дію» (підписаний `ApprovalResponse`). Цей шар додає третю відповідь — **хто має право вирішити**, коли `unresolvable` («streak вичерпано» — [graph.md](graph.md)) впирається не в баг, а в розвилку поза межами автономії агента. Без цього людина стає exception-handler-ом: отримує потік «зламалось, поможи» без контексту і без відчуття власності. Мета — зробити ескалацію ланцюгом **вкладених мандатів** (людина вирішує в межах своєї зони), а не ланцюгом ретрансляції; весь транспорт і формулювання розвилки бере на себе система.

## Карта мандатів (`.mt/mandates.yaml`)

Декларативний файл-репо, аналог CODEOWNERS для *рішень*, а не коду:

```yaml
mandates:
  - owner: olena
    scope:
      refs: ["refs/mt/tasks/design/**"]
      decision_types: [architecture, ux]
    thresholds: { budget_eur: 2000, risk: medium, irreversible: false }
    escalates_to: vitalii
  - owner: vitalii
    scope: { refs: ["refs/mt/**"], decision_types: ["*"] }
    thresholds: {}          # порожньо = кореневий мандат
    escalates_to: null
```

- **Явність меж** — агент детерміновано обчислює «своє vs чуже» рішення; зникає функція «попросити дозволу».
- **Зміна мандата = підписаний акт делегування** — розширення `scope`/`thresholds` лише commit-ом з Ed25519-підписом делегатора рівня вище; CI перевіряє підпис тим самим патерном, що boundary-enforcement `agent-core` ([stack.md](stack.md)).
- **Generation fencing** — кожна зміна файлу інкрементує generation; `decision-request` за застарілим generation фенситься й перераховується (той самий механізм, що claim generation — [git.md](git.md)).

## Профіль людини (приватний для хоста-селектора)

Компетенції і мандат — різні осі з різними правилами запису, читачами і **різною видимістю**:

| | Компетенції | Мандат |
| --- | --- | --- |
| Семантика | «вміє зробити» | «має право вирішити» |
| Оновлення | напівавтоматично, з run-історії | лише підписаний акт делегування |
| Читач | селектор призначення задач | маршрутизатор ескалацій |
| Зберігання | **приватно на хості селектора**, поза git-репо задач | публічно в репо (`mandates.yaml`) |
| Ризик злиття | найкращий інженер автоматично стає менеджером | ерозія відповідальності через автозростання мандата |

Мандати публічні, бо це права й відповідальність, не оцінки. Компетенції — **ні**: числові оцінки людей у спільному репо були б «рейтингом виконавців», який [retro.md](retro.md) нормативно забороняє. Профіль живе у приватному просторі хоста, що виконує селектор (той самий периметр, що retro-пропозиції — `~/.nitra/`); людина бачить зміни власних оцінок і може їх заперечити тим самим каналом, що retro-пропозиції. Маршрутизатору ескалацій профіль не потрібен — він читає `mandates.yaml` напряму.

```yaml
# ~/.nitra/people/olena.yaml — поза git-репо задач
competencies:
  rust: {level: 3, confidence: 0.9, last_confirmed: 2026-06-28}
  postgres: {level: 2, confidence: 0.3, last_confirmed: 2025-09-02}  # застаріла — не вирок
  updated_from_runs: 2026-07-10
growth_edge: [rust, postgres]   # опційно: людина хоче рости тут
```

Компетенція = `{level, confidence}`; **confidence старіє з часом без підтверджень** (не рівень — інакше застаріла висока оцінка мовчки гарантує призначення, застаріла низька назавжди «хрестить» людину). **Антиінфляція:** успіх на вузлі рівня N підтверджує компетенцію лише до N — серія перемог на легких вузлах не роздуває рівень (Гудхарт через рутину).

## Селектор призначення задач

Вхід — вузол з `required_competencies`. По-вузлове призначення в графі: селектор запускається **на кожному вузлі підграфа окремо**, тому в межах однієї задачі природно співіснують калібри виконавців. Алгоритм:

1. Відфільтрувати кандидатів, чиї компетенції ≥ вимог вузла.
2. Обрати **мінімально достатнього** (least-qualified-capable) — дешевше і тримає людей на межі зростання.
3. `growth_edge` + дефіцит рівня = 1 → призначення «на виріст» (`stretch: true`), щільніший супровід агента-виконавця в парі.
4. **Розвідувальна квота (probe):** на ε% вузлів, що одночасно низьковажільні, дешево верифіковані й оборотні, селектор свідомо ставить кандидата *нижче* формального мінімуму (`probe: true`) — пріоритет тим, чиї низькі оцінки найбільше застаріли. Probe ініціює система (на відміну від `stretch`, який декларує сама людина) — це і ламає «хрест» першого враження. Оновлення асиметричне для людей: успіх на probe підіймає сильно, невдача не опускає нижче поточного рівня (інакше probe стає екзаменом із покаранням, а не шансом). Прапорець `probe`/`stretch` завжди видимий виконавцю — прихованих перевірок немає.

**Власник вузла ≠ начальник виконавців підвузлів.** Призначення дає власність на *результат* вузла (стикування підвузлів, прийняття), а не владу над людьми піддерева — їх призначає селектор незалежно, їхні розвилки можуть їхати повз власника батьківського вузла (leverage-маршрутизація нижче). У графі ніхто ніким не командує, є лише власність на вузли.

## Артефакт `decision-request`

Розвилка, упакована для власника — живе в run branch поруч із `session.jsonl`:

```
refs/mt/runs/{run-id}/decisions/0001-decision-request.md
```

```markdown
---
type: decision-request
mandate_generation: 17
computed_owner: olena
escalation_chain: [olena, vitalii]
retry_history:
  - {agent: executor-sonnet, attempt: 1, outcome: failed-tests}
leverage_facets: {irreversible: true, blast_radius: repo, divergence: high}
deadline_cost: "затримка блокує 3 залежні задачі"
---
## Контекст
## Варіанти
### A. ...  — наслідки, ціна, ризик
## Рекомендація агента
Варіант B, тому що ...
```

**Інваріант:** до людини не доходить «run failed» — це вже покриває retry ladder ([graph.md](graph.md)); `decision-request` доповзає лише як нерозв'язана *розвилка*, коли ladder вичерпано (`unresolvable`) **і** причина — не баг, а вибір, що виходить за автономію агента. Відповідь власника — існуючий Ed25519-підписаний `ApprovalResponse` ([access.md](access.md)), розширений полем `chosen_option`; аудит-трейл = git log по `decisions/`.

## Escalation-intake

`decision-request` **ніколи не пишеться виконавцем напряму** — його пакує окремий агент, що перед пакуванням збагачує контекст: суміжні run branches, історія `decisions/` (прецеденти), vision/policy/ADR, карта мандатів, leverage-фасети. Та сама роль, що `task-intake → plan-dev` ([graph.md](graph.md)), тільки спрямована вгору: розвилка приходить до вирішувача сформульована на рівні найкращого аналітика системи, незалежно від кваліфікації того, хто в неї впирався.

## Маршрутизатор ескалацій

**Крок 1 — lookup.** Детермінований пошук власника в `mandates.yaml` (ref × decision_type × thresholds); немає відповіді за `sla_hours` → автопідйом по `escalation_chain` з нотифікацією обох сторін.

**Крок 2 — leverage-фасети.** Пороги мандата — груба сітка; поверх неї escalation-intake зобов'язаний заповнити окремі **декларативні фасети** — не перемножений бал (схлопнутий score неможливо оскаржити по частинах):

```yaml
leverage_facets:
  irreversible: false      # bool, не число
  blast_radius: subtree    # node | subtree | repo | company
  divergence: high         # чи варіанти реально розходяться в наслідках
  est_cost_eur: 800
```

Мапінг «фасети → режим» — детермінована таблиця в `mandates.yaml`: підписана політика, а не судження агента. Кожен фасет окремо аудитовний («сказав reversible — виявилось ні» — конкретна, виправна помилка калібрування), грубі enum-и помиляються рідше за формулу, і виконавець бачить фасети перед відправкою.

**Крок 3 — режим, з асиметричним суверенітетом.** Помилки маршрутизації принципово асиметричні: помилково «вгору» — спалена увага верху (обмежено, відновлювано, аналізатор це бачить як rubber-stamping); помилково «вниз» — наслідкове рішення виконано без власника (необмежено, виявляється пост-фактум). Тому машинне судження може лише **додавати** безпеку, ніколи знімати:

- **Вгору — вільно:** інтейк ескалює вище, ніж вимагає мапінг карти — високі фасети з «неважливого» scope спливають аж до кореня.
- **Вниз — лише підписаними актами:** нижче за мапінг карти рішення їде тільки через прецедент (підписаний колись), `delegate_down` (підписаний зараз) або policy-рядок `mandates.yaml` (підписаний назавжди). «Мовчання = згода» замінюється на «підпис один раз = згода тисячу разів». Сигнал «домінує жива спеціалізація» сам вниз не маршрутизує — він їде до обчисленого власника як готова пропозиція `delegate_down` в один підпис.

| Режим | Умова | Механіка |
| --- | --- | --- |
| ask-and-wait | високі фасети або `irreversible` | синхронна ескалація, run блокується до підпису |
| decide-and-inform | середні фасети **і лише reversible** | нижній рівень вирішує чернеткою й виконує; `veto_hours` у власника вище; мовчання матеріалізується як `auto-approved-by-policy {notified_at, veto_window_expired}` — чесний запис дії політики, не підроблений людський акт |
| local / agent | низькі фасети або прецедент | вирішується на місці, вгору йде лише notification |

**Інваріант reversible:** незворотне рішення ніколи не виконується в decide-and-inform — мінімум ask-and-wait, незалежно від решти фасетів. Мовчання може схвалити лише те, що можна відкотити: найгірша «тиха» помилка системи конструктивно обмежена вартістю відкату.

**Калібрувальний контур:** кожна ручна корекція режиму людиною (підняла «низьковажільне», опустила «завищене») — labeled data; аналізатор ескалацій рахує калібрування фасетів по класах рішень; систематичний перекіс класу → клас маршрутизується консервативно (детермінований мапінг без знижень) до перекалібрування.

**Де-ескалація** — `delegate_down`, повноцінний тип `ApprovalResponse`: власник рівня N вважає рішення нижчим → розвилка їде вниз з повним контекстом; після K повторів аналізатор ескалацій пропонує draft-PR на розширення мандата нижнього рівня (де-ескалація конвертується в делегування назавжди).

## Прецедентний рушій

Кожне підписане рішення в `decisions/` — кейс, не разовий акт: (1) перед ескалацією агент шукає аналогічну розвилку того самого/вищого власника — знайдено → рішення застосовується локально, вгору йде лише `«вирішено за прецедентом #NNNN»`; (2) аналізатор бачить повторюваний клас однотипних розвилок → пропонує власнику дистилювати в policy-рядок `mandates.yaml` (підписав → клас зникає з вхідних назавжди); (3) прецедент має `generation` карти мандатів і опційний `expires` — зміна vision/мандатів фенсить застарілі, вимагаючи повторного підтвердження. Ефект: судження власника — актив, що застосовується тисячі разів; нагору доходять тільки справді нові класи розвилок.

## Задачно-скоуплені (ефемерні) мандати

Народжуються з claim батьківського вузла графа, живуть у run branch (не в `mandates.yaml`), вмирають із закриттям вузла:

```yaml
ephemeral_mandate:
  owner: {виконавець батьківського вузла}
  scope: { refs: ["refs/mt/runs/{parent-run}/**"], decision_types: [integration] }
  thresholds: inherit_capped     # успадковані від постійного мандата власника, НЕ ширші
  expires_with: {parent-run}
```

`inherit_capped` — оркестратор піддерева (людина чи модель) отримує лише *локалізацію* інтеграційних розвилок свого піддерева, ніколи прав ширших за власні. Високоважільні й вузькоспеціалізовані розвилки з підвузлів і далі йдуть за загальними правилами маршрутизатора. Закриття батьківського вузла або зміна постійного мандата власника фенсить ефемерний мандат — застряглі `decision-request` перемаршрутовуються автоматично.

## Уніфікація виконавців: люди і моделі

Один селектор, один каскад. **Профіль моделі** (`.mt/models/{model}.yaml`) — та сама структура компетенцій, але **публічний у репо** (у моделей немає приватності); калібрується емпірично тим самим агрегатором (success rate по типах вузлів з історії run-ів цього репо, не зовнішні бенчмарки); нова версія моделі скидає confidence усіх її оцінок. Least-capable-sufficient по-вузлово = пряма token cost optimization (Haiku на рутину, Fable/Opus лише де глибоке міркування реально міняє результат).

Дві асиметрії проти людського випадку:

1. **Stretch — не інвестиція, а ставка.** Модель з невдачі нічого не виносить, тому stretch виправданий лише за дешевої верифікації: оптимістичний прогін дешевою моделлю → дешева перевірка (тести/lint/схеми) → ескалація каскадом при провалі — це retry-before-escalate, прочитаний як cost cascade знизу вгору.
2. **Верифікація дешевша за генерацію.** Слабша модель генерує, сильніша рецензує; approval-точка в межах порогів може бути підписом сильнішої моделі — людський підпис резервується для розвилок, що вийшли за пороги всього модельного каскаду.

Наскрізний ланцюг: `Haiku → Sonnet → Fable/Opus → людина-власник мандата`. Escalation-intake і прецедентний рушій працюють на всіх ярусах — прецедент людини застосовується моделями, розвилка, вирішена Fable в межах порогів, стає прецедентом для Sonnet/Haiku. Модель-оркестратор батьківського вузла отримує той самий `inherit_capped`: не може дозволити підвузлам більше, ніж дозволено їй самій.

## Process watcher

Мікроменеджер **процесу, не людей** — node actor, підписаний на relay-події (`ClaimChanged` — [access.md](access.md)): застряглі claim-и, задачі без комітів N днів, `decision-request` без відповіді біля SLA, run-и, що розповзлись за бюджет. Порядок сигналізації (критично для психології системи): (1) першим пінгується сам виконавець — «у тебе висить X, допомогти?», послуга, не нагляд; (2) тільки після grace period без руху — власник мандата вище, у форматі «X застрягло, Y в курсі з {дата}», завжди паралельно й прозоро — виконавець бачить, що і коли піде вгору.

## Агрегатор компетенцій і аналізатор ескалацій

- **Агрегатор компетенцій** — не окремий сервіс, а **другий вихід retro-рушія** ([retro.md](retro.md)): один читач run-історії з двома продуктами — приватні пропозиції виконавцю і оновлення компетенцій. Профілі людей оновлюються у приватному просторі хоста (людина бачить дельту і може заперечити — той самий канал, що retro-пропозиції); профілі моделей — звичайним комітом на review. Ніколи не торкається `mandates.yaml`.
- **Аналізатор ескалацій** — щотижневий прогін по git log `decisions/`: `>K%` розвилок scope X злітають вище **і** верх стабільно бере рекомендацію агента/варіант X → сигнал «мандат завузький» → draft-PR на `mandates.yaml`; обернено — часто відкочувані рішення в межах мандата → сигнал завузити. Розширення мандата лишається свідомим підписаним актом людини — система лише показує, де ієрархія тре.

| Generation fencing ([git.md](git.md)) | фенсить застарілі claim-и | + фенсить `decision-request` за застарілим generation карти мандатів |
| `unresolvable` / push тип 3 ([graph.md](graph.md), [access.md](access.md)) | алерт власнику «streak вичерпано» | збагачується escalation-intake у `decision-request` перед доставкою |
| Run branches | `session.jsonl` | + `decisions/` — аудит-трейл людських рішень |
| Derived-стани ([graph.md](graph.md)) | `unresolvable` — термінал | + `awaiting-decision`: вузол з відкритим `decision-request` без відповіді; блокує залежні (як `pending-audit`), видимий у `mt status` |
| Retro-рушій ([retro.md](retro.md)) | приватні пропозиції виконавцю | + другий вихід того самого прогону: агрегатор компетенцій |
| Retry policy | неявна | явний інваріант retry-before-escalate: до людини — лише розвилки |
| CI ([stack.md](stack.md)) | boundary для `agent-core` | + перевірка підпису делегатора на змінах `mandates.yaml` і секції `mandate` профілів |

Нових сервісів не потрібно — watcher і агрегатор такі самі node actors, як решта; substrate-first зберігається, усі нові сутності — файли в git.

## Місце в roadmap

Розгортання фазами по класах задач, не по людях: фаза 0 (mandates.yaml + decision-request з фасетами + retry-before-escalate + watcher) не потребує нової інфраструктури — вузьке коло виконавців на CLI одразу після dogfood-ядра; фаза 1 (власники мандатів середнього рівня, decide-and-inform, прецеденти) реалістично потребує dashboard; селектор/профілі/probe — після накопичення історії. Модельний каскад — паралельний трек з першого дня: та сама машинерія без соціальних ризиків. Деталі — [roadmap.md](../roadmap.md), M6.
