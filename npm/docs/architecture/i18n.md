# Багатомовність (i18n)

> Частина цільової архітектури **0.3.0-draft** — [зміст](index.md) · [огляд](overview.md). Реалізує крос-мовний вимір [мети](../vision.md): кожен учасник — людина чи агент — бачить контент своєю мовою, під капотом — автоматичний переклад і синхронізація.

## Суть

Цей документ визначає архітектурний підхід до підтримки багатомовності в системі, забезпечуючи, що кожна сторона взаємодії бачить контент у своїй рідній мові. Головним принципом є збереження єдиного канону — базової версії проєкту, яка ніколи не змінюється через переклади. Всі переклади розглядаються як похідні дані, прив'язані до базового стану за допомогою хешів, що гарантує їхню синхронізацію та валідність. Система підтримує коректну роботу навіть за відсутності перекладів, і всю роботу з мовами організовано через окреме, атомарно кероване сховище, відмінне від основного репозиторію.

## Принципи (нормативні)

1. **Один канон.** Base-мова проєкту (`i18n.base_lang`, default `en`) — єдине джерело істини. `fact`-hash, derived-стани, scanner і `## Check` читають **тільки** base; переклади ніколи не впливають на стан графа.
2. **Переклади — derived-дані.** Кожен переклад несе `source_hash` base-версії; hash розійшовся → переклад stale → регенерація (той самий CRC-патерн, що у файлової документації).
3. **Graceful degradation.** Немає перекладу / перекладач недоступний / схема файлу невідома → показується base. Система повністю коректна без жодного перекладу.
4. **Переклади — той самий контент для людей і ШІ.** Агент, що працює мовою користувача, читає ті самі матеріалізовані файли; кеш спільний, вартість перекладу платиться один раз.

## Три класи контенту

| Клас | Переклад | Back-sync |
| --- | --- | --- |
| **Immutable артефакти вузлів** (`task.md` після init, `plan/run/fact/audit`) | derived, read-only | ні — зміни йдуть штатним шляхом (`mt invalidate` + новий артефакт) |
| **Мутабельні доки** (README, `docs/`, чернетки `task.md` до init) | derived + authored | так — write path нижче |
| **Живий чат** (`session.jsonl`, Envelope) | переклад на поверхні (Шар 3), live | не застосовно |

## Сховище: `refs/mt/i18n/<lang>`

Переклади живуть **поза `main`** — в окремому custom ref на мову, за патерном `refs/mt/runs/*`/`refs/mt/archive/*`:

- дзеркальне дерево: `refs/mt/i18n/uk : docs/vision.md` ↔ `main : docs/vision.md`;
- `main` лишається чистим (без i18n-комітів у diff-ах), ref підтягується on-demand;
- «прихованість від інструментів» безкоштовна: ref не checkout-иться, поки не потрібен;
- GC: переклади мов без активних учасників — за `i18n_ttl_days` (як `archive_ttl_days`);
- **запис у i18n-ref — лише разом із fenced publish**: authored-переклад їде тим самим atomic push, що й base-правка (`--force-with-lease` і на i18n-ref); програш гонки = стандартний retry всього пакета. Окремого «пера» для перекладів не існує; фонові регенерації append-only — їхня гонка нешкідлива (виграє будь-яка свіжа за `source_hash`).

Директорія `.i18n/` у `main` **не використовується** (роздуває репо, засмічує diff, N перекладних комітів на кожну правку).

### Схема файлу перекладу

```yaml
# refs/mt/i18n/uk : docs/vision.md (frontmatter, далі — перекладений вміст)
schema_version: 1
source: docs/vision.md
source_hash: sha256:…      # base-версія, з якої згенеровано / яку згенерувала authored-правка
authored: false            # true = писала людина цією мовою; не перезаписувати
translated_at: ISO8601
model: MIM
```

**Staleness:** `source_hash` ≠ hash поточної base-версії → переклад застарілий. Поверхня показує stale-переклад з позначкою або base; регенерація — фоном.

## Read path: матеріалізація у worktree

Жодної віртуальної ФС (FUSE, git smudge) — wrapper/agent-server уже володіє обома точками:

```
claim → worktree від base_sha
      → overlay: файли мови учасника з refs/mt/i18n/<lang> поверх base
        (лише свіжі за source_hash; stale → base + фонова регенерація)
```

Мова учасника: обовʼязкове поле `lang` (BCP-47) у `ClientHello` протоколу v4 ([runtime.md](runtime.md)); fallback-джерело — атрибут акаунта на relay; для headless-агента — з `a.md`/конфігу вузла. Учасник реально бачить у worktree файли своєю мовою; у `main` завжди base.

**Що перекладається — триступенева схема:** default `**/*.md` → конфіг `i18n.include`/`i18n.exclude` (globs) → per-file opt-out `i18n: off` у frontmatter. Код і конфіги не перекладаються ніколи.

## Write path: компіляція в base перед publish

Правка мутабельного файлу мовою учасника (не base):

```
коміт у worktree → wrapper перед fenced publish:
  1. змінені людські файли → contract-aware переклад у base
  2. base → main (звичайний fenced publish, атомарність не змінюється)
  3. авторська мовна версія → refs/mt/i18n/<lang> (authored: true, source_hash = новий base-hash)
  4. решта мов: stale за hash → фонова регенерація
```

- **Захист від round-trip churn:** authored-версія **не перезаписується** зворотним перекладом, поки base не зміниться чужою правкою — авторський текст авторитетний для своєї мови.
- **Конкурентність:** правки файлів вузла йдуть через claim («одне перо» — [git.md](git.md)); для доків поза графом (README, `docs/`) серіалізація — через fenced publish (правило запису i18n-ref вище). Конфлікт «дві мови одночасно» неможливий за побудовою.
- **Фонова регенерація — системна черга agent-server, поза графом задач.** Переклади не матеріалізуються як MT-вузли: граф не засмічується службовим шумом, а вартість регенерацій не потрапляє в ledger задач — вона облікується в телеметрії хоста (лог провайдера, `model_tier` з `i18n`-конфігу).

## Contract-aware перекладач

Перекладач — не «переклади файл», а schema-aware трансформація. **Не перекладаються:**

- frontmatter-ключі та enum-значення (`result: success`, `mode: agent`);
- контрактні секції, які парсить скрипт: `## Task`, `## Done when`, `## Check`, `## Inputs`, `## Children`, `## Approvals`, `## Ref`, …;
- shell-команди (`## Check`), `ref:`-рядки, code fences, ідентифікатори/шляхи/dep-ids.

Перекладається лише людський текст між ними. **Fail closed:** файл з невідомою схемою або `schema_version` → не перекладається, показується base (креативний переклад `## Done when` не має шансу зламати scanner).

## Мовний набір: учасники, не топ-N

- **Eager** — мови активних учасників задачі (membership кореневого вузла вже знає акаунти; мова — атрибут акаунта).
- **Lazy** — будь-яка інша мова: переклад при першому відкритті, результат у кеш-ref.
- **Публікаційний набір** (напр. топ-40 для публічних проєктів) — опційний окремий крок (`i18n.publish_langs`), не режим за замовчуванням: квадратична вартість «усі мови × кожна правка» без читача — марнотратство.

## Live-шар: гібрид за capability

Повідомлення чату та події Envelope перекладаються при відображенні; у `session.jsonl` журналиться оригінал мовою автора. Точка перекладу — **negotiable**, через наявний механізм `client_capabilities`:

- **Default — хост (agent-server):** клієнт без LLM-доступу (телефон, viewer) отримує Envelope уже перекладені його `lang`; переклад робиться один раз на мову, кеш — у памʼяті run-а.
- **Клієнт із capability `self-translate`** (десктоп із власним провайдером/локальною моделлю) отримує оригінал і перекладає/кешує сам.

Кешує той, хто перекладає. Relay участі не бере — його межа «не парсить payload» недоторкана. Це окремий механізм від файлового шару: ефемерний, без git.

## Конфігурація

```jsonc
// .mt.json
{
  "i18n": {
    "base_lang": "en",
    "eager": "participants",        // participants | ["uk", "de", …]
    "publish_langs": [],            // опційний публікаційний набір
    "include": ["**/*.md"],         // default; звужується exclude-ом
    "exclude": [],                  // globs поверх include
    "model_tier": "MIM",            // профіль для системної черги регенерації
    "ttl_days": 90
  }
}
```
