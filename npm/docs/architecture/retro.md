# Мета-цикл: ретроспективне самопокращення

> Частина цільової архітектури **0.3.0-draft** — [зміст](index.md) · [огляд](overview.md). Реалізує мета-цикл [мети](../vision.md): платформа пропонує виконавцю кращі рішення, спираючись на дані, які граф уже фіксує задарма.

## Суть

Цей компонент реалізує ретроспективний цикл самопокращення, який автоматично аналізує історичні дані виконання. Він виявляє системні патерни невдач та пропонує конкретні оптимізації, адресовані безпосередньо виконавцю. Пропозиції не вносять змін у граф напряму, а слугують підставою для ручного впровадження, що гарантує збереження повного контролю. Успішно підтверджені зміни фіксуються як незмінні базовані лінії, що забезпечує чесне вимірювання ефекту.

## Концепція

Audit trail графа — готовий датасет ретроспективи: `run_NNN` (result, wall_sec, tokens, cost, retry-щаблі), плани та їх відхилення, аудит-вердикти, `run-summary` патернів невдач. Retro-аналіз читає цю історію і шукає закономірності: «задачі класу X із tool Y мають вищий failed_streak, ніж із Z», «щабель 2 retry ladder на вузлах типу W ніколи не рятує — одразу alternative-approach». Людина такої ретроспективи майже ніколи не робить — бо це окрема робота; тут вона побічний продукт виконання.

## Принципи (нормативні)

1. **Працює на виконавця.** Пропозиції адресовані виконавцю особисто; цикл **opt-in** (`retro.enabled`, default `false`). Іншим ролям пропозиції не видимі.
2. **Не інструмент нагляду.** Жодних персональних агрегатів для інших учасників чи «рейтингів виконавців»; крос-виконавча аналітика — свідомо поза 0.3.0.
3. **Пропозиція ≠ дія.** Retro нічого не змінює в графі й конфігах; застосування — лише штатні механізми (правка `a.md`/`.mt-override.json`/`surface_profiles` людиною, звичайний commit/PR).
4. **Дані не покидають периметр.** Аналіз виконує хост, який і так має доступ до репо; через relay їде лише push-нотифікація «є пропозиції» (тип 3), без вмісту.

## Механізм

- **Тригер:** `retro.schedule_days`, або поріг `retro.min_resolved` нових resolved-вузлів, або вручну `mt retro`.
- **Виконання:** системний фоновий прогін agent-server (поза графом задач — той самий патерн, що черга регенерації [i18n.md](i18n.md)): вибірка вузлів за період → LLM-аналіз (`retro.model_tier`) → пропозиції.
- **Вхід:** run/fact/plan/audit-файли вузлів, ledger-агрегати (cost/wall по піддеревах), чинні `surface_profiles`/`skill_profiles`/`retry_ladder`.
- **Вихід — suggestion:**

```yaml
suggestion:
  target: research/analyze          # вузол або клас задач (за патерном task.md)
  observed: '3 із 4 retry впирались у відсутність web-search'
  proposal: 'retry_ladder щабель 2: skills_add: [web-search]'
  evidence: [research/analyze/run_002.md, research/collect/run_003.md]
  impact_estimate: '−40% wall_sec на подібних задачах'
```

- **Зберігання:** приватний простір виконавця — `~/.nitra/retro/<period>.md` (поза git-репо задач); у спільний граф пропозиції **не** потрапляють.
- **Доставка:** push тип 3 ([access.md](access.md)) «задача X має пропозиції» → перегляд `mt retro show` або в UI; кожна пропозиція — прийняти (генерується конкретна правка конфігу/`a.md` на review) або відхилити з причиною (причина — теж вхід наступного циклу).

## Для агентських вузлів

Пропозиції щодо агентських вузлів (`model_tier`, `skills`, `retry_ladder`, вибір surface) адресуються **власнику задачі**; прийняті стають звичайними правками `a.md`/override-конфігу — git-історія показує еволюцію інструментів вузла. Це замикає крос-програмковий вимір ([surfaces.md](surfaces.md)): вибір тулів живиться даними, а не лише статичною конфігурацією.

## Інновації: прийняття і baseline

Цикл «аналіз → пропозиція → застосування» замикається третьою ланкою: «застосування → вимірювання → визнання». **Прийнята** оптимізація процесу (від retro, людини чи агента) матеріалізується immutable-файлом у корені графа:

```markdown
# mt/innovation_NNN.md
---
schema_version: 1
created_at: ISO8601
author: vkozlov                # handle людини | agent-конфігурація | retro
scope: 'research/*'            # клас задач (патерн) або конкретне піддерево
change: 'retry_ladder щабель 2: skills_add: [web-search]'
baseline:                      # зріз ledger по скоупу НА МОМЕНТ прийняття
  runs: 14
  avg_wall_sec: 4100
  avg_cost_usd: 0.61
  failed_streak_rate: 0.35
  evidence: [research/analyze/run_002.md, …]
---
```

Зафіксований baseline — те, що робить майбутню атрибуцію чесною: ефект порівнюється не з памʼяттю, а з зафіксованим зрізом. Приватність не порушується: **пропозиції приватні, впровадження публічне** — інновація змінила спільний процес (коміт у спільний репо), тому її вимірювання відкрите. Це і є межа з принципом «не нагляд»: вимірюється зміна процесу, а не робота людини.

## Impact: вимірювання і заохочення

**Вимірювання** — другий режим того самого retro-прогону: для задач у `scope` порівнюються метрики «після» проти `baseline`; періодичні `impact`-зрізи додаються до інновації (append-only):

```yaml
impact:
  period: 2026-07 … 2026-09
  runs: 22                      # confidence: нижче retro.impact_min_runs — зріз не публікується
  delta_wall_sec: '-38%'
  delta_cost_usd: '-31%'
  delta_failed_streak_rate: '-0.21'
```

**Заохочення** — різне для людей і агентів, з одного джерела даних:

- **Люди:** видимий профіль вкладу — сума підтверджених ефектів автора за період з evidence-посиланнями. Платформа дає **вимірювання**, не компенсацію: премії/визнання — політика команди поверх достовірних даних (платформа лишається нейтральною).
- **Агенти:** заохочення = **відбір**. Інновація з підтвердженим ефектом підвищує пріоритет свого патерну: кращі retry ladders/skills стають default для класу задач; конфігурації з вищим підтвердженим impact отримують більше задач свого класу. Еволюційний цикл: що доведено працює — вживається ширше.

**Anti-gaming:** порівняння лише в межах класу задач; `impact_min_runs` як поріг confidence; спірні зрізи проходять штатний аудит-механізм ([graph.md](graph.md)). Побічна цінність: `innovation`/`impact`-файли — датасет переносимості оптимізацій між проєктами.

## Конфігурація

```jsonc
// .mt.json
{
  "retro": {
    "enabled": false,          // opt-in per-виконавець
    "schedule_days": 7,
    "min_resolved": 10,        // не ганяти аналіз на порожньому періоді
    "impact_min_runs": 10,     // поріг confidence для impact-зрізів
    "model_tier": "AVG"
  }
}
```
