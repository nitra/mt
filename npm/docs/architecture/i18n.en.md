# Internationalization (i18n)

> A part of the target architecture **0.3.0-draft** — [content](index.md) · [overview](overview.en.md). Implements the cross-lingual dimension of the [vision](../vision.en.md): every participant—human or agent—sees the content in their native language, with automatic translation and synchronization under the hood.

## Essence

This document defines the architectural approach to supporting internationalization in the system, ensuring that every interacting party sees the content in their native language. The main principle is maintaining a single canon—a base version of the project that is never changed by translations. All translations are considered derived data, linked to the base state via hashes, which guarantees their synchronization and validity. The system supports correct operation even in the absence of translations, and all language handling is organized through a separate, atomically managed repository, distinct from the main repository.

## Principles (Normative)

1. **One Canon.** The base language of the project (`i18n.base_lang`, default `en`) is the single source of truth. `fact`-hash, derived states, scanner, and `## Check` read **only** from the base; translations never affect the graph state.
2. **Translations are derived data.** Each translation carries the `source_hash` of the base version; hash mismatch $\to$ translation stale $\to$ regeneration (the same CRC pattern as in file documentation).
3. **Graceful degradation.** No translation / translator unavailable / unknown file scheme $\to$ base is shown. The system is completely correct without any translation.
4. **Translations are the same content for humans and AI.** The agent working in the user's language reads the same materialized files; the cache is shared, and the cost of translation is paid once.

## Three Content Classes

| Class | Translation | Back-sync |
| --- | --- | --- |
| **Immutable node artifacts** (`task.md` after init, `plan/run/fact/audit`) | derived, read-only | no — changes follow the standard path (`mt invalidate` + new artifact) |
| **Mutable docs** (README, `docs/`, `task.md` drafts before init) | derived + authored | yes — write path below |
| **Live chat** (`session.jsonl`, Envelope) | surface translation (Layer 3), live | not applicable |

## Repository: `refs/mt/i18n/<lang>`

Translations live **outside of `main`**—in a separate custom ref for the language, following the pattern `refs/mt/runs/*`/`refs/mt/archive/*`:

- Mirror tree: `refs/mt/i18n/uk : docs/vision.md` ↔ `main : docs/vision.md`;
- `main` remains clean (without i18n commits in diffs), ref is pulled on-demand;
- "Invisibility to tools" is free: ref is not checked out until needed;
- GC: translations for languages without active participants $\to$ after `i18n_ttl_days` (like `archive_ttl_days`);
- **Writing to the i18n-ref — only together with fenced publish**: the authored translation goes through the same atomic push as the base change (`--force-with-lease` and to the i18n-ref); race condition loss = standard retry of the entire package. There is no separate "pen" for translations; background regenerations are append-only—their race is harmless (any fresh one based on `source_hash` wins).

The `.i18n/` directory in `main` is **not used** (bloats the repo, clutters diff, N translation commits for every change).

### Translation File Schema

```yaml
# refs/mt/i18n/uk : docs/vision.md (frontmatter, then — translated content)
schema_version: 1
source: docs/vision.md
source_hash: sha256:…      # base version from which the authored change was generated
authored: false            # true = written by a human in this language; do not overwrite
translated_at: ISO8601
model: MIM
```

**Staleness:** `source_hash` $\neq$ hash of the current base version $\to$ translation is stale. The surface shows the stale translation with a marker or the base; regeneration happens in the background.

## Read Path: Materialization in Worktree

No virtual FS (FUSE, git smudge) — the wrapper/agent-server already holds both points:

```
claim $\to$ worktree from base_sha
      $\to$ overlay: participant language files from refs/mt/i18n/<lang> on top of base
        (only fresh based on source_hash; stale $\to$ base + background regeneration)
```

Participant language: required `lang` field (BCP-47) in the `ClientHello` protocol v4 ([runtime.md](runtime.en.md)); fallback source $\to$ account attribute on the relay; for a headless agent $\to$ from `a.md`/node config. The participant actually sees files in their language in the worktree; in `main`, there is always the base.

**What is translated — three-stage scheme:** default `**/*.md` $\to$ config `i18n.include`/`i18n.exclude` (globs) $\to$ per-file opt-out `i18n: off` in frontmatter. Code and configurations are never translated.

## Write Path: Compilation to Base Before Publish

Change of a mutable file in the participant's language (not base):

```
commit in worktree $\to$ wrapper before fenced publish:
  1. changed human files $\to$ contract-aware translation to base
  2. base $\to$ main (normal fenced publish, atomicity unchanged)
  3. authored language version $\to$ refs/mt/i18n/<lang> (authored: true, source_hash = new base-hash)
  4. other languages: stale due to hash $\to$ background regeneration
```

- **Protection against round-trip churn:** the authored version is **not overwritten** by back-translation until the base is changed by another change $\to$ the author's text is authoritative for its language.
- **Concurrency:** node file changes go through claim ("one pen" — [git.md](git.en.md)); for docs outside the graph (README, `docs/`) serialization $\to$ through fenced publish (i18n-ref write rule above). "Two languages simultaneously" conflict is impossible by design.
- **Background regeneration $\to$ system queue of the agent-server, outside the task graph.** Translations are not materialized as MT-nodes: the graph is not cluttered with service noise, and the cost of regenerations does not end up in the task ledger—it is accounted for in the host telemetry (provider log, `model_tier` from `i18n`-config).

## Contract-aware Translator

The translator is not "translate the file," but schema-aware transformation. **The following are not translated:**

- frontmatter keys and enum values (`result: success`, `mode: agent`);
- contract sections parsed by the script: `## Task`, `## Done when`, `## Check`, `## Inputs`, `## Children`, `## Approvals`, `## Ref`, …;
- shell commands (`## Check`), `ref:` lines, code fences, identifiers/paths/dep-ids.

Only human text between them is translated. **Fail closed:** file with an unknown schema or `schema_version` $\to$ is not translated, base is shown (creative translation of `## Done when` has no chance of breaking the scanner).

## Language Set: Participants, not Top-N

- **Eager** — languages of active participants in the task (membership of the root node already knows the accounts; language $\to$ account attribute).
- **Lazy** — any other language: translation upon first opening, result in the cache-ref.
- **Publishing set** (e.g., top-40 for public projects) — optional separate step (`i18n.publish_langs`), not default mode: the quadratic cost of "all languages $\times$ every change" without a reader is waste.

## Live Layer: Hybrid by Capability

Chat messages and Envelope events are translated upon display; in `session.jsonl`, the original language of the author is logged. The translation point is **negotiable**, via the existing `client_capabilities` mechanism:

- **Default — host (agent-server):** a client without LLM access (phone, viewer) receives the Envelope already translated into its `lang`; translation is done once per language, cache $\to$ in the run memory.
- **Client with capability `self-translate`** (desktop with own provider/local model) receives the original and translates/caches it itself.

The one who translates caches. The relay of participation does not interfere—its boundary "does not parse payload" remains untouched. This is a separate mechanism from the file layer: ephemeral, without git.

## Configuration

```jsonc
// .mt.json
{
  "i18n": {
    "base_lang": "en",
    "eager": "participants",        // participants | ["uk", "de", …]
    "publish_langs": [],            // optional publishing set
    "include": ["**/*.md"],         // default; narrowed by exclude
    "exclude": [],                  // globs over include
    "model_tier": "MIM",            // profile for the system regeneration queue
    "ttl_days": 90
  }
}
```
