# Internationalization (i18n)

> A part of the target architecture **0.3.0-draft** — [content](index.md) · [overview](overview.en.md). Implements the cross-language dimension of the [goal](../vision.en.md): each participant — human or agent — sees content in their own language, with automatic translation and synchronization under the hood.

## Essence

This document defines the architectural approach to supporting multilingualism in the system, ensuring that each interacting party sees the content in their native language. The main principle is maintaining a single canon—the base version of the project, which is never changed through translations. All translations are considered derived data, linked to the base state via hashes, which guarantees their synchronization and validity. The system supports correct operation even without translations, and all language work is organized through a separate, atomically managed repository, distinct from the main repository.

## Principles (Normative)

1. **One Canon.** The base language of the project (`i18n.base_lang`, default `en`) is the single source of truth. `fact`-hash, derived states, scanner, and `## Check` read **only** the base; translations never influence the graph state.
2. **Translations are derived data.** Each translation carries the `source_hash` of the base version; hash mismatch → translation stale → regeneration (the same CRC pattern as in file documentation).
3. **Graceful degradation.** No translation / translator unavailable / unknown file schema → the base is shown. The system is fully correct without any translation.
4. **Translations are the same content for humans and AI.** An agent working in the user's language reads the same materialized files; the cache is shared, and the cost of translation is paid once.

## Three Content Classes

| Class | Translation | Back-sync |
| --- | --- | --- |
| **Immutable Node Artifacts** (`task.md` after init, `plan/run/fact/audit`) | derived, read-only | no — changes go through the standard path (`mt invalidate` + new artifact) |
| **Mutable Docs** (README, `docs/`, `task.md` drafts before init) | derived + authored | yes — write path below |
| **Live Chat** (`session.jsonl`, Envelope) | surface translation (Layer 3), live | not applicable |

## Repository: `refs/mt/i18n/<lang>`

Translations live **outside `main`** — in a separate custom ref for the language, following the pattern `refs/mt/runs/*`/`refs/mt/archive/*`:

- mirror tree: `refs/mt/i18n/uk : docs/vision.md` ↔ `main : docs/vision.md`;
- `main` remains clean (no i18n commits in diffs), the ref is pulled on-demand;
- "invisibility to tools" is free: the ref is not checked out until needed;
- GC: translations of languages with no active participants → after `i18n_ttl_days` (like `archive_ttl_days`);
- **writing to the i18n-ref — only with fenced publish**: the authored translation goes via the same atomic push as the base change (`--force-with-lease` and to the i18n-ref); race condition loss = standard retry of the entire package. No separate "pen" for translations exists; background regenerations are append-only — their race is harmless (any fresh one by `source_hash` wins).

The directory `.i18n/` in `main` **is not used** (it swells the repo, clutters the diff, N translation commits for every change).

### Translation File Schema

```yaml
# refs/mt/i18n/uk : docs/vision.md (frontmatter, then translated content)
schema_version: 1
source: docs/vision.md
source_hash: sha256:…      # base version from which the authored change was generated / generated
authored: false            # true = written by a person in this language; do not overwrite
translated_at: ISO8601
model: MIN
```

**Staleness:** `source_hash` $\neq$ hash of the current base version → translation is stale. The surface shows the stale translation with a marker or the base; regeneration happens in the background.

## Read path: Materialization in worktree

No virtual FS (FUSE, git smudge) — the wrapper/agent-server already owns both points:

```
claim → worktree from base_sha
      → overlay: participant language files from refs/mt/i18n/<lang> over base
        (only fresh by source_hash; stale → base + background regeneration)
```

Participant language: mandatory `lang` (BCP-47) field in `ClientHello` protocol v4 ([runtime.md](runtime.en.md)); fallback source → account attribute on the relay; for headless agents → from `a.md`/node config. The participant actually sees files in their language in the worktree; `main` is always the base.

**What is translated — three-stage scheme:** default `**/*.md` → config `i18n.include`/`i18n.exclude` (globs) → per-file opt-out `i18n: off` in the frontmatter. Code and configs are never translated.

## Write path: Compilation to base before publish

Editing a mutable file in the participant's language (not base):

```
commit in worktree → wrapper before fenced publish:
  1. modified human files → contract-aware translation to base
  2. base → main (standard fenced publish, atomicity remains unchanged)
  3. authored language version → refs/mt/i18n/<lang> (authored: true, source_hash = new base-hash)
  4. remaining languages: stale by hash → background regeneration
```

- **Protection against round-trip churn:** the authored version **is not overwritten** by reverse translation until the base changes by another change → the authored text is authoritative for its language.
- **Concurrency:** node file edits go through a claim ("one pen" — [git.md](git.en.md)); for docs outside the graph (README, `docs/`) serialization → via fenced publish (i18n-ref write rule above). "Two languages simultaneously" conflict is impossible by design.
- **Background regeneration → systemic queue of the agent-server, outside the task graph.** Translations are not materialized as MT nodes: the graph is not cluttered with service noise, and the cost of regenerations does not enter the task ledger — it is accounted for in the host telemetry (provider log, `model_tier` from `i18n`-config).

## Contract-aware Translator

The translator is not "translate file," but a schema-aware transformation. **Not translated:**

- frontmatter keys and enum values (`result: success`, `mode: agent`);
- contract sections parsed by the script: `## Task`, `## Done when`, `## Check`, `## Inputs`, `## Children`, `## Approvals`, `## Ref`, …;
- shell commands (`## Check`), `ref:` lines, code fences, identifiers/paths/dep-ids.

Only the human text between them is translated. **Fail closed:** file with unknown schema or `schema_version` → is not translated, the base is shown (creative translation of `## Done when` has no chance of breaking the scanner).

## Language Set: Participants, Not Top-N

- **Eager** — languages of active participants in the task (membership of the root node already knows the accounts; the language is an account attribute).
- **Lazy** — any other language: translation upon first opening, result in the cache-ref.
- **Publishing Set** (e.g., top-40 for public projects) — optional separate step (`i18n.publish_langs`), not the default mode: quadratic cost of "all languages $\times$ every change" without a reader → waste.

## Live Layer: Hybrid by Capability

Chat messages and Envelope events are translated upon display; `session.jsonl` logs the original language of the author. The translation point is **negotiable**, via the existing `client_capabilities` mechanism:

- **Default → host (agent-server):** client without LLM access (phone, viewer) receives the Envelope already translated into its `lang`; translation is done once per language, the cache is in the run's memory.
- **Client with capability `self-translate`** (desktop with own provider/local model) receives the original and translates/caches it itself.

The one who translates caches. The relay does not take part — its boundary "does not parse payload" remains untouched. This is a separate mechanism from the file layer: ephemeral, without git.

## Configuration

```jsonc
// .mt.json
{
  "i18n": {
    "base_lang": "en",
    "eager": "participants",        // participants | ["uk", "de", …]
    "publish_langs": [],            // optional publishing set
    "include": ["**/*.md"],         // default; narrowed by exclude
    "exclude": [],                  // globs over include
    "model_tier": "MIN",            // profile for system regeneration queue
    "ttl_days": 90
  }
}
```
