# Roadmap реалізації

> Продуктовий документ: **у якому порядку** будується [цільова архітектура 0.3.0-draft](architecture/index.md) і що вважається готовністю кожного кроку. Архітектура каже «що і як», roadmap — «що спочатку». Стратегічна рамка — [vision.md](vision.md): ядро + одна поверхня, доведені dogfooding-ом.

Кожен milestone має **demo-критерій** — сценарій, який можна показати за хвилину. Milestone не «готовий на 80%»: демо або відтворюється, або ні.

## M0 — Dogfood ядра (`@7n/mt` 0.2.x → зміцнення)

Розробка самого MT ведеться через MT: задачі цього репозиторію — вузли графа в `mt/`, агенти виконують їх автономно за контрактом 0.2.x.

- **Demo-критерій:** реальна фіча цього репо пройшла повний цикл `init → plan → spawn → run → audit → resolved` без ручних git-операцій; `mt status` показує живий граф.
- Покриває DoD-сценарій 1 (автономний headless).
- Вихід: список тертя контракту — вхідні дані для пріоритетів M1.

## M1 — agent-server локально (`mt serve` + `mt attach`)

Перший Rust-шар: `agent-protocol` + `agent-core` + мінімальний `agent-server` (одна машина, локальний WS, без relay). Інтерактивна сесія = run вузла: журнал у run ref, пуш кожен хід.

- **Demo-критерій:** `mt attach <node>` відкриває чат із задачею; закрив ноутбук — відкрив — сесія відновилась з реплеєм; `mt done` публікує fact у main тим самим fenced publish.
- Протокол подій v4 (включно з `lang` у ClientHello — поле є, переклад ще ні).

## M2 — Mission control (relay + телефон-approver)

Relay (акаунти, membership, presence, push) + мобільний тонкий клієнт. Це продукт із vision.md: «агенти на всіх твоїх пристроях під контролем з кишені».

- **Demo-критерій:** задача акаунта A виконується на сервері; телефон учасника B (роль approver, без git-доступу) отримує push, показує диф деструктивної дії, B підписує — підпис видно у `run_NNN.md ## Approvals`. Міграція сесії «перенести сюди» між двома машинами працює.
- Покриває DoD-сценарії 2, 3, 5.

## M3 — Dashboard і поверхні

`mt-dashboard` (підграф як одна картина) + перший спеціалізований surface за [surfaces.md](architecture/surfaces.md) (референс — designer із preview).

- **Demo-критерій:** dashboard показує live-стани всього піддерева (NodeState/PlanReview/Committed); у сесії designer користувач тицяє елемент у preview → `ContextSelected` → агент править саме його. Апрув плану з dashboard.
- Покриває DoD-сценарій 4.

## M4 — Файловий шар i18n

Реалізація [i18n.md](architecture/i18n.md): worktree-матеріалізація, contract-aware компіляція в base, `refs/mt/i18n/*`, системна черга регенерації, live-переклад за capability.

- **Demo-критерій** (він же критерій готовності глави i18n): учасник A (uk) і учасник B (de) правлять один README кожен своєю мовою; у `main` — лише англійський base; авторський текст A не зіпсовано зворотним перекладом; третя мова зʼявляється lazy при першому відкритті.

## M5 — Мета-цикл (retro)

Реалізація [retro.md](architecture/retro.md): фоновий аналіз audit trail, приватні пропозиції виконавцю, застосування через штатні правки конфігів; інновації з baseline та impact-вимірюванням.

- **Demo-критерій (пропозиції):** після ≥ `retro.min_resolved` resolved-задач `mt retro` видає щонайменше одну пропозицію з `evidence`-посиланнями на конкретні run-файли; прийняття пропозиції генерує конкретну правку `a.md`/конфігу на review.
- **Demo-критерій (impact):** прийнята інновація (`innovation_NNN.md` із baseline) через ≥ `retro.impact_min_runs` runs у скоупі отримує impact-зріз із виміряним Δwall/Δcost/Δfailed_streak — вклад автора видно з доказами.
- **MVP не чекає M1–M4:** аналіз працює на локальних даних одразу після M0 (dogfood) — без relay і сесій; на M2+ додається лише доставка на пристрої.

## Поза roadmap (свідомо)

- Hosted relay як сервіс (білінг, тенанти) — після першого зовнішнього користувача self-hosted.
- Перенесення контракту графа в Rust — окремий ADR після стабілізації протоколу ([stack.md](architecture/stack.md)).
- Кворумні криптографічні протоколи succession — 0.3.0 обходиться co-owner-практикою ([access.md](architecture/access.md)).
